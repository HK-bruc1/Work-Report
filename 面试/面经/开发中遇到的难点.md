# 双核 SDK 下软件 IIC 并发冲突导致心率功能异常

## 背景

产品是一款 TWS 智能耳机，基于杰理 BR52 双核芯片开发，有两个可选功能，都通过按键触发：

- **头部追踪空间音效**：开启后由内置 IMU（ICM42670P）实时采集头部姿态，驱动空间音效算法动态调整声场，默认关闭
- **心率血氧检测**：开启后由 PPG 芯片（HX3011）持续采集数据，默认关闭

两颗传感器都挂在同一组 GPIO 上，SDK 只提供一套软件模拟 IIC 接口：

```
SCL → IO_PORTC_06
SDA → IO_PORTC_07

├── ICM42670P（IMU）
└── HX3011（心率血氧）
```

这个项目最初是在单核 SDK 上开发完成的。在单核环境下，两个功能单独使用没有问题，同时开启也经过了长时间压力测试，稳定运行，没有发现任何异常。

后来产品切换到双核 SDK，业务代码和驱动配置没有做任何改动，直接迁移过来。

---

## 问题现象

双核环境下，两个功能单独使用依然正常。但在压力测试中先开启空间音效、再按键开启心率检测，心率功能出现偶发失效，表现出三种不同程度的异常：

- **LED 完全不亮**：按下心率按键后，PPG 传感器的发光 LED 没有点亮，心率检测根本没有启动。对应按键触发 `hx3011_init(HRS_MODE)` 时，LED 驱动电流等寄存器配置被 IIC 冲突打乱，芯片没有正确进入测量模式
- **LED 亮但数值异常**：LED 正常点亮，说明初始配置阶段完整，但后续周期性读取心率数据的 IIC 事务被打乱，读到乱序数据，心率值明显错误

问题复现率在 30%～60% 之间，连续频繁开关两个功能时几乎必现。同样的代码，单核没问题，双核出问题，而且只在两个功能同时开启时才触发——这是排查的起点。

---

## 排查过程

### 第一步：排除硬件和配置问题

首先确认硬件连接和驱动配置没有变动，检查供电、上拉电阻等硬件因素，均无异常。把两个功能的开启顺序互换，问题依然随机出现，排除顺序依赖。

### 第二步：查看驱动错误日志，定位失败点

规范的驱动对每次 IIC 事务都有 ACK 检测和错误输出。复现时串口已经有明确的输出：

```
hrSensor iic r err 0
```

对应的是读操作第一阶段——发出设备地址（写标志）后，从机没有回 ACK。

心率血氧的读操作是标准的 I2C 复合读时序（Repeated START）：

```
① START → 设备地址 + 写标志 → ACK   ← 失败点在这（err 0）
② 寄存器地址               → ACK
③ Repeated START → 设备地址 + 读标志 → ACK
④ 读取数据（逐字节 ACK，末字节 NACK）
⑤ STOP
```

冲突最容易暴露在①③这两个寻址阶段：从机依赖 START 条件同步自己的状态机，再靠 7 位地址判断是否应该响应。IMU 一旦在此时注入 GPIO 操作，START 条件被污染、或地址位被混淆，从机无法识别自己被寻址，直接不回 ACK——**问题在事务最前端就已经发生，还没到数据读取阶段**。

### 第三步：飞线接逻辑分析仪抓波形

两颗传感器都是封装芯片，引脚无法直接探测。在 PCB 上找到 SCL 和 SDA 线路上的上拉电阻焊盘，从焊盘飞线引出，接到逻辑分析仪的输入通道，同时抓两路信号。

问题复现时截到异常帧：SCL 保持高电平期间，SDA 出现了不应有的跳变。在 I2C 协议里，SCL 高期间 SDA 跳变意味着 START 或 STOP，从机状态机会因此复位，后续的地址帧全部解析错误，自然不会给 ACK。

**结论**：总线波形在传输过程中被"插入"了额外的电平跳变，最可能的原因是另一段代码在同一时刻也在操作同一组 GPIO。

### 第四步：审查代码，找到根因

重新审视两个功能的底层实现。

**硬件资源**：SDK 只提供一套软件模拟 IIC（Soft IIC 0），两颗传感器挂在同一组引脚上：

```
SCL → IO_PORTC_06
SDA → IO_PORTC_07

├── ICM42670P（IMU，空间音效）
└── HX3011（PPG，心率血氧）
```

**IMU 驱动**（`icm_42670p.c`）——直接裸调 IIC，没有任何锁保护：

```c
u32 icm42670p_I2C_Read_NBytes(unsigned char devAddr, ...)
{
    iic_start(icm42670p_info->iic_hdl);
    iic_tx_byte(icm42670p_info->iic_hdl, devAddr);
    // ...
    iic_stop(icm42670p_info->iic_hdl);
    // 无任何互斥保护
}
```

**HR 驱动**（`hrSensor_manage.c`）——每次 IIC 事务都加了 spinlock：

```c
u8 hrsensor_read_nbyte(u8 r_chip_id, ...)
{
    spin_lock(&sensor_iic);
    iic_start(hrSensor_info->iic_hdl);
    // ...
    iic_stop(hrSensor_info->iic_hdl);
    spin_unlock(&sensor_iic);
}
```

**问题一目了然**：HR 驱动用 spinlock 保护自己，但 IMU 驱动完全不知道这把锁的存在，从不持锁。HR 的锁对 IMU 没有任何约束力。

BR52 是真正的双核处理器（`CPU_CORE_NUM = 2`），空间音效的 IMU 数据采集运行在 Core 0，心率血氧运行在 Core 1，两个核心天然并行执行。一旦空间音效开启，IMU 任务就在 Core 0 上持续轮询 IIC，永不停歇。无论 HR 在什么时候操作 IIC，IMU 随时都可能同时在操作同一组 GPIO，波形互相叠加，从机完全无法解析。

### 第五步：对比单核 SDK，理解为何之前没有问题

单核下任务切换依赖 OS tick。OS tick 本质上是一个专门给 RTOS 用来做任务调度的硬件计数器，类似 FreeRTOS 里的系统滴答定时器（SysTick），按固定周期触发中断，OS 在中断里判断要不要切换任务。软件 IIC 在执行整个 IIC 事务期间会调用 `local_irq_disable` 关中断——关中断不是把计数器停掉，而是让 CPU 暂时不响应中断，OS tick 到点触发了中断信号，CPU 直接忽略，等开中断后再处理。这样整个 IIC 事务期间就不会发生任务切换，时序不会被打断。即便如此，整个事务执行时间本身也只有几微秒，OS tick 1ms 才触发一次，本来命中的概率就极低。

`local_irq_disable` 在双核下则完全失效——它只能屏蔽当前这个核的中断，对另一个核毫无约束。两个核天然同时执行，不需要等任何调度时机，只要 IMU 持续活跃，冲突就是必然发生的事。

---

## 根本原因

IMU 驱动与 HR 驱动共用同一套软件 IIC 总线，但 IMU 驱动的 IIC 操作完全没有互斥保护。双核环境下，两个核心同时操作同一对 GPIO 进行 bit-bang，波形互相叠加，从机状态机被异常跳变复位，通信失败。

HR 驱动虽然自身有 spinlock，但 IMU 驱动从不参与，这把锁形同虚设。

---

## 修复方案

找到根因后，第一个想法是沿用单核的思路——关中断。但双核下这条路走不通，原因有两点：一是没有"同时关掉两个核中断"的接口，每个核的中断控制器是独立的，两次调用之间本身就有时间差，不具备原子性；二是更根本的问题，关中断只能阻止中断触发的任务切换，但另一个核的正常代码执行完全不受影响，Core 0 关了自己的中断，Core 1 照样在跑。

多核环境下的互斥必须依赖硬件提供的原子操作。spinlock 正是基于此：底层用一条硬件原子指令（这颗 JL 芯片是 `testset`，单条指令完成读取并置位，硬件保证不可被打断），两个核同时去抢，只有一个能拿到，另一个原地空转等待。这是真正跨核的互斥，不依赖任何软件调度机制。

选 spinlock 而不是 OS 的 mutex，是因为 IIC 单次事务耗时极短（几十～几百微秒），spinlock 空转等待的代价远小于 mutex 触发 OS 任务调度的切换开销。

在总线层面建立统一的互斥锁，所有访问该 IIC 总线的驱动必须在操作前持锁。

**改动**：将 HR 驱动中已有的 `sensor_iic` 锁声明为全局锁，所有 IMU 驱动引入并统一使用：

```c
// hrSensor_manage.c：将锁改为全局可见
spinlock_t g_iic0_bus_lock;

// icm_42670p.c：引入同一把锁
extern spinlock_t g_iic0_bus_lock;

u32 icm42670p_I2C_Read_NBytes(unsigned char devAddr, ...)
{
    spin_lock(&g_iic0_bus_lock);      // 新增
    iic_start(icm42670p_info->iic_hdl);
    // ...
    iic_stop(icm42670p_info->iic_hdl);
    spin_unlock(&g_iic0_bus_lock);    // 新增
    return i;
}
```

对 SDK 中其余无保护的 IMU 驱动（`qmi8658c.c`、`lsm6dsl.c`、`sh3001.c`、`mpu9250.c`）做同样处理。`gSensor_manage.c` 中的 `extern spinlock_t sensor_iic` 声明同步改为新名称。

修复后，两个功能同时开启压力测试，心率数据稳定正常，0 失败。

---

## 事后复盘

排查这个问题让我意识到几件事：

- **单核验证通过不代表双核安全**。单核下靠时序侥幸掩盖的并发问题，在真正双核并行时会暴露得一览无余。
- **共享资源的锁必须对所有使用者生效**。只给一方加锁，而另一方完全不参与，这把锁就是摆设。
- **排查这类问题要先想清楚"谁在操作同一个硬件"**。波形异常直接定位到 GPIO 操作冲突，再从这个方向去找所有访问路径，比从业务逻辑层反推要快得多。
