## 定时器（Timer）与 PWM 在 TWS 耳机 SDK 中的应用

### 一、基础概念梳理

#### 定时器（Timer）

定时器的本质是一个计数器，由时钟源驱动，计数到预设值时触发中断或事件。

**硬件定时器（Hardware Timer）**

独立于 CPU 运行，精度高、抖动小，常用于 PWM 生成、边沿捕获、音频采样节拍等精确时序场合。

**时钟源链路（从源头到计数）：**

```
晶振（XTAL，如 24MHz）
  └─ PLL 倍频 / 总线分频器  →  系统时钟（如 192MHz）
       └─ 定时器时钟树分频  →  定时器输入时钟 TIMxCLK
            └─ 定时器内部预分频寄存器（PSC）
                 └─ 实际计数频率 = TIMxCLK / (PSC + 1)
                      └─ 计数器（CNT）向上/向下计数
                           └─ 溢出/比较匹配 → 触发中断或翻转 IO
```

三个核心寄存器：
- **PSC（预分频寄存器）**：对输入时钟再次分频，决定计数速率
- **ARR（自动重装寄存器）**：决定计数周期（溢出值），即 PWM 频率
- **CCR（捕获/比较寄存器）**：决定 PWM 占空比（输出比较）或记录边沿时刻（输入捕获）

**软件定时器（Software Timer）**

软件定时器的 Tick 本质上仍由硬件定时器提供——在 ARM Cortex-M 系列中，通常由片内**系统滴答定时器（SysTick）**充当这个角色：
- **裸机场景**：SysTick 每 1ms 产生一次中断，中断服务函数里递增全局计数，所有软件定时器在中断里遍历链表做倒计时；
- **引入 RTOS 后**：SysTick 同样作为系统时基，但中断里改为调用 OS 的 `xPortSysTickHandler`（FreeRTOS）或等价接口，OS 以此驱动任务调度和软件定时器服务。所以软件定时器是"搭 OS Tick 的便车"，并不是独立的硬件资源。

关键特点：
- 精度依赖 Tick 频率（通常 1ms），适用于 ms 级非实时场景
- 数量不受硬件限制，可按需注册
- 低功耗进睡眠后 Tick 可能停止，需区分"睡眠安全/不安全"两类接口
  - 这里参考FreeRTOS的低功耗机制，已经有文档了。


**SDK 中的软件定时器接口（以 JL SDK 为例）：**

```c
// ① 周期定时器（循环回调）——sys_timer_add
// 每隔 period_ms 触发一次 callback，返回 timer_id 用于后续管理
int timer_id = sys_timer_add(void *priv, void (*callback)(void *), u32 period_ms);

// 修改周期
sys_timer_modify(timer_id, new_period_ms);

// 删除定时器
sys_timer_del(timer_id);

// ② 单次超时回调（timeout）——sys_timeout_add / sys_hi_timeout_add
// 到期后只触发一次，不自动重复；hi 版本优先级更高，适合灯效状态机等时序敏感场景
int timeout_id = sys_timeout_add(void *priv, void (*callback)(void *), u32 delay_ms);
int hi_id      = sys_hi_timeout_add(void *priv, void (*callback)(void *), u32 delay_ms);
```

两者使用对比：

| 接口 | 触发方式 | 典型用途 |
|------|---------|---------|
| `sys_timer_add` | 周期重复 | 充电电流轮询、电量检测、ANC增益步进 |
| `sys_timeout_add` | 单次到期 | 按键长按判断、超时复位看门 |
| `sys_hi_timeout_add` | 单次到期（高优先级） | LED灯效状态机序列切换 |

#### PWM（脉冲宽度调制）

PWM 通过改变方波的**占空比**（高电平时间 / 周期）来等效调节输出"模拟量"。

- **频率**：决定周期，影响响应速度和可听噪声（LED驱动需 > 1kHz 防闪烁，音频需 > 20kHz）
- **占空比**：0%~100%，决定等效输出幅度（亮度、电流、转速）
- **分辨率**：计数位数决定，如 10-bit → 1024 档精度

**输出比较（Output Compare）原理——PWM 如何生成：**

PWM 的本质是硬件定时器利用**输出比较**功能，**周期性地输出方波**，每个周期内都有高电平和低电平两段。**周期固定不变**，通过调整高电平在周期中持续的时间长短来体现占空比：

```
CNT 从 0 计数到 ARR（周期固定）：

  0 ─────────────── CCR ─────────────── ARR
  |←── 高电平时间 ──→|←── 低电平时间 ──→|

  CNT < CCR  →  IO 输出高电平
  CNT ≥ CCR  →  IO 输出低电平
  CNT = ARR  →  溢出归零，进入下一个周期（波形重复）

周期 T  = (ARR + 1) × (PSC + 1) / TIMxCLK   ← ARR 决定，固定不变
占空比  = CCR / (ARR + 1) × 100%             ← 只改 CCR，不动 ARR
```

调节占空比时，**只修改 CCR**（高电平持续时间），ARR 和时钟不动，所以频率始终不变，变的只是每个周期内高电平占的比例。

**输入捕获（Input Capture）原理——测量外部脉冲宽度：**

输入捕获与输出比较使用同一组 CCR 寄存器，但方向相反——不是 CPU 写 CCR，而是硬件在检测到 IO 边沿时**自动把当前 CNT 值锁存进 CCR**，并触发中断：

```
外部信号边沿（上升/下降）到来
  → 硬件自动把 CNT 当前值锁存到 CCR
  → 产生捕获中断，CPU 读取 CCR
  → 两次捕获值之差 × 计数周期 = 脉冲宽度（时间）
```

典型用途：红外遥控解码（测量高/低电平持续时间）、超声波测距、频率测量。

**三种常见 PWM 生成方式：**

1. 硬件 PWM（定时器输出比较）：精度高，CPU 零负担，修改 CCR 即可调占空比
2. 软件 PWM（GPIO 翻转 + 定时中断）：灵活但 CPU 占用高，频率受中断延迟影响
3. PWM_LED 专用模块：JL 芯片内置，Sigma-Delta 架构，专门用于 LED 调光，支持呼吸渐变

### 二、在 TWS 耳机 SDK 中的具体应用场景

#### 场景1：LED 呼吸灯 / 闪烁灯效（PWM_LED 模块）

**原理：** JL 芯片内置专用 `PWM_LED` 硬件模块，支持呼吸变化、固定占空比、交替输出三种模式，通过配置 `pwm_led_platform_data` 结构体驱动。

**关键配置字段：**
```c
struct pwm_led_platform_data {
    u8  port0, port1;              // 输出引脚（支持双IO交替）
    u32 pwm_cycle;                 // PWM 底层周期，单位 10us
    u32 ctl_cycle;                 // 控制周期（一次呼吸/闪烁的总时长），单位 ms
    u8  out_mode;                  // 0:固定输出 1:输出两次 2:呼吸渐变模式
    u8  h_pwm_duty;                // 高亮占空比 0~100
    u8  l_pwm_duty;                // 低亮占空比 0~100
    u32 pwm_duty_max_keep_time;    // 最亮保持时间 ms（呼吸模式）
    u32 pwm_out_time;              // 单次呼吸上升时间 ms
    u8  alternate_out;             // 双IO交替输出使能
};
```

**上层 LED UI 状态机（led_ui_api.c）** 使用 `sys_hi_timeout_add()` 控制状态转移时序，实现"亮 500ms → 灭 200ms → 循环"等复合灯效序列：
```c
obj->timer = sys_hi_timeout_add((void *)uuid, led_do_next_action,
                                status->time_msec * LED_UI_FOLLOW_HW_TIME_MS);
```

**面试一句话总结：** LED 呼吸灯底层用 PWM_LED 硬件模块控制亮度渐变，上层用软件定时器驱动状态机切换灯效序列。

#### 场景2：触摸按键消抖、多击/长按识别（usr_timer）

**原理：** 触摸按键事件需要区分单击、双击、三击、长按、持续按住，这些都通过定时器窗口来判断。

```c
// 多击识别：首次触发启动定时器，计时窗口内再次触发则计数+1并刷新定时
if (arg->short_timer == 0) {
    arg->click_cnt = 1;
    arg->short_timer = usr_timeout_add((void *)ch_idx,
                        lp_touch_key_short_click_time_out_handle,
                        pdata->short_click_check_time, 1);  // e.g. 300ms 窗口
} else {
    arg->click_cnt++;
    usr_timer_modify(arg->short_timer, pdata->short_click_check_time); // 重置窗口
}

// 长按识别：按下时启动长按定时器，时间到触发长按事件
arg->long_timer = usr_timeout_add((void *)ch_idx,
                    lp_touch_key_trigger_long_click_event,
                    pdata->long_click_check_time, 1);  // e.g. 800ms
```

**面试一句话总结：** 按键多击靠"计时窗口+计数器"实现，长按靠"超时未松手"判断，全部基于软件定时器。

---

#### 场景4：ANC 动态增益平滑（sys_timer + usr_timer）

**ANC 动态增益平滑（audio_anc_common_plug.c）：**
模式切换时（ANC 开/关/环境音），增益不能突变否则会有"嗡"声，用 10ms 定时器做渐变：

```c
// 每 10ms 步进 5 个增益单位，逐步逼近目标增益
music_dyn_hdl->loud_timeid = sys_timer_add(NULL,
                    audio_anc_music_dyn_gain_resume_timer, 10);
```

**面试一句话总结：** 音频增益突变会产生爆破声，用高频定时器做分步渐变是消除"pop noise"的常见手段。

#### 场景5：电池电量检测（usr_timer）

```c
// 周期性采样 VBAT，做滤波防抖后上报电量百分比
vbat_fast_timer = usr_timer_add(NULL, vbat_check, VBAT_DETECT_TIME, 1);
```

- 充电状态：较高频率（如 200ms）检测，快速响应电量变化

#### 场景7：红外光感入耳检测（JSA1221）——IIC + 内部 ADC，非输入捕获

> 容易混淆：红外**光感传感器**（JSA1221）虽然也叫"红外"，但其工作方式与遥控解码完全不同。

JSA1221 是一颗集成了红外 LED 发射器和 ADC 的近接传感器，芯片内部完成光电转换和 AD 采样，MCU 只需通过 **IIC 总线**读取 16 位 ADC 结果：

```c
// jsa1221.c：通过 IIC 读取近接传感器 ADC 值
u16 jsa1221_read_pxs_data(void)
{
    u8 lower_data, upper_data;
    irSensor_read(JSA_1221_READ_ADDR, JSA_PXS_DATA1, &lower_data, 1); // IIC 读低字节
    irSensor_read(JSA_1221_READ_ADDR, JSA_PXS_DATA2, &upper_data, 1); // IIC 读高字节
    return (upper_data * 256 + lower_data);  // 合成 16-bit ADC 值
}

// 检测逻辑：ADC 值与阈值比较判断入耳/离耳
enum irSensor_event jsa1221_check(void *para)
{
    if (jsa1221_pxs_int_flag()) {
        u16 pxs_data = jsa1221_read_pxs_data();
        return (pxs_data < jsa_low_th) ? IR_SENSOR_EVENT_FAR : IR_SENSOR_EVENT_NEAR;
    }
    return IR_SENSOR_EVENT_NULL;
}
```

SDK 还支持另一种不带专用传感器的方案，直接读 ADC 通道或 GPIO 电平（`in_ear_detect.c`）：

```c
// 方案1：ADC 模式——采样红外接收管的电压值与阈值比较
u32 ear_ad = adc_get_value(TCFG_EAR_DETECT_AD_CH);
if (ear_ad >= TCFG_EAR_DETECT_AD_VALUE) { /* 入耳 */ }

// 方案2：GPIO 模式——直接读接收管输出的数字电平
if (gpio_read(TCFG_EAR_DETECT_IRO2) == TCFG_EAR_DETECT_DET_LEVEL) { /* 入耳 */ }
```

### 四、面试回答思路（参考）

> **问：你对定时器和 PWM 了解多少？在 TWS SDK 中有哪些应用？**

**可以这样回答：**

定时器分硬件和软件两类。硬件定时器独立计数、精度高，适合 PWM 生成和信号捕获；软件定时器依赖系统 Tick，而 Tick 本身也是由硬件定时器（如 SysTick）产生的——裸机下 SysTick 中断里直接维护软件定时器链表，引入 RTOS 后 SysTick 转交给 OS 作为任务调度时基，软件定时器则由 OS 统一管理。软件定时器的优点是数量不受硬件限制，缺点是精度受调度抖动影响，且进低功耗睡眠后 Tick 可能停止。在 SDK 里，软件定时器有两类接口：`sys_timer_add` 用于**周期回调**（如轮询充电电流），`sys_timeout_add` / `sys_hi_timeout_add` 用于**单次超时回调**（如按键长按判断、LED 灯效序列切换）。

PWM 本质是通过调节占空比来等效输出模拟量，关键参数是频率和分辨率。

在我们的 TWS 耳机 SDK 里，这两者的应用非常广泛：

1. **LED 灯效**：底层用 JL 专用 PWM_LED 模块实现呼吸渐变，上层用软件定时器驱动状态机切换灯效序列；
2. **按键识别**：触摸多击用"计时窗口+计数"配合 usr_timer 实现，长按用超时检测实现；
4. **音频处理**：切歌或 ANC 模式切换时用 2ms/10ms 高频定时器步进调整增益，消除爆破声；

核心体会是：**软件定时器适合业务逻辑（状态转移、超时保护），硬件定时器/PWM 适合需要精确时序的硬件控制（波形生成、信号捕获）。两者结合才能高效完成嵌入式设计。**
