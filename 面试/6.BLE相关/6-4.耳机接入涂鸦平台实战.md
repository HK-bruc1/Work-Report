# 6-4. 耳机接入涂鸦平台实战

> **定位**：面向嵌入式音频方向的面试复习文档。以 JL 双核 SDK（AC709N / BR52）为实战案例，
> 深度剖析耳机如何将涂鸦 BLE SDK 移植到 JL 平台，以及涂鸦平台与 JL RCSP 协议的本质差异。

---

## 零、为什么要接入涂鸦平台？

| 维度 | 仅使用 RCSP | 接入涂鸦平台 |
|------|------------|------------|
| 生态 | JL 私有生态，只能配合 JieLi App | 涂鸦全球生态，支持 Smart Life / OEM App |
| 云能力 | 无云端 | 设备上云，支持场景联动、OTA 云端管理 |
| 安全 | 无加密（RCSP 明文传输） | AES-128 + 设备三元组认证 |
| 开发成本 | JL SDK 直出，完全开源 | 需要移植涂鸦 BLE SDK + 适配层 |

---

## 一、涂鸦 BLE 协议 vs RCSP 协议：横向对比

### 1.1 协议栈层次对比

```
┌──────────────────────────────────────────────────────────────────┐
│                         手机 App 层                               │
│     JieLi App (RCSP)          │       Smart Life / OEM App       │
└─────────────┬─────────────────┴────────────┬─────────────────────┘
              ▼                               ▼
┌─────────────────────────┐   ┌──────────────────────────────────┐
│       RCSP 协议层         │   │       Tuya BLE 协议层 v4.0       │
│  OpCode: 0xE1~0xE9 OTA  │   │  帧: HEAD(0x0055)+SEQ+TYPE+DATA  │
│  OpCode: 0x40~0x4F EQ   │   │  AES-128-ECB 加密               │
│  明文传输，JL 私有          │   │  三元组认证 (PID+DevID+AuthKey) │
└─────────────┬───────────┘   └─────────────────┬────────────────┘
              ▼ GATT ATT                         ▼ GATT ATT
┌─────────────────────────┐   ┌──────────────────────────────────┐
│  Service: AE00 (16-bit)  │   │  Service: FD50 (16-bit)          │
│  Char Notify:  AE01      │   │  Char Write:  128-bit UUID 0001  │
│  Char Write:   AE02      │   │  Char Notify: 128-bit UUID 0002  │
│  (16-bit UUID)           │   │  Char Read:   128-bit UUID 0003  │
└─────────────┬───────────┘   └─────────────────┬────────────────┘
              └──────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│              JL BLE GATT 框架层（两协议共用底层）                  │
│  app_ble_hdl_alloc() / app_ble_profile_set()                    │
│  app_ble_att_write_callback_register()                          │
│  app_ble_att_send_data()                                        │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 关键差异汇总

| 对比项 | RCSP 协议 | 涂鸦 BLE 协议 |
|--------|----------|--------------|
| **UUID 类型** | 16-bit 短 UUID | 128-bit 全 UUID（厂商自定义）|
| **Service UUID** | `0xAE00` | `0xFD50` |
| **加密** | 无（明文） | AES-128-ECB/CBC（mbedtls）|
| **身份认证** | 无 | 三元组（PID + DeviceID + AuthKey）|
| **SDK 性质** | 完全开源，在 JL SDK 内 | 核心闭源（`libtuya_lib.a`）+ 开放适配层 |
| **任务模型** | 跑在现有 JL 任务中 | 涂鸦 SDK 自建独立 RTOS 任务 |
| **云端对接** | 无 | 涂鸦云（设备绑定、OTA 推送）|
| **三元组存储** | 无此概念 | License 区（Flash 末尾 256B）或 VM 分区 |

> **为什么用 128-bit UUID？**
> 16-bit UUID 需要向蓝牙 SIG 申请付费分配，128-bit 是厂商自定义空间，涂鸦自行生成。
> 代价：广播数据额外消耗 16 字节，ATT PDU 中 UUID 字段更长。

## 二、涂鸦 BLE SDK 的组成结构（移植前要搞清楚）

```
tuya_protocol/
├── tuya_ble_config.h              ← [第1步] 配置宏，移植的起点
├── tuya_protocol.c                ← [第2步] GATT Profile 注册 + ATT 回调
├── tuya_event.c                   ← 涂鸦事件处理入口
├── tuya_ble_sdk_version.h         ← SDK 版本号
│
├── port/                          ← [第3步] 平台适配层（核心移植工作）
│   ├── tuya_ble_port.h            ← 定义所有需要实现的接口（约30个函数）
│   ├── tuya_ble_port_JL.c         ← JL 平台的具体实现（约1170行）← 重点文件
│   ├── tuya_ble_port.c            ← 通用的 port 实现（部分函数）
│   └── tuya_ble_port_peripheral.c ← 外设（UART等）适配
│
├── sdk/
│   ├── include/                   ← 涂鸦 SDK 头文件（开放）
│   │   ├── tuya_ble_api.h         ← 对外 API（init、dp_report 等）
│   │   ├── tuya_ble_main.h        ← SDK 初始化
│   │   └── tuya_ble_type.h        ← 类型定义
│   ├── src/                       ← 部分开源的 SDK 源码
│   │   ├── tuya_ble_main.c        ← SDK 主逻辑（含任务调度）
│   │   ├── tuya_ble_event.c       ← 事件分发
│   │   └── tuya_ble_storage.c     ← 存储抽象层
│   └── lib/
│       └── libtuya_lib.a          ← 闭源核心库（协议帧、安全、状态机）
│
└── app/
    └── demo/
        ├── tuya_ble_app_demo.c    ← [第4步] 业务逻辑 + 三元组读取 + SDK 初始化
        ├── tuya_app_func.c        ← DP 数据处理、TWS 同步
        └── tuya_ota.c             ← OTA 处理
```

> **面试要点**：移植涂鸦 SDK 并不是"引入一个静态库那么简单"。
> `libtuya_lib.a` 只是协议核心的闭源部分，实际还需要：
> 1. **约 30 个平台接口的具体实现**（`tuya_ble_port_JL.c`，1170 行）
> 2. **GATT Profile 的注册**（对接 JL BLE 框架）
> 3. **三元组的获取与存储方案**（Flash License 区 / VM 分区）
> 4. **业务层的 DP 事件回调实现**（耳机功能与涂鸦 DP 点的映射）

## 三、移植全过程：Step by Step

### Step 1：配置 `tuya_ble_config.h`

这是移植的起点，需要根据平台能力设置以下关键宏：

```c
// 是否使用 OS（JL 平台有 RTOS，设为 1）
#define TUYA_BLE_USE_OS  1

// 涂鸦 SDK 是否自建任务（设为 1，SDK 会调用 tuya_ble_os_task_create 创建自己的任务）
#define TUYA_BLE_SELF_BUILT_TASK  1
#define TUYA_BLE_TASK_PRIORITY    1
#define TUYA_BLE_TASK_STACK_SIZE  (256 * 10)  // 2560 字节

// 设备接入方式（纯 BLE + 可通过 BLE 注册到涂鸦云）
#define TUYA_BLE_DEVICE_COMMUNICATION_ABILITY \
    (TUYA_BLE_DEVICE_COMMUNICATION_ABILITY_BLE | \
     TUYA_BLE_DEVICE_COMMUNICATION_ABILITY_REGISTER_FROM_BLE)

// 安全类型：使用 AuthKey 加密（最常用）
#define TUYA_BLE_SECURE_CONNECTION_TYPE  TUYA_BLE_SECURE_CONNECTION_WITH_AUTH_KEY

// NV 存储的地址映射（虚拟地址，port 层会映射到实际 VM 分区 ID）
#define TUYA_BLE_AUTH_FLASH_ADDR         0x00   // 鉴权数据主区
#define TUYA_BLE_AUTH_FLASH_BACKUP_ADDR  0x100  // 鉴权数据备份区
#define TUYA_BLE_SYS_FLASH_ADDR          0x200  // 系统数据主区
#define TUYA_BLE_SYS_FLASH_BACKUP_ADDR   0x300  // 系统数据备份区

// MTU 最大值
#define TUYA_BLE_DATA_MTU_MAX  243

// 固件信息（用于涂鸦云识别版本）
#define TUYA_BLE_APP_VERSION_STRING  "1.0.0"
#define APP_PRODUCT_ID               "xxxxxxxx"  // 8字节 PID，在涂鸦 IoT 平台获取
```

### Step 2：注册 GATT Profile（`tuya_protocol.c`）

涂鸦 v4.0 的 GATT Profile 是一段编译时确定的二进制描述数组，由 JL 工具生成：

```c
// tuya_protocol.c
// 条件编译：v3 和 v4 的 Profile 不同
#if (TUYA_BLE_PROTOCOL_VERSION_HIGN == 0x04)

const uint8_t tuya_profile_data[] = {
    // Primary Service: 0xFD50
    // ...（JL GATT 格式的二进制描述，由工具生成）

    // Characteristic 1: Write/Write-Without-Response（手机→耳机）
    // UUID: 00000001-0000-1001-8001-00805F9B07D0
    // Handle: 0x0006

    // Characteristic 2: Notify（耳机→手机）
    // UUID: 00000002-0000-1001-8001-00805F9B07D0
    // Handle: 0x0008, CCCD: 0x0009

    // Characteristic 3: Read（手机主动读设备信息）
    // UUID: 00000003-0000-1001-8001-00805F9B07D0
    // Handle: 0x000b
};

// 对应 Handle 宏定义
#define ATT_CHARACTERISTIC_00000001_VALUE_HANDLE  0x0006
#define ATT_CHARACTERISTIC_00000002_VALUE_HANDLE  0x0008
#define ATT_CHARACTERISTIC_00000002_CCCD_HANDLE   0x0009
#define ATT_CHARACTERISTIC_00000003_VALUE_HANDLE  0x000b
#endif

// 向 JL BLE 框架注册
void tuya_ble_profile_init(void)
{
    tuya_ble_hdl = app_ble_hdl_alloc();             // 申请协议句柄
    app_ble_profile_set(tuya_ble_hdl, tuya_profile_data);  // 注册 Profile
    app_ble_hdl_uuid_set(tuya_ble_hdl, 0xFD50);     // 绑定 Service UUID
    app_ble_att_read_callback_register(tuya_ble_hdl, tuya_att_read_callback);
    app_ble_att_write_callback_register(tuya_ble_hdl, tuya_att_write_callback);
    app_ble_att_server_packet_handler_register(tuya_ble_hdl, tuya_cbk_packet_handler);
}
```

### Step 3：实现平台适配层 `tuya_ble_port_JL.c`（核心移植工作）

`tuya_ble_port.h` 定义了约 30 个接口，全部需要在 `tuya_ble_port_JL.c` 中实现。
这是移植的主体工作，涵盖以下 7 个类别：

#### 3.1 BLE 收发接口

```c
// 【发送】涂鸦 SDK 调此接口发送 Notify 数据
// 实现：通过 operation table 转发给 JL BLE 框架
__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_gatt_send_data(const uint8_t *p_data, uint16_t len)
{
    // ble_operation 是在 tuya_protocol.c 注册的回调表
    if (ble_operation && ble_operation->send_data) {
        int ret = ble_operation->send_data(NULL, (void *)p_data, len);
        return ret ? TUYA_BLE_ERR_UNKNOWN : TUYA_BLE_SUCCESS;
    }
    return TUYA_BLE_ERR_INVALID_STATE;
}
// send_data 最终调用：
// app_ble_att_send_data(tuya_ble_hdl, 0x0008, data, len, ATT_OP_NOTIFY)

// 【接收】JL ATT 写回调中调此接口，将数据送入涂鸦 SDK
u8 JL_tuya_ble_gatt_receive_data(u8 *p_data, u16 len)
{
    return tuya_ble_gatt_receive_data(p_data, len);  // 进入 libtuya_lib.a 处理
}
```

#### 3.2 广播控制接口

```c
// 涂鸦 SDK 会在鉴权成功后动态修改广播包（加入 binding 状态）
__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_gap_advertising_adv_data_update(
    uint8_t const *p_ad_data, uint8_t ad_len)
{
    tuya_set_adv_disable();              // 暂停广播
    tuya_set_adv_data((u8*)p_ad_data, ad_len);  // 更新广播数据
    tuya_set_adv_enable();               // 重启广播
    return TUYA_BLE_SUCCESS;
}
```

#### 3.3 定时器接口（映射到 JL 系统定时器）

```c
// JL 平台没有像 FreeRTOS 那样的通用定时器 API，
// 需要区分单次和周期两种定时器，分别映射到不同 JL 函数

__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_timer_create(
    void **p_timer_id, uint32_t timeout_value_ms,
    tuya_ble_timer_mode mode, tuya_ble_timer_handler_t timeout_handler)
{
    *p_timer_id = malloc(2);       // 用于存储 JL timer ID（u16）
    // 保存定时器参数供 start 时使用
    ble_timer.mode = mode;
    ble_timer.timeout_handler = timeout_handler;
    ble_timer.timeout_value_ms = timeout_value_ms;
    return TUYA_BLE_SUCCESS;
}

__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_timer_start(void *timer_id)
{
    // 根据模式选择单次或周期定时器
    u16(*sys_timer_create)(void *, void(*)(void *), u32);
    sys_timer_create = (ble_timer.mode == TUYA_BLE_TIMER_SINGLE_SHOT)
                     ? sys_timeout_add    // 单次
                     : sys_timer_add;     // 周期
    *(u16*)timer_id = sys_timer_create(NULL, ble_timer.timeout_handler,
                                       ble_timer.timeout_value_ms);
    return TUYA_BLE_SUCCESS;
}
```

#### 3.4 NV 存储接口（映射到 JL VM 分区）

涂鸦 SDK 用虚拟地址访问 NV，JL 的适配层把虚拟地址映射为 VM 分区 ID：

```c
// 虚拟地址 → VM 分区 ID 的映射（在 tuya_ble_app_demo.c 中注册）
static uint8_t app_tuya_get_vm_id(u32 addr)
{
    switch (addr) {
    case TUYA_BLE_AUTH_FLASH_ADDR:        return CFG_USER_TUYA_INFO_AUTH;
    case TUYA_BLE_AUTH_FLASH_BACKUP_ADDR: return CFG_USER_TUYA_INFO_AUTH_BK;
    case TUYA_BLE_SYS_FLASH_ADDR:         return CFG_USER_TUYA_INFO_SYS;
    case TUYA_BLE_SYS_FLASH_BACKUP_ADDR:  return CFG_USER_TUYA_INFO_SYS_BK;
    default: return TUYA_BLE_ERR_INVALID_ADDR;
    }
}

// NV 写入实现
__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_nv_write(
    uint32_t addr, const uint8_t *p_data, uint32_t size)
{
    u8 index = tuya_get_vm_id_func(addr);   // 虚拟地址 → VM ID
    if (syscfg_write(index, p_data, size) != size) {
        return TUYA_BLE_ERR_INVALID_ADDR;
    }
    // 写入鉴权数据时，触发 TWS 同步
    if (addr == TUYA_BLE_AUTH_FLASH_ADDR) {
        os_taskq_post_msg(ble_task.ble_task_name, 1, BLE_TASK_MSG_INFO_SYNC_AUTH);
    } else if (addr == TUYA_BLE_SYS_FLASH_ADDR) {
        os_taskq_post_msg(ble_task.ble_task_name, 1, BLE_TASK_MSG_INFO_SYNC_SYS);
    }
    return TUYA_BLE_SUCCESS;
}

// NV 读取实现
__TUYA_BLE_WEAK tuya_ble_status_t tuya_ble_nv_read(
    uint32_t addr, uint8_t *p_data, uint32_t size)
{
    u8 index = tuya_get_vm_id_func(addr);
    syscfg_read(index, p_data, size);
    return TUYA_BLE_SUCCESS;
}
```

> **注意**：JL SDK 提供了另一种 NV 存储方式——文件模式（`NV_MODE_FILE`），
> 把数据存到 VFS 文件系统中的 `USERIF` 文件。
> Flash 空间较大时推荐文件模式，Flash 空间紧张时用 VM 模式（默认）。

#### 3.5 OS 任务与消息队列接口（最复杂的部分）

涂鸦 SDK 需要一个独立的 RTOS 任务来运行其事件循环。JL 没有 FreeRTOS 的标准接口，适配层把涂鸦的任务/队列 API 映射到 JL 的任务系统：

```c
// 涂鸦 SDK 启动时调此接口创建自己的任务
__TUYA_BLE_WEAK bool tuya_ble_os_task_create(
    void **pp_handle, char *p_name,
    void (*p_routine)(void *), void *p_param,
    uint16_t stack_size, uint16_t priority)
{
    // 映射到 JL 的 task_create()
    int ret = task_create(p_routine, p_param, p_name);
    if (ret == 0) {
        ble_task.ble_task_name = p_name;   // 保存任务名，后续 post 消息用
        os_sem_create(&ble_task.ble_sem, 0); // 用信号量模拟 suspend/resume
    }
    return ret ? false : true;
}

// 消息入队（涂鸦 SDK 内部事件分发）
__TUYA_BLE_WEAK bool tuya_ble_os_msg_queue_send(
    void *p_handle, void *p_msg, uint32_t wait_ms)
{
    if ((uint32_t)p_handle == m_cb_queue_table[0]) {
        // 业务回调队列：投递到 JL app_core 任务（而非涂鸦任务）
        int argv[3] = {(int)tuya_app_cb_handler, 1, (int)p_msg};
        os_taskq_post_type("app_core", Q_CALLBACK, 3, argv);
    } else {
        // SDK 内部事件队列：写入环形缓冲，通知涂鸦任务
        cbuf_write(&ble_queue.queue_hdl, p_msg, ble_queue.msg_size);
        os_taskq_post_msg(ble_task.ble_task_name, 2,
                          BLE_TASK_MSG_MSG_COMES, (u32)p_handle);
    }
    return true;
}
```

> **设计要点**：涂鸦的业务回调（DP 数据、OTA 数据）被投递到 `app_core` 任务执行，
> 而 SDK 内部的协议事件在涂鸦自建任务中处理。两个任务通过 JL 消息队列解耦。

#### 3.6 加密接口（对接 mbedtls）

```c
// AES-128-ECB 加密（用于数据帧加密）
__TUYA_BLE_WEAK bool tuya_ble_aes128_ecb_encrypt(
    uint8_t *key, uint8_t *input, uint16_t input_len, uint8_t *output)
{
    if (input_len % 16) { return false; }   // 必须是 16 的整倍数
    mbedtls_aes_context aes_ctx;
    mbedtls_aes_init(&aes_ctx);
    mbedtls_aes_setkey_enc(&aes_ctx, key, 128);  // key 就是 AuthKey 的前16字节
    while (input_len > 0) {
        mbedtls_aes_crypt_ecb(&aes_ctx, MBEDTLS_AES_ENCRYPT, input, output);
        input += 16; output += 16; input_len -= 16;
    }
    mbedtls_aes_free(&aes_ctx);
    return true;
}
// 同样还需要实现：AES-ECB 解密、AES-CBC 加解密、MD5、SHA256、HMAC-SHA256
// 全部基于 JL SDK 内置的 mbedtls 库实现
```

### Step 4：三元组读取与 SDK 初始化（`tuya_ble_app_demo.c`）

#### 4.1 三元组的存储位置

JL 平台有两种三元组存储方案，初始化时按优先级依次尝试：

```
Flash 布局（三元组相关）

Flash 末尾 256 字节（License 区）：
┌──────────────────────────────────────────────────────┐
│  flash_capacity - 256 + 0x50 (LIC_PAGE_OFFSET=80)   │
│                                                      │
│  _flash_of_lic_para_head:                            │
│    crc(2B) + string_len(2B) + ...                    │
│                                                      │
│  三元组字符串（ASCII，逗号分隔）:                       │
│    "tuya52c534229871,gqGPQQl4n540dc6sVPoGoh4fO7a8DzED,DC234E3EBD3D"
│     ←── DeviceID(16) ──→←────── AuthKey(32) ────→←MAC(12)→
└──────────────────────────────────────────────────────┘

VM 分区（备用，Flash License 区读取失败时使用）：
  syscfg_read(VM_TUYA_TRIPLE, ...)
```

#### 4.2 三元组读取流程

```c
// 先从 Flash License 区读取
static uint8_t read_tuya_product_info_from_flash(uint8_t *read_buf, u16 buflen)
{
    u32 flash_capacity = sdfile_get_disk_capacity();
    u32 flash_addr = flash_capacity - 256 + LIC_PAGE_OFFSET;
    u8 *lic_ptr = (u8*)sdfile_flash_addr2cpu_addr(flash_addr);

    // CRC 校验 License 头
    if (!license_para_head_check(lic_ptr)) { return FALSE; }

    lic_ptr += sizeof(_flash_of_lic_para_head);

    // 依次解析：DeviceID(16字符), AuthKey(32字符), MAC(12字符)
    // 格式：字段间用逗号分隔，ASCII 字符串
    tuya_get_one_info(lic_ptr, device_id);   // 16 bytes
    lic_ptr += 17;  // 16 + ','
    tuya_get_one_info(lic_ptr, auth_key);    // 32 bytes
    lic_ptr += 33;  // 32 + ','
    tuya_get_one_info(lic_ptr, mac_str);     // 12 bytes（十六进制字符串）
    parse_mac_data(mac_str, mac_addr);       // ASCII → 6字节二进制
    return TRUE;
}

void tuya_ble_app_init(void)
{
    device_param.device_id_len = 16;
    device_param.use_ext_license_key = 1;  // 使用外部 License（非 SDK 自管理）

    // 优先从 Flash License 区读取三元组
    flash_ret = read_tuya_product_info_from_flash(read_buf, sizeof(read_buf));

    if (flash_ret == TRUE) {
        // Flash 读取成功
        memcpy(device_param.device_id, read_buf, 16);
        memcpy(device_param.auth_key,  read_buf + 16, 32);
        parse_mac_data(read_buf + 48, device_param.mac_addr.addr);
    } else {
        // 回退：从 VM 分区读取（升级后 Flash 被擦除的情况）
        vm_read_result = syscfg_read(VM_TUYA_TRIPLE, &vm_triple, TUYA_TRIPLE_LENGTH);
        if (vm_read_result == TUYA_TRIPLE_LENGTH) {
            memcpy(device_param.device_id, &vm_triple, 16);
            memcpy(device_param.auth_key,  &vm_triple + 16, 32);
            parse_mac_data(&vm_triple + 48, device_param.mac_addr.addr);
        }
        // 若两者都失败，triple_info.flag = 0（设备无法接入涂鸦云）
    }

    // 设置 BLE MAC 地址（使用三元组中的 MAC，而非芯片默认 MAC）
    app_ble_set_mac_addr(tuya_ble_hdl, device_param.mac_addr.addr);

    // 设置 PID
    device_param.p_type = TUYA_BLE_PRODUCT_ID_TYPE_PID;
    device_param.product_id_len = 8;
    memcpy(device_param.product_id, APP_PRODUCT_ID, 8);  // 在配置文件中定义

    // 固件版本（涂鸦云 OTA 版本比对用）
    device_param.firmware_version = TY_APP_VER_NUM;
    device_param.hardware_version = TY_HARD_VER_NUM;

    // 注册 VM 地址映射函数
    tuya_get_vm_id_register(app_tuya_get_vm_id);

    // ★ 核心：初始化涂鸦 SDK
    // 内部会调用 tuya_ble_os_task_create() 创建独立任务
    // 并调用 tuya_ble_nv_init()、tuya_ble_rand_generator() 等接口
    tuya_ble_sdk_init(&device_param);

    // 注册业务回调（DP 数据接收、OTA、鉴权结果等）
    tuya_ble_callback_queue_register(tuya_app_cb_handler);
}
```

### Step 5：实现业务回调 `tuya_app_cb_handler()`

涂鸦 SDK 通过回调函数将所有事件上报给应用层：

```c
void tuya_app_cb_handler(tuya_ble_cb_evt_param_t *event)
{
    switch (event->evt) {
    case TUYA_BLE_CB_EVT_CONNECTE_STATUS:
        // BLE 连接/断开状态变化
        break;

    case TUYA_BLE_CB_EVT_DP_DATA_RECEIVED:
        // 收到手机下发的 DP 数据（功能控制）→ 解析并执行
        tuya_data_parse(event);
        break;

    case TUYA_BLE_CB_EVT_DP_QUERY:
        // 手机查询设备当前状态 → 上报所有 DP 点当前值
        tuya_info_indicate();
        break;

    case TUYA_BLE_CB_EVT_OTA_DATA:
        // 收到 OTA 固件数据 → 写入 Flash
        tuya_ota_proc(event->ota_data.type,
                      event->ota_data.p_data,
                      event->ota_data.data_len);
        break;

    case TUYA_BLE_CB_EVT_UNBOUND:
        // 解绑请求（用户删除设备）
        break;

    case TUYA_BLE_CB_EVT_DEVICE_RESET:
        // 恢复出厂设置
        break;

    case TUYA_BLE_CB_EVT_TIME_STAMP:
        // 时间同步（涂鸦云下发 Unix 时间戳）
        break;
    }
    tuya_ble_inter_event_response(event);  // 必须调用，SDK 内部状态更新
}
```

### Step 6：DP 数据处理（功能 → DP 点的映射）

```c
// DP（Data Point）是涂鸦平台对设备功能的抽象
// 每个功能对应一个 DP ID，格式：[DP_ID(1B) + Type(1B) + Len(2B) + Data(nB)]

void tuya_data_parse(tuya_ble_cb_evt_param_t *event)
{
    uint8_t dp_id   = event->dp_received_data.p_data[0];
    uint8_t type    = event->dp_received_data.p_data[1];
    uint16_t len    = TWO_BYTE_TO_DATA(&event->dp_received_data.p_data[2]);
    uint8_t *data   = &event->dp_received_data.p_data[4];

    switch (dp_id) {
    case 5:   // 音量设置
        tuya_set_music_volume(data[3]);
        tuya_sync_info_send(&data[3], APP_TWS_TUYA_SYNC_VOLUME); // 同步给从耳
        break;
    case 8:   // 降噪模式（ANC）
        anc_mode_switch(data[0], 1);
        break;
    case 12:  // 寻找设备（播放提示音）
        tuya_find_device(data[0]);
        break;
    case 18:  // EQ 参数设置（10 段 EQ）
        tuya_eq_info_deal(tuya_info, (char *)data);
        break;
    case 19~24: // 按键功能自定义（左右耳 × 3种操作）
        key_table_l/r[x] = tuya_key_event_swith(data[0]);
        syscfg_write(TUYA_SYNC_KEY_INFO, key_value_record, ...);
        tuya_sync_info_send(...);  // TWS 同步
        break;
    case 43:  // 修改蓝牙名称
        tuya_bt_name_deal(data, data_len);
        break;
    }

    // 所有 DP 操作后，上报当前状态给手机（确认执行结果）
    tuya_ble_dp_data_send(sn, DP_SEND_TYPE_ACTIVE,
                          DP_SEND_FOR_CLOUD_PANEL,
                          DP_SEND_WITH_RESPONSE,
                          event->dp_received_data.p_data, data_len + 4);
}
```

## 四、涂鸦 SDK 的任务模型（与 RCSP 的本质区别）

```
JL 系统任务全景（接入涂鸦后）

┌──────────────┐  ┌────────────────────┐  ┌──────────────────────┐
│  app_core    │  │  tuya_ble_task     │  │  btstack_task        │
│  (JL 主任务)  │  │  (涂鸦自建任务)     │  │  (BLE 协议栈任务)     │
│              │  │                    │  │                      │
│  处理业务逻辑  │  │  涂鸦 SDK 事件循环  │  │  HCI/LL 协议处理     │
│  DP 数据回调  │  │  协议帧收发        │  │  ATT 读写分发        │
│  RCSP 处理   │  │  NV 存储操作       │  │                      │
│  按键/UI 等   │  │  TWS 同步触发      │  │                      │
└──────┬───────┘  └─────────┬──────────┘  └──────────┬───────────┘
       │                    │                         │
       │ os_taskq_post      │ os_taskq_post_msg       │ ATT Write
       │ (Q_CALLBACK)       │ (BLE_TASK_MSG_*)        │ callback
       └────────────────────┴─────────────────────────┘
                    JL 任务间消息队列通信
```

**关键设计**：`tuya_ble_sdk_init()` 内部调用 `tuya_ble_os_task_create()` 创建了一个独立任务。这个任务专门跑涂鸦 SDK 的事件循环（协议帧解析、状态机驱动、NV 读写）。业务回调则被投递到 `app_core` 任务执行，避免在协议任务中执行耗时操作。

RCSP 没有独立任务，所有 RCSP 处理都在 `btstack_task` 或 `app_core` 中完成，调用链短但耦合高。

```c
// nv_write 时触发同步
// 当 TUYA_BLE_SYS_FLASH_ADDR 被写入（鉴权成功，SDK 更新 sys_settings）
// → 发消息给涂鸦任务 → 调用 tuya_tws_bind_info_sync()

void tuya_tws_bind_info_sync()
{
    if (tws_api_get_role() == TWS_ROLE_MASTER) {
        // 打包：login_key + device_virtual_id + bound_flag
        memcpy(tuya_tws_sync_info.login_key,
               tuya_ble_current_para.sys_settings.login_key, LOGIN_KEY_LEN);
        memcpy(tuya_tws_sync_info.device_virtual_id,
               tuya_ble_current_para.sys_settings.device_virtual_id,
               DEVICE_VIRTUAL_ID_LEN);
        tuya_tws_sync_info.bound_flag =
               tuya_ble_current_para.sys_settings.bound_flag;

        // 通过 JL TWS 私有链路发送给从耳
        tws_api_send_data_to_sibling(&tuya_tws_sync_info,
                                     sizeof(tuya_tws_sync_info),
                                     TWS_FUNC_ID_TUYA_AUTH_SYNC);  // = 'T','U','A','U'
    }
```

## 六、涂鸦 OTA vs RCSP OTA

| 对比项 | RCSP OTA | 涂鸦 OTA |
|--------|---------|---------|
| **固件来源** | 手机本地文件 / RCSP Server | 涂鸦云端（云端版本管理）|
| **触发方式** | App 手动选择文件触发 | 云端推送 / App 手动触发 |
| **传输协议** | RCSP OpCode 0xE1~0xE9 | 涂鸦 BLE 帧 TYPE=OTA |
| **加密** | 无（明文传输固件）| AES-128 加密 |
| **写 Flash** | 调用 JL OTA API | 在 `tuya_ota.c` 中调用 JL OTA API |
| **重启触发** | 写 UPDATA_PARM + 重启 | 写 UPDATA_PARM + 延时重启 |

两种 OTA 的底层写 Flash 和触发升级机制完全一致，都走 JL 的 uboot → ota.bin 流程。

## 七、面试高频考点

### Q1：移植涂鸦 SDK 需要做哪些工作？只引入 .a 库就行了吗？

**A**：不行，`.a` 只是协议核心，完整移植包含 4 个层次：
1. **配置**（`tuya_ble_config.h`）：裁剪功能、设置 OS 模式、NV 地址映射
2. **GATT 注册**（`tuya_protocol.c`）：向 JL BLE 框架注册 Service 0xFD50 和 3 个特征值，绑定 ATT 读写回调
3. **平台适配**（`tuya_ble_port_JL.c`，约 1170 行）：实现约 30 个接口，覆盖 BLE 收发、定时器、NV 存储、OS 任务/队列、加密算法、MAC 操作、临界区、随机数、延时
4. **业务层**（`tuya_ble_app_demo.c` + `tuya_app_func.c`）：三元组读取、SDK 初始化、DP 事件处理、TWS 同步

### Q2：涂鸦和 RCSP 的任务模型有什么不同？

**A**：涂鸦 SDK 在初始化时通过 `tuya_ble_os_task_create()` 自建一个独立的 RTOS 任务，专门跑协议事件循环（解帧、状态机、NV 读写）。业务回调通过消息队列投递到 `app_core` 任务执行。
RCSP 没有独立任务，所有处理都在现有的 JL 任务中完成，实现更简单但 SDK 与系统耦合更紧。

### Q3：三元组存在哪里？优先级如何？

**A**：JL 平台有两个存储位置：
1. **Flash License 区**（优先）：Flash 末尾 256 字节，偏移 80 字节处，带 CRC 头校验。这是产线工具写入的，格式为逗号分隔的 ASCII 字符串（DeviceID,AuthKey,MAC）
2. **VM 分区**（备用）：`syscfg_read(VM_TUYA_TRIPLE, ...)`，用于 License 区被擦除（如 OTA 后 Flash 全擦）时的回退

### Q4：涂鸦的 NV 存储如何与 JL VM 分区对接？

**A**：涂鸦 SDK 用虚拟地址（`TUYA_BLE_AUTH_FLASH_ADDR` 等 4 个地址）访问存储。JL 适配层维护一个地址→VM ID 的映射表（通过 `tuya_get_vm_id_register()` 注册），把虚拟地址转换为 `syscfg_write/read()` 的 VM ID。此外，`tuya_ble_nv_write()` 在写入鉴权数据时还会触发 TWS 同步消息，这是涂鸦 TWS 方案的关键链路。

### Q5：为什么涂鸦需要在广播包里更新数据？

**A**：涂鸦 BLE 广播包中包含设备的绑定状态标志（是否已绑定到涂鸦账号）。鉴权成功后，SDK 调用 `tuya_ble_gap_advertising_adv_data_update()` 更新广播包，让手机 App 能从广播中快速判断设备是否已绑定，决定是否发起连接。JL 适配层实现这个接口时需要先停广播、更新数据、再重启广播。
