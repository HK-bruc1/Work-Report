# 一、结论先行

在“文件系统 + 连续写录音”的场景下：

> **SD NAND 比 SPI NAND 快的根本原因，不是接口，而是：
>  内部硬件 FTL + 缓存 + 异步机制，显著减少了“写等待”和“写放大”**

即：

- **SD NAND：写入“表面完成”，实际后台慢慢写**
- **SPI NAND：写入“真正完成”，每一步都在等 NAND**

------

# 二、两种系统结构对比

## 1）SD NAND（贴片式 Managed NAND）

典型器件：

- SanDisk iNAND

系统结构：

```
文件系统
   ↓
块设备接口（512B扇区）
   ↓
【内部控制器】
   ├── FTL（地址映射）
   ├── 写缓存（SRAM/DRAM）
   ├── 垃圾回收（GC）
   ├── 磨损均衡
   └── ECC
   ↓
NAND
```

------

## 2）SPI NAND（外部 FTL）

典型器件：

- Winbond W25N01GV

系统结构：

```
文件系统
   ↓
你实现的 FTL（软件）
   ↓
SPI 驱动
   ↓
NAND
```

------

# 三、在“写录音文件”时发生了什么

录音文件通常是：

- 连续写
- 数据块较小（例如 512B / 1KB / 几KB）
- 高频写入

------

## 1）在 SD NAND 上的写路径

```
写请求（文件系统）
    ↓
写入内部缓存
    ↓
立即返回
    ↓
（后台）
合并写 → 顺序写 NAND → GC → 擦除
```

### 关键特征

- MCU 几乎不等待 NAND
- 小写被合并成整页写
- 擦除在后台完成
- 写操作“被平滑”

------

## 2）在 SPI NAND + 自研 FTL 上的写路径

典型流程（即使你已经做了 FTL）：

```
写请求
   ↓
FTL 映射
   ↓
Program Load
   ↓
Program Execute
   ↓
等待 Busy
   ↓
返回
```

如果涉及：

- page 未对齐
- block 用尽

还会触发：

```
读改写（RMW）
或
GC（搬数据 + 擦除）
```

### 关键特征

- 每次写都可能等待 NAND
- 写放大明显
- GC 可能阻塞前台
- 擦除可能同步发生

------

# 四、为什么“SD NAND 明显更快”（核心原因）

## 1）写操作是否“阻塞”

- **SD NAND：非阻塞（缓存 + 异步）**
- **SPI NAND：阻塞（等 Program / Erase）**

👉 决定了 MCU 是否被“卡住”

------

## 2）写合并能力（Write Coalescing）

- **SD NAND：自动把小写合并成整页甚至多页**
- **SPI NAND：取决于你的 FTL 是否实现**

👉 直接影响写放大

------

## 3）垃圾回收（GC）执行时机

- **SD NAND：后台执行**
- **SPI NAND：常见为前台触发**

👉 会不会突然卡几毫秒

------

## 4）擦除操作（Erase）是否隐藏

- **SD NAND：提前/后台擦除**
- **SPI NAND：可能写路径中触发**

👉 擦除一次就是毫秒级延迟

------

## 5）缓存规模差异

- **SD NAND：KB ~ MB 级缓存**
- **SPI NAND：通常只有 MCU RAM（很小）**

👉 决定能否“吸收突发写入”

------

## 6）调度与并行（工业级优化）

SD NAND 内部通常具备：

- 写队列
- pipeline（写/擦并行）
- 多 plane 操作

👉 软件 FTL 很难完全做到

------

# 五、为什么“SPI 24MHz 仍然输给 1MHz”

关键点：

> **瓶颈不在 SPI 带宽，而在 NAND 内部时间**

典型量级：

- SPI 传输（2KB@24MHz）：< 1 ms
- NAND Program：0.3 ~ 1 ms
- Block Erase：2 ~ 5 ms

👉 即使 SPI 提速：

> **也无法消除 Program / Erase 延迟**

而 SD NAND：

> **把这些延迟隐藏在后台**

------

# 六、文件系统视角的本质差异

## SD NAND

```
文件系统 → “理想块设备”
```

👉 行为符合文件系统预期

------

## SPI NAND + FTL

```
文件系统 →（不完全匹配）→ NAND物理特性
```

👉 即使有 FTL，也可能存在：

- 同步写
- 小写未合并
- GC 阻塞

------

# 七、总结（工程视角）

在“挂文件系统写录音”场景下：

> **SD NAND 快的本质是：
>  把 NAND 的所有慢操作（Program / Erase / GC）隐藏、合并、延迟**

而 SPI NAND（即使有自研 FTL）：

> **如果没有做到“缓存 + 异步 + 后台 GC + 顺序化”，就仍然在直接承受 NAND 的物理延迟**

------

# 最后一条判断标准

如果一个存储器：

- 支持 512B 扇区
- 支持随意覆盖写
- 写入几乎不阻塞

👉 那它一定：

> **内部已经实现了完整且高度优化的 FTL（通常远强于软件实现）**
