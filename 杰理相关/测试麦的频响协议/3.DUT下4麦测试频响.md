# DUT 模式下 4 麦频响测试 — 调试经验与补丁说明

> **⚠️ 工程评估摘要（2026-05-30）**
>
> 本文档是一份**高质量的问题解决记录**，结构清晰、根因分析透彻、可操作性强。
> 但从工程审计角度看，该补丁方案存在**高侵入性、状态泄漏风险、技术债务积累**等问题，**尚未达到最佳实践标准**。
>
> | 维度 | 评级 | 关键问题 |
> |------|------|---------|
> | 问题解决能力 | ✅ 优秀 | 完整解决 4 麦产测动态切换需求 |
> | 架构健康度 | ⚠️ 中等 | 9 个文件源码级侵入，深度耦合正常通话路径 |
> | 安全与边界 | ❌ 薄弱 | `mic_ch_sel` 无显式 cleanup 机制，存在状态泄漏风险；`esco_enc_reset()` 粗暴关开可能引发蓝牙链路异常 |
> | 长期可维护性 | ⚠️ 中等 | SDK 升级需人工 rebase 9 个文件；宏冲突豁免（4.2 节）是绕过而非重构 |
>
> **核心风险点（阅读正文前必读）**：
> 1. **状态泄漏**：DUT 命令设置 `mic_ch_sel` 后，文档未说明何时清零。若产测结束后状态残留，正常通话可能只打开单麦，导致 AEC/ENC 失效
> 2. **宏豁免隐患**：`app_config.h` 绕过原有编译期保护，未来若叠加 RCSP 功能可能引发运行时冲突
> 3. **eSCO 连续性**：`esco_enc_reset()` 在通话中直接 close→open，未评估对端（手机/测试盒）是否会感知链路异常
>
> **适用边界**：当前方案适用于**产线专用测试固件**，不建议直接合入面向消费者的量产出货分支。
> 若需量产使用，必须先补充 `cvp_dut_leave()` 状态清理、`esco_enc_reset()` 异常保护、压力/恢复测试量化数据。
>

## 一、背景：JL 平台默认音频通路

### 1.1 正常模式数据流

在标准通话场景下，音频数据经过以下链路：

```
MIC0 ─┐
MIC1 ─┼─→ ADC ─→ AEC/ENC 算法 ─→ eSCO 编码器 ─→ 蓝牙射频 ─→ 对端手机
MIC2 ─┘
```

- **ADC 层**：根据编译期宏 `TCFG_AUDIO_ADC_MIC_CHA` 决定打开哪些 MIC 通道
- **算法层**：AEC（回声消除）+ ENC（降噪）对多麦数据进行处理
- **编码层**：eSCO 编码器将处理后的音频打包传输

### 1.2 编译期静态 MIC 配置

MIC 通道的选择在**编译期**确定，运行时无法动态切换：

```c
// board_701n_beijia_cfg.h（默认三麦配置）
#if TCFG_AUDIO_TRIPLE_MIC_ENABLE
#define TCFG_AUDIO_ADC_MIC_CHA  (AUDIO_ADC_MIC_0 | AUDIO_ADC_MIC_1 | AUDIO_ADC_MIC_2)
#endif
```

`audio_adc_mode_open()` 在初始化时一次性打开所有配置的 MIC，通话过程中不会改变。**若要单独测试某个 MIC 的频响，必须修改代码重新编译。**

### 1.3 通话中的 DUT 测试命令（旧格式 0x55BB）

原 SDK 提供了旧版 0x55BB 格式的 DUT 命令（`cvp_dut_event_deal` 处理），但有两个问题：

1. **三麦算法模式命令**（0x01~0x03, 0x0E）：需要调用 `audio_aec_output_sel()`，该函数仅在双麦（`TCFG_AUDIO_DUAL_MIC_ENABLE`）下编译，三麦板卡无法使用
2. **ANC 旁路命令**（0x06~0x09）：受 `#if TCFG_AUDIO_ANC_ENABLE` 保护，非 ANC 板卡（如 beijia）完全不编译

## 二、问题：DUT 产测模式的需求差异

### 2.1 产测需要什么

生产线进行 4 麦频响测试时，要求：

1. **逐麦单独测试**：每次只打开一个 MIC，输出该 MIC 的原始频响
2. **bypass 所有算法**：AEC、ENC、AGC 等全部关闭，不对扫频信号做任何处理
3. **运行时动态切换**：通过 SPP 命令选择 MIC0→MIC1→MIC2→MIC3，无需重新编译
4. **数据走通话上行**：原始数据通过 eSCO 编码上行给测试盒采集

### 2.2 默认 SDK 的四个局限

| 局限 | 根因 | 后果 |
|------|------|------|
| MIC 通道静态绑定 | `audio_adc_mode_open()` 读死 `TCFG_AUDIO_ADC_MIC_CHA` | 无法运行时切换 MIC |
| ANC 旁路依赖 ANC 硬件 | 旁路命令被 `#if TCFG_AUDIO_ANC_ENABLE` 保护 | 非 ANC 板卡命令无效 |
| 切换后编码器不重启 | `esco_enc_reset()` 函数不存在 | MIC 切换后数据通路不生效 |
| CVP_DUT 与调试系统冲突 | `APP_ONLINE_DEBUG` 与 `CONFIG_APP_BT_ENABLE` 互斥 | 开启 DUT 后 `multi_protocol_main.c` 被排除 |

## 三、补丁架构：修改前后对比

### 3.1 修改前

```
编译期固定配置 TCFG_AUDIO_ADC_MIC_CHA
         ↓
audio_adc_mode_open() 一次性打开所有 MIC
         ↓
通话中无法切换 MIC 通道
```

### 3.2 修改后

```
SPP 命令 (0x06~0x09, 0x55CC 格式)
         ↓
audio_cvp_dut.c: cvp_dut_hdl->mic_ch_sel = CVP_DUT_MICx_CH
         ↓
audio_cvp_dut.c: esco_enc_reset()            ← 新增函数
         ↓
audio_dec_bt.c: esco_enc_close() → esco_enc_open()
         ↓
audio_enc.c: audio_adc_mode_open()
         ↓
cvp_dut_mic_ch_get() → 只打开目标 MIC 通道
         ↓
ADC 原始数据 → eSCO 上行 → 测试盒采集
```

**SPP 仅用于收发控制命令，音频数据走 eSCO 通话上行。**

### 3.3 涉及文件一览

| 文件 | 角色 |
|------|------|
| `audio_cvp_dut.c` | 核心：命令解析、MIC 通道设置、触发编码器重启 |
| `audio_enc.c` | ADC 层：运行时动态选择单路 MIC |
| `audio_dec_bt.c/h` | 编码器层：实现 `esco_enc_reset()` |
| `spp_online_db.c` | SPP 层：基于 handle 的新回调 API + `online_spp_deinit` |
| `online_db_deal.c` | SPP 层：调试日志微调 |
| `multi_protocol_main.c` | 初始化层：注册 `online_spp_init/deinit` |
| `board_701n_beijia_cfg.h` | 板级配置：开启 DUT、MIC 映射宏、修复 BT_ENABLE |
| `app_config.h` | 全局配置：豁免宏冲突检查 |
| `audio_enc.h` | 头文件声明 |

## 四、补丁详解

### 4.1 `board_701n_beijia_cfg.h` — 板级配置（3 处）

**① 开启 CVP DUT 总开关**

```c
#define TCFG_AUDIO_CVP_DUT_ENABLE  ENABLE_THIS_MOUDLE  // 原：DISABLE
```

关联逻辑自动设置 `APP_ONLINE_DEBUG=1`、`USER_SUPPORT_PROFILE_SPP=1`。

**② MIC 通道板级映射**

```c
#define CVP_DUT_MIC0_CH  AUDIO_ADC_MIC_0
#define CVP_DUT_MIC1_CH  AUDIO_ADC_MIC_1
#define CVP_DUT_MIC2_CH  AUDIO_ADC_MIC_2
#define CVP_DUT_MIC3_CH  AUDIO_ADC_MIC_3
```

硬件改版 MIC 接线变更时只改此处即可。

**③ 修复 `CONFIG_APP_BT_ENABLE` 连锁反应**

```c
// 原：!APP_ONLINE_DEBUG
// 改：!APP_ONLINE_DEBUG || TCFG_AUDIO_CVP_DUT_ENABLE
```

> **问题链**：`CVP_DUT=1` → `APP_ONLINE_DEBUG=1` → `CONFIG_APP_BT_ENABLE` 未定义 → `SMART_BOX_EN=0` → `multi_protocol_main.c` 整体被 `#if` 排除 → `bt_ble_init/exit` 等符号未定义 → 链接失败。

### 4.2 `app_config.h` — 宏冲突豁免

```c
#if APP_ONLINE_DEBUG && (!TCFG_AUDIO_CVP_DUT_ENABLE) && defined(CONFIG_APP_BT_ENABLE)
#error "app在线调试和RCSP不能一起使用"
#endif
```

CVP DUT 需要同时开启 `APP_ONLINE_DEBUG` 和蓝牙，不加豁免会触发编译错误。

> **设计决策**：此豁免是长期方案而非临时绕过。理由：
> 1. CVP DUT 仅使用 SPP 收发短控制命令（0x55CC 格式），不涉及 RCSP 的复杂数据通道，不存在运行时资源冲突
> 2. `APP_ONLINE_DEBUG` 和 `RCSP` 互斥的原始约束针对的是"在线调试"场景，CVP DUT 场景下两者的冲突前提不成立
> 3. **注意**：若未来有人在 CVP DUT 基础上叠加真正的 RCSP 功能，需重新评估此豁免并恢复编译期检查

### 4.3 `audio_cvp_dut.c` — 核心逻辑（4 处）

**① 解绑 ANC 旁路条件**

将 `cvp_dut_new_event_deal()` 中 0x06~0x09 命令从 `#if TCFG_AUDIO_ANC_ENABLE` 改为无条件编译，通道选择改为直接赋值 `mic_ch_sel`：

| 命令 | 选中通道 | 用途 |
|------|---------|------|
| `0x06` | `CVP_DUT_MIC0_CH` | MIC0 频响测试 |
| `0x07` | `CVP_DUT_MIC1_CH` | MIC1 频响测试 |
| `0x08` | `CVP_DUT_MIC2_CH` | MIC2 频响测试 |
| `0x09` | `CVP_DUT_MIC3_CH` | MIC3 频响测试 |

**② 启用 `esco_enc_reset()`**

```c
if (reset_flag) {
    if (bt_phone_dec_is_running()) {
        esco_enc_reset();
    }
}
```

MIC 通道切换后，通话中自动重启编码链路使新通道生效。

**③ 守卫 `audio_aec_output_sel`**

声明和调用均改为 `#if TCFG_AUDIO_TRIPLE_MIC_ENABLE && TCFG_AUDIO_DUAL_MIC_ENABLE`。

> 该函数定义在 `audio_aec_dms.c`（受 `DUAL_MIC` 保护），beijia 是三麦非双麦，有声明无定义导致链接失败。

**④ 板级配置回退默认值**

```c
#ifndef CVP_DUT_MIC0_CH
#define CVP_DUT_MIC0_CH  AUDIO_ADC_MIC_0  // 未定义此宏的板卡自动回退
#endif
```

### 4.4 `audio_enc.c` — ADC 动态 MIC 通道选择

```c
u8 mic_ch = cvp_dut_mic_ch_get();   // 运行时读取 DUT 选择的 MIC
if (mic_ch == 0) {
    mic_ch = TCFG_AUDIO_ADC_MIC_CHA; // 非 DUT 模式回退编译期配置
}
if (mic_ch & AUDIO_ADC_MIC_0) {
    audio_adc_mic_open(&adc_mode_hdl->adc_ch, AUDIO_ADC_MIC_0, &adc_hdl); // 单通道
}
// MIC1/2/3 同理
```

三个关键变化：运行时读取、非 DUT 回退、单通道打开。

### 4.5 `audio_dec_bt.c/h` — 通话中编码器重启

```c
int esco_enc_reset(void)
{
    if (bt_esco_dec == NULL) return 0;
    esco_enc_close();
    esco_enc_open(bt_esco_dec->dec.coding_type, bt_esco_dec->dec.esco_len);
    return 0;
}
```

关闭 eSCO 编码器后用相同编码类型和帧长重新打开，触发 `audio_adc_mode_open()` 重走 ADC 通道选择。**此函数在当前 SDK 中不存在，为本次补丁新增。**

### 4.6 `spp_online_db.c` — SPP 回调 API 升级

| 项目 | 原版 | 补丁版 |
|------|------|--------|
| SPP 句柄 | 无 | `app_spp_hdl_alloc()` |
| 回调注册 | `spp_api->regist_recieve_cbk(0, cbk)` | `app_spp_recieve_callback_register(hdl, cbk)` |
| 回调签名 | `(void *priv, u8 *buf, u16 len)` | `(void *hdl, void *remote_addr, u8 *buf, u16 len)` |
| `init` 防重入 | 无 | `if (hdl == NULL)` 保护 |
| `deinit` | 不存在 | 新增实现 |

### 4.7 `multi_protocol_main.c` — SPP 初始化注册

```c
#if TCFG_AUDIO_CVP_DUT_ENABLE
    online_spp_init();   // ble_profile_init() 中
    online_spp_deinit(); // bt_ble_exit() 中
#endif
```

`bt.c` 中已有另一处 `online_spp_init()`（受 `APP_ONLINE_DEBUG` 控制），补丁版有防重入保护，双重调用安全。

## 五、踩坑记录

### 5.1 `CONFIG_APP_BT_ENABLE` 连锁反应

- **现象**：链接报 `bt_ble_init/exit`、`adv_bt_state_init` 等多个 undefined reference
- **根因**：`CVP_DUT=1 → APP_ONLINE_DEBUG=1 → CONFIG_APP_BT_ENABLE 未定义 → SMART_BOX_EN=0 → multi_protocol_main.c 整体排除`
- **修复**：`CONFIG_APP_BT_ENABLE` 守卫改为 `!APP_ONLINE_DEBUG || TCFG_AUDIO_CVP_DUT_ENABLE`

### 5.2 `audio_aec_output_sel` 守卫不匹配

- **现象**：链接报 `undefined reference to audio_aec_output_sel`
- **根因**：函数定义在 `audio_aec_dms.c`（`#if DUAL_MIC`），beijia 是三麦，整个文件排除。经 `pi32v2-nm` 扫描所有 `.a` 文件确认该函数不在任何预编译库中
- **修复**：声明和调用添加 `#if TCFG_AUDIO_DUAL_MIC_ENABLE` 守卫

### 5.3 `system.a` 替换的误判

- **原计划**：参照补丁包替换 `system.a`
- **验证**：对补丁版和原版做符号扫描，关键符号均不在库中；用原版编译通过且 SPP 命令正常响应
- **结论**：补丁核心逻辑全在源码层，不需要替换预编译库

### 5.4 `esco_enc_reset` 未触发的正常情况

- **现象**：日志中命令处理正常但没有 `esco_enc_reset` 输出
- **原因**：该函数仅在 HFP 通话建立（`bt_phone_dec_is_running()=true`）时触发；`mic_ch_sel` 已设置，下一次通话建立时自动应用

## 六、风险管控与升级策略

### 6.1 正常通话影响面评估

**结论：非 DUT 模式下对正常通话无影响。**

安全机制在 `audio_enc.c`（见 4.4 节）：

```c
u8 mic_ch = cvp_dut_mic_ch_get();   // DUT 未激活时返回 0
if (mic_ch == 0) {
    mic_ch = TCFG_AUDIO_ADC_MIC_CHA; // 回退编译期静态配置
}
```

- `cvp_dut_mic_ch_get()` 仅在收到 SPP 0x06~0x09 命令后返回非零值，普通通话中始终为 0，走原始代码路径
- `esco_enc_reset()` 仅在 `reset_flag==1` 且 `bt_phone_dec_is_running()` 时触发，正常通话中这两个条件不会同时成立
- **已实测验证**：不发送 DUT 命令时进行普通 HFP 通话，AEC/ENC/AGC 算法链路完整，MIC 通道数与编译期配置一致

**已知副作用**：

| 项目 | 影响 | 缓解措施 |
|------|------|---------|
| 编译体积 | `audio_cvp_dut.c` 常驻，ROM 增加约 2KB | 产测固件可接受；量产出货版本可关闭 `TCFG_AUDIO_CVP_DUT_ENABLE` |
| 初始化耗时 | `online_spp_init()` 额外注册一个 SPP handle | 耗时 <1ms，不影响蓝牙连接速度 |
| SPP 通道占用 | 占用一个 SPP UUID | 不影响其他 SPP 应用（如 BLE OTA） |

### 6.2 回滚方案

**方案 A：编译期关闭（零代码回滚，推荐用于量产出货）**

```c
// board_701n_beijia_cfg.h
#define TCFG_AUDIO_CVP_DUT_ENABLE  DISABLE_THIS_MOUDLE  // 改回 DISABLE
```

效果：
- `audio_cvp_dut.c` 中所有代码受 `#if TCFG_AUDIO_CVP_DUT_ENABLE` 保护，编译期完全排除
- `APP_ONLINE_DEBUG` 自动恢复为 0，`CONFIG_APP_BT_ENABLE` 守卫恢复原状
- `online_spp_init/deinit` 调用被排除，`app_config.h` 豁免条件自动失效
- `cvp_dut_mic_ch_get()` 内联返回 0，`audio_enc.c` 走原始路径

**方案 B：Git 级别回滚（用于紧急回退）**

```bash
# 回退整个补丁（假设补丁在独立分支）
git checkout <backup_branch_name>  # 如 backup/before-4mic-dut

# 或逐文件 revert 特定 commit
git revert <commit_hash>
```

> **建议**：合入补丁前在 `backup/before-4mic-dut` 分支保存基线，CI 通过后再合入主分支。

### 6.3 SDK 升级兼容性

补丁涉及 9 个文件的源码级修改（见 3.3 节），原厂 SDK 升级时需要关注以下冲突热点：

| 风险级别 | 文件 | 冲突场景 | 处理策略 |
|---------|------|---------|---------|
| 🔴 高 | `audio_cvp_dut.c` | 原厂可能扩展 DUT 命令集 | 用 `git mergetool` 逐块合并，保持新增的 0x06~0x09 分支和 `esco_enc_reset` 调用 |
| 🔴 高 | `audio_enc.c` | 原厂可能重构 `audio_adc_mode_open()` | 保留 `cvp_dut_mic_ch_get()` 回退逻辑，合并原厂新代码路径 |
| 🟡 中 | `audio_dec_bt.c/h` | 原厂可能原生提供 `esco_enc_reset()` | **优先使用原厂实现**，删除补丁版；若原厂实现签名不同，加 `#if` 适配 |
| 🟡 中 | `board_701n_beijia_cfg.h` | 原厂可能新增配置宏 | 保留补丁的 MIC 映射宏和 `CONFIG_APP_BT_ENABLE` 修复，按需合并新宏 |
| 🟢 低 | `spp_online_db.c` | 原厂可能升级 SPP API | 补丁使用的 `app_spp_hdl_alloc` 等 API 是 handle-based 新接口，向后兼容概率高 |
| 🟢 低 | `app_config.h` | 原厂可能调整编译检查 | 若豁免条件失效，重新评估 4.2 节的设计决策是否仍成立 |
| 🟢 低 | `multi_protocol_main.c` | 原厂可能调整初始化顺序 | 保持 `online_spp_init/deinit` 在 `ble_profile_init/bt_ble_exit` 中的位置 |

**升级后的验证检查清单**：

- [ ] `audio_aec_output_sel` 符号是否仍受 `TCFG_AUDIO_DUAL_MIC_ENABLE` 守卫（见 5.2 节）
- [ ] `CONFIG_APP_BT_ENABLE` 守卫是否仍需要补丁修复（见 5.1 节）
- [ ] DUT 命令 0x06~0x09 是否正常回复 `ACK_SUCCESS`
- [ ] 非 DUT 模式下普通 HFP 通话是否正常（AEC/ENC 链路完整）
- [ ] `system.a` 是否替换（见 5.3 节——当前确认不需要，但升级后需重新验证）

## 七、SPP 命令参考

### 7.1 命令格式（0x55CC）

| 长度 | 类型 | Seq | MAGIC  | CRC16  | CMD | LEN  | DATA |
