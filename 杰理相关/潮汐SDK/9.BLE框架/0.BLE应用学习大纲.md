# BLE 应用开发学习大纲：面向自定义 GATT Service 对接

## 1. 学习目标

本文档面向使用 BLE SDK 进行应用开发的工程师，重点不是从零实现 BLE 协议栈，而是掌握如何在现有 SDK 基础上完成：

- 自定义 BLE Service / Characteristic 设计
- GATT Server 与 GATT Client 的交互
- 手机 App、网关、MCU 设备之间的 BLE 数据通信
- Read / Write / Notify / Indicate 的使用
- BLE 连接、广播、订阅、上报、分包、MTU、安全等应用层开发要点
- BLE 调试、抓包、问题定位与工程化设计

适用场景包括：

- MCU BLE 外设开发
- 手机 App 对接自定义蓝牙设备
- BLE 传感器、手环、控制器、配置工具、网关设备
- 自定义私有 BLE 协议设计

------

# 2. 总体学习顺序

推荐学习顺序如下：

```text
1. BLE 基本角色与整体流程
2. GAP：广播、扫描、连接
3. GATT：Service / Characteristic / Descriptor
4. ATT 操作：Read / Write / Notify / Indicate
5. 自定义 Service 设计
6. SDK 中的 GATT Server 开发
7. SDK 中的 GATT Client 开发
8. 数据协议封装与分包
9. MTU、连接参数、吞吐与功耗
10. 配对、绑定、加密与权限
11. 调试工具与抓包分析
12. 工程化设计与常见问题
```

对于应用开发来说，不建议一开始深入 Link Layer、PHY、L2CAP 底层实现。应先掌握 SDK 提供的接口和回调模型，再逐步理解底层机制对应用的影响。

------

# 3. BLE 应用开发的核心理解

## 3.1 BLE 不等于 GATT

BLE 是完整的低功耗蓝牙通信体系，GATT 只是其中用于应用数据交互的一层。

BLE 包括：

```text
应用层
  └── 业务协议、自定义数据格式

GATT
  └── Service / Characteristic / Descriptor

ATT
  └── Read / Write / Notify / Indicate

GAP
  └── 广播、扫描、连接、角色管理

SMP
  └── 配对、绑定、加密

L2CAP
  └── 分包、复用、通道管理

Link Layer / PHY
  └── 射频、跳频、连接事件、空口收发
```

但在 SDK 应用开发中，最常接触的是：

```text
GAP + GATT + 业务逻辑
```

------

## 3.2 BLE 应用开发的典型模型

大多数 BLE 设备开发，本质是：

```text
GATT Server 暴露 Service / Characteristic
GATT Client 发现 Service / Characteristic
GATT Client 对 Characteristic 进行 Read / Write / Notify / Indicate 操作
```

典型交互如下：

```text
设备端：GATT Server
手机端：GATT Client

1. 设备开始广播
2. 手机扫描到设备
3. 手机发起连接
4. 手机发现服务
5. 手机发现特征值
6. 手机读取设备信息
7. 手机写入配置或控制命令
8. 手机订阅 Notify
9. 设备主动上报数据
```

------

# 4. 第一阶段：BLE 基础角色与整体流程

## 4.1 必须掌握的角色

### Central / Peripheral

这是 GAP 层角色。

| 角色       | 说明               | 常见设备               |
| ---------- | ------------------ | ---------------------- |
| Peripheral | 广播自己，等待连接 | MCU 设备、传感器、手环 |
| Central    | 扫描并发起连接     | 手机、网关、电脑       |

### GATT Server / GATT Client

这是 GATT 层角色。

| 角色        | 说明                                    | 常见设备       |
| ----------- | --------------------------------------- | -------------- |
| GATT Server | 保存数据，暴露 Service / Characteristic | BLE 外设       |
| GATT Client | 访问 Server 数据                        | 手机 App、网关 |

常见组合：

```text
Peripheral = GATT Server
Central    = GATT Client
```

但这不是强制关系，某些复杂设备也可以既是 Client 又是 Server。

------

## 4.2 应用开发必须理解的完整流程

```text
设备初始化
  ↓
配置 GAP 参数
  ↓
配置广播数据
  ↓
注册 GATT Service
  ↓
开始广播
  ↓
等待 Central 连接
  ↓
连接成功
  ↓
Client 发现 Service / Characteristic
  ↓
Client Read / Write / Subscribe
  ↓
Server Notify / Indicate 上报
  ↓
断开连接
  ↓
重新广播
```

学习重点：

- 设备什么时候开始广播
- 手机如何识别目标设备
- 连接成功后如何发现服务
- Client 如何写数据给 Server
- Server 如何主动上报数据
- 断线后如何恢复广播

------

# 5. 第二阶段：GAP 广播、扫描与连接

GAP 负责设备发现和连接管理。

## 5.1 广播 Advertising

Peripheral 通过广播告诉外界：

- 设备名称
- 设备类型
- 支持的 Service UUID
- 厂商自定义数据
- 是否可连接

常见广播内容：

```text
Device Name
Service UUID
Manufacturer Specific Data
Tx Power
Appearance
Flags
```

应用开发重点：

- 广播包最大长度限制
- 广播间隔对功耗和连接速度的影响
- 广播数据和扫描响应数据的区别
- 如何让 App 通过名称、MAC、Service UUID 或厂商数据识别设备

------

## 5.2 扫描 Scanning

Central 通过扫描发现 Peripheral。

App 或网关通常会根据以下信息过滤设备：

```text
设备名
MAC 地址
Service UUID
Manufacturer Data
RSSI
```

工程建议：

- 不要只依赖设备名识别设备
- 更推荐使用 Service UUID 或 Manufacturer Data
- 若设备使用随机地址，需要考虑 MAC 地址变化问题

------

## 5.3 连接 Connection

连接建立后，双方进入连接态。

连接后需要关注：

- 连接间隔 Connection Interval
- Slave Latency
- Supervision Timeout
- MTU
- PHY
- 是否加密
- 是否绑定

应用开发中常见事件：

```text
Connected
Disconnected
Connection Parameter Updated
MTU Updated
Security Completed
```

------

# 6. 第三阶段：GATT 数据模型

GATT 是 BLE 应用数据交互的核心。

## 6.1 GATT 层级结构

```text
GATT Server
  └── Service
        └── Characteristic
              └── Descriptor
```

示例：

```text
Device Information Service
  ├── Manufacturer Name String
  ├── Model Number String
  └── Firmware Revision String

Custom Sensor Service
  ├── Sensor Data Characteristic
  ├── Config Characteristic
  └── Control Characteristic
```

------

## 6.2 Service

Service 用来表示一组相关功能。

例如：

```text
Battery Service
Device Information Service
Heart Rate Service
Custom UART Service
Custom Sensor Service
```

Service 通过 UUID 标识。

常见 UUID 类型：

| 类型         | 长度    | 用途              |
| ------------ | ------- | ----------------- |
| 16-bit UUID  | 2 字节  | 蓝牙 SIG 标准服务 |
| 128-bit UUID | 16 字节 | 厂商自定义服务    |

自定义服务通常使用 128-bit UUID。

------

## 6.3 Characteristic

Characteristic 是实际读写数据的对象。

一个 Characteristic 通常包含：

```text
UUID
Properties
Permissions
Value
Descriptors
```

常见属性：

| 属性                   | 说明                                   |
| ---------------------- | -------------------------------------- |
| Read                   | Client 可以读取                        |
| Write                  | Client 可以写入，需要响应              |
| Write Without Response | Client 可以写入，不需要响应            |
| Notify                 | Server 可以主动通知 Client             |
| Indicate               | Server 可以主动通知 Client，并要求确认 |
| Broadcast              | 很少用                                 |
| Extended Properties    | 较少用                                 |

------

## 6.4 Descriptor

Descriptor 是 Characteristic 的附加描述信息。

最常见的是：

```text
CCCD: Client Characteristic Configuration Descriptor
```

CCCD 用于控制 Notify / Indicate 是否开启。

Client 订阅 Notify，本质上是写 CCCD。

```text
0x0000: 关闭 Notify / Indicate
0x0001: 开启 Notify
0x0002: 开启 Indicate
```

------

# 7. 第四阶段：ATT 操作方式

GATT 的交互最终会落到 ATT 操作。

## 7.1 Read

Client 主动读取 Server 的 Characteristic。

适合：

- 读取设备信息
- 读取电量
- 读取当前状态
- 读取配置参数

示例：

```text
Client -> Server: Read Battery Level
Server -> Client: 85%
```

------

## 7.2 Write

Client 向 Server 写数据。

适合：

- 控制命令
- 参数配置
- 开始采集
- 停止采集
- 设备校准
- Wi-Fi 配网数据下发

示例：

```text
Client -> Server: Write Control Command
Server -> Client: Write Response
```

------

## 7.3 Write Without Response

Client 向 Server 写数据，但 Server 不返回 ATT Write Response。

适合：

- 高速数据传输
- 分包写入
- 对实时性要求高的场景

注意事项：

- 速度快
- 可靠性依赖链路层
- 应用层通常需要自己设计 ACK 或序号
- 不要无限制连续发送，需要考虑 SDK 缓冲区和流控

------

## 7.4 Notify

Server 主动向 Client 上报数据，不需要 Client 确认。

适合：

- 传感器周期上报
- 实时状态变化
- 日志输出
- UART 透传
- 波形数据
- 设备事件通知

流程：

```text
Client 写 CCCD 开启 Notify
Server 检测到 Notify 已开启
Server 调用 SDK API 发送 Notify
Client 收到 Notify 数据
```

------

## 7.5 Indicate

Server 主动向 Client 上报数据，需要 Client 确认。

适合：

- 关键状态
- 重要事件
- 不适合丢失的数据
- 配置结果
- 告警信息

流程：

```text
Client 写 CCCD 开启 Indicate
Server 发送 Indicate
Client 收到后自动回 Confirm
Server 收到 Confirm 后才能继续发送下一包 Indicate
```

------

## 7.6 Notify 与 Indicate 对比

| 项目           | Notify               | Indicate                 |
| -------------- | -------------------- | ------------------------ |
| 是否需要确认   | 不需要               | 需要                     |
| 吞吐           | 高                   | 低                       |
| 延迟           | 低                   | 较高                     |
| 可靠性         | 较低                 | 较高                     |
| 适合场景       | 实时数据流           | 关键事件                 |
| 是否可连续发送 | 可以，但受缓冲区限制 | 通常一次只能等待一个确认 |

工程建议：

- 大量数据上报优先使用 Notify
- 关键结果或告警可使用 Indicate
- 若 Notify 也需要可靠性，应在业务协议中增加 ACK、序号、重传机制

------

# 8. 第五阶段：自定义 BLE Service 设计

这是 BLE 应用开发最重要的部分。

## 8.1 自定义服务设计原则

设计自定义 GATT Service 时，应先回答以下问题：

```text
1. 设备需要暴露哪些功能？
2. 哪些数据需要被读取？
3. 哪些参数需要被写入？
4. 哪些数据需要主动上报？
5. 数据是否需要确认？
6. 是否需要加密？
7. 是否需要分包？
8. 是否需要兼容未来版本？
```

------

## 8.2 推荐的自定义 Service 结构

对于普通 MCU 设备，推荐设计为：

```text
Custom Device Service
  ├── TX Characteristic: Notify
  ├── RX Characteristic: Write / Write Without Response
  ├── Status Characteristic: Read / Notify
  ├── Config Characteristic: Read / Write
  └── Control Characteristic: Write / Indicate
```

或者采用类似 UART 透传模型：

```text
Custom UART Service
  ├── RX Characteristic: Write Without Response
  └── TX Characteristic: Notify
```

其中：

```text
RX: 从 Client 到 Server
TX: 从 Server 到 Client
```

注意：TX / RX 命名要从设备角度定义，还是从 App 角度定义，需要在协议文档中明确。

------

## 8.3 示例：自定义传感器服务

```text
Service UUID:
12345678-1234-5678-1234-56789abcdef0

Characteristic 1: Sensor Data
UUID:
12345678-1234-5678-1234-56789abcdef1
Properties:
Notify, Read

Characteristic 2: Config
UUID:
12345678-1234-5678-1234-56789abcdef2
Properties:
Read, Write

Characteristic 3: Control
UUID:
12345678-1234-5678-1234-56789abcdef3
Properties:
Write, Indicate
```

交互示例：

```text
App 连接设备
App 发现 Custom Sensor Service
App 订阅 Sensor Data Notify
App 写 Config 设置采样周期
App 写 Control 启动采集
设备通过 Sensor Data Notify 周期上报数据
```

------

## 8.4 Characteristic 设计建议

### 状态类数据

适合使用：

```text
Read + Notify
```

例如：

```text
设备状态
电量
传感器当前值
工作模式
错误码
```

### 配置类数据

适合使用：

```text
Read + Write
```

例如：

```text
采样周期
阈值
工作模式
设备参数
```

### 控制类数据

适合使用：

```text
Write
```

例如：

```text
启动
停止
重启
校准
恢复出厂
```

### 数据流

适合使用：

```text
Write Without Response + Notify
```

例如：

```text
串口透传
日志输出
文件传输
OTA 数据传输
波形数据
```

------

# 9. 第六阶段：BLE 业务协议设计

GATT 只定义了数据通道，不定义你的业务数据格式。

因此，自定义 BLE 服务通常还需要设计一层业务协议。

## 9.1 为什么需要业务协议

例如，只有一个 RX Characteristic 和一个 TX Characteristic：

```text
RX: App 写入设备
TX: 设备 Notify 给 App
```

但里面的数据可能包括：

```text
读取版本
设置参数
启动采集
停止采集
上传传感器数据
返回执行结果
错误码
```

所以需要在 Value 中定义自己的数据格式。

------

## 9.2 简单命令格式示例

```text
Byte 0: Command ID
Byte 1: Sequence ID
Byte 2: Payload Length
Byte 3..N: Payload
Byte N+1: Checksum
```

示例：

```text
01 10 02 03 E8 FE
```

含义：

```text
Command ID: 0x01
Sequence ID: 0x10
Payload Length: 0x02
Payload: 0x03E8
Checksum: 0xFE
```

------

## 9.3 推荐字段

一个较完整的 BLE 应用协议可包含：

```text
Header
Version
Command ID
Sequence ID
Payload Length
Payload
Checksum / CRC
```

如果需要分包，可增加：

```text
Packet Type
Frame ID
Total Length
Offset
Fragment Index
Fragment Count
```

------

## 9.4 是否需要 ACK

如果使用 Notify / Write Without Response 传输关键数据，建议设计应用层 ACK。

例如：

```text
App -> Device: CMD_SET_CONFIG
Device -> App: RSP_SET_CONFIG_OK

Device -> App: EVENT_ALARM
App -> Device: ACK_EVENT_ALARM
```

ACK 中建议带上：

```text
Sequence ID
Command ID
Result Code
Error Code
```

------

## 9.5 错误码设计

建议定义统一错误码：

```text
0x00: OK
0x01: Unknown Command
0x02: Invalid Length
0x03: Invalid Parameter
0x04: Busy
0x05: Permission Denied
0x06: Not Supported
0x07: CRC Error
0x08: Timeout
```

------

# 10. 第七阶段：SDK 中的 GATT Server 开发

这是 MCU BLE 外设开发的重点。

## 10.1 GATT Server 开发流程

一般 SDK 开发流程如下：

```text
1. 初始化 BLE 协议栈
2. 配置设备名
3. 配置广播数据
4. 创建 GATT Service
5. 添加 Characteristic
6. 添加 CCCD
7. 注册回调函数
8. 开始广播
9. 等待连接
10. 处理 Read / Write / CCCD / Notify Complete 等事件
```

------

## 10.2 必须理解的回调事件

不同芯片 SDK 名称不同，但核心事件类似：

```text
BLE Stack Ready
Advertising Started
Connected
Disconnected
MTU Updated
Connection Parameter Updated
Read Request
Write Request
CCCD Changed
Notification Sent
Indication Confirmed
Security Completed
```

------

## 10.3 Server 端核心逻辑

Server 端通常需要维护：

```text
当前连接状态
当前连接句柄
Notify 是否使能
当前 MTU
当前安全状态
发送队列
业务状态机
```

示例逻辑：

```text
if connected:
    if notify_enabled:
        send_notify(sensor_data)
```

不要在未连接或未订阅 Notify 时直接发送 Notify。

------

## 10.4 Write 回调处理

Client 写入 Characteristic 后，Server 会收到 Write 回调。

处理步骤：

```text
1. 判断写入的是哪个 Characteristic
2. 判断长度是否合法
3. 解析命令
4. 校验参数
5. 执行业务动作
6. 返回结果或通过 Notify / Indicate 上报结果
```

工程建议：

- Write 回调中不要做太耗时的操作
- 复杂操作交给任务队列或主循环处理
- 长命令需要考虑分包
- 不可信任 Client 输入，必须检查长度和参数

------

## 10.5 Notify 发送注意事项

发送 Notify 前要检查：

```text
1. 是否已连接
2. Client 是否已经写 CCCD 开启 Notify
3. 当前是否允许发送
4. 数据长度是否小于等于 ATT_MTU - 3
5. SDK 发送缓冲区是否可用
```

Notify 单包有效数据长度通常是：

```text
ATT Payload = MTU - 3
```

例如：

```text
MTU = 23 时，有效数据长度为 20 字节
MTU = 247 时，有效数据长度为 244 字节
```

------

# 11. 第八阶段：SDK 中的 GATT Client 开发

如果你的设备或网关需要主动连接其他 BLE 设备，就需要掌握 GATT Client。

## 11.1 GATT Client 开发流程

```text
1. 初始化 BLE 协议栈
2. 开始扫描
3. 根据广播数据筛选目标设备
4. 发起连接
5. 等待连接成功
6. 发现 Service
7. 发现 Characteristic
8. 发现 Descriptor
9. 读取需要的数据
10. 写入配置或控制命令
11. 写 CCCD 开启 Notify / Indicate
12. 接收 Notify / Indicate
13. 处理断线重连
```

------

## 11.2 Client 侧必须掌握的操作

```text
Scan
Connect
Discover Services
Discover Characteristics
Discover Descriptors
Read Characteristic
Write Characteristic
Write CCCD
Receive Notification
Receive Indication
Disconnect
Reconnect
```

------

## 11.3 Client 侧订阅 Notify 的本质

Client 开启 Notify 不是调用一个抽象的“订阅 topic”，而是写入该 Characteristic 的 CCCD。

```text
写入 0x0001: 开启 Notify
写入 0x0002: 开启 Indicate
写入 0x0000: 关闭
```

------

# 12. 第九阶段：MTU、分包与吞吐

## 12.1 MTU 是什么

MTU 表示 ATT 层单次能承载的数据大小。

默认 MTU 通常是：

```text
23 字节
```

但 ATT Value 实际有效载荷是：

```text
MTU - 3
```

所以默认情况下单包通常只能传：

```text
20 字节
```

------

## 12.2 MTU 协商

连接后，Client 和 Server 可以进行 MTU 交换。

例如：

```text
Client 请求 MTU = 247
Server 支持 MTU = 247
最终 MTU = 247
单包有效数据 = 244 字节
```

注意：

```text
最终 MTU = 双方支持值中的较小值
```

------

## 12.3 什么时候需要分包

以下情况需要考虑分包：

```text
数据长度 > MTU - 3
文件传输
OTA
日志批量上传
图片数据
大块配置
证书下发
```

------

## 12.4 简单分包协议示例

```text
Frame ID: 1 byte
Total Length: 2 bytes
Offset: 2 bytes
Payload: N bytes
CRC: 1 byte
```

接收方根据：

```text
Frame ID
Total Length
Offset
```

进行重组。

------

## 12.5 吞吐优化关注点

影响 BLE 吞吐的因素包括：

```text
MTU
Connection Interval
PHY 1M / 2M
Data Length Extension
每个连接事件可发送包数量
Write Without Response
Notify 发送队列
SDK 缓冲区
手机系统限制
```

应用开发建议：

- 大量数据下发使用 Write Without Response
- 大量数据上报使用 Notify
- 增大 MTU
- 适当减小连接间隔
- 使用发送完成回调做流控
- 不要在应用层盲目 while 连续发送

# 13. 第十阶段：安全、配对、绑定与权限

## 13.1 基本概念

| 概念           | 说明                         |
| -------------- | ---------------------------- |
| Pairing        | 配对，建立安全关系           |
| Bonding        | 绑定，保存密钥，下次自动加密 |
| Encryption     | 加密链路                     |
| Authentication | 认证，防止中间人攻击         |
| Authorization  | 授权，应用层允许访问         |

------

## 13.2 Characteristic 权限

Characteristic 不仅有 Properties，还有 Permissions。

例如：

```text
Read
Read Encrypted
Read Authenticated
Write
Write Encrypted
Write Authenticated
```

应用场景：

```text
普通传感器数据：可直接 Read / Notify
设备序列号：可 Read
控制命令：可要求加密 Write
敏感配置：可要求绑定后访问
OTA：可要求加密连接
```

------

## 13.3 工程建议

- 普通调试阶段可先关闭复杂安全机制
- 产品阶段根据数据敏感性开启加密或绑定
- 不要把关键控制命令完全暴露在无权限 Characteristic 上
- 对恢复出厂、升级、解锁等危险操作增加应用层认证

# 15. 第十二阶段：工程化设计

## 15.1 推荐的软件架构

```text
BLE Stack / SDK
  ↓
BLE Adapter Layer
  ↓
GATT Service Layer
  ↓
Protocol Parser
  ↓
Business Logic
  ↓
Hardware Driver
```

建议不要让业务逻辑直接依赖 SDK API。

可以封装一层：

```text
ble_send_notify()
ble_is_connected()
ble_is_notify_enabled()
ble_get_mtu()
ble_start_adv()
ble_disconnect()
```

这样可以降低 SDK 迁移成本。

------

## 15.2 BLE 事件处理建议

BLE 回调中只做轻量处理：

```text
保存状态
复制数据
投递消息
设置标志位
```

不要在 BLE 回调中做：

```text
长时间阻塞
Flash 大量擦写
复杂计算
等待信号量
无限循环发送
```

------

## 15.3 推荐状态机

```text
INIT
  ↓
ADVERTISING
  ↓
CONNECTED
  ↓
SERVICE_READY
  ↓
NOTIFY_ENABLED
  ↓
DATA_TRANSFER
  ↓
DISCONNECTED
  ↓
ADVERTISING
```

每个状态要明确允许哪些操作。

例如：

```text
未连接：不允许 Notify
未订阅：不允许上报业务数据
MTU 未完成：使用默认单包长度
正在发送：等待发送完成回调
```

# 17. 实战练习路线

## 17.1 入门练习


## 17.2 自定义 Service 练习

目标：能创建自己的 Service。

练习内容：

```text
1. 新建一个 128-bit UUID Service
2. 添加 RX Characteristic，支持 Write
3. 添加 TX Characteristic，支持 Notify
4. App 写入 RX
5. 设备收到后解析数据
6. 设备通过 TX Notify 返回结果
```

完成标准：

```text
App 写 0x01，设备 Notify 返回 0x81
```

------

## 17.3 配置参数练习

目标：掌握配置类数据交互。

练习内容：

```text
1. 添加 Config Characteristic
2. 支持 Read 当前配置
3. 支持 Write 修改配置
4. 修改采样周期
5. 保存到 RAM 或 Flash
6. Notify 当前状态
```

完成标准：

```text
App 能读取和修改设备参数
```

------

## 17.4 数据上报练习

目标：掌握 Notify。

练习内容：

```text
1. 添加 Sensor Data Characteristic
2. Client 开启 Notify
3. 设备定时上报模拟传感器数据
4. 断开连接后停止上报
5. 重连后重新等待订阅
```

完成标准：

```text
只有在连接且 Notify 使能后，设备才会上报数据
```

------

## 17.5 分包传输练习

目标：掌握 MTU 与分包。

练习内容：

```text
1. 获取当前 MTU
2. 计算单包最大数据长度
3. 发送超过 MTU 的大数据
4. 设计 Frame ID / Offset / Length
5. 接收端重组数据
6. 校验 CRC
```

完成标准：

```text
能够稳定传输超过 1 KB 的数据
```

------

## 17.6 安全连接练习

目标：掌握配对和加密。

练习内容：

```text
1. 设置某个 Characteristic 需要加密访问
2. 手机访问时触发配对
3. 配对成功后允许 Read / Write
4. 断开重连后测试绑定恢复
```

完成标准：

```text
未加密时访问失败，加密后访问成功
```

# 18. 常见问题与排查方向

## 18.1 手机能扫描到，但连接不上

检查：

```text
广播类型是否可连接
连接参数是否合法
是否已达到最大连接数
是否白名单限制
是否 SDK 状态异常
```

------

## 18.2 能连接，但看不到自定义 Service

检查：

```text
Service 是否注册成功
UUID 字节序是否正确
App 是否使用缓存
Android / iOS 是否需要刷新 GATT 缓存
是否连接到了旧固件设备
```

------

## 18.3 Notify 收不到

检查：

```text
Characteristic 是否有 Notify 属性
是否添加 CCCD
Client 是否写 CCCD 开启 Notify
Server 是否记录 notify_enabled
发送长度是否超过 MTU - 3
SDK notify API 是否返回成功
是否等待发送完成回调
```

------

## 18.4 Write 收不到

检查：

```text
Characteristic 是否有 Write 属性
权限是否要求加密
写入方式是否匹配
UUID 是否正确
Handle 是否正确
数据长度是否超过限制
回调是否注册成功
```

------

## 18.5 数据超过 20 字节失败

原因通常是：

```text
默认 MTU = 23
有效数据 = 20 字节
```

解决：

```text
进行 MTU 协商
应用层分包
检查双方最大 MTU 支持
```

------

## 18.6 Android 能用，iOS 不正常

检查：

```text
iOS 对后台 BLE 限制
iOS GATT 缓存
iOS 对连接参数的限制
iOS 对 Notify 频率的实际限制
是否使用标准服务更容易被系统识别
```

# 22. 一句话总结

BLE 应用开发的核心不是从零实现协议栈，而是在 SDK 提供的 GAP / GATT 基础设施之上，设计清晰的 Service / Characteristic / 业务协议，并正确处理连接、订阅、读写、上报、分包、安全和调试问题。

对于自定义 BLE 服务对接来说，最重要的是：

```text
广播让对方找到设备
GATT Service 定义功能
Characteristic 定义数据通道
Write 承载控制和配置
Notify 承载主动上报
MTU 和分包决定数据传输能力
CCCD 决定订阅是否开启
业务协议决定真正的数据含义
```
