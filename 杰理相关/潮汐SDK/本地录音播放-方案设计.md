# 本地录音文件播放 — 方案设计

## 1. 总览

```
录音: MIC → ADC(L+R) → OPUS Encoder(16kbps,mono) → 40B/帧 → rdx_uxfile_raw_write → .raw
回放: .raw → fopen → 40B 帧 → source_dev0_input_write → cbuf(2048B)
          → source_dev0_get_packet(40B) → OPUS Decoder(1ch) → DAC
```

全链路单声道 40B OPUS 闭环。不依赖 `file_player` / `TCFG_MUSIC_PLAYER_ENABLE`，使用 `dev_flow_player` 自定义数据流播放器。

### 2.4 帧大小计算

```
opus_stenc_api.h:43-47
  OPUS_CR = 16                                       // 压缩率
  OPUS_SR = 16000                                    // 采样率 Hz
  OPUS_FS = 20                                       // 帧间隔 ms
  FRAME_POINT = 16000 / (1000/20) = 320              // 每帧采样点
  FRAME_SIZE = 320 * sizeof(s16) = 640               // 每帧输入字节 (20ms PCM)

帧大小 = bit_rate × 帧时长 / 8
       = 16000 × 0.020 / 8
       = 40 字节
```

### 2.5 ADC 路由

```
translation_ear_recoder.c:240-242
  case MIC_TO_MONO_OPUS:
    #if (RDX_SUPPORT_ALGORITHM == 1)
        ch = AUDIO_CH_LR;                             // 双路 ADC 输入, 供降噪算法
    #else

```

`ch` 通过 `set_global_ch_mode(ch)` → `fmt.channel_mode = global_ch_mode` → `SINK_DEV0/SINK_DEV1` 的 `NODE_IOC_SET_FMT`。

**这控制的是 ADC 采样路由**（双 mic 采入），**不影响 encoder 声道数**。Encoder 声道数由上面的 `AUDIO_CH_NUM = 1` 决定。

### 2.8 板端日志验证

以下为实际录音时的 pipeline 格式协商日志，**直接证明 encoder 输出是单声道**：

```
[STREAM]format negotiation: suss
  { sink_dev0,          16000, 2 ch, 16bit }         ① MIC 采入 — 2ch 双路（供降噪算法）
  ...
[STREAM]format negotiation: suss
  { sink_dev1,          16000, 1 ch, 16bit }         ② Encoder 输出 — 1ch 单声道！
```

关键原始日志行：

```
adc num: 2 , channel_mode: 25              ← 双 ADC 通道
sink_dev0 nego,type=100,sr=16000,ch_mode=25   ← MIC 采入 2ch
sink_dev1 nego,type=100000,sr=16000,ch_mode=11 ← Encoder 输出 1ch
```

**结论**：① 和 ② 来自同一次录音，ADC 采入双路（供 `RDX_SUPPORT_ALGORITHM` 降噪），但 encoder 输出只有 1 路。录音产的就是 40B 单声道 OPUS。

## 3. 回放侧参数及证据

### 3.0 接口契约

回放本地录音文件只需两个 API，以下为完整的接口要求和约束。

#### 3.0.1 `dev_flow_player_open(ch_num, source_uuid)`

| 参数 | 类型 | 取值 | 说明 |
|------|------|------|------|
| `ch_num` | `u8` | `1` | 单声道 → 内部选 `PIPELINE_UUID_TRANSLATION_MONO` (0x9463) |
| | | `2` | 立体声 → 内部选 `PIPELINE_UUID_TRANSLATION` (0x218B) |
| `source_uuid` | `u16` | `NODE_UUID_SOURCE_DEV0` (0x8FC4) | 绑定 `source_dev0_file` 源节点 |

**调用后行为：**
1. 内部自动 `dev_flow_player_close()` 关闭旧实例（幂等，可重复调用）
2. 创建 jlstream pipeline，协商解码链：`SOURCE_DEV0 → Decoder(OPUS) → DAC`
3. 调用 `source_dev0_open()` → `output_buff_init()` 分配 2048B cbuf
4. 源节点声明格式：16kHz / OPUS / 单声道（`AUDIO_CH_MIX`）
5. 启动 jlstream 线程，开始轮询 cbuf 取数据
6. **返回后不会自动播放**，需要立即调用 `source_dev0_input_write()` 喂数据

#### 3.0.2 `source_dev0_input_write(data, len)`

| 参数 | 类型 | 要求 |
|------|------|------|
| `data` | `u8 *` | 指向 **OPUS 编码**的音频数据（非 PCM） |
| `len` | `u16` | 字节数，**必须是 40 的整数倍**（调用方对齐后传入） |
| 返回值 | `u32` | 实际写入 cbuf 的字节数 |

**数据格式约束：**
- 必须是 OPUS 编码帧，与录音侧产出的 `.raw` 文件格式一致
- 采样率 16kHz，单声道，16kbps 码率
- 每帧标准 40 字节（= 16000bps × 20ms / 8）

**缓冲区机制：**
```
cbuf 总大小：    OUTPUT_BUFF_SIZE = 2048 字节
解码启动阈值：   cbuf ≥ 512 字节 (OUTPUT_BUFF_SIZE/4) 才开始取数据
取数据粒度：    每次取 40 字节（对齐到 40 的倍数）
```

**背压处理（重要）——先问后写，不依赖返回值兜底：**
调用方写入前，**必须**先通过 `source_dev0_get_free_space()` 查询 cbuf 空闲空间：

```
// pb_write_pending() 中的写前检查
free_space = source_dev0_get_free_space();
if (free_space < 40)        → 不写，return 0，下个 tick 再试

write_len = min(remain, free_space)
write_len = (write_len / 40) * 40      // 强制 40 对齐
source_dev0_input_write(data, write_len) // 精确写入，这次一定成功
```

- 空闲不足 40 字节（一整帧）时，直接跳过，不尝试写入
- 空闲够写多少，就写多少——然后对齐到 40 边界，绝不留碎字节
- `cbuf_write` 返回 ≠ 请求量只发生在调用方违反契约时，正常路径永不打 `N`
- 若绕过检查写入非 40 对齐的数据，尾部不足 40 字节的残留会永远卡在 cbuf 中无法被消费

**低水位处理：**
- cbuf 取空时，源节点返回 `NODE_STA_SOURCE_NO_DATA`，打印 `K`
- 首次取空后 `data_ok` 重置为 0，需要重新累积 ≥ 512 字节才能恢复解码
- **所以持续喂数据不能断**，否则会有可闻的播放卡顿

#### 3.0.3 辅助函数

```
source_dev0_file.c:
u32 source_dev0_get_free_space(void)
{
    if (output_buff == NULL) return 0;
    return OUTPUT_BUFF_SIZE - cbuf_get_data_len(&output_cbuf_h);
}

bool source_dev0_is_empty(void)
{
    if (output_buff == NULL) return true;
    return cbuf_get_data_len(&output_cbuf_h) == 0;
}
```

#### 3.0.4 `dev_flow_player_close()`

停止并释放 pipeline，释放 cbuf。调用后如需再播放需重新 `dev_flow_player_open()`。

#### 3.0.5 典型调用序列

```
// 1. 启动播放器
dev_flow_player_open(1, NODE_UUID_SOURCE_DEV0);

// 2. 立即填充 cbuf 到满（≥512B 以触发解码）
while (source_dev0_get_free_space() >= 40) {
    fread(frame, 1, 40, file);
    source_dev0_input_write(frame, 40);
}

// 3. 启动定时器持续喂数据（10ms 周期）
sys_timer_add(NULL, feed_timer_cb, 10);
// feed_timer_cb: while (free >= 40) { read; write; }

// 4. 播放完毕
dev_flow_player_close();
```


**每个 stream_frame 固定 40 字节**，与录音侧 40B/帧完全对齐。

### 3.4 数据泵策略：先问后写，对齐灌入

**写侧（泵 → cbuf）**：`pb_write_pending()` 先查询 cbuf 空闲空间，再决定写多少：

```
pb_pump_fill() 入口
  │
  ├─ 有 pending 数据？
  │   └─ pb_write_pending():
  │         free_space = source_dev0_get_free_space()
  │         │
  │         ├─ free_space < 40B  →  return 0（不写，退出，下个 tick 再试）
  │         │
  │         └─ free_space >= 40B:
  │               write_len = min(remain, free_space)
  │               write_len = (write_len / 40) * 40   // 对齐到完整帧
  │               source_dev0_input_write(..., write_len)  // 精确写入
  │
  ├─ 无 pending + 未 EOF → pb_load_next_chunk(520):
  │         fread 520B → pending_len = 520 → 回到上面写出去
  │
  └─ EOF + pending 为空 → 停止灌入，等待排空
```

**读侧（cbuf → 解码器）**：`source_dev0_get_packet()` 每 8ms 被源节点调用：

```
source_dev0_get_packet() 每 8ms
  │
  ├─ data_ok == 0 && cbuf 数据量 < 512B
  │   └─ return NULL（一次性门槛）
  │
  ├─ data_ok == 1（已通过门槛）:
  │     source_input_read(output, 40)
  │       ├─ cbuf 不足 40B → 读到 0 → 打 K → data_ok 归零
  │       └─ cbuf >= 40B  → 读 40 字节送入解码器
```

**背压联动**：泵写（10ms tick，520B/块）速度远超源节点消费（8ms，40B/帧）。修复前是先写再看能不能塞进去，导致 `N` 标记从头打到尾。修复后先问空闲空间再对齐写入，`cbuf_write` 每次都能完全写入，不再产生部分写入。

| 参数 | 写侧 | 读侧 |
|------|------|------|
| 单位 | fread 520B，写入量 = min(剩余, 空闲) 对齐 40B | 固定 40B |
| 触�� | 10ms 定时器 | 8ms 帧中断 |
| 门槛 | cbuf 空闲 ≥ 40B 才写 | cbuf ≥ 512B 才启动（仅初始态） |
| 速率 | 受背压自然限速 | ~5 B/ms |

### 3.5 预填充

```
rdx_playback.c:387    pb_pump_fill();   // dev_flow_player_open 后立即填 cbuf
```

`pb_pump_fill()` 循环读写，每次写前检查空闲空间。填到 cbuf 空闲不足 40 字节时自动停止。通常首次能灌入约 2040 字节（51 帧），远超 512 字节解码阈值。管道启动后立刻有数据可消费。

### 3.6 持续喂数据

```
rdx_playback.c:23     #define PB_PUMP_INTERVAL_MS   (10u)  // 10 ms 定时器
rdx_playback.c:292    pb_schedule_pump(PB_PUMP_INTERVAL_MS)
```

- 10ms 周期 `sys_timeout_add_2_task` 投递到 `app_core`
- 每次回调调用 `pb_pump_fill()`：先写完 pending 数据，再尝试读下一块
- cbuf 空闲不足 40 字节时，`pb_write_pending` 的写前检查直接返回 0，循环自然退出
- 管道每 20ms 只消费 1 帧（40B），泵 10ms tick 一次但大部分 tick 因空闲不足而空转

### 3.7 EOF 排空

不数 tick，直接 poll cbuf：

```
rdx_playback.c:262    if (source_dev0_is_empty()) → pb_finish_natural()
rdx_playback.c:267    if (++drain_ticks > 300)   → 看门狗强停 (3s)
```

- 正常情况：cbuf 空了说明管道消费完毕，立即结束
- 异常情况：3s 看门狗防止管道卡死导致永不结束

### 3.8 回放侧参数总结

| 参数 | 值 | 说明 |
|------|-----|------|
| 播放管道 | `PIPELINE_UUID_TRANSLATION_MONO` | `dev_flow_player_open(1, ...)` |
| 源节点格式 | 16kHz / OPUS / 单声道 | `source_dev0_get_fmt()` |
| 泵出周期 | **10 ms** | `PB_PUMP_INTERVAL_MS` |
| 泵出策略 | **先问后写，对齐灌入** | `pb_write_pending()`, 写前查 `free_space`, clamp+align 到 40B |
| fread 块 | 520 字节 (13 帧) | `PB_CHUNK_BYTES` |
| cbuf 大小 | 2048 字节 | `OUTPUT_BUFF_SIZE` |
| 解码阈值 | 512 字节 | `OUTPUT_BUFF_SIZE/4` |

| EOF 排空 | poll `source_dev0_is_empty()` | 精确、无猜测 |
| 排空看门狗 | 3 秒 | `PB_DRAIN_TIMEOUT_MS` |
