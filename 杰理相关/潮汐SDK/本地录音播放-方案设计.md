# 本地录音文件播放 — 方案设计

## 1. 原理

```
┌──────────┐    ┌─────────────┐    ┌──────────┐    ┌──────┐    ┌─────┐
│ SD NAND  │ →  │ DAT 缓存     │ →  │ fopen    │ →  │ OPUS │ →  │ DAC │
│ .raw 文件│    │ (开机已加载)  │    │ 文件句柄  │    │ 解码  │    │ 输出 │
└──────────┘    └─────────────┘    └──────────┘    └──────┘    └─────┘
                                        ▲
      ┌─────────────────────────────────┘
      │  music_file_play(FILE *, ...) / music_file_play_callback(FILE *, priv, cb, ...)
      │  读文件 → jlstream 管道 → 解复用 → 解码 → 音效 → DAC
      │
      ├── music_file_player_ff(step_s, player)  快进 N 秒
      ├── music_file_player_fr(step_s, player)  快退 N 秒
      ├── music_file_player_pp(player)          暂停/恢复
      └── music_file_player_stop()              停止
```

JL BR28 平台已有的 `file_player` 模块封装了"读文件→解复用→解码→DAC"全链路。OPUS 解码器已随录音功能一起启用（`TCFG_DEC_OPUS_ENABLE=1`）。录音和播放用的是同一个 codec——编码存文件，解码放文件。

**不需要**开启 `TCFG_APP_MUSIC_EN`（那是给 U 盘/SD 卡音乐浏览模式用的）。本地录音播放只需要 `TCFG_MUSIC_PLAYER_ENABLE`，它控制的是底层 `file_player` 本地文件播放引擎，和蓝牙音乐（`a2dp_player`）是两套不同的模块。直接用 `file_player.h` 的底层 API 即可，更轻量，完全可控。

> 关于 `TCFG_MUSIC_PLAYER_ENABLE` 的说明：SDK 原生在 `app_config.h` 里有一个兜底定义 `#define TCFG_MUSIC_PLAYER_ENABLE TCFG_APP_MUSIC_EN`，也就是说它原本只是完整音乐 App 的附属开关。当前方案在 `sdk_config.h` 里把它**显式定义为独立的 `1`**，目的就是把本地文件播放引擎从完整音乐 App 模式里剥离出来：不开 `TCFG_APP_MUSIC_EN`，只开 `TCFG_MUSIC_PLAYER_ENABLE`，就能独立启用 `file_player` 播放录音文件。

## 2. 现状

### 2.1 基础条件已就绪

| 组件 | 状态 | 证据 |
|------|------|------|
| SD 挂载 | ✅ | `sd0 mount ok`，由 `rdx_app_emmc_poweron()` 控制供电 |
| DAT 缓存 | ✅ | 开机 4s 完成同步，23 个文件、329920 字节 |
| 文件元数据 | ✅ | `uxfile_data_t` 含 `sn`、`filename`、`data_len` |
| OPUS 解码 | ✅ | `TCFG_DEC_OPUS_ENABLE = 1` |
| file_player API | ✅ | `music_file_play()` / `ff()` / `fr()` / `pp()` / `stop()` 已编译进工程 |
| APP_MSG handler | ✅ | `APP_MSG_REC_PREV/NEXT/FR/FF` 四个 case 已存在于 `rdx_app.c` |

### 2.2 尚未验证 — 播放链路的关键不确定点

| 不确定点 | 说明 |
|----------|------|
| **文件路径格式** | 日志存在性检查显示 `/287875.RAW` 风格（大写扩展名、带斜杠）。DAT 中可能是带路径的完整格式，`mnt/sd0/%s` 仅是猜测 |
| **`.raw` OPUS 格式能否被 file_player 识别** | `file_player` 依赖 `jlstream` 做格式检测，OPUS-in-raw 是否能正确解复用未验证 |
| **music_file_play(FILE*) 的文件句柄生命周期** | `file_player.c` 的 `music_file_close()` 依赖 `music_player_get_file_hdl()`，裸用 `music_file_play(FILE*)` 时不能假设 FILE* 一定会被正确关闭 |
| **回调事件名** | `file_player.c` 对外回调的是 `STREAM_EVENT_START`、`STREAM_EVENT_STOP`、`STREAM_EVENT_NONE`，不是 `PLAY_END` |
| **STREAM_EVENT_STOP 的多重语义** | 正常播完、主动 stop、读错误、快进越过文件尾都会触发 STOP。不能简单收到 STOP 就自动下一首 |
| **music_file_player_stop() 后 pb.fp 失效** | `stop()` 内部 `music_player_free(player)`，之后 pb.fp 指向已释放内存，必须立即置 NULL |
| **DAT 与文件系统不一致** | 日志中既有 `File not exist ... marking invalid` 又有 `Mismatch: cache=11, file=~24`，播放列表不能直接信任 DAT，必须以 `fopen` 成功为准 |

> **结论：当前日志没有任何 `APP_MSG_REC_*`、解码 START/STOP 或 `music_callback` 证据。只能说基础设施在，不能说 `.raw` 已能被 `music_file_play()` 正常解码出声。**

### 2.3 按键映射现状

当前 `rdx_key.c` 中五个 IO 按键的实际映射：

| 按键 | 短按 | 长按 |
|------|------|------|
| NUM0 (上一个) | `APP_MSG_NULL` | `APP_MSG_REC_FR` (快退) |
| NUM1 (下一个) | `APP_MSG_NULL` | `APP_MSG_REC_FF` (快进) |
| NUM2 (音量加) | `APP_MSG_VOL_UP` | `APP_MSG_VOL_UP` |
| NUM3 (音量减) | `APP_MSG_RECORD_SWITCH` | `APP_MSG_RECORD_SWITCH` |
| NUM4 (录音开关) | `APP_MSG_NULL` | `APP_MSG_RECORD_SWITCH` (flag) |

**NUM0/NUM1 的短按当前是 `APP_MSG_NULL`**，尚未映射到 `APP_MSG_REC_PREV/NEXT`。

修改按键映射前，需先用以下命令定位具体行号：
```bash
grep -n "APP_MSG_NULL" SDK/apps/common/third_party_profile/rdx_protocol/rdx_key.c
```

并确认：NUM0/NUM1 短按改为 PREV/NEXT 后，与长按 FR/FF 不会冲突。`rdx_key.c` 中长按通常通过定时器在 `key_driver` 层判断，短按和长按是 `KEY_ACTION_CLICK` vs `KEY_ACTION_LONG` 两个不同 action，不会互相覆盖。

### 2.4 播放互斥条件

封装为单一检查函数：

```c
static bool rdx_playback_can_start(void)
{
    RecordStatus *rp = rdx_record_get_status();
    if (rp->run == RECORD_STATE_START || rp->run == RECORD_STATE_RESUME) {
        log_info("playback blocked: recording");
        return false;
    }
    if (rdx_uxfile_is_sync_in_progress()) {
        log_info("playback blocked: dat sync in progress");
        return false;
    }
    if (rdx_uxfile_is_datFileInfo_loading()) {
        log_info("playback blocked: dat file info loading");
        return false;
    }
    return true;
}
```

以下条件有一个为真则不应启动播放：
- 正在录音（`rp->run == RECORD_STATE_START \|\| rp->run == RECORD_STATE_RESUME`）
- DAT 同步正在进行（`rdx_uxfile_is_sync_in_progress()`）
- DAT 分包传输中（`rdx_uxfile_is_datFileInfo_loading()`）
- SD 未挂载或正在格式化

## 3. 设计

### 3.1 新增数据结构和消息

```c
// rdx_playback.c
typedef enum {
    PB_INTENT_NONE = 0,   // 无意图（正常播完可自动下一首）
    PB_INTENT_STOP,       // 人为停止（不自动切歌）
    PB_INTENT_SWITCH,     // 切歌中（不自动切歌，等切换完成后清零）
} pb_intent_t;

typedef struct {
    u32  cur_sn;             // 当前播放文件的 SN，0 = 未在播放
    u32  total_count;        // 缓存中文件总数
    struct file_player *fp;  // music_file_play_callback 返回的播放器句柄
    u8   status;             // 0=STOP, 1=PLAYING, 2=PAUSE
    pb_intent_t intent;      // 当前操作意图，用于 STOP 回调区分是否自动下一首
} rdx_playback_t;

static rdx_playback_t pb = { 0 };
```

新增消息定义（`app_msg.h`）：

```c
APP_MSG_REC_AUTO_NEXT,    // 当前文件播完，自动下一首（回调→app_core 内部消息）
```

### 3.2 文件路径构造

日志显示录音文件名格式为 **不定长十六进制 SN + `.raw`**（DAT 中可能为大写 `.RAW` 且带 `/` 前缀）。

**关键约束**：
- 使用 DAT 缓存中的 `filename` 字段原样构造路径，不要自己拼扩展名
- 先检查 `filename[0]` 是否为 `'/'`，避免双斜杠

```c
const char *fname = fi->filename;   // DAT 中的原始值，如 "/287875.RAW" 或 "287875.raw"

// 候选路径按优先级尝试：
const char *candidates[] = {
    fname,              // 候选 0: DAT 原始值（可能已含路径）
    // 如果 fname 不以 '/' 开头，追加以下前缀：
    // "mnt/sd0/%s"     // 候选 1
    // "/mnt/sd0/%s"    // 候选 2（某些 SDK 配置）
    // "/%s"            // 候选 3（兜底）
};
```

以 `fopen(path, "r")` 成功且 `flen(f)` 返回非零为准，硬编码第一个有效的路径格式。FAT 文件系统不区分大小写，但路径前缀区分。

### 3.3 四个按键处理函数

#### PREV — 上一个录音文件

```
前提: rdx_playback_can_start() == true
1. 在 DAT 缓存中找 SN < pb.cur_sn 的最大 SN
2. 如果 pb.cur_sn == 0（未播放），选最大 SN（最新文件）
3. rdx_uxfile_get_file_data_by_sn(target_sn, ...) → 拿到 filename
4. 逐个尝试候选路径 fopen(path, "r")
5. 如果 fopen 失败 → 跳过，找下一个更小的 SN，重复
6. 如果 pb.fp != NULL:
     pb.intent = PB_INTENT_SWITCH
     music_file_player_stop()
     pb.fp = NULL
7. pb.fp = music_file_play_callback(file, &pb, pb_callback, NULL)
8. pb.cur_sn = target_sn, pb.status = PLAYING, pb.intent = PB_INTENT_NONE
9. 找不到任何有效文件 → 停止，打印提示
```

#### NEXT — 下一个录音文件

```
同 PREV，方向相反：找 SN > pb.cur_sn 的最小 SN
fopen 失败时跳过，尝试下一个 SN
```

#### FR — 快退

```
if (pb.fp && pb.status == PLAYING) {
    music_file_player_fr(3, pb.fp);  // 步长 2-3 秒
}
```

#### FF — 快进

```
if (pb.fp && pb.status == PLAYING) {
    music_file_player_ff(3, pb.fp);
}
```

### 3.4 播放结束回调

```c
// 所有 playback 日志统一前缀
#define PB_LOG(fmt, ...)  log_info("[PB] " fmt, ##__VA_ARGS__)

static int pb_callback(void *priv, int parm, enum stream_event event)
{
    rdx_playback_t *pb_ctx = (rdx_playback_t *)priv;  // priv 是 &pb

    switch (event) {
    case STREAM_EVENT_START:
        PB_LOG("cb START");
        pb_ctx->status = PB_STATUS_PLAYING;
        break;

    case STREAM_EVENT_STOP: {
        PB_LOG("cb STOP");
        pb_ctx->status = PB_STATUS_STOP;

        // 访问 fp 前先判空——stop() 内部可能已 free
        struct file_player *p = pb_ctx->fp;
        if (p == NULL) break;

        // 仅当"正常播完"且"非人为停止/切歌"时自动下一首
        if (p->read_err == 0
            && pb_ctx->intent != PB_INTENT_STOP
            && pb_ctx->intent != PB_INTENT_SWITCH) {
            PB_LOG("auto next");
            // 投递消息到 app_core，不在回调里做 fopen / music_file_play
            os_taskq_post_type("app_core", Q_MSG, 1,
                (int[]){ APP_MSG_REC_AUTO_NEXT });
        }

        // stop 后 fp 已失效
        pb_ctx->fp = NULL;
        break;
    }

    default:
        break;
    }
    return 0;
}
```

**约束**：
- `priv` 传 `&pb`，不是 `struct file_player *`
- 回调在 file_player 的 stream 上下文执行，**禁止在回调里直接 `fopen`、遍历 DAT、调用 `music_file_play`**
- `STREAM_EVENT_STOP` 后 `pb.fp` 已被 `music_player_free()` 释放，回调里判空后不再访问
- `APP_MSG_REC_AUTO_NEXT` 在 `rdx_app_msg_handler` 中处理，调用 `rdx_playback_next()`，与手动 NEXT 共用同一套导航逻辑

### 3.5 主动停止

```c
void rdx_playback_stop(void)
{
    if (pb.fp) {
        pb.intent = PB_INTENT_STOP;
        music_file_player_stop();
        pb.fp = NULL;      // stop() 已 free，立即置空
        pb.status = PB_STATUS_STOP;
        pb.intent = PB_INTENT_NONE;
    }
}
```

### 3.6 文件组织

| 职责 | 位置 | 说明 |
|------|------|------|
| 播放状态 & 核心逻辑 | `rdx_playback.c/h` (新建) | `rdx_playback_init()`、`prev()`、`next()`、`ff()`、`fr()`、`stop()`、`can_start()` |
| 消息 handler 对接 | `rdx_app.c` | 四个 `case APP_MSG_REC_*` + `APP_MSG_REC_AUTO_NEXT` 改为调用 `rdx_playback_*()` |
| 按键映射 | `rdx_key.c` | NUM0 短按 → `APP_MSG_REC_PREV`，NUM1 短按 → `APP_MSG_REC_NEXT` |
| 消息定义 | `app_msg.h` | 新增 `APP_MSG_REC_AUTO_NEXT` |
| 编解码器 | 无需改动 | TCFG_DEC_OPUS_ENABLE 已开启 |
| file_player | 无需改动 | 使用现有 API |

### 3.7 边界条件

| 场景 | 处理 |
|------|------|
| 缓存为空（0 个文件） | 直接返回，打印提示 |
| DAT 与文件系统不一致 | 以 `fopen` 成功为准，失败则跳过并尝试下一个 |
| SD 未挂载 | 播放前检查，未挂载则 `rdx_app_emmc_poweron()` 后重试 |
| 播放中收到新按键 | 设 `intent = SWITCH`，`stop()` 后再 `play()` 新文件 |
| 快进超过文件末尾 | `file_player` 触发 `STREAM_EVENT_STOP`，`intent` 为 `NONE` → 自动下一首 |
| 快退到文件开头 | `file_player` 内部限制到 0 |
| 录音进行中 | `can_start()` 返回 false，忽略播放键 |
| DAT 同步/传输中 | 同上 |

## 4. 不在此次范围

- **手机端远程控制播放**：需要通过 BLE 协议命令触发，属于二期
- **音量加减（NUM2）的 handler**：当前 `APP_MSG_VOL_UP/DOWN` 在 `rdx_app_msg_handler` 中无处理，需单独补齐
- **播放列表的复杂排序/过滤**：当前按 SN 线性排列即可
- **NUM3/NUM4 的录音控制语义修正**：属于录音流程

## 5. 实施策略：分两阶段

### 阶段 1：最小可验证版本（验证播放链路）

**目标**：用一个 `.raw` 文件跑通 `fopen → music_file_play_callback → 解码 → DAC` 全链路。

**入口**：阶段 1 不需要等 NUM0/NUM1 映射改完。可临时用已有按键触发（如 `APP_MSG_RECORD_SWITCH`），或在 BLE 连接成功后自动触发一次，减少同时修改的变量数。

**验证清单**：

| # | 验收项 | 证据 |
|---|--------|------|
| 1 | 候选路径中有一个能 `fopen` 成功，`flen()` 非零 | `[PB] path=/..., flen=...` |
| 2 | `music_file_play_callback()` 返回非 NULL | `[PB] fp=0x...` |
| 3 | 回调收到 `STREAM_EVENT_START` | `[PB] cb START` |
| 4 | 回调收到 `STREAM_EVENT_STOP`，`read_err=0` | `[PB] cb STOP, read_err=0` |
| 5 | DAC 有音频输出 | 听到声音，或用示波器/逻辑分析仪抓 I2S/SPK 信号 |
| 6 | 播放过程中不 crash、不卡死 | 整机正常响应其他事件 |

**阶段 1 不要求**：按键映射修改、PREV/NEXT 导航、FF/FR、自动切歌、互斥守卫。

### 阶段 2：完整播放功能

阶段 1 通过后，依次补齐：

- NUM0 短按 `APP_MSG_REC_PREV`，NUM1 短按 `APP_MSG_REC_NEXT`（不改长按 FR/FF）
- PREV/NEXT 完整导航（跳过 `fopen` 失败的文件）
- 播放结束自动切下一个（`APP_MSG_REC_AUTO_NEXT`）
- 录音互斥守卫（`rdx_playback_can_start()`）
- FR/FF 快进快退
- 播放状态管理（STOP/PLAYING/PAUSE）

**阶段 2 验收清单**：

| # | 验收项 |
|---|--------|
| 1 | NUM0 短按 → PREV，NUM1 短按 → NEXT |
| 2 | PREV/NEXT 能跳过 `fopen` 失败的文件 |
| 3 | 正常播完后自动切下一个 |
| 4 | 正在录音时按播放键被忽略，打印 `[PB] blocked: recording` |
| 5 | FR/FF 前进/后退 3 秒 |
| 6 | 多次快速按键不 crash、不同步卡死 |

## 6. 实战踩坑记录（2026-07-03）

> 这部分记录从首次落地到串口复测中实际遇到的问题。后续遇到新日志时，优先按“按键入口 → DAT/SN → fopen → pipeline UUID → file_player → decoder START/STOP → DAC”的顺序定位，不要跨层猜测。

| # | 现象 / 日志证据 | 根因 | 修复方案 | 经验 |
|---|---|---|---|---|
| 1 | `APP_MSG_REC_NEXT` 后长时间没有 `[PB] next: trying`，串口出现大量 DAT hex dump 和 `timer_no_response: app_core` | `find_max_sn()` 从 `1..4096` 逐个调用 `rdx_uxfile_get_file_data_by_sn()`，这个 API 会触发较重的 DAT 访问，放在 app_core 按键链路里会阻塞 | 删除固定范围扫描，改用库里已有的 `rdx_uxfile_get_the_largest_fileSn()` 获取最大 SN；在 `rdx_uxfile.h` 补声明 | 不要在实时消息处理路径里暴力遍历 DAT。隐藏库 API 名字看起来像查询，实际可能会读文件、dump、加锁 |
| 2 | `total_count=6`，但实际 SN 是 `2/4/6/10/12/14`；用文件总数当 SN 上限会漏文件 | DAT 的 `total_count` 是记录数量，不是最大 SN；SN 是稀疏递增序列 | 新增 `pb_max_sn`，由 `rdx_uxfile_get_the_largest_fileSn()` 维护；PREV/NEXT 以 `pb_max_sn` 为边界 | `count` 和 `id/max_sn` 必须分开建模。任何删除、异常恢复、旧文件同步都会让 SN 不连续 |
| 3 | 首次按 PREV 时可能从 `0xFFFFFFFF` 开始扫，表现为卡死或极慢 | `cur_sn == 0` 时做 `start - 1`，无符号下溢 | `cur_sn == 0` 时 PREV 从 `pb_max_sn` 往下找，NEXT 从 1 往上找 | 所有无符号倒序循环都要单独处理 0 边界，不能直接 `sn >= 1` 后自减 |
| 4 | 遇到第一个 DAT 记录打不开就失败，后续可播放文件没有机会尝试 | `rdx_playback_play_file()` 不返回打开失败状态，PREV/NEXT 无法继续遍历 | `rdx_playback_play_file()` 改成 `bool`，`false` 表示打不开；PREV/NEXT 在循环内 `continue` 尝试下一个 SN | DAT 和实际文件系统可能不一致，播放导航必须以 `fopen + flen > 0` 为准，不能只信 DAT |
| 5 | 初版切歌/停止后可能残留 FILE 句柄或 player 句柄 | `music_file_play_callback()` 失败、STOP 回调、手动 stop、切歌各自清理，职责分散 | 引入 `pb_file` 跟踪当前 FILE；用 `cleanup_playback()` 统一释放 player/file | 资源清理要集中。尤其是 SDK player 内部会管理链表，外部再持有 FILE 时必须有单一 owner |
| 6 | 初版 `cleanup_playback()` 先 `fclose(pb_file)` 再 `music_file_player_stop()`，存在 stop 阶段访问已关闭 FILE 的风险 | file_player 停管线时仍可能读/seek 当前 FILE | 顺序改为：保存 `old_file` → `pb_file=NULL` → `music_file_player_stop()` → `pb.fp=NULL` → 最后 `fclose(old_file)` | stop 管线先于关闭底层文件句柄，顺序要和官方 `music_player.c` 模式一致 |

### 当前验证边界

截至 2026-07-06，已验证到以下边界：

```text
APP_MSG_REC_NEXT
  → [PB] lazy init: total=2 files, max_sn=4
  → [PB] next: trying SN=2
  → [PB] path match[1]: storage/sd0/C/46d3ca.raw, flen=78800
  → pipeline_uuid: d96f
  → [JLSTREAM]create_stream: 430ab0, 0       ← 流创建成功
  → demuxer_not_match: 46D3CA.RAW             ← demuxer 不认识 .raw 格式
  → music_file_player_start: start failed, err=-2
```

已确认：

- 独立 tab 方案有效：`本地音乐(0x6FEC) → 解码器(0x1A85) → BitWidth Convert(0x7DE5) → DAC(0xDCCD)` 节点链路已拉通。
- `jlstream_pipeline_parse(0xD96F, 0x6FEC)` 成功，`create_stream` 返回有效指针。
- **下一道关卡**：`jldemuxer_get_tone_file_fmt()` 在预编译库里无法识别无头 `.raw` OPUS 数据。当前已开始方案 A，在 `file_player.c:music_file_get_fmt()` 加 fallback：当 demuxer 失败且文件名以 `.raw` 结尾时，手工设置 `coding_type=AUDIO_CODING_OPUS / sample_rate=16000 / channel_mode=AUDIO_CH_LR`；下一轮板端日志重点看是否打印 `RDX raw OPUS fallback` 并推进到 decoder START/协商阶段。

已确认：

- 按键入口有效。
- DAT 缓存可加载，最大 SN 可获得。
- SN 稀疏遍历有效。
- 文件路径已能打开到真实 `.raw` 文件。
- `"music"` pipeline 名称已经能返回非 0 UUID。
- `pipeline_uuid: d96f` 来自 `jlstream_event_notify(STREAM_EVENT_GET_PIPELINE_UUID, "music")` 的返回值；当前 `"music"` 被映射到 `PIPELINE_UUID_A2DP (0xD96F)`，这不是从录音文件里解析出来的。
- 这个 UUID 属于蓝牙音乐的 A2DP pipeline；本地录音播放如果复用它，就必须保证该 pipeline 里同时有 `NODE_UUID_MUSIC (0x6FEC)` 源节点。如果替换了 A2DP 的源节点，蓝牙音乐会受影响。
- 更干净的做法是新建一条专门给本地录音播放的 pipeline（例如 UUID `0xE000`），源节点为 `本地音乐 (0x6FEC)`，链路走 `音量控制器 → EQ → BITWIDTHCONVERT → DAC`，并在 `get_pipeline_uuid("music")` 里返回这个新 UUID。

## 最终结论：解码器无法从无头裸流确定双声道

### 核心死穴

经过 11 轮迭代验证（方案A 五轮 + 方案B 四轮 + 编码器 format_mode 两轮），所有路径汇聚到同一个瓶颈：

```
{ decoder,            16000, 1 ch, 16bit }
```

**SDK OPUS 解码器（无论标准 `opus_dec_plug` 还是 `opus_stenc_dec_plug`）无法从 format_mode=0/1 的无头裸流中确定双声道。** 解码器始终协商为单声道输出，导致 packet 边界错位 → 杂音。

### 完整迭代记录

| # | 方案 | 改动 | 结果 |
|---|------|------|------|

| 2 | A | x6flow 新建独立 tab，`0x6FEC→0x1A85→0x7DE5→0xDCCD` | `create_stream` OK，`demuxer_not_match` |

### 根因分析

```
录音侧                                     播放侧
──────────────────────────────────────     ──────────────────────────────────────
STENC OPUS encoder (format_mode=0/1)      file_player / source_dev0
    ↓                                         ↓
写入裸 OPUS 帧（无容器头/无声道元数据）     jldemuxer / decoder 读帧
    ↓                                         ↓
.raw 文件                                  解码器需要从数据中确定声道数
                                               ↓
                                          format_mode=0: 无 TOC → 默认单声道
                                          format_mode=1: 有 range 头但解码器不认
                                          format_mode=2: OGG 但 jldemuxer 不认 .raw 扩展名
                                               ↓
                                          协商: decoder = 1ch（单声道）
                                               ↓
                                          双声道 80B 数据按单声道 40B 解 → 杂音
```

### SDK 文档已确认的方案（当前版本不可用）

`media_flow.html §3.5.8` 记载了：

```c
// audio_general_config.h
#define TCFG_DECODER_OUTPUT_CH_SEL    AUDIO_CH_LR   // 强制解码器输出立体声
```

但此宏在 SDK 3.0.0 的 `audio_general_config.h` 中不存在——这是更高版本 SDK 的功能。

### 可用的解决路径

| 路径 | 工作量 | 说明 |
|------|--------|------|
| **A. 升级 SDK** | 大 | 升级到支持 `TCFG_DECODER_OUTPUT_CH_SEL` 的版本，一行宏解决 |
| **B. 录音改用 OGG + 改扩展名** | 中 | `format_mode=2` + 文件名 `.ogg`（需改 uxfile 预编译库或后处理重命名），`file_player` 原生支持 OGG→自动识别双声道 |
| **C. 软件重编码** | 大 | 播放前在应用层做 STENC decode → PCM → 再走标准音频路径，完全绕过 SDK 解码器 |
