# BLE 单连接多服务共存与统一广播方案——RDX 私有服务与 HOGP

> 架构名称：Single-Connection Multi-Service GATT Server with Unified Advertising。
>
> 文档定位：基于当前 T2620 RDX + HOGP 代码，取消 CONFIG/HOGP 分时广播和 KEY5 三击切换，改为一个统一广播入口、一套多服务 GATT Profile、最多一条 BLE 连接。
>
> 完成状态：2026-07-17 已完成 Phase 0～4 的代码实施、兼容清理、host contract、Windows 固件构建和目标 App/PC 回归；该架构现为唯一正式 BLE 运行模型。

## 1. 结论

本方案已经完成并作为正式架构落地。实施前识别的三个前置问题均已收敛：

1. 单连接竞争采用 first-come-first-served，不做中心身份猜测或固件强制抢占；需要切换中心时，由用户断开/关闭当前占用方后让另一方连接；
2. 安全策略固定为 `IO_CAPABILITY_NO_INPUT_NO_OUTPUT + Just Works + Bonding + Secure Connections`，不虚假声明 MITM；
3. bonded HID CCC 不依赖 JL runtime 表，已按 SM peer identity 持久化和匹配恢复，不同 peer 不继承订阅状态。

产品安全决策（2026-07-17）：RDX 采用“连接即有权访问”模型。任意 BLE 中心只要建立当前唯一连接并按协议格式访问 RDX ATT 句柄，即可调用包括名称修改、录音、文件操作、OTA、格式化、恢复出厂、绑定/解绑和 HOGP keymap 修改在内的敏感 RDX 命令。产品明确接受该暴露面，不再把远端身份鉴权、owner 或 session auth gate 作为 Phase 2B 前置条件。单连接、ATT 参数校验、协议帧校验、busy 状态、OTA 完整性校验和当前连接句柄检查仍必须保留，但这些机制不得被描述为远端身份认证。

删除 owner 不等于删除在线/离线功能区分。产品继续以 HID capability 状态区分按键路径：只有当前链路满足 `HID Input CCC enabled && encrypted && !suspended`、即 `rdx_hogp_keyboard_is_ready()==1` 时才进入在线 HOGP 按键功能；其他情况，包括 App 只访问 RDX、普通 BLE 已连接但未订阅 HID、配对/加密未完成、CCC 关闭或 HID suspend，均走原有离线功能。这里的“HID 连接”是同一 BLE 链路上的 HID ready 状态，不是新的 owner，也不是第二条连接。

目标运行模型如下：

```text
                 统一 connectable legacy advertising
        主 ADV：Flags + 统一名称
        Scan Response：RDX 私有数据 + HID UUID
                               │
                ┌──────────────┴──────────────┐
                │                             │
          Windows / PC                  手机 BLE App
          以 HOGP 连接                   以 RDX 连接
                │                             │
                └──────────────┬──────────────┘
                               │
                      任一中心连接成功
                               │
                        停止 BLE 广播
                               │
                  同一 GATT Profile 全部可发现
                 RDX 与 HID 按会话能力独立激活
                               │
                           连接断开
                               │
                        恢复统一广播
```

本方案明确不支持手机和 PC 同时保持两条 BLE 连接。工程继续保持：

```c
config_le_hci_connection_num = 1;
config_le_gatt_server_num = 1;
```

这里的“统一入口”不是在广播阶段判断连接方属于 APP 还是 HID，而是让任意中心从同一广播进入同一条连接。连接建立后，根据对端实际访问和订阅的服务决定启用哪些能力；RDX 命令不再附加远端身份鉴权条件。

## 2. 当前方案存在的问题

当前代码虽然已经把 RDX Service 和 HID Service 放进同一个 `rdx_profile_data[]`，但运行时仍然人为分成两个互斥身份：

- `rdx_ble_mode_controller.c/.h` 保存 `requested_mode`、`advertised_mode` 和 `connection_owner`；
- CONFIG 广播只放 RDX 名称和私有 Manufacturer Data；
- HOGP 广播放 HID Service UUID、Keyboard Appearance 和名称；
- 连接建立时，根据连接前的广播模式把连接标记为 `RDX_BLE_OWNER_CONFIG` 或 `RDX_BLE_OWNER_HOGP`；
- ATT read/write、RDX notify、HID notify 和配键协议继续按 owner 拒绝另一侧访问；
- KEY5 三击调用 `rdx_ble_mode_request_toggle()`，有连接时还要先断开再切换广播。

这套 owner 模型解决的是“两个广播身份分时复用”，不是 BLE 栈的硬限制。统一入口后，广播身份和连接归属都不再成立，应直接删除，而不是增加第三种 `BOTH` 模式。

## 3. 目标与非目标

### 3.1 目标

1. 上电后始终发送一种统一广播，不需要 KEY5 切换。
2. PC 能根据标准 HID 信息发现并连接键盘。
3. 私有 BLE App 能根据既有 RDX Manufacturer Data 发现并连接设备。
4. 任一中心连接后停止广播，只维持一条 BLE 连接。
5. 同一连接上的 RDX Service 和 HID Service 都保持可发现。
6. RDX 数据通道与 HID 键盘通道各自根据实际使用状态启用。
7. 手机仅连接 RDX 时不主动弹出系统 BLE 配对提示。
8. PC 使用 HID 时完成配对/加密，只有 ready 后才能发送键盘报告。

### 3.2 非目标

- 不支持手机和 PC 双中心并发连接；
- 不增加第二个 `app_ble_hdl`；
- 不增加第二套 GATT Server；
- 不使用 Extended Advertising；
- 不根据远端 MAC、操作系统或连接时序猜测对端类型；
- 不隐藏同一 Profile 中的另一套 Service；
- 不改变现有 HID Report Map、Report 长度和 handle 布局。

### 3.3 必须先确认的单连接产品策略

统一广播不能解决两个中心设备对同一条连接的竞争。已绑定的 Windows 可能在设备恢复广播后自动回连，并长时间占用唯一 BLE 连接；手机 App 的后台/快速重连也可能反向阻止 PC 进入 HOGP。

因此实施前必须明确并写入产品规格：

- 是否接受严格的 first-come-first-served；
- 是否需要保留物理按键或其他本地操作，用于强制断开当前 BLE 连接；
- 是否需要 PC 自动回连退避窗口或手机配置窗口；
- 用户在 PC 已自动回连时，如何可预期地转入手机配置。

如果上述策略没有明确答案，即使技术上实现了统一广播，产品也可能因“已绑定 PC 始终抢到连接”而无法正常使用私有 App。

## 4. 统一广播设计

### 4.1 Phase 1 广播方案尝试记录

继续使用一个 `ADV_IND` legacy advertising set，Advertising Data 和 Scan Response Data 各自最多 31 字节。Phase 1 已完成两种统一布局的实机尝试，但两种布局均未满足正式 App 的发现要求，因此本节记录实验事实，不将其中任一种描述为最终方案。

#### 4.1.1 基线：原 CONFIG/HOGP 分时广播

原方案通过 KEY5 在两个互斥广播身份间切换：

| 模式 | Advertising Data | Scan Response | 已验证发现方 |
|---|---|---|---|
| CONFIG | Flags + 完整名称 | RDX Manufacturer Data | 私有 App |
| HOGP | Flags + HID UUID + Appearance + 名称 | 空 | Windows |

HOGP 启动路径会调用 `app_ble_rsp_data_set(..., NULL, 0)` 清空 Scan Response，因此 HOGP 能将所有 PC 发现字段放入主 ADV，是因为该时刻完全不携带 RDX Manufacturer Data。这个基线只能证明两个身份分别可用，不能证明一个统一广播能同时兼容两端扫描器。

#### 4.1.2 尝试 A：HID 主 ADV，RDX Scan Response

第一次统一布局为：

```text
Advertising Data（31 bytes）:
Flags + HID UUID 0x1812 + Appearance 0x03C1 + 完整名称

Scan Response（27 bytes）:
RDX Manufacturer Data
```

实机结果：

- nRF Connect 能看到完整 ADV 和 Scan Response；
- Windows 能发现设备并识别 HOGP；
- 正式手机 App 无法发现设备；
- nRF Connect 显示的 Manufacturer Data 为 `0x363037E8392C53D3C44245414E544B00A2060000184B314E56`，证明固件确实发送了 RDX Scan Response。

当时的初步假设是“正式 App 不获取或不解析 Scan Response”，因此决定互换两包内容。该假设在尝试 B 后被证明不完整。

#### 4.1.3 尝试 B：RDX 主 ADV，HID Scan Response

第二次统一布局为：

```text
Advertising Data（30 bytes）:
Flags 3 + RDX Manufacturer AD Structure 27

Scan Response（28 bytes）:
HID UUID 4 + Appearance 4 + 完整名称 20
```

nRF Connect 捕获到的合并 Raw Data 为：

```text
0x02010A1AFF363037E8392C53D3C44245414E544B00A2060000184B314E56
  030312180319C10313094265616E7374616C6B20524B422030303032
```

按 AD Structure 长度拆分后，前 30 字节是主 ADV：

```text
02 01 0A
1A FF 36 30 37 E8 39 2C 53 D3 C4 42 45 41 4E 54 4B
00 A2 06 00 00 18 4B 31 4E 56
```

后 28 字节是 Scan Response：

```text
03 03 12 18
03 19 C1 03
13 09 42 65 61 6E 73 74 61 6C 6B 20 52 4B 42 20 30 30 30 32
```

这份 Raw Data 已证明互换代码生效，不是增量编译或旧固件造成的假象。实机结果仍然是 nRF Connect 和 Windows 可以发现，正式 App 无法发现。

因此可以排除“只要把 RDX Manufacturer Data 放进主 ADV，App 就能恢复发现”这一假设。当前更符合证据的判断是：App 的过滤或解析还依赖原 CONFIG 广播的其他特征，例如名称位于主 ADV、名称紧跟 Flags、Manufacturer Data 位于 Scan Response，或固定字段顺序/偏移/总长度。没有 App 扫描源码或日志时，不能继续把问题简化为 active scan 与 passive scan 的差异。

主 ADV 也无法同时容纳当前完整名称和完整 Manufacturer Data：

```text
Flags 3 + Complete Name 20 + Manufacturer Data 27 = 50 bytes
```

#### 4.1.4 候选尝试 C：保持 RDX 原布局，仅追加 HID UUID

候选 C 已在开发开关下实现，目标是最大限度保持原 CONFIG 广播的字节顺序：

```text
Advertising Data（23 bytes）:
Flags + 完整名称

Scan Response（31 bytes）:
RDX Manufacturer Data 27 + HID UUID 0x1812 4
```

方案 B 与 C 都保留相同的 RDX GATT Service 和 Manufacturer Data 内容，但空口分包位置不同，不能把两者都理解为“RDX 放在主 ADV”：

| 方案 | 主 ADV | Scan Response | 实机结果 |
|---|---|---|---|
| B | Flags + RDX Manufacturer Data | HID UUID + Appearance + Name | PC 可发现，正式 App 不可发现 |
| C | Flags + Name | RDX Manufacturer Data + HID UUID | PC 与正式 App 均可发现并分别连接 |

C 保留的是原 CONFIG 广播布局：主 ADV 仍为 `Flags + Name`，Manufacturer Data 仍是 Scan Response 的第一个 AD Structure，只在其末尾追加 HID UUID。B 则把 Manufacturer Data 移到了主 ADV，并把名称移到了 Scan Response；这正是正式 App 能否发现的关键实机差异。

由于 C 的 Scan Response 31 字节预算已用满，广播 Appearance `0x03C1`（Keyboard）被删除。实机仍能识别为 HID 键盘，说明当前 Windows 环境主要通过以下信息判定：

1. 扫描阶段从 Scan Response 中的 HID Service UUID `0x1812` 发现 HOGP 能力；
2. 连接后从 GATT HID Service `0x1812`、HID Information、Report Map、Input Report 等属性确认键盘能力。

Appearance 是设备类别提示，可能影响扫描列表中的分类、图标或部分主机的预连接过滤，但不是 HOGP Service 和键盘 Report Map 的替代品，也不是本次 Windows 环境中的必要发现条件。

当前名称和 Manufacturer Data 下，预期 Raw Data 分包为：

```text
Advertising Data:
02 01 0A
13 09 42 65 61 6E 73 74 61 6C 6B 20 52 4B 42 20 30 30 30 32

Scan Response:
1A FF 36 30 37 E8 39 2C 53 D3 C4 42 45 41 4E 54 4B
00 A2 06 00 00 18 4B 31 4E 56
03 03 12 18
```

尝试 C 的 Go/No-Go 条件：

1. 正式 App 恢复前台、后台和快速重连发现；
2. Windows 能从 Scan Response 中的 `0x1812` 发现并识别键盘；删除旧配对记录、重开蓝牙后全新扫描和配对仍须成功，不能只使用历史缓存结果；
3. nRF Connect 拆包结果与上述字节顺序一致；
4. 如果 App 仍失败，必须取得 App 扫描过滤/解析代码或运行日志，再决定是否继续调整固件包序，禁止继续盲猜字段位置。

2026-07-17 实机验证结果：候选 C 已通过发现阶段。正式 App 与 PC 均能扫描到设备，PC 也能建立连接。这说明保持原 RDX 广播布局并在 Scan Response 末尾追加 HID UUID 的 31 字节方案可行，Windows 在本次环境中不依赖广播 Appearance。正式发布前仍要按上述无缓存流程覆盖目标 Windows 版本和蓝牙主控。

#### 4.1.5 候选 C 首次连接问题及修复验证

`COM3_2026-07-17_12-18-34.log` 记录了正式 App 扫描成功后的首次连接。连接完成时固件仍沿用旧分时模式，根据当前 advertised mode 直接设置 `owner=HOGP`：

```text
[BLE_MODE] connected req=HOGP adv=HOGP owner=HOGP
[BLE_MODE] RDX CCC write rejected: owner=HOGP hdl=0x0009
[BLE_MODE] RDX write rejected: owner=HOGP hdl=0x0006
...约 5 秒后重复写入仍被拒绝...
att disconn
HCI_EVENT_DISCONNECTION_COMPLETE: 13
```

因此断开不是广播、射频或 GATT Service 缺失导致的。正式 App 已连上并开始访问 RDX 句柄，但旧 owner gate 拒绝了全部 RDX CCC/value 写入；App 重试后以 HCI reason `0x13`（Remote User Terminated Connection）结束连接。

当前开发开关路径采用过渡兼容策略：普通和 Enhanced Connection Complete 均先设置 `owner=NONE`，不再从统一广播身份推断连接方；首次访问 RDX value/CCC 时认领 `CONFIG` 并初始化 RDX runtime，首次访问 HID read/write 时认领 `HOGP` 并初始化 HOGP runtime。已经认领后再访问冲突服务仍按旧 gate 拒绝。该策略还避免了手机 App 连接完成时无条件执行 HOGP 配对请求。

这只是 Phase 1 实机验证期间的受开关过渡实现，目的是让当前 owner 架构能够分别验证正式 App 和 PC。最终模型仍按第 5、6、10 节建立独立的 RDX/HID session，并在 Phase 2B 删除互斥 owner，而不是长期保留“首次访问决定唯一 owner”。

2026-07-17 重新编译、烧录上述修复后，正式 App 已能够成功建立并保持连接，不再出现连接后约 5 秒断开的现象。结合此前结果，候选 C 的 Phase 1 核心路径验证成功：PC 与正式 App 均能从同一个广播入口发现设备，PC 能连接 HOGP，正式 App 能连接 RDX。

因此候选 C 确认为本阶段采用的统一广播布局，不再继续尝试 A/B 的字段互换方案。后台扫描、快速重连、RDX 完整业务功能、HID 键盘输入和双端抢占顺序仍按第 11 节验收矩阵继续回归，不影响候选 C 的广播方案结论。

Phase 1 验证期间曾使用 `TCFG_RDX_HOGP_UNIFIED_ENTRY_ENABLE` 隔离开发路径；Phase 4 已删除该宏及 legacy fallback，候选 C 现为唯一广播构包路径。

### 4.2 统一名称

广播名称和 GAP Device Name 必须保持一致，统一使用 `rdx_ble_server_get_local_name()` 的结果。不要再使用 HOGP 独立名称或按模式切换名称。

建议产品名称继续使用：

```text
Beanstalk RKB [认证尾码]
```

原 `RDX_HOGP_CUSTOM_NAME`、`RDX_HOGP_NAME_SOURCE` 和 `RDX_HOGP_APPEARANCE` 均已在 Phase 4 删除，禁止重新作为独立 HOGP 广播配置引入。

当前 BLE 改名路径只更新 RAM/VM，没有直接调用 `rdx_ble_server_adv_data_changed()`。统一入口必须明确名称更新策略：

- 无连接时改名：写入成功后立即停止并重建统一广播，使广播名称与 GAP Device Name 同步；
- 已连接时改名：更新 RAM/VM，标记 `adv_refresh_pending`，当前连接断开后使用新名称恢复广播；
- 认证尾码变化：与显式 BLE 改名走同一刷新入口，不允许只改 GAP 名称或只改广播名称；
- 已配对主机可能缓存旧名称，量产验收需区分固件未刷新与主机缓存两类现象。

同时必须修复现有名称存储路径，不能只改广播构包：

- `ble_local_name` 当前只有 `BLE_LOCAL_NAME_MAX_LEN` 字节，应为结尾 `\0` 保留空间；
- `rdx_ble_server_set_local_name()` 当前虽然截断了 `len`，却仍使用无界 `sprintf()` 复制原字符串，实际没有应用截断结果；
- 默认名称写 VM 时必须使用“基础名称 + 空格 + 认证尾码”的实际长度，不能只写基础名称长度；
- 所有 RAM/VM/GAP/ADV 路径必须共用同一个有界名称生成函数，并显式补 `\0`。

### 4.3 广播生命周期

统一生命周期只有三条规则：

1. 无连接且未被 DUT、关机、Wi-Fi 传输或格式化等条件抑制时，发送统一广播；
2. 任一 BLE 中心连接成功后，停止广播，不再启动其他身份广播；
3. 当前连接断开且广播未被抑制时，恢复统一广播。

`rdx_ble_server_adv_data_changed()` 只重建统一广播数据，不再根据 HOGP/CONFIG mode 分流。

## 5. 单连接会话模型

### 5.1 删除 connection owner

统一入口后，连接本身不应再具有 `CONFIG` 或 `HOGP` owner。一个连接只保存链路级状态：

```c
typedef struct {
    u16 con_handle;
    u16 mtu_size;
    u8 connected;
    u8 encrypted;
} rdx_ble_link_state_t;
```

在这条链路之上维护两个互不排斥的能力状态：

```c
typedef struct {
    u8 ccc_configured;
    u8 stream_tx_ready;
} rdx_ble_config_session_t;

typedef struct {
    u8 input_ccc_enabled;
    u8 encrypted;
    u8 suspended;
    u8 ready;
} rdx_ble_hid_session_t;
```

状态关系为：

```text
link.connected
├─ config_session.active     = 当前连接首次访问 RDX 私有 value/CCC
├─ config_session.command_rx = active && 当前连接可写 RDX ATT value
├─ config_session.stream_tx  = active && RDX CCC/发送状态满足要求
└─ hid_session.ready     = HID CCC enabled && encrypted && !suspended
```

两项可以都为 0、单独为 1，也可以在同一条连接上同时为 1。不要再引入 `OWNER_BOTH`，否则 owner 互斥思路仍会渗透到发送、按键和清理路径。

在线/离线功能只按 `hid_session.ready` 路由：

| 当前链路状态 | RDX 命令 | 物理按键路径 |
|---|---|---|
| 未连接 | 不可用 | 离线功能 |
| 已连接，HID not ready | 可用 | 离线功能 |
| 已连接，HID ready | 可用 | 在线 HOGP keymap |

同一连接可以保持纯 link、RDX-only、HID-only 或 RDX+HID 状态。RDX capability 在首次访问私有 value/CCC 时幂等激活，不是身份授权；HID capability 因 HID 访问 attach，并由 CCC、加密状态进入 ready。任一 capability 状态变化都不应断开连接，也不应改变另一侧的准入语义。

### 5.2 连接建立

`HCI_SUBEVENT_LE_CONNECTION_COMPLETE` 和 `HCI_SUBEVENT_LE_ENHANCED_CONNECTION_COMPLETE` 统一执行：

1. 先检查 event 长度和 Connection Complete `status`，只有 `status == 0` 才能进入 connected helper；
2. 从 event 解析并保存唯一 `con_handle`；
3. 设置 `ble_conn = TRUE`；
4. 只初始化共享 link session、MTU、自动关机、广播定时器和公共 LED 状态；
5. 不启动 RDX 录音、EMMC、OTA、stream 或其他 App 业务；
6. 不 attach HOGP capability，也不读取/恢复 HID CCC；
7. 不设置 connection owner；
8. 不在连接完成时调用 `sm_api_request_pairing()`；
9. 不重新启动任何广播。

普通与 enhanced connection complete 已收敛到同一个内部 helper，避免两条路径继续漂移。RDX capability 仅在首次访问 RDX 私有 value/CCC 时初始化；HOGP capability 仅在首次 HID read/write 时 attach。

`HCI_SUBEVENT_LE_CONNECTION_UPDATE_COMPLETE` 已改为从 event 解析 update handle，并与当前全局链路句柄比较，不再使用每次 callback 初始为 0 的局部变量。

### 5.3 连接断开

`HCI_EVENT_DISCONNECTION_COMPLETE` 必须使用事件里的原连接句柄完成清理，不能先把本地句柄写成 0 再传给子模块。当前顺序：

1. 从 event packet 解析 `disconnected_handle`；
2. 在清理前快照 `rdx_active`，并记录 handle、status、reason、RDX/HID capability 状态；
3. 调用 `rdx_hogp_on_disconnected(disconnected_handle)`；未 attach 时该调用幂等 no-op，已 attach 时先释放按键并清理 HID runtime；
4. 仅当 `rdx_active` 时停止 RDX 发送、同步、强制断开、流和业务 runtime；纯 HID 链路不执行 RDX disconnected 业务副作用；
5. 执行公共 link cleanup，清空唯一连接句柄和 session 临时状态；不把旧 handle 的 runtime CCC 继承给下一连接，bonded peer 的跨上电订阅由独立持久记录恢复；
6. 若未命中广播抑制条件，恢复统一广播与自动关机策略。

不再根据 previous owner 分支；清理范围由连接实际激活的 capability 决定。

建议把断开顺序收敛为一个内部 helper，伪代码如下：

```c
static void rdx_ble_link_disconnected(u16 disconnected_handle)
{
    bool rdx_active = rdx_ble_session_is_rdx_active(disconnected_handle);
    rdx_hogp_on_disconnected(disconnected_handle);
    if (rdx_active) {
        rdx_ble_server_rdx_disconnected_cleanup_internal();
    }
    rdx_ble_server_link_disconnected_cleanup_internal();
    rdx_ble_link_state_reset(disconnected_handle);
    rdx_ble_unified_adv_restart_if_allowed();
}
```

关键约束是先保留并使用事件中的 `disconnected_handle` 完成 capability-scoped 清理，最后再清空全局连接句柄并恢复广播。

## 6. ATT 访问模型

### 6.1 服务发现

保留当前 `rdx_profile_data[]`，其中：

- GAP 和既有 RDX Service 保持原 handle；
- HID Service 保持 `0x0016–0x0025`；
- HID Report Map、HID Information、Input/Output Report 字节契约保持不变。

同一 GATT Server 下，连接方能够发现完整数据库是正常行为。广播只负责帮助中心发现产品能力，不负责隐藏 Service。

### 6.2 owner gate 删除原则

以下判断最终全部取消，不再替换为远端身份授权 gate：

```c
rdx_ble_connection_owner_get() != RDX_BLE_OWNER_CONFIG
!rdx_ble_connection_owner_is_hogp()
```

替换原则：

| 路径 | 新准入条件 |
|---|---|
| HID 静态 read | 连接有效；按原 offset/length 读取 |
| HID Control Point / Protocol Mode / Output Report write | 连接有效；必要时检查 HID session |
| HID Input Report notify | `rdx_hogp_keyboard_is_ready()` |
| RDX ATT write | 当前连接有效；通过长度、offset、帧格式和命令参数校验 |
| RDX notify/indicate | 对应 RDX CCC 和发送状态满足要求 |
| HOGP keymap 配置 | 当前连接有效，再检查参数、事务和 busy 状态 |
| OTA / bulk send | RDX session 已建立且对应 CCC ready |

特别注意：广播中包含 HID UUID 不等于任意连接都已获得 HID 输入能力，HID report 仍要求 CCC、加密和非 suspend；但连接中存在 RDX Service 时，产品设计就是允许当前中心调用 RDX 命令。

owner 删除后仍必须保留 HID capability 判定。HID 静态属性可被读取不代表进入在线功能；只有 `rdx_hogp_keyboard_is_ready()` 为 1 才允许发送 HID report 并把物理按键从离线路径切到在线 keymap。

### 6.3 RDX 采用连接级访问，不做远端身份鉴权

当前仓库没有可直接复用的 `rdx_protocol_is_authorized()` 一类接口。`connection_owner` 仅由连接前的广播模式赋值，本来也不是远端身份认证；`PROTOCOL_EVENT_CMD_AUTH_SN` 只是设备 Auth SN 查询，也不证明远端持有秘密凭据。

产品决定不改造预编译 `librdxApp.a` 和手机 App 来增加挑战/响应。Phase 2B 删除 owner 后采用以下明确语义：

1. 当前唯一 `con_handle` 对 RDX ATT read/write 具有访问资格；
2. `rdx_app_protocol_handle()` 中的敏感命令不检查远端身份或 session auth 状态；
3. OTA value handle 可按现有协议进入 `rdx_protocol_ota_handle()`，但必须保留长度、状态和固件完整性校验；
4. keymap 修改不检查 CONFIG owner 或远端身份，只保留协议参数、事务一致性、产品功能开关和 busy 检查；
5. RDX notify/indicate 仍按当前连接句柄、CCC 和发送状态发送，避免跨连接或未订阅发送；
6. 断开后清空所有 per-link runtime，下一条连接不会继承 MTU、发送中、事务、HID ready 或按键状态。

Phase 2A 曾将 `TCFG_RDX_SESSION_AUTH_GATE_ENABLE` 保持为 `0`；Phase 2B 已删除该宏、`IDENTIFIED/AUTHORIZED` 状态和授权 API，避免后续维护者误以为量产固件存在远端鉴权。不得通过连接成功、HID 加密、bond、Auth SN 查询或私有协议格式将日志标记为“RDX 已认证”。

该决策意味着通用 GATT 客户端或逆向实现协议的中心也可能执行敏感操作。保留的检查只用于内存安全、状态机一致性和镜像完整性，不构成访问控制。若未来产品重新要求身份鉴权，应作为新的协议/App 联合版本实现，不得仅把现有 gate 宏改为 `1`。

### 6.4 CCC 必须按连接句柄处理

当前 HID CCC 已经使用：

```c
multi_att_get_ccc_config(connection_handle, att_handle);
multi_att_set_ccc_config(connection_handle, att_handle, cfg);
```

统一入口后保持这一做法。RDX 侧仍使用 `att_set_ccc_config()` 的路径应核对是否隐含当前连接；即使当前只允许一条连接，也建议所有新的 session 判断都使用 callback 传入的 `connection_handle`，避免以后出现陈旧连接状态。

CCC 写入还必须严格验证 `offset == 0`、`buffer_size == 2`，且 Input Report CCC 只允许 `0x0000` 和 `0x0001`。只有合法的 `0x0001` 才能设置 `input_ccc_enabled` 并触发延迟配对；非法长度、offset 或配置值必须返回对应 ATT Error。

#### Bond 与 CCC 持久化边界

`multi_att_clear_ccc_config(connection_handle)` 和 `multi_att_get/set_ccc_config()` 管理的是按当前 connection handle 索引的运行期 CCC。Phase 3 实机回归已经证明：本项目的自定义 HOGP profile 在设备重新上电后，JL 栈不会仅凭既有 bond 自动把 Input Report CCC 恢复到该运行期表。因此，“断连时不主动清 CCC”只能避免同一次上电期间过早丢失状态，不能替代跨上电、按 peer 的 CCC 持久化。

实现和验收必须区分：

- 必须清理的当前 connection handle 临时状态；
- 需要由产品固件按 bonded peer identity 持久化和恢复的 HID 订阅状态；
- 绝对不得继承给不同 peer 的 `input_ccc_enabled`、`encrypted`、`ready` 和 held-key runtime。

最佳实践不是在“链路已加密”或“存在 bond”时无条件把 CCC 设为 `0x0001`。加密只证明链路拥有密钥，不证明该 peer 订阅了 HID Input Report；这样处理会把手机或普通 BLE 中心误判为 HID 在线。正式修复必须：

1. 仅在当前链路已加密、合法 CCC `0x0001` 写入成功后，按 SM 解析得到的稳定 peer identity 保存订阅记录；CCC `0x0000` 时删除该 peer 的记录；
2. 持久记录采用 magic、schema/version、有效标志和完整性校验，使用有界 peer slot，不能只保存一个无地址归属的全局布尔值；
3. bonded reconnect 加密成功后解析当前 peer identity，仅在记录匹配时恢复 `multi_att` CCC 和 HOGP notify runtime；不同 peer 绝不继承；
4. 同时覆盖“Encryption Change 早于 HOGP attach”和“HOGP attach 早于 Encryption Change”两种事件顺序，恢复操作必须幂等且限定当前 `con_handle`；
5. bond 删除、密钥丢失、CCC disable、恢复出厂以及持久记录损坏时安全收敛为 CCC disabled、HID not ready；
6. 修复版本之前创建的 bond 没有可靠的历史 CCC 意图，默认迁移策略应要求删除 PC bond 并重新配对一次。不得仅凭旧 bond 自动推断已订阅；如果产品必须无感迁移，需定义额外、可审计的兼容规则并单独做安全评审。

已绑定 Windows 自动重连时可能完全不再写 CCC。固件必须依靠上述 peer-scoped 持久记录重建 ready，不能继续假设主机会重写或 JL 栈会替产品恢复。

## 7. HID 配对与加密策略

### 7.1 推荐策略：按 HID 使用延迟配对

早期 `rdx_hogp_on_connected()` 曾在连接完成后立即调用：

```c
sm_api_request_pairing(con_handle);
```

统一广播后不能继续这样做，否则手机 BLE App 只要连接 RDX，也可能立即收到系统配对请求，破坏私有 App 体验。Phase 2B 已删除该连接即配对路径。

当前实现为：

1. 连接完成时只初始化共享 link；首次 HID read/write 才 attach HID runtime，静态读取不主动请求配对；
2. 对端启用 `HID_INPUT_REPORT_CLIENT_CONFIGURATION_HANDLE` 时记录 `input_ccc_enabled`；
3. 如果链路尚未加密，调用 `sm_api_request_pairing(connection_handle)`；
4. 在 `HCI_EVENT_ENCRYPTION_CHANGE` 成功后设置 `encrypted`；
5. 只有 `input_ccc_enabled && encrypted && !suspended` 时，`rdx_hogp_keyboard_is_ready()` 返回 1；
6. 加密失败、CCC 关闭或断开连接时立即清除 ready，并发送/收敛本地 release 状态。

延迟配对不等于只保护 notify。必须为 HID Protocol Mode、Input Report、Output Report、Control Point 和 CCC 逐项定义链路安全条件，并核对 JL ATT table flags 能否表达这些条件。当前 HID value flags 只表达 read/write/notify/dynamic，没有明确的加密权限，因此不能只在 `rdx_hogp_keyboard_report_send()` 前检查 encrypted，却默认允许未加密远端写 Output Report 或 Control Point。

参考状态转换：

```text
CONNECTED
   │
   ├─ RDX App 写私有特征 ──> RDX session active，不触发 HID 配对
   │
   └─ Host 写 HID Input CCC
          │
          ├─ 已加密 ─────────> HID READY
          │
          └─ 未加密 ─────────> request pairing
                                  │
                                  ├─ encryption success -> HID READY
                                  └─ failure            -> HID NOT READY
```

### 7.2 主机兼容性验证点

Windows、macOS、iOS/iPadOS、Android 对 HOGP 的配对发起时机可能不同。部分主机可能在写 CCC 前主动发起 SM，也可能在发现 HID Service 后先读取 Report Map。实现应允许主机主动配对，并把所有 Encryption Change 都按当前连接句柄更新。

Phase 3 正式选择 `IO_CAPABILITY_NO_INPUT_NO_OUTPUT + Just Works + Bonding + Secure Connections`。原因是当前产品没有显示、键盘输入、数字比较确认等可认证配对交互，继续请求 MITM 既无法达到对应安全级别，也会让日志和产品声明与实际能力不一致。`multi_protocol_main.c` 中 RDX + HOGP 分支因此删除 `SM_AUTHREQ_MITM_PROTECTION`，保留 Bonding、Secure Connections 和最小密钥长度 7；`RDX_HOGP_PAIRING_MODE=0` 继续作为编译期约束。该策略提供链路加密，但不提供对远端身份的 MITM 认证，也不改变“任意 BLE 中心可调用 RDX 命令”的产品决策。

HID ATT 权限按“发现信息公开、运行数据加密”划分：

| 属性 | 未加密访问 | Phase 3 权限 |
|---|---|---|
| HID Service、Characteristic Declaration、Report Reference | 允许发现 | 静态公开 |
| Report Map、HID Information | 允许读取 | 静态公开，保证主机可在配对前完成 HOGP 发现 |
| Protocol Mode | 拒绝读写 | encrypted read + encrypted write |
| Input Report | 拒绝读取和 notify 使用 | encrypted read；notify 仍由 ready 的加密条件约束 |
| Input Report CCC | 拒绝读写 | encrypted read + encrypted write |
| Control Point | 拒绝写入 | encrypted write |
| Output Report | 拒绝读写 | encrypted read + encrypted write |

JL/BTstack ATT 表的 read/write security permission bits 是主安全边界；HOGP write callback 同时保留未加密拒绝和 `0x0F Insufficient Encryption` 作为纵深防御。由于 ATT 层可能在进入动态 callback 前就拒绝未加密 CCC 写入，目标主机应在收到安全错误后发起配对并重试；若某个 JL 栈版本仍将该写入分发到 callback，callback 会请求 `sm_api_request_pairing()` 后拒绝本次明文写入。两条路径都不得恢复“连接即配对”或“静态 HID 读取即配对”。

实机必须记录目标主机实际协商的 Just Works、加密状态和 bond/重连结果，不得把 Secure Connections 或 encrypted 日志描述为已获得 MITM。

如果实机发现某个目标 Windows 版本不会走到 CCC 写入，可采用以下兼容回退，而不是恢复“连接即配对”：

1. 在首次明确的 HID 动态特征访问时触发配对；或
2. 在 HID CCC 写入返回安全错误并触发配对，配对完成后由主机重试。

最终选择以目标 Windows 版本、手机 App 和实际 JL ATT 回调行为的联调结果为准。

日志中若出现失败，必须记录完整事件、阶段和 HCI reason。`0x05` 可表示 Authentication Failure；`0x3E` 通常属于 Connection Failed to be Established/同步超时类原因，不能直接归因为配对失败。

## 8. 物理按键路由

这是统一入口最容易引入产品回归的地方。

当前 `rdx_app_earphone_key_remap()` 使用 `rdx_hogp_keyboard_is_connected()` 决定是否吞掉所有按键。统一入口后，如果它仍等价于“BLE 已连接”，手机 App 一连接，KEY1～KEY5 就会被当作 HID 按键处理，即使手机从未订阅 HID。

必须改成：

```c
if (rdx_hogp_keyboard_is_ready()) {
    // 执行 HID keymap，并消费对应按键事件
} else {
    // 继续走原有本地/离线产品按键路径
}
```

同时删除 KEY5 三击的以下专用路径：

```c
if (num_idx == 4 && index == KEY_ACTION_TRIPLE_CLICK) {
    rdx_ble_mode_request_toggle();
    ...
}
```

删除后 KEY5 三击是恢复到现有离线 key table、改作其他产品功能还是显式忽略，应由产品按键表决定；它不再承担 BLE 身份切换。

HID ready 从 1 变 0 时必须：

- 取消延迟 release timer；
- 发送或至少在本地收敛全零 report；
- 清除 held key 状态；
- 防止旧连接的回调继续影响新连接。

#### Phase 2A 实机回归发现：既有异步删除缓存缺陷（非 Phase 2 引入）

`COM3_2026-07-17_15-27-32.log` 中 App 已经在 `con_handle=0x0050`、`owner=CONFIG` 的正常 RDX 会话中下发 `delfile`，命令也进入了 UXFILE 删除队列，没有出现 owner 或 session 授权拒绝。实际失败链路为：

```text
[DELETE] File not exist (clean record): storage/sd0/C/9c5bb6.raw
Cache not available, cannot remove entry!
[DAT_SAVE] ERROR: Cache not initialized, abort save!
```

该现象会使物理文件已不存在时的 DAT 索引无法移除，App 继续显示旧记录；重新上电触发启动期 DAT 重载/修复后，列表才更新。

根因不是 Phase 2 的 BLE session、统一广播或授权 gate 改动。`rdx_uxfile_recordFile_delete_handle()` 是异步入队 API，非负返回值只表示队列接受了请求；2026-07-15 的本地播放适配提交 `7ef8a9c` 在入队成功后立即调用 `rdx_uxfile_invalidate_dat_cache()`，在删除 worker 执行前就破坏了它仍需使用的 DAT cache。Phase 2A 只是在扩大 RDX 实机回归时发现了这个既有缺陷。

修复原则和实现如下：

1. 删除请求入队后，应用层不再调用 UXFILE DAT cache 的 invalidate/free；
2. UXFILE 删除 worker 独占该 cache 的生命周期，直到完成文件处理、DAT 记录移除和持久化；
3. 应用层只在入队成功后保守地调用 `rdx_playback_on_file_deleted()`，收敛与 UXFILE cache 独立的本地播放选中/导航状态；
4. 录音完成或 SN 复用的内容提交路径仍保留 `rdx_uxfile_invalidate_dat_cache()`，因为该回调发生在内容已更新的边界，不与删除 worker 竞争；
5. `rdx_uxfile.h` 已明确该 API 的“入队成功不等于删除完成”契约，host contract 禁止在删除 case 中提前失效/释放 DAT cache；
6. 当前 `librdxApp.a` 未导出删除完成回调或可靠状态接口，所以不使用固定延时猜测完成时机。后续升级协议库时，应以 worker completion 事件驱动 App ACK 和播放状态的最终确认。

软件验证已完成：新增 `DELETE_QUEUE_PRESERVES_UXFILE_CACHE` 契约，全部 8 组 host software tests 通过，Windows 固件构建通过。随后 App 实机回归确认删除后无需重新上电即可刷新列表，日志不再出现 `Cache not available` 和 `DAT_SAVE ERROR`。

#### Phase 2A 实机回归发现：统一广播断连后未重挂自动关机（非 Phase 2 引入）

`COM3_2026-07-17_15-59-21.log` 确认 App 成功将待机关机时间设为 1 分钟。设置时 BLE 仍连接，电源策略正常拒绝启动 timer；随后断连已完成并恢复广播，但没有再次调用 `sys_auto_shut_down_enable()`，日志持续超过 1 分钟仍未关机。

该问题由 Phase 1 统一广播入口提交 `34c3590` 引入，不是 session 授权或文件缓存修复引入。旧代码把 `rdx_ble_server_auto_shut_down_enable(1)` 放在 CONFIG 广播恢复函数中；统一入口仍保留 `advertised=HOGP`，因此 App 断连后走 HOGP 广播恢复分支，跳过了自动关机重挂。Phase 2A 的统一断连 helper 延续了该分支行为，在本阶段实机回归中发现。

修复后广播恢复只负责广播，自动关机属于断连收尾策略：所有断连入口在清空链路状态、完成录音清理并恢复广播后，统一调用 `rdx_ble_server_disconnected_idle_policy_resume()`。该 helper 先复用 Wi-Fi 传输、DUT、显式关机和 SD 格式化的公共抑制策略，再调用 `rdx_ble_server_auto_shut_down_enable(1)`；HOGP 名称刷新或普通广播重启不会重置待机倒计时。

自动契约增加两个边界：断连必须在链路状态清空后重挂 idle policy；通用 HOGP/CONFIG 广播刷新函数不得操作自动关机 timer。实机验收要求覆盖“BLE 连接中设置 1 分钟 → 断开 App → 约 60 秒后关机”，并确认连接、录音、Wi-Fi 传输、DUT 或格式化期间仍不会误关机。

#### Phase 2A 阻断项：Windows HOGP bonded reconnect 循环断连/重连

App 回归全部通过后，Windows 实机出现持续断开和自动回连。`COM3_2026-07-17_16-21-02.log` 截取到其中一个完整周期：

```text
[BLE_MODE] disconnected ... owner=NONE con=0x0000 conn=0
rdx_ble_server_auto_shut_down_enable: 1
[POWEROFF]sys_auto_shut_down_enable
[BLE_MODE] connected ... owner=NONE con=0x0050 conn=1
[HOGP] encryption_change hdl=0x0050 enabled=1 status=0
[HOGP_ERR] encryption_change ignored: stale handle
[HOGP] conn complete hdl=0x0050
[HOGP] state mode=1 conn=1 con=0x0050 ccc=0 enc=0
```

该回连同时违反三个 Phase 2 生命周期不变式：

1. 自动关机是 link-level policy，任意成功的普通/Enhanced Connection Complete 都必须立即暂停 timer，不能等待 CONFIG/HOGP owner claim；
2. Encryption Change 可能早于首次 HID ATT 访问。统一 session 必须先按 `con_handle` 保存加密状态，HOGP runtime 后续 attach 时必须同步已发生的状态，不能依赖再来一次事件；
3. bonded CCC 不属于通用 per-link runtime。断连必须清除本地 `connected/encrypted/ready/key` 状态，不能把旧 connection handle 的 CCC 直接继承给新连接；Phase 2A 当时仅用当前 `con_handle` 查询 JL runtime CCC，Phase 3 已证明该表不会跨设备重启恢复，正式方案必须增加 peer-scoped 持久化；
4. 统一断连日志必须在清理前记录 `handle/status/reason/rdx_active/hid_ready`，以区分远端主动断开、supervision timeout、安全失败和本机策略断开。

Phase 2A 当时的实现为：连接成功的统一 helper 立即调用 `rdx_ble_server_auto_shut_down_enable(0)`；HCI Encryption Change 先写统一 session，HOGP capability attach 读取该 session，再将当前加密值作为初始状态传入 `rdx_hogp_on_connected()`；删除统一断连中的 `multi_att_clear_ccc_config()`，并在 HOGP attach 时查询当前 handle 的 runtime CCC。该方案解决了加密事件早到和同次上电断连重连问题，但没有解决设备重新上电后的 bonded CCC 恢复，后者以 Phase 3 回归结论和 6.4 的 peer-scoped 方案为准。

该项当时属于 Phase 2A 统一链路生命周期的核心验收门禁。修复后的 Windows reconnect 已能收敛为 `conn=1 ccc=1 enc=1`，PC 连接期间自动关机 timer 保持关闭，断开 PC 后按配置时间重新启动；Phase 2B/3 回归已验证该状态模型。

- 按当前 `con_handle` 保存和清理 RDX 链路、CCC、MTU、发送与事务 runtime；
- 产品已明确接受 RDX“连接即有权访问”，预编译协议库/手机 App 不再增加鉴权依赖；

- 合并普通/增强 Connection Complete 与断开生命周期，增加 status/长度检查并修复 Connection Update handle 比较；
- 实现公共 link/HID runtime 的幂等清理和 RDX capability-scoped 业务清理；不再把 runtime CCC 当成跨连接全局状态，跨上电的 bonded CCC 后续按 peer identity 持久化；

#### Phase 2B 实机回归发现：断连后统一广播未恢复

`COM3_2026-07-17_19-43-00.log` 显示连接 `0x0050` 以 reason `0x13` 正常断开后，固件已完成 link/RDX 清理，但广播刷新只打印 `refresh deferred until disconnect`，之后没有 `rdx_adv_enable = 1`。这不是中心设备扫描缓存，而是断连回调与 JL `app_ble` wrapper 清理之间的时序竞争。

协议 packet handler 收到 `HCI_EVENT_DISCONNECTION_COMPLETE` 时，RDX 自己的 `ble_conn/session` 已可清零，但 `app_ble_get_hdl_con_handle()` 在回调返回前仍可能保留旧连接句柄。若此时同步调用 `rdx_ble_server_adv_data_changed()`，它会把刷新标记为 pending；旧实现没有消费该 pending 的后续事件，因此广播永久停止。Phase 2B 删除专用 HOGP 广播重启路径后，所有断连都收敛到统一刷新入口，使这个潜在时序缺陷稳定暴露。

修复采用模块私有的一次性延迟任务：断连回调只完成状态清理并安排广播恢复，回调返回后再检查 wrapper 句柄、重建数据并启动统一广播；若 wrapper 尚未释放则以 10 ms 间隔有限重试。新连接建立或 BLE server 退出时必须取消待处理任务，回调自身也要复查当前 link，防止旧断连任务关闭或重建新连接对应的广播状态。不得通过删除 `app_ble_get_hdl_con_handle()` 保护来修复，否则会重新引入连接建立/断连交错时的广播竞争。

host contract 已增加断言，冻结“断连回调不直接启动广播、异步等待 wrapper 清理、有限重试、新连接取消任务”四个边界。实机复测应在断连日志后看到 `[BLE_SESSION] advertising restart after disconnect` 和 `rdx_adv_enable = 1 ret = 0`，并确认 App 主动断开、PC 主动断开及远端 reason `0x13` 后均可重新扫描和连接。

- 删除 mode controller、KEY5 三击切换和 DUT mode request；
- 删除 HID 与 RDX 全部 ATT/send/keymap owner gate，不增加远端身份授权替代条件；
- 用当前链路状态、CCC、HID ready、协议参数和业务状态替代 owner；
- 保留在线/离线按键分流：HID ready 时走在线 HOGP keymap，HID not ready 时走原离线功能；RDX ATT 访问本身不得把链路标记为 HID ready；
- Phase 2B 当时启用 `TCFG_RDX_HOGP_UNIFIED_ENTRY_ENABLE=1` 并删除 auth gate 宏及未使用的授权状态/API；该统一入口宏最终在 Phase 4 删除；
- 验证正式 App 与任意非 App BLE 中心均可调用 RDX 命令，且非法包、错误事务、busy 状态和 OTA 完整性检查仍然有效；
- 覆盖 PC 和手机的两个方向连接，确认不再存在“因统一广播被赋成某个固定 owner，导致另一侧必然被拒绝”的中间状态；
- Phase 2B 已完成 CCC 驱动的延迟配对与 ready 状态收敛；Phase 3 单独处理全局 SM auth flags 和动态属性加密权限，不再混入连接生命周期重构。

#### Phase 3 实机回归发现：设备重启后 bonded HID CCC 未恢复（非 Phase 3 引入）

`COM3_2026-07-17_20-25-43.log` 记录了 Windows 已保留配对记录、设备重新上电后立即自动回连的路径：

```text
[BLE_SESSION] connected hdl=0x0050 enhanced=0 rdx=0 hid_ready=0
sm_ltk: ...
[HOGP] conn complete hdl=0x0050 restored_ccc=0x0000
[HOGP] state conn=1 con=0x0050 ccc=0 enc=1 suspend=0 ready=0
```

日志证明 link 已连接、LTK 已用于回连、加密状态已恢复且 HOGP capability 已 attach；缺失的是 Input Report CCC。Windows 对既有 bond 没有再次写 CCC，`multi_att_get_ccc_config()` 在本次上电只返回 `0x0000`，所以 `rdx_hogp_keyboard_is_ready()` 正确保持 0，物理按键按设计进入离线功能。

该缺陷不是 Phase 3 的 Just Works auth flags 或 ATT 加密权限引入。它源于更早的 Phase 2A/2B 实现只保留了 per-handle runtime CCC，并错误假设 JL 栈会为自定义 HOGP profile 跨上电恢复 bonded CCC；Phase 3 的“设备重启 + Windows 自动回连”回归首次提供了反证。在线/离线判定本身没有错，禁止通过删除 CCC 条件、仅看 `encrypted`、仅看 bond 或连接成功来绕过。

修复必须采用 6.4 定义的 peer-scoped CCC 持久化方案：加密链路上的合法 CCC 写入按稳定 identity 保存；回连加密后仅为匹配 peer 恢复；覆盖 attach/encryption 两种时序；CCC disable、bond 失效、恢复出厂和记录损坏均收敛为 disabled。host contract 需要冻结“按 peer 保存、不同 peer 不继承、无记录不猜测、两种时序均可恢复”的边界。修复前已有 bond 默认要求在刷入修复版本后删除并重新配对一次，以建立可信订阅记录。

解决方案已按上述边界实现：

- 新增独立 `rdx_hogp_subscription_store.c/.h`，使用 VM ID `165/166` 保存 A/B 双副本；每条记录包含 `HCS1` magic、schema、递增 revision、4 个有界 peer slot 和 CRC32，写入后立即 readback、逐字节比较并重新校验，掉电或单副本损坏时保留上一有效 revision；
- peer key 只来自 JL 正式接口 `get_sm_peer_address()` 返回的 SM 真实地址，不使用 connection handle、临时 RPA、bond 全局布尔值或 App/RDX owner；全零、全 `0xFF`、重复 peer 和损坏记录均拒绝；
- 加密链路上的合法 CCC `0x0001/0x0000` 分别保存/删除当前 peer 的订阅意图。identity 暂不可用时保留当前连接内的 pending update；VM 硬失败时 enable 回滚并返回 `ATT 0x0E Unlikely Error`，disable 保持本地关闭并返回错误，防止静默形成错误状态；
- `rdx_hogp_on_connected()` 与 `rdx_hogp_on_encryption_change()` 都执行幂等 restore，覆盖“先加密后 attach”和“先 attach 后加密”；Encryption Change 若发现当前 peer 已有持久订阅，可直接 attach HOGP capability，避免依赖 Windows 重写 CCC 或重新访问 GATT；
- 恢复只对当前加密 `con_handle` 且 peer 记录匹配时调用 `multi_att_set_ccc_config(..., 0x0001)`，随后仍由 `CCC && encrypted && !suspended` 决定 ready；不同 peer、无记录 peer 和普通 App 链路保持 HID not ready；
- 所有删除 BLE bond 的 RDX 恢复出厂/解绑路径统一调用 `rdx_vm_ble_pairing_state_reset()`，同时清空持久订阅；断连即使尚未 attach HOGP 也会清理该 handle 缓存的 peer identity；
- host contract 已新增 A/B VM、schema/CRC/readback、显式 CCC 意图、稳定 peer identity、双事件时序、不同 peer 隔离和 bond reset 边界；8 组 host software tests 与 Windows 固件完整构建通过。

修复已完成软件验证，但 Phase 3 仍需实机验收。由于当前 PC bond 建立于修复之前，刷入新版本后必须先在 Windows 删除设备并重新配对一次，看到 `subscription persisted enabled=1` 后再重启设备。重启自动回连应出现 `subscription restored`、`restored_ccc=0x0001` 和 `ready=1`；随后验证持续输入、Caps Lock Output Report、20 次跨上电回连，以及关闭 PC 蓝牙后由手机 App 连接时不会继承 PC 的 HID 订阅。

2026-07-17 Windows 实机复测已通过：删除旧设备并重新配对后成功建立 peer-scoped 订阅记录；设备重新上电时 Windows 立即自动回连，HID CCC、加密和 ready 均正确恢复，物理按键进入在线功能。本问题的 PC 跨上电恢复路径至此完成修复验收。手机 BLE App 按不同 peer 隔离，未订阅 HID CCC 时保持 `ready=0` 并执行离线功能；目标 App 全功能回归也已通过。Caps Lock、配对取消/失败和更多 OS/蓝牙主控组合继续列入发布级矩阵，不影响本方案代码架构收尾。

## 11. 验收标准

### 11.1 广播

- 上电不按 KEY5 即可同时被 Windows 键盘发现流程和私有 App 扫描发现；
- 候选 C 的主 ADV 与原 CONFIG 的 Flags + Name 字段顺序一致；
- Scan Response 先完整保留原 RDX Manufacturer Data，再追加 HID UUID `0x1812`；
- 候选 C 不广播 Appearance；必须删除 Windows 旧 bond、重开蓝牙并完成全新扫描/配对，确认不是历史设备类别缓存造成的假阳性；
- ADV 和 Scan Response 各自不超过 31 字节；
- 设备只显示一个统一名称；
- 任一连接建立后设备不再可被第二个中心连接；
- 断开后恢复同一种统一广播。

### 11.2 手机 App 路径

- App 前台、后台和快速重连场景均能按原 CONFIG 广播兼容路径发现设备；
- 手机连接后不因固件主动请求而立即弹出系统配对；
- RDX CCC、命令、状态同步、录音、文件传输和 OTA 行为不回归；
- 手机未启用 HID 时，物理按键继续走产品原有路径，不被 HOGP 吞掉；
- App 仅访问 RDX、未启用 HID CCC 时必须保持 `HID not ready`，物理按键继续执行离线功能；
- 当前唯一 BLE 连接不需要 RDX 身份授权即可执行名称修改、录音控制、文件操作、OTA、格式化、恢复出厂、绑定/解绑和 HOGP keymap 修改；
- 正式 App 与非 App BLE 中心访问相同 RDX 句柄时，不因 owner、bond、HID 加密或 session auth 状态得到不同命令权限；
- 非法长度、offset、CRC、事务版本、busy 状态及 OTA 镜像完整性失败时仍必须安全拒绝，不产生越界或部分提交；
- 上述访问语义和失败边界必须包含设备黑盒证据，不能只由 PowerShell 源码正则断言代替。

### 11.3 PC HID 路径

- Windows 能发现 HID Service、读取 Report Map 并完成配对；
- HID Input CCC 打开且链路加密成功后 ready；
- HID ready 后物理按键进入在线 HOGP keymap；CCC 关闭、suspend、加密失效或断开后退出在线状态并恢复离线功能；
- KEY1～KEY5 按当前 keymap 发送 8-byte Input Report；
- CCC 未打开、未加密、suspend 或断开时禁止发送；
- HID CCC 对非法 offset、非 2 字节长度以及 `0x0000/0x0001` 以外值返回正确 ATT Error；
- Protocol Mode、Input/Output Report、Control Point 和 CCC 的加密权限与正式 HOGP 安全策略一致；
- 断开、配对失败或取消订阅时按键状态收敛为 release；
- 重连后不会把 CCC、加密和 ready runtime 错误继承给不同 peer；
- 已绑定 Windows 自动重连时，分别覆盖“主机重写 CCC”和“主机不重写 CCC”的实际路径，键盘都能依据栈/bond 语义恢复 ready。

### 11.4 主机实机矩阵

自动化 PowerShell 文本检查不能证明配对时序、主机缓存和空口行为。Phase 3/4 至少记录以下矩阵的设备型号、操作系统版本、蓝牙主控/驱动、首次配对和重连日志：

| 主机类别 | 最低覆盖 | 验证内容 |
|---|---|---|
| Windows | Win10、Win11；至少 2～3 种不同蓝牙主控 | 发现、首次配对、删除 bond 后重配、自动回连、输入、Caps Lock Output Report、断连 reason |
| Apple（若为产品目标） | macOS；需要移动端支持时增加 iOS/iPadOS | 发现、配对方法、输入、重连、系统软键盘/物理键盘行为 |
| Android（若为产品目标） | 至少 2 个厂商版本 | HOGP 配对、输入、重连，以及普通 RDX App 连接不主动弹配对 |
| 私有 App | 正式发布版本 | 按原 CONFIG 广播兼容路径发现设备、RDX 命令、OTA、手机连接时按键不被吞 |
| 非 App BLE 中心 | 通用 GATT 工具或独立测试端 | 连接后可调用代表性 RDX 读写及敏感命令；非法包、错误事务和 busy 状态仍被安全拒绝 |

每个主机至少覆盖：主机先发起配对、固件因明确 HID 访问请求配对、已有 bond 重连、配对取消/失败、连接中关闭 CCC、异常断开六类路径。失败时记录 ATT 操作顺序、SM event、Encryption Change 和 HCI reason，不能只记录“连接失败”。

### 11.5 单连接边界

- PC 已连接时手机不能再建立第二条 BLE 连接；
- 手机已连接时 PC 不能再建立第二条 BLE 连接；
- 已绑定 PC 自动回连抢占唯一连接时，用户可通过 Phase 0 确定的产品操作转入手机配置；
- 手机 App 后台/快速重连抢占唯一连接时，用户仍能按产品策略恢复 PC HOGP；
- 对 PC 先连、手机先连、PC 自动回连与手机后台回连同时竞争四类顺序进行实机验收；
- `config_le_hci_connection_num` 和 `config_le_gatt_server_num` 均保持 1；
- 代码中不存在第二个 RDX/HOGP `app_ble_hdl`。

| 手机通用 GATT 浏览器误开 HID CCC | 手机出现配对提示 | 这是明确 HID 使用行为，可接受；普通私有 App 不访问 HID CCC |
| 手机连接导致按键被吞 | 产品功能失效 | 按键只看 `rdx_hogp_keyboard_is_ready()`，禁止只看 BLE connected |
| 后续维护者重新引入 session auth 脚手架并把它当成量产安全边界 | 错误宣称 RDX 已认证，或误开 gate 导致正式 App 命令全部失效 | Phase 2B 已删除 auth gate 宏、授权状态/API 和相关测试；未来若重新要求鉴权，必须联合升级协议库与 App |
| 动态名称增长导致主 ADV 超长 | 广播启动失败或字段截断 | 候选 C 下完整名称最多 26 字节，构包逐项检查并在超限时使用 shortened name |
| BLE 名称 setter 仍使用无界 `sprintf()` 或 VM 写错长度 | 内存越界、认证尾码丢失或 GAP/广播名称不一致 | Phase 1 先收敛为单一有界名称 API，补 NUL 并按实际长度写 VM |
| 改名后未刷新统一广播 | 广播名称与 GAP Device Name 不一致 | 无连接立即重建广播；已连接设置 pending 并在断连后刷新 |
| 断连清理仍按旧 owner 分支 | timer、按键或流状态残留 | 公共 link/HID runtime 幂等清理；RDX 业务按 `rdx_active` 清理，并增加断连回归测试 |
| 历史 bond 与 Service Changed 缓存 | PC 使用旧 GATT 缓存 | 开发阶段删除主机 bond；若量产 Profile handle 不变则不主动改变数据库版本 |

开发期回退开关 `TCFG_RDX_HOGP_UNIFIED_ENTRY_ENABLE`、旧分时实现和兼容测试已在 Phase 4 删除。后续若需改变产品模型，应以新的显式方案重新评审，不得恢复双状态机或通过隐藏宏长期维护两套行为。

## 13. 最终落地结果

Phase 0 前置条件已经确认，产品已接受 RDX“连接即有权访问”，并已采用“单 BLE 连接、私有广播与 HID 广播统一入口”作为正式产品模型：

- 广播统一，不再让用户理解或操作 BLE 身份；
- Profile 统一，沿用当前一个 Server、一套 handle；
- 连接唯一，符合当前 RAM、栈配置和产品需求；
- 能力按实际协议行为激活，而不是按广播身份互斥；
- RDX 路径以当前连接和协议/业务校验为准，不做远端身份鉴权；HID 路径以 CCC + Encryption 为准；
- KEY5 不再承担模式切换。

最终实现没有 mode controller、owner、专用 HOGP 广播或统一入口兼容宏；配对、按键路由、RDX 参数/事务边界、断连清理和 bonded CCC 恢复均已收敛到 per-link/per-peer 状态。至此本方案完成，后续维护以本模型和 host contracts 为基线。
