# AC701N PC 模式与 SD NAND 文件访问

## 1. 文档目的

本文说明 T2620 如何通过 USB 连接 Windows PC，并把设备上的 SD NAND 暴露为可移动磁盘。

本文只讨论当前仓库已经具备基础驱动的方案：

- 芯片：AC701N / BR28
- USB 角色：Device（SDK 中也称 Slave）
- USB Class：Mass Storage Class（MSC/MSD）
- 存储设备：SD0，即通过 SD 接口连接的 SD NAND
- PC 访问方式：PC 直接读写 SD NAND 的逻辑扇区

当前仓库没有完整的 SPI NAND 设备注册、FTL 和板级接入，不能仅通过打开
`TCFG_NANDFLASH_DEV_ENABLE` 切换到 SPI NAND。因此本文不把 SPI NAND 列为可用方案。

## 2. 结论

使用 USB MSC 暴露 `sd0` 的技术路线是正确的，但必须同时解决两个不同层面的问题：

1. USB 枚举：让 PC 识别 AC701N 为 USB Mass Storage 设备。
2. 存储所有权：确保 PC 访问期间，设备侧 VFS、录音、播放和文件传输不再访问同一 SD NAND。

USB MSC 只是块设备传输协议，并不理解 FAT32、目录或文件。Windows 通过 MSC 读取扇区，
再由 Windows 自己解析文件系统。固件退出 PC 模式后若还要读取这些文件，磁盘格式必须是
固件 `fat` 文件系统驱动能够挂载的格式。对本项目应把 FAT32 作为产品格式约束。

## 3. 当前产品行为

T2620 以独立 DIP 电源开关表达用户是否允许设备进入完整工作状态。

| DIP 状态/操作 | USB 对端 | 当前行为 |
|---|---|---|
| OFF | 普通充电器 | 只充电，不启动 PC 模式 |
| OFF | PC 数据口 | 只充电，不枚举 MSC |
| ON | 未连接 USB | 正常进入蓝牙模式 |
| ON | 普通充电器 | 保持产品定义的开机充电行为，不进入 PC 模式 |
| ON | PC 数据口 | 切换到 PC 模式并暴露 SD NAND |
| PC 模式中 DIP 切到 OFF | PC 数据口仍连接 | 先安全停止 MSC，再进入关机/充电状态 |
| PC 模式中拔出 USB，DIP 仍为 ON | USB 断开 | 退出 PC、恢复设备侧文件系统并回到蓝牙模式，不关机 |

当前代码在 OTG 上线事件和 `pc_mode_try_enter()` 两处检查 DIP，保证 DIP OFF 时只充电。
DIP OFF 充电期间再切到 ON 时，DIP 处理路径会主动检查已有 USB Slave 状态并请求 PC 模式。
USB 拔出时，RDX 充电事件仅在 DIP OFF 时执行关机；DIP ON 会保持开机，由应用模式切换恢复 BT。

## 4. 四个相关宏的职责

### 4.1 `TCFG_APP_PC_EN`

PC 应用模式总开关：

```c
#define TCFG_APP_PC_EN 1
```

它会推导出：

```text
TCFG_APP_PC_EN
  -> TCFG_PC_ENABLE
  -> TCFG_OTG_MODE_SLAVE = OTG_SLAVE_MODE
  -> TCFG_USB_SLAVE_ENABLE = 1
```

只有打开该宏，`pc.c`、USB Device Stack 和 OTG Slave 检测才会形成完整入口。

### 4.2 `TCFG_USB_SLAVE_MSD_ENABLE`

Mass Storage 子功能开关：

```c
#define TCFG_USB_SLAVE_MSD_ENABLE 1
```

打开后，`task_pc.c:usb_start()` 会启用 `MASSSTORAGE_CLASS`，并注册：

```c
msd_register_disk("sd0", NULL);
```

该宏决定 PC 是否出现磁盘。它不决定设备是否因插 USB 而开机。

### 4.3 `TCFG_CHARGE_POWERON_ENABLE`

该宏控制开机充电策略，不是 USB MSC 的依赖：

```c
#define TCFG_CHARGE_POWERON_ENABLE 0
```

对“DIP OFF 时插 USB 只充电”的目标，建议保持为 `0`。将它改为 `1` 会扩大开机充电行为，
普通充电器也可能使系统保持完整运行，不能用它区分 PC 数据口和普通充电器。

准确关系如下：

| 目标 | 必要配置/逻辑 |
|---|---|
| 启用 PC 应用模式 | `TCFG_APP_PC_EN = 1` |
| PC 显示 SD NAND 磁盘 | `TCFG_USB_SLAVE_MSD_ENABLE = 1` |
| 关机插线后允许完整开机充电 | `TCFG_CHARGE_POWERON_ENABLE = 1`，但不是 PC 模式必需 |
| DIP OFF 时禁止进入 PC | 入口处检查 `get_power_on_status()`，宏本身无法保证 |

### 4.4 `TCFG_SD_ALWAY_ONLINE_ENABLE`

本项目的 SD0 是焊接在板上的 SD NAND，不是可插拔 SD 卡：

```c
#define TCFG_SD_ALWAY_ONLINE_ENABLE 1
```

该宏让设备管理器启动时强制登记 `sd0` 并停止 SD 插拔检测定时器。它不代表 VFS 永远保持挂载；
进入 PC 模式时仍会卸载 FAT，只是介质不会因 MSD 关闭产生的延迟 `DEVICE_EVENT_OUT` 被误判为拔卡。

## 5. 当前项目配置事实

JL 工具生成的 `sdk_config.h` 默认值与项目覆盖后的有效值如下：

| 配置 | 工具生成值 | 当前有效值 |
|---|---:|---:|
| `TCFG_CHARGE_POWERON_ENABLE` | 0 | 0 |
| `TCFG_APP_PC_EN` | 0 | 1 |
| `TCFG_USB_SLAVE_MSD_ENABLE` | 1 | 1 |
| `TCFG_USB_SLAVE_HID_ENABLE` | 1 | 0 |
| `TCFG_USB_SLAVE_AUDIO_SPK_ENABLE` | 1 | 0 |
| `TCFG_USB_SLAVE_AUDIO_MIC_ENABLE` | 1 | 0 |
| `TCFG_SD_ALWAY_ONLINE_ENABLE` | SDK 默认 | 1 |
| `TCFG_BT_BACKGROUND_ENABLE` | 0 | 0 |

SD0 本身保持 `TCFG_SD0_ENABLE = 1`、1-bit 数据模式和 24 MHz 时钟。

SD0 引脚为：

| 信号 | GPIO |
|---|---|
| DATA0 | PC03 |
| CLK | PC04 |
| CMD | PC05 |

当前 SD0 是 1-bit、24 MHz，不是 1 MHz。实际吞吐还受 SD NAND、驱动、缓存大小和
USB 调度影响，必须通过硬件测试测量，不能只由总线时钟推算。

当前调试 UART TX 是 `PA3`，不是 USB DP；`TCFG_CFG_TOOL_ENABLE` 也为 `0`。因此当前有效
配置没有让调试 UART 占用 DP/DM。以后修改配置时仍应检查所有已启用外设，不能仅根据宏名
存在就判断引脚冲突。

## 6. 不要直接修改生成配置

`SDK/apps/earphone/board/br28/sdk_config.h` 和 `sdk_config.c` 由 JL 配置工具生成。
项目级开关应放在：

```text
SDK/apps/earphone/include/t2620_project_config.h
```

`app_config.h` 会在包含 `sdk_config.h` 后立即包含该覆盖文件。建议配置为：

```c
/* Enable PC mode without changing the charge-only power policy. */
#undef TCFG_APP_PC_EN
#define TCFG_APP_PC_EN                            1

#undef TCFG_CHARGE_POWERON_ENABLE
#define TCFG_CHARGE_POWERON_ENABLE                0

#undef TCFG_USB_SLAVE_MSD_ENABLE
#define TCFG_USB_SLAVE_MSD_ENABLE                 1

/* SD0 is soldered SD NAND, not removable media. */
#undef TCFG_SD_ALWAY_ONLINE_ENABLE
#define TCFG_SD_ALWAY_ONLINE_ENABLE                1
```

当前产品只要求 MSC 文件访问，因此关闭与目标无关的 UAC 和 USB HID，减少端点、音频和内存变量：

```c
#undef TCFG_USB_SLAVE_AUDIO_SPK_ENABLE
#define TCFG_USB_SLAVE_AUDIO_SPK_ENABLE           0

#undef TCFG_USB_SLAVE_AUDIO_MIC_ENABLE
#define TCFG_USB_SLAVE_AUDIO_MIC_ENABLE           0

#undef TCFG_USB_SLAVE_HID_ENABLE
#define TCFG_USB_SLAVE_HID_ENABLE                 0
```

这些是当前产品的固定覆盖值。除非产品需求变化，否则不恢复这些复合设备功能。

## 7. PC 模式进入路径

### 7.1 已开机时插入 PC

```text
PC 数据线接入
  -> OTG 检测到 SLAVE_MODE
  -> usb_stack 发布 DEVICE_EVENT_IN
  -> pc_device_event_handler() 检查 DIP ON 并返回进入请求
  -> app_common_device_event_handler() 请求 APP_MODE_PC
  -> pc_mode_try_enter() 再次检查 DIP、OTG 和 RDX 存储忙状态
  -> pc_mode_init()
  -> dev_manager_takeover("sd0") 卸载 FAT 并阻止后台重新挂载
  -> pc_task_start()
  -> USBSTACK_START
  -> usb_start()
  -> usb_device_mode() + msd_register_disk("sd0")
```

只有取得 SD0 的 Host 所有权后才允许启动 USB Stack；接管或卸载失败会终止 PC 模式进入。

### 7.2 充电在线启动

启动时若 `get_charge_online_flag()` 为真，`app_task_init()` 会优先选择：

```text
APP_MODE_IDLE + IDLE_MODE_CHARGE
```

`TCFG_CHARGE_POWERON_ENABLE == 0` 保持 DIP OFF 的关机充电策略，但 PC 数据口在 DIP ON 时需要
让系统保持运行。充电事件会等待 OTG 角色稳定，只有检测到 `SLAVE_MODE` 且 DIP ON 时才临时允许
PC 开机路径。PC 模式入口还有两层保护：

1. 收到 OTG Slave 上线事件时，DIP OFF 不发送进入 PC 的消息。
2. `pc_mode_try_enter()` 再次检查 DIP 状态，防止其他路径绕过事件门控。

项目已有统一状态函数：

```c
u8 get_power_on_status(void);
```

在 T2620 上，PB1 为低表示 DIP ON，函数返回 `1`。

对于 DIP OFF 充电时 USB 已连接、之后再切到 ON 的边界，DIP 处理路径会主动检查
`usb_otg_online(0)`；若已经是 `SLAVE_MODE` 或等待确认状态，则主动请求进入 `APP_MODE_PC`。

## 8. 蓝牙与 PC 模式的关系

应用层同一时刻只运行一个前台模式。当前：

```c
#define TCFG_BT_BACKGROUND_ENABLE 0
```

因此从 BT 切到 PC 时会走非后台退出流程，停止扫描/连接并退出 BT Stack；RDX BLE 也会退出。
这是当前项目配置的行为，不是 AC701N 平台永远不能同时保留蓝牙和 USB 的理论限制。

本文按当前配置设计：PC 模式期间不提供 RDX BLE 控制。若以后要求 BLE 与 MSC 共存，需要单独
设计蓝牙后台、资源占用和模式事件处理，不能只修改一个宏。

## 9. MSC 与文件系统的正确边界

### 9.1 MSC 暴露的是裸块设备

PC 访问路径为：

```text
Windows 文件系统
  -> SCSI READ(10) / WRITE(10)
  -> USB Bulk-Only Transport
  -> msd.c
  -> dev_bulk_read() / dev_bulk_write()
  -> sd0 块设备
  -> SD NAND
```

MSC 不解析 FAT，也不通过设备侧 `fopen()` 访问文件。因此：

- PC 能否枚举出磁盘，不要求设备侧先成功挂载 FAT32。
- Windows 可以把裸磁盘识别为未格式化介质并提示格式化。
- 固件退出 PC 模式后需要读取文件时，磁盘必须采用固件支持的格式。
- 对本产品应规定只使用 FAT32，避免 Windows 格式化为固件不能挂载的 exFAT/NTFS。

当前 `msd.c` 在启用设备管理器时，还要求 `sd0` 已存在于设备管理器列表，之后才会
`dev_open("sd0")`。这里检查的是设备登记/在线状态，不等于 MSC 需要 VFS 保持挂载。

### 9.2 为什么日志是 1 个文件，Windows 却显示 3 个项目

RDX 日志里的文件总数是业务索引口径，只统计 DAT 索引中的录音文件，不等于 FAT 根目录项总数。
在 `COM3_2026-07-21_15-04-30.log` 中，业务扫描结果只有：

```text
2 | b2e01a.raw | 766400 | 0
```

行首的 `2` 是录音序号 SN，不是文件数量；该表只有这一行，因此业务录音数是 1。

同一份日志还显示固件访问了 `/UXFILE.DAT`，并显示 `/PENDING.DAT` 不存在。磁盘上可能出现：

| 项目 | 来源 | 是否计入 RDX 录音数 |
|---|---|---|
| `B2E01A.RAW` | 用户录音 | 是 |
| `UXFILE.DAT` | RDX 文件索引/元数据 | 否 |
| `PENDING.DAT` | 录音异常恢复的临时状态，仅需要时存在 | 否 |
| `System Volume Information` 等 | Windows 自动创建的卷管理项目 | 否 |

因此“日志 1 个业务文件、资源管理器 3 个项目”本身不表示重复文件或 MSC 读错目录。要精确核对
第三个项目，应在对应盘符执行 `dir /a X:\`，因为资源管理器默认可能隐藏系统项。

### 9.3 不能同时由 PC 和设备侧挂载

FAT 文件系统会缓存 FAT 表、目录项和空闲簇信息。若 Windows 与设备侧 VFS 同时读写同一卷，
即使底层驱动支持并发，也会造成文件丢失、交叉链或整卷损坏。

所以 PC 模式必须拥有排他的存储所有权：

```text
DEVICE_OWNED_MOUNTED
  -> QUIESCING
  -> HOST_OWNED_MSD
  -> RESTORING
  -> DEVICE_OWNED_MOUNTED
```

## 10. 当前存储所有权交接实现

### 10.1 进入 PC 前

当前实现按以下顺序接管：

1. `rdx_pc_storage_is_busy()` 检查录音、播放、文件传输、DAT 扫描/加载、格式化、OTA 和 Wi-Fi 传输。
2. 任一业务仍忙时拒绝切换，不强行停止正在进行的文件操作。
3. `dev_manager_takeover("sd0")` 设置 `mount_blocked` 并卸载 FAT。
4. 卸载成功后将存储状态切换为 `PC_STORAGE_HOST_OWNED`。
5. 只有 Host 已取得所有权才允许 `USBSTACK_START` 和 `msd_register_disk("sd0")`。

接管内部使用 unmount，不删除设备条目。MSD 在线判断依赖 `sd0` 仍位于设备管理器列表；
删除条目会让 `check_disk_status()` 把介质视为离线。

当前忙检查是进入时的快照，并不是完整的跨任务锁。`mount_blocked` 能阻止设备管理器后台重新挂载，
而模式切换会退出 RDX/BT；后续若允许 RDX 与 PC 模式并存，必须再引入覆盖所有文件入口的显式
所有权状态或锁，不能只依赖一次忙检查。

### 10.2 PC 模式期间必须保证

- `mount_blocked` 禁止设备管理器自动 mount `sd0`。
- `TCFG_SD_ALWAY_ONLINE_ENABLE = 1` 保持焊接式 SD NAND 的介质在线状态稳定。
- 当前 `TCFG_BT_BACKGROUND_ENABLE = 0`，RDX 随 BT 前台模式退出，不在 PC 模式继续接受文件命令。
- USB Stack 启动前存储状态必须已经是 `PC_STORAGE_HOST_OWNED`。

`storage_state` 和 `mount_blocked` 是当前实现的所有权边界，不只依赖
`app_in_mode(APP_MODE_PC)`。

### 10.3 退出 PC 后必须完成

1. 停止 USB MSC 数据通路。
2. 注销 LUN 并关闭 MSD 持有的块设备句柄。
3. 确认 PC 已安全弹出，或确认 USB 已断开。
4. 重新 `mount` SD0。
5. 检查 mount 结果；失败时上报错误，不要直接擦盘。
6. mount 成功后，才恢复录音、播放和文件传输。

当前 `pc_task_stop()` 同步请求 `USBSTACK_STOP`。只有 USB Task 已执行完
`usb_stop()`、注销 LUN 并关闭 MSD 块设备句柄后，才调用 `dev_manager_restore("sd0")`。
STOP 投递失败或等待超时时保持 `HOST_OWNED` 和 `mount_blocked`，禁止设备侧重新挂载；
restore 失败则进入 `RESTORE_FAILED`，SD0 继续保持 blocked。正常日志依次出现：

```text
App Stop - PC
[PC-STORAGE] USB resources released
[PC-STORAGE] HOST_OWNED -> DEVICE_OWNED, sd0 remount ok
```

SD0 为焊接式 SD NAND 并启用常在线后，不应再出现紧随其后的 SD0 `DEVICE_EVENT_OUT` 和
`__dev_manager_del, sd0 del ok`。

Windows 的“弹出设备”可能只发送 SCSI START STOP UNIT，而 USB 线仍保持连接。当前产品契约是：
安全弹出只让 Host 停止磁盘读写，设备继续保持 PC 模式和 Host 所有权；用户随后拔线，固件才恢复
设备侧 VFS 并返回蓝牙模式。“弹出但不拔线后立即恢复设备侧文件访问”不属于当前产品需求。

## 11. 自动格式化保护

当前板级配置启用了：

```c
#define TCFG_SD0_FORMAT_ON_BOOT       ENABLE_THIS_MOUDLE
#define TCFG_SD0_FORCE_FORMAT_ON_BOOT DISABLE_THIS_MOUDLE
```

原 SDK 路径在 SD0 mount 失败后读取 VM 标志，但无论 VM 标志有效还是无效，最终都可能调用：

```c
f_format("storage/sd0/C/", "fat", 0);
```

VM 标志只改变日志内容，并不会阻止再次格式化。因此以下情况原本都可能导致用户数据被擦除：

- Windows 把磁盘格式化为 exFAT 或 NTFS。
- USB 非安全拔出导致 FAT 损坏。
- SD NAND 暂时上电或通信异常，造成一次 mount 失败。
- VM 被擦除或升级流程改变。

当前项目增加了独立保护并在覆盖配置中关闭它：

```c
#define TCFG_SD0_AUTO_FORMAT_ON_MOUNT_FAIL_ENABLE 0
```

`dev_manager.c` 只有在板级格式化开关和该项目保护同时为 1 时才会走 mount 失败自动格式化。
这保证 PC 非安全拔出或用户误格式化为其他文件系统时，固件不会因为一次挂载失败立即擦除数据。

对量产产品仍应遵守：

1. 量产关闭自动格式化，或重构为明确的首次初始化流程。
2. 只有确认介质从未初始化且没有用户数据时才自动格式化。
3. 已使用过的介质 mount 失败时进入故障状态，等待用户确认或维护工具修复。
4. 永远保持 `TCFG_SD0_FORCE_FORMAT_ON_BOOT = 0`。

## 12. USB 参数事实

当前代码配置的是 Full-Speed 12 Mbps，设备描述符中的关键值为：

| 项目 | 当前值 |
|---|---|
| `FUSB_MODE` | `1` |
| `bcdUSB` | USB 1.1 (`0x0110`) |
| EP0 最大包长 | 64 bytes |
| Bulk 最大包长 | 64 bytes |
| 配置描述符 MaxPower | 100 mA |
| BR28 `USB_MAX_HW_EPNUM` | 5，即 EP0 到 EP4 |

Full-Speed 设备通过设备端 D+ 上拉向 Host 表示连接，Host 检测该状态并开始总线复位；不是
PC 主动把设备端 D+ 拉高。

当前主要端点分配为：

| Class | 端点 |
|---|---|
| MSD | EP1 Bulk IN/OUT |
| HID | EP2 Interrupt IN |
| UAC Speaker/Mic | 使用同号的不同方向端点，具体以描述符为准 |

不要在文档中给出未经实测的 SD NAND 读写速度。当前 `MSD_BLOCK_NUM = 1`，USB、块缓存、
文件系统簇大小和 SD NAND 写放大都会影响结果。

## 13. 当前实施状态

### 13.1 当前产品范围

T2620 当前 USB 产品需求仅包含 MSC 文件访问。USB HID、UAC Speaker、UAC Microphone 及其他
复合设备 Class 明确保持关闭，不计入当前产品完成度。只有未来产品需求变化时才评估启用，并重新
验证端点、SRAM、时钟和枚举稳定性。

### 13.2 已完成功能

1. 在 `t2620_project_config.h` 打开 PC 和 MSD，保持充电开机宏为 `0`。
2. DIP ON 双层门控及 USB 已连接时的主动重试。
3. 进入 PC 前卸载 FAT 并阻止重挂载，取得所有权后才启动 MSC。
4. 退出 PC 时确认 USB/LUN 资源释放后再恢复 SD0；失败时保持 blocked。
5. DIP OFF 通过应用消息先停止 PC/MSC，再进入关机流程。
6. 存储忙时拒绝进入 PC，PC 模式退出 BT/RDX 前台。
7. mount 失败自动格式化已关闭，焊接式 SD NAND 配置为始终在线。
8. Windows 可识别磁盘并读写文件；安全弹出后拔线可恢复设备侧 VFS。

### 13.3 后续工作：可靠性验证

当前产品范围内不再有 USB Class 功能开发项。后续工作是量产可靠性验证，包括连续插拔、
大文件校验、写入期间意外拔线、STOP 超时故障注入及 remount 失败恢复策略验证。

## 14. 验证矩阵

| 场景 | 预期结果 |
|---|---|
| DIP OFF + 普通充电器 | 只充电，无 USB 枚举 |
| DIP OFF + PC 数据口 | 只充电，无 MSC 枚举 |
| DIP ON + 普通充电器 | 不进入 PC；具体开机充电行为符合产品定义 |
| DIP ON + PC 数据口 | 进入 PC，Windows 出现磁盘 |
| 正在录音时连接 PC | 明确拒绝进入；录音停止后重新触发才能进入 |
| PC 写文件后安全弹出并拔线 | 设备重新 mount，文件可由设备正常读取 |
| PC 写文件时直接拔线 | 不崩溃；下次 mount 失败不得自动擦除用户数据 |
| PC 格式化为非 FAT32 | 固件 mount 失败并保持存储不可用，不得自动重格式化；日志不推断具体失败原因 |
| DIP 在 PC 模式切到 OFF | MSC 安全停止后进入关机/充电状态 |
| DIP ON，PC 模式中拔出 USB | 恢复 SD0 并回到 BT，不执行 `rdx_app_normal_poweroff` |
| DIP OFF 充电时切到 ON，USB 已连接 PC | 主动检测现有 Slave 状态并进入 PC |
| 退出 PC 并恢复 SD0 后 | 不出现延迟 SD0 `DEVICE_EVENT_OUT`，不删除 `sd0` |
| 连续插拔 100 次 | 无死锁、无文件句柄泄漏、无 FAT 损坏 |

验证时应同时检查：

- Windows 设备管理器中的描述符和驱动状态。
- 磁盘管理中的容量、分区和文件系统。
- 大文件循环读写及校验值。
- 每次退出 PC 后设备侧 mount 结果。
- 录音、播放、文件传输是否在 PC 模式期间被可靠阻止。
- `dir /a X:\` 显示的所有目录项，区分录音、RDX 元数据和 Windows 系统项目。
