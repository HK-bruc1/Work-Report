# 本地录音文件播放调试记录

> 文档状态：阶段性结案存档
>
> 最后更新：2026-07-13
>
> 适用范围：RDX 本地录音、BLE 文件传输、耳机端 JLStream 播放链路

## 1. 调试目标与最终结论

本次目标是在设备端读取 SD 卡中的 RDX 本地录音文件，通过 JLStream 解码并由 DAC 播放，同时支持按键切换录音。

最终设备验证表明，该功能已经达到核心目标：本地录音能够完整解码，播放速度和音调正确，没有持续声音变形。问题并非一个单点故障，而是播放入口、Codec 路由、解码输出采样率，以及录音端历史遗留问题叠加造成的。

最终确认的格式契约如下：

```text
录音输入 PCM : 16000 Hz, 2 ch, 16 bit
PCM 帧布局   : L[320] + R[320]，每声道 320 点/20 ms
Opus 文件     : headerless stereo Opus raw
Opus 帧大小   : 80 bytes/20 ms
Opus 码率     : 32000 bit/s
文件数据率    : 4000 bytes/s
解码输出 PCM : 48000 Hz（JL stereo Opus decoder 固定输出）
本地 DAC      : 48000 Hz, 1 ch
```

最新文件 `a171c0.raw` 的验证结果：

```text
文件大小       : 34240 bytes
理论时长       : 34240 / 4000 = 8.56 s
实测播放时长   : 约 8.63 s
文件读取/写入  : read=34240, written=34240
播放收尾       : Source_Dev0 缓存正常排空
解码状态       : 无 dec_err
设备听感       : 速度和音调正常，持续变形基本消失
```

当前剩余问题主要是少量录音缓存丢弃、停止阶段尾帧、重复初始化导致 SN 跳号，以及进一步的音质优化。这些问题不需要推翻现有 Stereo Opus 方案。

## 2. 问题边界

本次排查最终识别出两类相互独立、但会在耳机本地播放时叠加的问题。

| 问题归属 | 具体问题 | BLE APP | 耳机本地播放 |
|---|---|---|---|
| 本地播放端 | 单声道流程、40 字节送帧、普通 Opus Decoder | 不受影响 | 无法正确播放或立即解码失败 |
| 本地播放端 | Stereo Opus 解码输出 48 kHz，但播放流声明 16 kHz | 不受影响 | 三倍慢放、音调降低、声音变形 |
| 录音端 | PCM 平面布局曾被错误理解和处理 | 可解码，但文件内容可能变形 | 可解码，但文件内容同样可能变形 |
| 录音端 | MIC/DAC 双路同步、缓存满和停止时序不完善 | 可能出现缺口或尾部缺失 | 同样可能出现缺口或尾部缺失 |

因此，“BLE APP 能播放”与“录音内容完全正确”不是同一个结论。BLE APP 使用自己的 Stereo Opus 解码链路，只要文件结构合法，就可能正常解码；但如果编码前 PCM 排列错误或双路不同步，APP 播放出来的内容仍可能有噪声或变形。

## 5. 阶段一：建立本地播放链路

最初工程没有完整的本地录音播放实现。本次新增 `rdx_playback.c/.h`，建立了以下流程：

1. 按键发送 `APP_MSG_REC_NEXT`。
2. 从 UXFILE/DAT 缓存中查找录音 SN 和文件名。
3. 打开 `storage/sd0/C/<filename>.raw`。
4. 调用 `dev_flow_player_open(..., NODE_UUID_SOURCE_DEV0)` 创建 JLStream 播放流。
5. 使用非阻塞定时数据泵从文件读取数据，并写入 Source_Dev0 环形缓冲区。
6. 通过可写空间检查实施背压。
7. 文件读取完成后等待 Source_Dev0 缓冲区排空，再关闭播放器。
8. 支持上一条、下一条、自动切换和错误文件跳过。

同时增加了播放与录音、文件同步、文件传输之间的互斥检查，以及以下 Source_Dev0 辅助接口：

```text
source_dev0_get_free_space()
source_dev0_is_empty()
```

第一份日志证明基础链路已经打通：

```text
[PB]open ok: storage/sd0/C/da573.raw, flen=67680
source  : 16000 Hz, 1 ch, 16 bit
decoder : 16000 Hz, 1 ch, 16 bit
dac     : 16000 Hz, 1 ch, 16 bit
```

文件可以完整读取、写入并启动 DAC，但没有正常声音。这说明 SD 卡、文件访问、播放器创建和 DAC 启动都不是首要故障，排查重点转向录音格式与播放参数。

## 6. 阶段二：对齐录音格式与播放参数

录音日志给出的编码参数为：

```text
formate = 2
sr = 16000
br = 32000
ch = 2
fs = 20
```

`formate = 2` 对应 `RECORD_FORMATE_OPUS_16K_STERO`。DAT 中的 `opus` 字段为：

```text
1093142528 = 0x41280800
```

按 RDX 协议位域解析，同样表示 16 kHz、双声道、20 ms、32 kbps。固定码率下每帧大小为：

```text
32000 bit/s / 8 × 20 ms = 80 bytes
```

最初播放端存在三处明显不一致：

1. 使用 `dev_flow_player_open(1, ...)`，选择了单声道流程。
2. Source_Dev0 声明为普通 `AUDIO_CODING_OPUS`。
3. 文件读取和 Source_Dev0 输出均按 40 字节对齐。

第一次修正包括：

- 播放改用双声道流程；
- 文件和 Source_Dev0 改为按 80 字节完整帧投递；
- Source_Dev0 输出缓存扩展到至少一个 80 字节帧；
- 仅在缓存中存在完整帧时向 Decoder 输出；
- 保留普通单声道 Opus 分支，避免影响其他调用场景。

修正后日志显示 Source 已经进入双声道：

```text
source_dev0 NODE_IOC_SET_PRIV_FMT 2
source  : 16000 Hz, 2 ch, 16 bit
decoder : 16000 Hz, 1 ch, 16 bit
dac     : 16000 Hz, 1 ch, 16 bit
```

但此时仍有明显杂音。这个结果说明声道数和帧边界只是第一层问题，参数表面对齐并不等于实际 Codec 路由已经正确。

## 7. 阶段三：对齐 Stereo Opus Codec 路由

新文件 `aeee28.raw` 的日志使问题更加明确：

```text
[PB]open ok: storage/sd0/C/aeee28.raw, flen=38880
source_dev0 NODE_IOC_SET_PRIV_FMT 2
source  : 16000 Hz, 2 ch, 16 bit
decoder : 16000 Hz, 1 ch, 16 bit
dac     : 16000 Hz, 1 ch, 16 bit
[JLSTREAM]dec_err: 40
[JLSTREAM]dec_end: 0
```

`38880 / 80 = 486` 帧，理论时长约为 `9.72` 秒，但播放约 `0.2` 秒就结束。源码检查发现：

- Source_Dev0 双声道分支仍上报 `AUDIO_CODING_OPUS`；
- `t2620_project_config.h` 尚未启用 Stereo Opus 编解码插件；
- `sdk_used_list.used` 中只有 `opus_dec_plug` 和 `opus_encoder_plug`；
- 缺少 `opus_stenc_plug` 和 `opus_stenc_dec_plug`。

因此，80 字节 Stereo Opus 裸帧实际被送入了普通 Opus Decoder。`TCFG_DEC_OGG_OPUS_ENABLE` 不能替代 Stereo Opus Decoder 插件。

对应修正为：

```c
#ifndef TCFG_STENC_OPUS_ENABLE
#define TCFG_STENC_OPUS_ENABLE                    1
#endif

#ifndef TCFG_DEC_STENC_OPUS_ENABLE
#define TCFG_DEC_STENC_OPUS_ENABLE                1
#endif
```

Source_Dev0 双声道分支改为上报：

```text
coding_type  = AUDIO_CODING_STENC_OPUS
channel_mode = AUDIO_CH_LR
frame_dms    = 200
bit_rate     = 32000
```

构建后需要确认生成的 `SDK/apps/earphone/sdk_used_list.used` 至少包含：

```text
opus_stenc_plug
opus_stenc_dec_plug
```

这一阶段解决了 Decoder 选择错误和 `dec_err: 40`。设备从无法正常解码或纯杂音，进展到能够辨识录音内容，但仍存在速度、音调和声音变形。

## 8. 阶段四：修正本地播放的 48 kHz 输出

新录音 `8cac51.raw` 大小为 `18880` 字节：

```text
18880 / 80 = 236 帧
236 × 20 ms = 4.72 s
```

本地播放不再出现 `dec_err: 40`，但实测播放约 `14.16` 秒，恰好是理论时长的三倍。

通过固件反汇编中的 `opus_dec.c` 确认，JL 的 `opus_stenc_dec_plug` 固定输出 48 kHz PCM。此前 Source_Dev0 仍向播放流声明 16 kHz，导致每个 20 ms Opus 包解出的 960 个样点被 DAC 按 16 kHz 播放：

```text
960 / 48000 = 20 ms   // 正确
960 / 16000 = 60 ms   // 错误，恰好三倍时长
```

修正方式是在 Stereo Opus 播放分支声明 48 kHz：

```text
sample_rate = 48000
coding_type = AUDIO_CODING_STENC_OPUS
frame_dms   = 200
bit_rate    = 32000
```

这里的 48 kHz 是 Decoder/DAC 侧的 PCM 输出采样率。录音编码输入仍为 16 kHz，文件仍为 80 字节/20 ms、32 kbps，协议格式没有变化。

这是本地播放专属问题。BLE APP 使用自己的 Opus Decoder，并按正确的解码输出采样率播放，所以不会出现耳机固件中的三倍慢放问题。

## 9. 阶段五：定位录音端历史遗留问题

本地播放 Codec 路由和采样率修正后，已经可以听到录音内容，但新旧日志和 BLE APP 听感仍提示文件内容本身可能存在问题。排查因此继续向编码前的 PCM 数据链路推进。

### 9.1 历史假设：输入是交织 PCM

早期参考 JLStream 常见布局和 `opus_st_enc_demo.c`，曾假定 `EffectDev2` 输入为：

```text
L0 R0 L1 R1 ...
```

并按偶数、奇数样点拆分左右声道。该版本使设备从纯杂音或无法辨识，进展到可以大致听清 DAC 内容，因此是有效的阶段性突破；但它不是最终正确实现。

### 9.2 源码证据：实际输入是平面 PCM

继续检查实际生产者 `source_dev1_file.c` 后确认，Source_Dev1 已经把 MIC 和 DAC 两个缓存组装为：

```text
L0 L1 ... L319 R0 R1 ... R319
```

也就是 `L[320] + R[320]` 的平面布局。若再按偶数、奇数样点拆分，实际效果会近似变成：

```text
编码左声道 = 原左声道偶数样点
编码右声道 = 原左声道奇数样点
原右声道数据没有被正确使用
```

最终修正为直接向 Stereo Opus Encoder 传入前后两个平面声道：

```c
s16 *pcm_l = tst_buf_LR;
s16 *pcm_r = tst_buf_LR + FRAME_POINT;

memcpy(pcm_l, indata, FRAME_SIZE);
memcpy(pcm_r, indata + FRAME_POINT, FRAME_SIZE);

ret = hdl->opus_stenc->run_wb(
    (void *)run_buf,
    pcm_l,
    pcm_r,
    (u8 *)outdata,
    &encLen
);
```

每个编码周期的预期数据量为：

```text
输入 PCM = 16000 × 20 ms × 2 ch × 2 bytes = 1280 bytes
输出 Opus = 80 bytes
```

### 9.3 MIC/DAC 双路同步与缓存问题

录音日志中的大量 `M` 和 `W` 表明，MIC 和 DAC 两个生产者到达时间存在抖动。旧逻辑没有始终保证两路都具备完整 20 ms 数据，缓存满时还可能发生部分写入，进而破坏样点边界或声道时间关系。

本轮修正包括：

1. 仅当 MIC 和 DAC 两个缓存均达到 640 字节时，才组装一个 Stereo PCM 帧。
2. 数据写入后，一旦两路均满足一帧，主动唤醒编码线程。
3. 每路缓存从约 4 帧扩大到 8 帧，以吸收短时调度抖动。
4. 缓存空间不足时放弃完整输入块，不再进行破坏帧边界的部分写入。
5. 去掉逐帧字符刷屏，在停止时统一输出统计：

```text
source_dev1 stats: frames=... wait=... drop=mic/dac remain=mic/dac
```

其中：

- `frames`：向后级提供的完整 Stereo PCM 帧数；
- `wait`：查询时两路尚未同时具备完整帧的次数，不等于丢帧；
- `drop`：缓存满导致整块未写入的次数，目标为 `0/0`；
- `remain`：停止时两个输入缓存的剩余数据量。

这些是录音端遗留问题，并非本地播放专属。错误一旦被编码进文件，BLE APP 和耳机本地都会解出同样的内容差异。

## 12. BLE APP 兼容性结论

### 12.1 为什么修改前 BLE APP 仍能播放

修改前的 `.raw` 文件在许多阶段仍然是结构合法的 Stereo Opus 码流。BLE APP 使用自己的 Opus Decoder，能够按照双声道、80 字节/20 ms 和正确输出采样率进行解码，因此不会受到耳机本地播放端以下问题的影响：

- 单声道播放流程；
- 40 字节错误帧边界；
- 普通 Opus 与 Stereo Opus Decoder 路由错误；
- 48 kHz 解码输出被本地 DAC 按 16 kHz 播放。

但是，能被 APP 解码不代表文件中的音频内容一定正确。编码前的 PCM 排列、MIC/DAC 同步或缓存丢弃问题已经写入文件时，APP 仍可能播放出可辨识但带噪声、缺口或变形的声音。

### 12.2 本次修改是否影响原有 BLE APP

当前修改保持了 BLE APP 所依赖的文件格式和传输协议：

```text
headerless stereo Opus raw
16000 Hz、双声道 PCM 编码输入
20 ms/帧
80 bytes/帧
32000 bit/s
约 4000 bytes/s
```

本地播放端修改只作用于耳机内部 JLStream 和 DAC，不改变 BLE 下载文件。录音端修改会改善新文件中的音频样本和双路时间关系，但不会改变 Opus 帧大小、码率、文件布局或 BLE 传输方式。

新固件录制的文件已经通过 BLE APP 实听验证，未发现兼容性回退。因此当前可以得出以下结论：

> 本次修复保持原有 BLE 文件协议兼容。录音端修复改善新文件的音频内容，本地播放端修复不参与 BLE APP 解码；设备实测未发现对 BLE APP 播放造成负面影响。

### 12.3 新旧文件的影响范围

- 固件升级不会重写 SD 卡中的旧 `.raw` 文件；
- 旧文件中已经存在的声道排列、同步或缺帧问题不会自动修复；
- 本地播放器可以按正确 Codec 和采样率播放旧文件，但无法恢复已经错误编码的内容；
- 只有使用新固件重新录制的文件，才会应用 PCM 平面直传和双路同步修复。

## 13. 已排除问题与次要日志

目前已排除：

- SD 卡未挂载或文件无法打开；
- 播放器接口完全不可用；
- DAC 未启动；
- 文件未被完整送入 Source_Dev0；

- EOF 后立即关闭导致 Source 缓存尾部被截断；
- 当前版本仍按 16 kHz 播放 48 kHz 解码输出。

以下日志不是本次核心故障的直接根因：

- `dev online aready, err` / `sd add fail`：设备已经在线时重复调用 `dev_manager_add()`；
- `flash info not found`：启动阶段 Flash 信息提示；
- `tone_player_faild`：提示音资源问题；
- `timer_no_response`：同步文件访问或停止流程造成的短时任务延迟；
- 关机提示音之后的 `dec_err: 40`：与本地录音文件解码无关。

## 14. 剩余优化项

### P1：录音连续性

1. 使用 30 秒、1 分钟和 5 分钟录音观察 `drop` 是否随时长增长，目标为 `drop=0/0`。
2. 若仍发生缓存满，优先检查编码线程唤醒和 SD 写入阻塞，再评估是否继续扩大缓存。
3. 若必须丢弃数据，应成对丢弃 MIC/DAC 的同一时间段，避免双声道时间轴逐渐错位。
4. 减少无效轮询，降低 `wait` 数量和线程调度开销。

### P1：停止阶段尾帧

当前 Source_Dev1 比最终文件多产生约 11 帧。建议将停止顺序收敛为：

```text
停止 MIC/DAC 生产者
→ 排空 Source_Dev1
→ 排空 EffectDev2 和文件写缓存
→ 关闭文件并保存 DAT
```

目标是使 `source frames × 80` 与最终文件大小更加接近，并避免录音末尾缺少约 200 ms。

### P1：重复初始化与 SN 消耗

一次录音启动日志先创建了 SN=1 的 `5a608e.raw`，随后又创建实际保存的 SN=2 `a171c0.raw`，最终文件列表只有一条记录。需要检查 `rdx_record_run_init()` 与录音任务消息的调用时序，避免空文件、孤立文件和 SN 跳号。

### P2：Codec 配置收敛

Stereo 本地播放的核心路由是 `AUDIO_CODING_STENC_OPUS`，不是普通 `AUDIO_CODING_OPUS`。后续应使用同一文件做启用/关闭 A/B 验证，再确认以下全局配置是否仍有必要：

```text
TCFG_DEC_OGG_OPUS_ENABLE
CONFIG_OGG_OPUS_DEC_SET_CBR_PACKET_LEN = 80
```

稳定版本中不应未经验证直接删除，避免影响工程内其他普通 Opus 场景。

### P2：剩余音质

使用同一固定测试音分别进行 MIC-only、DAC-only 和 MIC+DAC 三组录音，区分剩余噪声来自输入源、AGC/DNS、模拟增益还是 Opus 32 kbps。完成隔离前不要同时修改码率、增益和降噪参数。

### P3：可靠性回归

- 连续录音 5 分钟；
- 连续播放全部文件；
- 播放中切换上一条和下一条；
- 验证录音与播放互斥；
- 文件传输期间禁止播放；
- 损坏文件或长度非 80 字节对齐时安全跳过；
- 多次录音后重启并重新播放。

必须使用新固件重新录制的新文件，不能仅用修复前的 `da573.raw`、`aeee28.raw` 或 `8cac51.raw` 判断录音端修复效果。

录音端预期：

```text
sr : 16000, br : 32000, ch : 2, fs : 20
文件大小约为 4000 bytes/s
source_dev1 stats: ... drop=0/0 ...
```

本地播放端预期：

```text
source_dev0 NODE_IOC_SET_PRIV_FMT 2
source  : 48000 Hz, 2 ch
decoder : 48000 Hz, 1 ch
dac     : 48000 Hz, 1 ch
无 dec_err
read = written = 文件长度
Source_Dev0 缓存正常排空
```

任意文件的理论播放时长为：

```text
播放时长（秒）≈ 文件大小（bytes）/ 4000
```

BLE APP 与耳机本地应能辨识同一段内容，速度和音调基本一致。

## 16. 经验总结

1. 参数一致性应同时检查声道数、帧大小、Codec 类型和 Decoder 实际输出采样率，不能只看日志中的输入采样率。
2. “文件可解码”只证明码流结构有效，不证明编码前 PCM 内容正确。
3. 判断 PCM 是交织还是平面布局时，应以当前链路的实际生产者代码为准，不能仅套用 SDK 示例。
4. 修复播放端后暴露录音端问题并不矛盾：播放端负责正确还原文件，录音端负责保证写入文件的内容正确。
5. 旧文件无法验证录音端新修复；任何编码输入修改都必须重新录制后再比较。
6. 文件大小、固定帧长和实测时长之间的数学关系，是定位 Codec 和采样率问题最可靠的证据之一。
