# AI翻译与录音功能详细文档

---

## 一、芯片选型建议

> **做opus相关的AI翻译功能，701 RAM 640K，708 RAM 384K，709 RAM 320K，功能较多的项目，建议选择701系列芯片开发AI翻译类应用。**

音频翻译涉及大量音频数据的接收、编解码与中转，对算力和RAM有较高要求。

```mermaid

    A[701N<br/>RAM 640K] -->|推荐| D[AI翻译应用]
    B[708N<br/>RAM 384K] -->|可用| D
    C[709N<br/>RAM 320K] -->|RAM较紧张| D
    style A fill:#4CAF50,color:#fff
    style B fill:#FF9800,color:#fff
    style C fill:#f44336,color:#fff
    style D fill:#2196F3,color:#fff
```

## 二、功能概览与芯片支持矩阵

| 功能 | 通信方式 | 支持芯片 | 备注 |
|------|----------|----------|------|
| 普通录音 | BLE、SPP | 697N、700N、701N、708N、709N、689N、710N | — |
| 普通录音 | ESCO | 所有支持蓝牙通话的芯片 | 走HFP，原理同手机语音助手，需APP获得手机权限 |
| 通话录音 | BLE、SPP | 701N、708N、709N、689N、710N | 710N仅支持录单路且仅支持单mic通话 |

| 音频翻译 | ESCO | 所有支持蓝牙通话的芯片 | 仅支持mic翻译，走HFP，需APP获得手机权限 |
| 通话翻译 | BLE、SPP | 701N、708N、709N、689N | — |

## 三、核心原理

### 3.1 整体协同架构

AI翻译不在耳机本地运算，本质是**耳机 + 手机APP 的协同流水线**：

```mermaid
flowchart LR
    subgraph 耳机
        MIC[MIC采集]
        ENC[Opus编码]
        DEC[Opus解码]
        DAC[DAC播放]
        MIC --> ENC
        DEC --> DAC
    end

    subgraph 手机APP
        AI[AI翻译引擎\n云端 / 本地]
    end

    ENC -->|BLE / SPP / ATT over EDR| AI
    AI -->|翻译后Opus| DEC

    style AI fill:#9C27B0,color:#fff
    style MIC fill:#03A9F4,color:#fff
    style DAC fill:#03A9F4,color:#fff
```

### 3.2 为什么用Opus而不是AAC/mSBC？

| 特性 | Opus | AAC | mSBC |
|------|------|-----|------|
| 设计目标 | 低延迟实时传输 | 高音质存储/流媒体 | 通话语音 |
| 延迟 | 极低（最低2.5ms帧长） | 较高 | 中等 |
| 码率灵活性 | 6kbps～510kbps任意调 | 固定档位 | 固定约64kbps |
| 网络抗丢包 | 内置FEC前向纠错 | 无 | 无 |
| 语音+音乐 | 两者都擅长 | 仅音乐 | 仅语音 |
| 授权费 | 完全免费开源 | 需授权费 | 免费 |
| 传输通道 | SPP/BLE/ATT over EDR | A2DP专用 | **eSCO专用，无法走SPP/BLE** |

> mSBC只能跑在eSCO链路，根本无法传给APP；AAC码率不够灵活；Opus低延迟+抗丢包+免费，天然适合实时AI流水线。

### 3.3 音质影响分析

```mermaid
flowchart TD
    Q[用哪种翻译模式?]
    Q --> A[录音 / 通话翻译]
    Q --> B[音视频字幕模式]
    Q --> C[音视频语音转译模式]

    A --> A1[几乎无感知\n人声 16kbps Opus 清晰可懂]
    B --> B1[完全无损\nDAC始终播放原始A2DP音频]
    C --> C1[有牺牲\n原声切断 改播TTS合成语音\n用户主动取舍]

    style B1 fill:#4CAF50,color:#fff
    style A1 fill:#8BC34A,color:#fff
    style C1 fill:#FF9800,color:#fff
```

## 四、普通录音

非通话状态下三个通道（BLE/SPP/ESCO）均未占用，可任意选择。

```mermaid
flowchart LR
    MIC[MIC / ADC]
    CH{传输通道}
    APP[手机APP\n录音存储]

    MIC --> CH
    CH -->|BLE| APP
    CH -->|SPP| APP
    CH -->|ESCO\n走HFP 需权限| APP

    style MIC fill:#03A9F4,color:#fff
    style APP fill:#4CAF50,color:#fff
```

> ESCO方式实现原理同手机语音助手，需要APP获得手机录音权限。

## 五、通话录音

通话中ESCO已被通话数据占用，**只能通过BLE或SPP**将上下行音频同时中转到APP。

```mermaid
flowchart LR
    subgraph 耳机
        MIC[MIC / ADC\n上行采集]
        ESCO_TX[ESCO_TX]
        ESCO_RX[ESCO_RX\n下行接收]
        DAC[DAC]
    end

    PHONE[手机\n通话对端]
    APP[手机APP\n录音存储]

    MIC -->|通话上行| ESCO_TX --> PHONE
    PHONE --> ESCO_RX -->|通话下行| DAC
    MIC -->|上行旁路\nBLE / SPP| APP
    ESCO_RX -->|下行旁路\nBLE / SPP| APP

    style MIC fill:#03A9F4,color:#fff
    style DAC fill:#03A9F4,color:#fff
    style APP fill:#4CAF50,color:#fff
    style PHONE fill:#9E9E9E,color:#fff
```

> 710N目前仅支持**录单路**（上行或下行其中一路），且仅支持单mic通话；其他芯片上下行均可录制。

## 六、音频翻译（录音翻译 / 面对面翻译）

三个通道均未被占用，可选BLE、SPP（推荐）或ESCO（仅mic翻译）。

```mermaid
flowchart LR
    subgraph 耳机
        MIC[MIC / ADC]
        ENC[Opus编码\nai_recorder.c]
        AITX[AI_TX节点\nai_tx_node.c]
        AIRX[AI_RX节点\nai_rx_file.c]
        DEC[Opus解码\nai_player.c]
        DAC[DAC播放]
    end

    subgraph 手机APP
        AI[AI翻译引擎]
    end

    MIC --> ENC --> AITX -->|BLE / SPP| AI
    AI -->|翻译后Opus| AIRX --> DEC --> DAC

    style AI fill:#9C27B0,color:#fff
    style MIC fill:#03A9F4,color:#fff
    style DAC fill:#03A9F4,color:#fff
```

**SDK配置要点：**
- "智能语音"页面选择 **AI翻译**，使能右上角按钮
- 录音模式：仅用 `ADC → AI_TX` 单向流
- 录音翻译模式：`ADC → AI_TX` 和 `AI_RX → DAC` 双向流同时使能

## 七、音视频翻译

### 7.1 字幕模式（TWS耳机_AI翻译）

DAC始终播放原声，翻译结果以**字幕**形式在APP显示，**对音质零影响**。

```mermaid
flowchart TD
    subgraph 翻译关闭
        direction LR
        RX1[A2DP_RX] --> DAC1[DAC\n播放原声]
        SW1[Switch\n断开 ✗] -.-x AITX1[AI_TX]
    end

    subgraph 翻译打开
        direction LR
        RX2[A2DP_RX] --> DAC2[DAC\n播放原声]
        RX2 --> SW2[Switch\n接通 ✓] --> AITX2[AI_TX] --> APP[APP\n字幕显示]
    end

    style DAC1 fill:#4CAF50,color:#fff
    style DAC2 fill:#4CAF50,color:#fff
    style APP fill:#9C27B0,color:#fff
    style SW2 fill:#4CAF50,color:#fff
    style SW1 fill:#9E9E9E,color:#fff
```

### 7.2 语音转译模式（TWS耳机_AI翻译（语音转译））

DAC切换播放原声或翻译后语音，用户主动取舍音质。

```mermaid
flowchart TD
    subgraph 翻译关闭
        direction LR
        RX1[A2DP_RX] --> SW1[Switch1 ✓] --> DAC1[DAC\n播放原声]
        SW2OFF[Switch2 ✗] -.-x AITX1[AI_TX]
    end

    subgraph 翻译打开
        direction LR
        RX2[A2DP_RX] --> SW2[Switch2 ✓] --> AITX2[AI_TX] --> APP[APP翻译] --> AIRX[AI_RX] --> DEC[Opus解码] --> DAC2[DAC\n播放翻译语音]
        SW1OFF[Switch1 ✗] -.-x DAC_OFF[DAC 原声断开]
    end

    style DAC1 fill:#4CAF50,color:#fff
    style DAC2 fill:#FF9800,color:#fff
    style APP fill:#9C27B0,color:#fff
    style SW1 fill:#4CAF50,color:#fff
    style SW2 fill:#4CAF50,color:#fff
    style SW1OFF fill:#9E9E9E,color:#fff
    style SW2OFF fill:#9E9E9E,color:#fff
```

## 八、通话翻译

通话翻译是最复杂的场景，**上下行各自独立**走翻译链路。

### 8.1 完整数据流

```mermaid
flowchart TB
    subgraph 下行翻译 - 听对方说话
        direction LR
        PH1[手机] -->|mSBC / eSCO| ERX[ESCO_RX\n耳机解码]
        ERX -->|Opus编码| APP1[APP翻译]
        APP1 -->|Opus| DEC1[耳机Opus解码]
        DEC1 --> DAC1[DAC播放]
    end

    subgraph 上行翻译 - 说话给对方
        direction LR
        MIC2[MIC] -->|Opus编码| APP2[APP翻译]
        APP2 -->|Opus| DEC2[耳机Opus解码]
        DEC2 -->|eSCO编码| ETX[ESCO_TX]
        ETX --> PH2[手机]
    end

    style APP1 fill:#9C27B0,color:#fff
    style APP2 fill:#9C27B0,color:#fff
    style DAC1 fill:#03A9F4,color:#fff
    style MIC2 fill:#03A9F4,color:#fff
```

### 8.2 Switch路由逻辑

```mermaid
flowchart LR
    subgraph 翻译关闭 - 正常通话
        ERXA[ESCO_RX] -->|SW1通| DACA[DAC]
        MICA[MIC/ADC] -->|SW3通| ETXA[ESCO_TX]
    end

    subgraph 翻译打开
        ERXB[ESCO_RX] -->|SW2通| AITXB[AI_TX → APP → AI_RX → DAC]
        MICB[MIC/ADC] -->|SW4通| AITXC[AI_TX → APP → AI_RX → ESCO_TX]
    end

    style AITXB fill:#9C27B0,color:#fff
    style AITXC fill:#9C27B0,color:#fff
```

| 状态 | Switch(1) | Switch(2) | Switch(3) | Switch(4) |
|------|-----------|-----------|-----------|-----------|
| 翻译关闭 | 通（ESCO_RX→DAC） | 断 | 通（ADC→ESCO_TX） | 断 |
| 翻译打开 | 断 | 通（ESCO_RX→AI_TX） | 断 | 通（ADC→AI_TX） |

### 8.3 编码模型选择

```mermaid
flowchart TD
    Q[通话翻译编码模型选择]
    Q --> A[单声道编码模型]
    Q --> B[立体声编码模型 推荐]

    A --> A1[2组单声道编码\n2组单声道解码]
    A1 --> A2[CPU占用高\n切换JLA-V2\n时钟提升192MHz]

    B --> B1[1组立体声编码\n2组单声道解码]
    B1 --> B2[MIPS和RAM更低\nRCSP协议必选此模型]

    style B fill:#4CAF50,color:#fff
    style A fill:#FF9800,color:#fff
    style B2 fill:#4CAF50,color:#fff
    style A2 fill:#f44336,color:#fff
```

**立体声编码开启方式：**

```c
// 开启opus立体声编码（celt mode）
#define TCFG_ENCODER_CHANNEL_NUM (BIT(0) | BIT(1))
```

### 8.4 带宽调优

出现通话翻译卡顿（如打印 `spp OVERFLOW, please check======`）时，适当调大ACL链路带宽比例。

## 十、源文件架构

![image-20260317185507039](./assets/image-20260317185507039.png)

- SDK默认将ai_translator.c挂靠到rcsp_translator.c中，如果有其他协议需求，可以将ai_translator.c中的接口集成到其他协议中。
  - **opus数据可以挂靠到不同的BLE协议。目前是依靠RCSP的杰理官方BLE协议发送到杰理之家APP。**
- ai_player.c自身可以脱离ai_translator.c单独使用，用于播放opus数据流。
- ai_recorder.c自身可以脱离ai_translator.c单独使用，用于采集编码opus数据流。

```mermaid
flowchart TB
    subgraph 协议层
        RCSP[rcsp_translator.c\nRCSP翻译协议]
        TRANS[ai_translator.c\n翻译模型主体]
        RCSP --> TRANS
    end

    subgraph 音频采集链路
        REC[ai_recorder.c\n采集+Opus编码]
        VOICE[ai_voice_recoder.c\n智能语音页面流]
        TX[ai_tx_node.c\nSPP/BLE/ATT发送]
        VOICE --> REC --> TX
    end

    subgraph 音频播放链路
        RX[ai_rx_file.c\n接收编码数据]
        RXPLAY[ai_rx_player.c\nOpus解码]
        PLAYER[ai_player.c\n解码+TWS同步播放]
        RX --> RXPLAY --> PLAYER
    end

    subgraph 原有音频流接入
        A2DP[a2dp_player.c\n蓝牙音乐]
        ESCOP[esco_player.c\n通话播放]
        ESCOR[esco_recoder.c\n通话录制]
    end

    subgraph 基础设施
        COMMON[ai_audio_common.c\n公共函数/数据结构]
        CONFIG[ai_audio_config.h\n功能配置宏]
        EVENT[jlstream_event_handler.c\n流事件/Switch/CPU时钟]
    end

    TRANS --> REC
    TRANS --> PLAYER
    A2DP --> TX
    ESCOP --> TX
    ESCOR --> TX

    style TRANS fill:#9C27B0,color:#fff
    style CONFIG fill:#607D8B,color:#fff
```

| 文件 | 职责 |
|------|------|
| `ai_translator.c` | AI翻译功能主体，内置常用翻译模型，可对接不同客户协议 |
| `ai_player.c` | 接收Opus数据，解码播放，分发给TWS从机同步 |
| `ai_recorder.c` | 采集音频，编码成Opus，发送给APP |
| `ai_audio_common.c` | 公共函数与数据结构 |
| `ai_audio_config.h` | 功能配置宏定义 |
| `rcsp_translator.c` | 对接杰里之家APP的RCSP翻译协议 |
| `ai_tx_node.c` | 通过SPP/BLE/ATT over EDR发送编码音频 |
| `ai_rx_file.c` | 接收编码音频，传给后级节点 |
| `ai_voice_recoder.c` | 智能语音页面录音数据流（ADC→AI_TX） |
| `ai_rx_player.c` | AI_RX相关解码数据流（AI_RX→DAC） |
| `a2dp_player.c` | 蓝牙音乐页面A2DP播放流控制 |
| `esco_player.c` | 蓝牙通话页面eSCO播放流控制 |
| `esco_recoder.c` | 蓝牙通话页面eSCO录制流控制 |
| `jlstream_event_handler.c` | 音频流事件响应，CPU时钟分配，Switch节点行为 |

> `ai_player.c` 和 `ai_recorder.c` 均可脱离 `ai_translator.c` 单独使用。

## 十二、调试技巧

### 缓存空间调试
```c
// 打印写入/读取缓存长度及当前缓存空间
#define AI_AUDIO_RECV_SPEED_DEBUG  0  // 改为1开启
```

### RCSP收发数据打印
在 `rcsp_interface.c` 中开启调试打印。

### 屏蔽干扰打印
调试APP→耳机下行时，若 `APP_SPP_ONLINE_DEBUG` 已开启，`spp_online_db.c` 的收数打印会造成翻译卡顿，建议注释掉该打印。

```mermaid
flowchart LR
    DEBUG[APP_SPP_ONLINE_DEBUG 开启]
    SPP[spp_online_db.c\n大量收数打印]
    LAG[翻译卡顿]
    FIX[注释收数打印]

    DEBUG --> SPP --> LAG
    FIX -->|解决| LAG

    style LAG fill:#f44336,color:#fff
    style FIX fill:#4CAF50,color:#fff
```
