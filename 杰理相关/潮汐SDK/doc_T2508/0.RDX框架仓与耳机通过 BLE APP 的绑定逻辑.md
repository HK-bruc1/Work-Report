# 仓与耳机通过 BLE APP 的绑定逻辑

## 1. 核心结论

BLE 读特征 95 字节协议格式固定为 `role,Lmac,Rmac,Cmac,SN,label sn,wifi Mac`，APP 通过读特征中的固定字段判断仓是否已绑定耳机。

通过编译宏 `RDX_READCHAR_CASE_CENTRIC`（`rdx_app_config.h`）控制两套数据源模式：

| 模式 | SN / label_sn 来源 | Lmac / Rmac 来源 | Cmac 来源 |
|------|-------------------|-------------------|-----------|
| **仓为主** (`=1`, 默认) | flash 认证码 (`auth_key` / `fac_sn`) | VM 耳机 MAC | flash (`ble_mac_str`) |
| 耳机为主 (`=0`) | VM 耳机信息 (`ep_info.sn` / `label_sn`) | VM 耳机 MAC | flash (`ble_mac_str`) |

两套模式共享同一套绑定/解绑/冷启动流程，仅数据源路由不同。数据源切换收敛在单一函数 `rdx_vm_get_sn_and_label_sn()` 中，所有调用点通过该函数获取 SN/label_sn 指针。

## 2. 协议格式

```
role,Lmac,Rmac,Cmac,SN,label sn,wifi Mac
```

| 字段 | 长度 | 含义 |
|------|------|------|
| role | 1 | 角色标识，仓固定为 `C` |
| Lmac | 12 | 左耳 MAC，无冒号 |
| Rmac | 12 | 右耳 MAC，左耳 MAC 最后一字节 +1 |
| Cmac | 12 | 仓自身 BLE MAC，无冒号 |
| SN | 24 | 设备认证标识 |
| label sn | 16 | 盒标 SN |
| wifi Mac | 12 | Wi-Fi MAC；未就绪时固定 `000000000000` |

总长度：1 + 12 + 12 + 12 + 24 + 16 + 12 + 6 逗号 = 95 字节

## 3. 数据源路由

BLE 读特征由 `rdx_app_earphone_pack_readchardata()` 统一拼装：

```c
"C,%s,%s,%s,%s,%s,%s"
```

SN 和 label_sn 通过 `rdx_vm_get_sn_and_label_sn()` 获取，该函数内部按 `RDX_READCHAR_CASE_CENTRIC` 路由：

```c
void rdx_vm_get_sn_and_label_sn(const char **sn, const char **label_sn)
{
#if (RDX_READCHAR_CASE_CENTRIC == 1)
    // 仓为主：从 flash rdx_auth_info 取
    if (p_auth->auth_key[0] != '\0') *sn = p_auth->auth_key;
    if (p_auth->fac_sn[0]   != '\0') *label_sn = p_auth->fac_sn;
#else
    // 耳机为主：从 VM EarphoneInfo 取
    if (ep_info->sn[0]       != '\0') *sn = ep_info->sn;
    if (ep_info->label_sn[0] != '\0') *label_sn = ep_info->label_sn;
#endif
    // 无数据时不改写指针，保持调用方默认值
}
```

各字段来源对照：

| 字段 | 仓为主 (`RDX_READCHAR_CASE_CENTRIC=1`) | 耳机为主 (`=0`) |
|------|------|------|
| role | 固定 `C` | 固定 `C` |
| Lmac | VM `EarphoneInfo.ep_mac_str` | VM `EarphoneInfo.ep_mac_str` |
| Rmac | Lmac 派生 +1 | Lmac 派生 +1 |
| Cmac | Flash `rdx_auth_info.ble_mac_str` | Flash `rdx_auth_info.ble_mac_str` |
| SN | Flash `rdx_auth_info.auth_key` | VM `EarphoneInfo.sn` |
| label sn | Flash `rdx_auth_info.fac_sn` | VM `EarphoneInfo.label_sn` |
| wifi Mac | 固定 `000000000000` | 固定 `000000000000` |

无回退路径。数据源无效时填充全零占位。

## 4. flash 与 VM 的关系

### 4.1 flash：仓本机认证信息

结构体 `rdx_auth_info_t`（`rdx_vm.h`）：

```c
typedef struct {
    u8   ble_mac_hex[6];
    char ble_mac_str[13];
    char auth_key[25];     // flash 第一段，24 字节
    char fac_sn[17];       // flash 第三段，16 字节
} rdx_auth_info_t;
```

flash 中烧录格式为 `auth_key,cmac,fac_sn`（逗号分隔），严格 24/12/16 字节。

初始化时 `rdx_vm_parse_product_info()` 用手动逗号解析（非 `strtok`，RTOS 线程安全）拆出三段：
- `auth_key` → `rdx_auth_info.auth_key`
- `cmac` → `rdx_auth_info.ble_mac_str` + 通过 `le_controller_set_mac()` 写入 BLE 控制器
- `fac_sn` → `rdx_auth_info.fac_sn`

Cmac 复用 `ble_mac_str` / `ble_mac_hex` 字段，无单独字段。

### 4.2 VM：绑定后耳机的 MAC 信息

```c
typedef struct {
    u8   sn[25];
    u8   ep_mac[6];
    char ep_mac_str[13];
    u8   label_sn[17];
} EarphoneInfo;
```

**仓为主模式**下仅 `ep_mac` / `ep_mac_str` 有效，`sn` / `label_sn` 由 memset 保持全零。冷启动时若检测到残留的 `sn` / `label_sn`，自动清零写回 VM。

**耳机为主模式**下所有字段均持久化。

### 4.3 仓为主模式实测示例

```
flash 烧录:  ZENCHORDC202500000000003,50EE40783319,ZCE1250000000003
VM 耳机信息: mac=7878359A614E

BLE 读特征:  C,7878359A614E,7978359A614E,50EE40783319,ZENCHORDC202500000000003,ZCE1250000000003,000000000000
              │ └─VM MAC─┘└VM派生─┘└FLASH cmac┘└── flash auth_key ──┘└─ flash fac_sn ─┘└ 占位 ┘
```

SN 和 label_sn 全部来自 flash，不再依赖 APP 同步的耳机数据。

## 5. APP 如何判断是否已绑定

APP 通过 BLE 读特征中固定位置的字段判断：

- `Lmac` 非全零 + `SN` 非全零 → 已绑定（耳机 MAC 已存入 VM）
- `Lmac` 全零 → 未绑定，需发起配对流程

系统另有 `bound_state` 广播位，属另一套账号 bound/unbound 机制，不在本文讨论范围。

## 6. 绑定时序

### 6.1 绑定前（冷启动）

`rdx_vm_auth_info_init()` 流程：

1. `rdx_vm_read_ep_info_fromVM()` — 读 VM 中的 `EarphoneInfo`
2. **仓为主**：检查 VM 中是否有残留的 `sn`/`label_sn`，有则清零写回
3. `rdx_vm_read_product_info_from_flash()` — 读 flash 认证码
4. `rdx_vm_parse_product_info()` — 解析 AuthKey + Cmac + FacSN
5. 若 flash 无效且 `RDX_READCHAR_SIMULATE_ENABLE=1`，回退模拟数据
6. `rdx_app_earphone_pack_readchardata()` — 拼装 95 字节读特征

### 6.2 APP 点击绑定

协议层回调 `rdx_app_device_pair_handle(au_code, mac_str, label_sn)`：

**仓为主模式**：
- au_code、label_sn 仅做校验 + 日志，**不持久化**
- mac_str 写入 VM 的 `ep_mac_str` / `ep_mac`
- 立即重拼读特征

**耳机为主模式**：
- au_code、mac_str、label_sn 全部写入 VM
- 立即重拼读特征

### 6.3 绑定后即时效果

VM 写入成功后读特征立即刷新，无需等重启。

仓为主实测日志（flash 有效）：

```
rdx_app_device_pair_handle --> au_code validated: ZENCHORDC202507070011825
rdx_app_device_pair_handle --> label_sn validated: ZCE1250825011825
=== write earphone info to vm, result: 61
===rdx_app_earphone_pack_readchardata --> readchardata: C,7878359A614E,7978359A614E,50EE40783319,ZENCHORDC202500000000003,ZCE1250000000003,000000000000
```

SN / label_sn 来自 flash，MAC 来自 VM。

## 7. 冷启动恢复逻辑

仓冷启动后从 VM 读回耳机 MAC，从 flash 读回认证信息，重新拼装读特征。

仓为主实测冷启动日志：

```
=== read ep info from vm, result: 61
=== read ep info from vm success, mac: 7878359A614E
----- read auth data from flash success, auth key: ZENCHORDC202500000000003
rdx_vm_parse_product_info --> Cmac: 50EE40783319
---------- rdx_auth_info.fac_sn: ZCE1250000000003
===rdx_app_earphone_pack_readchardata --> readchardata: C,7878359A614E,7978359A614E,50EE40783319,ZENCHORDC202500000000003,ZCE1250000000003,000000000000
```

## 8. 解绑与重新绑定

### 8.1 解绑入口

解绑通过两个路径触发（互备）：

1. **`rdx_ble_server_gatt_receive_data` 拦截**：匹配 `*APP#devunpair#` 字符串，直接调用 `rdx_app_device_unpair_handle()`
2. **`rdx_vm_unbound_cb` 预留清理**：仓为主模式下清除 earphone MAC（路径 1 已覆盖，此路径为冗余保障）

### 8.2 解绑行为

`rdx_app_device_unpair_handle()` 将 `EarphoneInfo` 全部字段 memset 清零并写回 VM：

- **仓为主**：Lmac/Rmac 变全零；SN/label_sn 来自 flash，不受影响
- **耳机为主**：所有字段变全零

## 9. flash 有效/无效对读特征的影响

### 9.1 仓为主模式

**flash 有效**：SN / Cmac / label_sn 来自 flash，Lmac / Rmac 来自 VM

**flash 无效**：

| 受影响项 | 表现 |
|---------|------|
| `rdx_auth_info.auth_key` | 保持全零 |
| `rdx_auth_info.ble_mac_str` | 全零 |
| Cmac | `000000000000` |
| SN | `000000000000000000000000` |
| label sn | `0000000000000000` |
| Lmac / Rmac | 来自 VM（不受 flash 影响） |

若 `RDX_READCHAR_SIMULATE_ENABLE=1`，flash 无效时兜底使用模拟数据（见第 10 节）。

### 9.2 耳机为主模式

Lmac / SN / label_sn 来自 VM，不受 flash 影响。Cmac 无有效来源时填全零。

## 10. 模拟开关

`RDX_READCHAR_SIMULATE_ENABLE`（默认 0）用于 flash 无效时的兜底路径。

仓为主模式的模拟数据提供完整的 flash 三字段：

```c
// 格式: auth_key(24),cmac(12),fac_sn(16)
"ZENCHORDC202500000000001,50EE407836FD,ZCE1250000000001"
```

耳机为主模式的模拟数据仅提供 Cmac：

```c
"50EE407836FD"
```

模拟开关不参与正常的配对/解绑逻辑，仅影响冷启动 flash 无效场景。

## 11. 与旧版实现的关键差异

| 方面 | 旧版 | 当前 |
|------|------|------|
| 读特征长度 | 39 或 24 | 95 |
| 读特征内容 | 仅 flash AuthKey | 完整 7 字段 |
| 数据主权 | 单一路径 | 仓为主 / 耳机为主双模式，编译宏切换 |
| SN 来源 | flash AuthKey（旧）或 VM（中） | 宏控制：flash auth_key 或 VM ep_info.sn |
| label_sn 来源 | 无或 VM | 宏控制：flash fac_sn 或 VM ep_info.label_sn |
| flash 解析 | 仅取 Cmac | 完整三段解析（AuthKey + Cmac + FacSN） |
| VM 持久化 | 三元组 (SN + MAC + label_sn) | 仓为主仅 MAC；耳机为主三元组 |
| 模拟数据 | 仅 Cmac | 仓为主三字段；耳机为主 Cmac |
| 逗号解析 | strtok（非线程安全） | 手动实现 |
| 宏分布 | 无 | 收敛到 `rdx_vm_get_sn_and_label_sn()` 单点 |
