# AC701N AI翻译 — 录音模式原理与调用链

> 本文档基于杰理 AC701N TWS耳机 SDK v3.0.0（patch 3.3.4）源码与官方文档整理，聚焦 `AI_TRANSLATOR_MODE_RECORD`（录音模式）的完整数据流与调用链。

---

## 第1章 概述

### 1.1 功能定位

AI翻译是杰理SDK中基于 **RCSP协议** 扩展的一项功能。手机APP通过BLE（或SPP）与耳机建立RCSP连接后，可发送翻译相关指令，控制耳机进入不同翻译模式并传输音频数据。

录音模式（`AI_TRANSLATOR_MODE_RECORD`）的典型场景：
- 用户在APP中点击"录音翻译"
- APP通过BLE下发模式切换指令
- 耳机打开麦克风，采集环境音
- 音频经JLStream Pipeline编码（默认Opus）
- 编码后的音频帧通过BLE回传至APP
- APP将音频发给云端ASR/翻译服务

### 1.2 核心文件

| 层级 | 文件路径 | 职责 |
|------|----------|------|
| 协议层 | `SDK/apps/common/third_party_profile/jieli/rcsp/server/functions/translator/rcsp_translator.c` | RCSP指令解析、封包、发送 |

## 第2章 架构总览

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                        手机APP                               │
│  (通过BLE发送RCSP指令 / 接收编码音频数据)                      │
└──────────────────────────┬──────────────────────────────────┘
                           │ BLE (RCSP Protocol)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Protocol Layer                                            │
│  rcsp_translator.c                                         │
│  • RCSP指令解析：SET_MODE / START_RECORD / STOP_RECORD     │
│  • 数据封包：op03格式 (source/encode/count/CRC/len)        │
│  • 数据发送：JL_DATA_send / JL_CMD_send                    │
└──────────────────────────┬──────────────────────────────────┘
                           │ 调用 ai_translator / ai_recorder API
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  App Layer (ai_translator.c)                               │
│  • 翻译状态机管理（IDLE ↔ RECORD）                          │
│  • 模式切换冲突检测（a2dp/esco运行中）                       │
│  • TWS主从角色约束（从机不执行模式切换）                    │
└──────────────────────────┬──────────────────────────────────┘
                           │ 启动/停止 ai_recorder
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Recorder Layer (ai_recorder.c)                            │
│  • 双通道管理（ch0/ch1）                                    │
│  • 编码参数配置（Opus：16kHz, 16kbps, 20ms帧）              │
│  • 环形缓冲区帧对齐                                         │
│  • 注册AI_TX节点回调到音频Pipeline                          │
│  • TWS状态同步（主机→从机）                                 │
└──────────────────────────┬──────────────────────────────────┘
                           │ 打开/关闭 ai_voice_recoder
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Audio Pipeline (JLStream)                                 │
│  ai_voice_recoder.c 负责构建Pipeline：                      │
│  ┌─────┐    ┌──────┐    ┌─────────┐    ┌───────┐         │
│  │ ADC │───→│ CVP  │───→│ Encoder │───→│ AI_TX │         │
│  │节点 │    │节点  │    │(Opus)   │    │节点   │         │
│  └─────┘    └──────┘    └─────────┘    └───┬───┘         │
│                                             │ 回调        │
└─────────────────────────────────────────────┼──────────────┘
                                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Data Send (回调到 Protocol Layer)                          │
│  recorder_send_for_dev_mic → ai_recorder_data_send         │
│  → 环形buf满帧 → _translator_op_send_audio_data            │
│  → JL_DATA_send → BLE → APP                                │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 各层职责

| 层级 | 核心职责 |
|------|----------|
| **Protocol Layer** | RCSP协议指令的"入口"和"出口"。负责把APP的二进制指令解析为内部API调用，以及把编码后的音频数据按RCSP格式封包发回APP。 |
| **App Layer (ai_translator)** | 翻译业务的"状态机"。管理当前处于哪种翻译模式，处理模式切换时的冲突（如正在播歌/通话），通知APP模式变化。 |
| **Recorder Layer (ai_recorder)** | 录音数据的"调度中心"。管理录音通道的生命周期，维护环形缓冲区做帧对齐，根据TWS主从角色决定数据发给谁。 |
| **Audio Pipeline (JLStream)** | 音频处理的"硬件抽象"。负责把麦克风模拟信号经过ADC采样、CVP降噪、Opus编码，最终从AI_TX节点输出。 |
| **Data Send** | 编码数据的"最后一步"。在环形缓冲区攒够一帧后，封装RCSP op03包头，通过BLE发送。 |

## 第3章 各层模块详解

### 3.1 Protocol Layer（RCSP协议层）

#### 3.1.1 RCSP翻译操作码

定义于 `rcsp_translator.h`：

```c
#define RCSP_TRANSLATOR_OP_GET_MODE_INFO           0x00
#define RCSP_TRANSLATOR_OP_SET_MODE                0x01
#define RCSP_TRANSLATOR_OP_INFORM_MODE_INFO        0x02
#define RCSP_TRANSLATOR_OP_SEND_AUDIO_DATA         0x03
#define RCSP_TRANSLATOR_OP_GET_CACHE_FREE_SIZE     0x04
#define RCSP_TRANSLATOR_OP_INFORM_CACHE_FREE_SIZE  0x05
```

录音模式涉及的关键指令：
- `SET_MODE` (0x01)：切换翻译模式，录音模式对应 `RCSP_TRANSLATOR_MODE_RECORD` (0x01)
- `OpCode 0x04` (独立指令)：开始录音
- `OpCode 0x05` (独立指令)：停止录音

#### 3.1.2 模式与编码映射

RCSP协议层的模式和编码类型与内部AI层不同，通过映射表转换：

```c
// 模式映射（RCSP → AI）
RCSP_TRANSLATOR_MODE_RECORD  →  AI_TRANSLATOR_MODE_RECORD

// 编码映射（RCSP → AUDIO_CODING）
RCSP_TRANSLATOR_ENCODE_TYPE_OPUS  →  AUDIO_CODING_OPUS
```

#### 3.1.3 入口函数

```c
int JL_rcsp_translator_functions(void *priv, u8 OpCode, u8 OpCode_SN,
                                 u8 *data, u16 len, u16 ble_con_handle, u8 *spp_remote_addr);
```

该函数是RCSP协议层收到翻译相关指令时的统一入口。根据 `OpCode` 分发到不同处理逻辑：
- `JL_OPCODE_TRANSLATOR` (0x34)：内部子操作码（SET_MODE / SEND_AUDIO_DATA 等）
- `0x04`：开始录音（独立指令）
- `0x05`：停止录音（独立指令）

#### 3.1.4 音频数据封包格式（op03）

`translator_op_03_audio_format` 是 RCSP `op03` 的包头结构，但它**不用于录音模式**。当前源码中：
- `AI_TRANSLATOR_MODE_RECORD`
- `AI_TRANSLATOR_MODE_RECORD_TRANSLATION`

这两种模式都直接通过 DATA 通道发送编码后的裸音频数据；只有通话翻译、A2DP翻译、面对面翻译等模式才会封装 `op03` 包头后走 CMD 通道。

```c
struct translator_op_03_audio_format {
    u8  source;   // 音频来源：RCSP_TRANSLATOR_AUDIO_SOURCE_DEV_MIC (0x01)
    u8  encode;   // 编码类型：RCSP_TRANSLATOR_ENCODE_TYPE_OPUS (0x02)
    u8  count;    // 计数（倒序，0表示结束包）
    u16 crc;      // 音频数据CRC16
    u16 len;      // 音频数据长度
    // u8 buf[];  // 实际的Opus编码数据
};
```

对应发送路径如下：
- 录音模式 / 录音翻译模式：`JL_DATA_send(JL_OPCODE_DATA, 0x04, buf, len, ...)`
- 其他翻译发送模式：封装 `translator_op_03_audio_format` 后，`JL_CMD_send(JL_OPCODE_TRANSLATOR, param, param_len, ...)`

### 3.2 App Layer（ai_translator 翻译状态机）

#### 3.2.1 状态与数据结构

```c
struct ai_translator_handle {
    u8 init;                   // 是否已初始化
    u8 state;                  // 运行状态（0=停止, 1=运行中）
    u8 call_translate_step;    // 通话翻译步骤标记
    u8 stop_rec_for_call;      // 是否因来电暂停录音
    struct ai_trans_mode mode_info;  // 当前模式信息
    struct ai_translator_ops ops;    // 回调函数（协议层注册）
};
```

#### 3.2.2 翻译模式枚举

```c
enum {
    AI_TRANSLATOR_MODE_IDLE,
    AI_TRANSLATOR_MODE_RECORD,              // 录音模式（本文档重点）
    AI_TRANSLATOR_MODE_RECORD_TRANSLATION,  // 录音翻译（边录边播译文）
    AI_TRANSLATOR_MODE_CALL_TRANSLATION,    // 通话翻译
    AI_TRANSLATOR_MODE_CALL_TRANSLATION_STEREO_ENC,
    AI_TRANSLATOR_MODE_CALL_RECORD_STEREO_ENC,
    AI_TRANSLATOR_MODE_A2DP_TRANSLATION,    // A2DP音频翻译
    AI_TRANSLATOR_MODE_FACE_TO_FACE_TRANSLATION, // 面对面翻译
};
```

#### 3.2.3 模式切换核心逻辑

`ai_translator_set_mode()` 是模式切换的核心函数。对于 **录音模式** (`AI_TRANSLATOR_MODE_RECORD`)：

1. **冲突检测**：检查当前是否在播放A2DP音乐或处于通话ESCO状态
   - 如果A2DP运行中：发送暂停指令 `bt_cmd_prepare(USER_CTRL_AVCTP_OPID_PAUSE, ...)`
   - 如果ESCO运行中：根据配置返回错误或继续
2. **状态标记**：`tlr_hdl.state = 1`
3. **注意**：录音模式在此阶段**不直接启动录音器**，仅设置模式状态。录音的正式启动由独立的 `cmd 0x04` 触发。

#### 3.2.4 TWS主从约束

```c
if (tws_api_get_role() == TWS_ROLE_SLAVE) {
    return -AI_TRANS_SET_MODE_STATUS_FAIL;
}
```

**关键原则**：翻译模式切换只能在 **TWS主机** 上执行。从机通过TWS同步命令被动接收状态。

### 3.3 Recorder Layer（ai_recorder 录音调度中心）

#### 3.3.1 双通道设计

```c
#define AI_AUDIO_CH_MAX  2

struct ai_recorder_handle {
    struct ai_recorder_ch recorder_ch[AI_AUDIO_CH_MAX];  // ch0, ch1
    struct ai_recorder_ops ops;
    u8 init;
    // ...
};
```

录音模式仅使用 **ch0**。ch1用于通话翻译等需要双路录音的场景。

#### 3.3.2 录音通道结构

```c
struct ai_recorder_ch {
    u8 ch;           // 通道号
    u8 state;        // 运行状态
    u8 host_state;   // TWS主机状态（用于同步）
    u8 media_type;   // 媒体类型（录音模式为 AI_AUDIO_MEDIA_TYPE_VOICE）
    u8 tws_rec;      // 是否TWS双端控制
    u8 *cbuf_buf;    // 环形缓冲区指针
    u32 buf_size;    // 缓冲区大小
    u32 offset;      // 当前写入偏移
    OS_MUTEX mutex;  // 互斥锁
    struct ai_audio_format format;  // 编码格式
};
```

#### 3.3.3 环形缓冲区与帧对齐

`ai_recorder_start()` 根据编码参数计算帧大小：

```c
// Opus帧大小计算
fsize = (fmt->bit_rate / 8) * fmt->frame_dms / 10000;
// 例如：16000bps, 20ms(=200dms) → (16000/8)*200/10000 = 40 bytes/帧
```

缓冲区大小按帧大小取整：
```c
size = (200 + fsize / 2) / fsize * fsize;  // 约200字节对齐
```

`ai_recorder_data_send()` 的核心逻辑：
1. 接收AI_TX节点传来的编码数据（长度可能不是完整帧的整数倍）
2. 写入环形缓冲区
3. 当内部写指针达到缓冲区末尾（`rec_ch->offset == buf_size`）时，调用协议层发送回调
4. 如果数据跨边界，处理剩余数据的回环写入

**注意**：这里对齐的是**发送批次大小**，不是“每个Opus帧单独发送”。在默认录音配置下：
- `fsize = 40 bytes/帧`
- `buf_size ≈ 200 bytes`

因此录音模式通常会累计约 5 帧后再交给协议层发送一次。

#### 3.3.4 录音ops回调注册

`ai_recorder_ops` 定义了6种音频来源的发送回调：

```c
struct ai_recorder_ops {
    // 协议层发送接口（TWS主机/从机）
    int (*recorder_send_by_protocol_layer_host)(u8 *buf, u32 len, u32 offset, u32 priv);
    int (*recorder_send_by_protocol_layer_slave)(u8 *buf, u32 len, u32 offset, u32 priv);
    
    // 各音频流页面AI_TX节点的回调
    int (*recorder_send_for_dev_mic)(u8 *buf, u32 len);       // <智能语音>页面
    int (*recorder_send_for_esco_upstream)(u8 *buf, u32 len);  // <蓝牙通话>上行
    int (*recorder_send_for_esco_downstream)(u8 *buf, u32 len);// <蓝牙通话>下行
    int (*recorder_send_for_esco_mix)(u8 *buf, u32 len);       // <蓝牙通话>混合
    int (*recorder_send_for_a2dp)(u8 *buf, u32 len);           // <蓝牙音乐>页面
};
```

录音模式使用的是 `recorder_send_for_dev_mic`，最终映射到 `translator_send_ch_for_dev_mic`。

### 3.4 Audio Pipeline（JLStream 音频流水线）

#### 3.4.1 Pipeline构建

`ai_voice_recoder_open()` 负责构建和启动录音Pipeline：

```c
// 1. 获取Pipeline UUID（由音频流程图工具生成）
u16 uuid = jlstream_event_notify(STREAM_EVENT_GET_PIPELINE_UUID, (int)"ai_voice");

// 2. 解析Pipeline，尝试不同的音频源
recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_ADC);      // 优先ADC
recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_PDM_MIC);  // 其次PDM
recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_IIS0_RX);  // 最后IIS
```

**关键原理**：同一个"ai_voice" Pipeline可能配置了多个输入源节点，代码按优先级尝试，哪个能解析成功就用哪个。

#### 3.4.2 节点配置

**Encoder节点配置（Opus）**：
```c
struct encoder_fmt enc_fmt = {
    .bit_rate = 16000,     // 码率16kbps
    .complexity = 0,       // 编码复杂度（0最低，3最高）
    .sample_rate = 16000,  // 采样率16kHz
    .format = 0,           // 封装格式（0=百度无头格式）
};
jlstream_node_ioctl(stream, NODE_UUID_ENCODER, NODE_IOC_SET_PRIV_FMT, (int)(&enc_fmt));
```

**注意**：源码注释表明 `struct encoder_fmt` 的私有参数里，`complexity`、`format`、`frame_dms` 属于可生效项；但当前录音路径里实际显式赋值的只有 `complexity` 和 `format`，并**没有**给 `frame_dms` 赋值。`bit_rate` 和 `sample_rate` 仍仅作阅读参考，实际采样率、码率、声道、帧长应以音频流程图工具中的 Encoder 节点配置为准。

**AI_TX节点配置**：
```c
struct stream_fmt fmt = {
    .coding_type = AUDIO_CODING_OPUS,
};
jlstream_node_ioctl(stream, NODE_UUID_AI_TX, NODE_IOC_SET_FMT, (int)(&fmt));
```

**SOURCE节点（ADC中断点数）**：
```c
// 默认320采样点
jlstream_node_ioctl(stream, NODE_UUID_SOURCE, NODE_IOC_SET_PRIV_FMT, 320);
// 如果启用了CVP，改为256采样点
jlstream_node_ioctl(stream, NODE_UUID_SOURCE, NODE_IOC_SET_PRIV_FMT, 256);
```

#### 3.4.3 CVP节点连接

CVP（Clear Voice Processing）是通话降噪处理节点。如果Pipeline中存在CVP节点：

```c
u16 node_uuid = get_cvp_node_uuid();
if (node_uuid) {
    // 设置CVP的参考采样率（与DAC采样率一致）
    jlstream_node_ioctl(stream, node_uuid, NODE_IOC_SET_FMT, (int)ref_sr);
    // 设置CVP的输入源（ADC/PDM/IIS）
    jlstream_node_ioctl(stream, node_uuid, NODE_IOC_SET_PRIV_FMT, source_uuid);
}
```

#### 3.4.4 Pipeline启动

```c
// 设置场景（用于音频流程图中的条件编译）
jlstream_set_scene(stream, STREAM_SCENE_AI_VOICE);

// 启动Pipeline
jlstream_start(stream);
```

#### 3.4.5 AI_TX节点回调注册

Pipeline启动后，需要把数据发送回调注册到AI_TX节点：

```c
void ai_voice_recoder_set_ai_tx_node_func(int (*func)(u8 *, u32))
{
    jlstream_node_ioctl(stream, NODE_UUID_AI_TX, NODE_IOC_SET_PRIV_FMT, (int)func);
}
```

录音模式注册的回调是 `recorder_send_for_dev_mic` → `translator_send_ch_for_dev_mic`。

### 3.5 Data Send（数据发送）

#### 3.5.1 发送路径

```
AI_TX节点 → recorder_send_for_dev_mic
    → translator_send_ch_for_dev_mic
        → ai_recorder_data_send(ch=0, buf, len)
            → 环形buf累积
                → buf满 → recorder_send_by_protocol_layer_host
                    → _translator_op_send_audio_data
                        → JL_DATA_send → BLE → APP
```

#### 3.5.2 录音模式的发送特点

在 `_translator_op_send_audio_data()` 中，录音模式走独立分支：

```c
case RCSP_TRANSLATOR_MODE_RECORD:
case RCSP_TRANSLATOR_MODE_RECORD_TRANSLATION:
    ret = JL_DATA_send(JL_OPCODE_DATA, 0x04, buf, len, JL_NOT_NEED_RESPOND, 0, NULL);
    break;
```

**关键区别**：
- 录音模式使用 **DATA通道** (`JL_OPCODE_DATA`, subtype `0x04`) 直接发送裸编码数据
- 其他模式（通话翻译等）使用 **CMD通道** (`JL_OPCODE_TRANSLATOR`, op `0x03`) 发送带包头的数据

**原因**：录音模式产生的数据量大、频率高，使用DATA通道可以避免阻塞RCSP控制通道，且不需要每帧都带包头（APP端已知参数）。

## 第4章 录音模式深度调用链

### 4.1 阶段一：模式设置

**触发条件**：APP下发 `JL_OPCODE_TRANSLATOR` + `RCSP_TRANSLATOR_OP_SET_MODE`，参数 mode=RECORD。

```
[JL_rcsp_translator_functions]
  │ OpCode == JL_OPCODE_TRANSLATOR, op == RCSP_TRANSLATOR_OP_SET_MODE
  ▼
[_translator_op_set_mode(&minfo)]
  │ 1. 将RCSP模式/编码/声道/采样率映射为内部格式
  │    mode: RCSP_TRANSLATOR_MODE_RECORD → AI_TRANSLATOR_MODE_RECORD
  │    encode: RCSP_TRANSLATOR_ENCODE_TYPE_OPUS → AUDIO_CODING_OPUS
  │ 2. 填充 struct ai_trans_mode
  ▼
[ai_translator_set_mode(&ai_minfo)]
  │ 1. 参数校验（NULL检查、相同模式检查）
  │ 2. 状态检查：当前必须为IDLE才能切换到RECORD
  │ 3. TWS角色检查：从机直接返回FAIL
  │ 4. 冲突检测：
  │    - a2dp_player_runing() ? → ai_translator_stop_a2dp_player() 暂停音乐
  │    - esco_player_runing() ? → 根据配置返回IN_CALL或继续
  │ 5. tlr_hdl.state = 1;  // 标记翻译功能已激活
  │ 6. 更新 minfo 到 tlr_hdl.mode_info
  │ 7. bt_sniff_disable();  // 关闭sniff以保证BLE吞吐
  │ 8. TWS同步：主机发送 SYNC_TLR_INFO 到从机
  ▼
[返回成功] → [_translator_op_set_mode] → [JL_CMD_response_send] → APP收到响应
```

**阶段一关键原理**：
- 模式设置**不打开麦克风**，只是为后续录音做准备（停音乐、占资源、同步TWS）
- 从机不执行任何模式切换操作，只被动接收主机的同步信息
- `bt_sniff_disable()` 是为了防止BLE进入低功耗模式影响音频数据传输带宽

### 4.2 阶段二：录音启动

**触发条件**：APP下发 `OpCode = 0x04`（start record）。

```
[JL_rcsp_translator_functions]
  │ OpCode == 0x04
  ▼
[_translator_op_record_start]
  │ 1. 获取当前翻译模式信息 ai_translator_get_mode_info()
  │ 2. 填充 struct ai_audio_format
  │    coding_type = ai_minfo.coding_type  (OPUS)
  │    sample_rate = ai_minfo.sr           (16000)
  │    bit_rate    = 16000
  │    channel     = ai_minfo.ch           (1)
  │    frame_dms   = 200                   (20ms)
  ▼
[ai_recorder_start(ch=0, &format, AI_AUDIO_MEDIA_TYPE_VOICE, tws_rec=0)]
  │ 1. 通道校验（ch < AI_AUDIO_CH_MAX）
  │ 2. 状态检查（防止重复启动）
  │ 3. 计算帧大小：fsize = (16000/8) * 200/10000 = 40 bytes
  │ 4. 计算缓冲区大小：size = round_up_to_frame(200) ≈ 200 bytes (5帧)
  │ 5. malloc(cbuf_buf)
  │ 6. ai_recorder_try_to_set_func(rec_ch)
  │    │ media_type == AI_AUDIO_MEDIA_TYPE_VOICE
  │    │ → ai_voice_recoder_open(OPUS, 0)
  │    │ → ai_voice_recoder_set_ai_tx_node_func(recorder_send_for_dev_mic)
    │    │ → 直接打开录音Pipeline并注册AI_TX回调
    │    │ → 本路径不会进入A2DP/ESCO场景的20ms延迟重试分支
  │ 7. rec_ch->state = 1
  │ 8. TWS同步：主机发送 SYNC_REC_INFO 到从机
  ▼
[ai_voice_recoder_open]
  │ 1. 获取Pipeline UUID：jlstream_event_notify(STREAM_EVENT_GET_PIPELINE_UUID, "ai_voice")
  │ 2. 解析Pipeline：
  │    jlstream_pipeline_parse(uuid, NODE_UUID_ADC)      // 尝试ADC
  │    jlstream_pipeline_parse(uuid, NODE_UUID_PDM_MIC)  // 尝试PDM
  │    jlstream_pipeline_parse(uuid, NODE_UUID_IIS0_RX)  // 尝试IIS
  │ 3. 配置Encoder节点（Opus参数）
  │ 4. 配置AI_TX节点（coding_type = AUDIO_CODING_OPUS）
  │ 5. 配置SOURCE节点（ADC中断点数：320或256）
  │ 6. 如果存在CVP节点，配置其参考采样率和输入源
  │ 7. jlstream_set_callback(stream, ..., ai_voice_recoder_callback)
  │ 8. jlstream_set_scene(stream, STREAM_SCENE_AI_VOICE)
  │ 9. jlstream_start(stream)  ← Pipeline开始运行！
  ▼
[返回成功] → APP收到 cmd 0x04 的响应
```

**阶段二关键原理**：
- 模式设置和录音启动是**分离的两个步骤**。这是为了支持"先设模式、后按需启停录音"的交互逻辑
- 录音模式对应的 `AI_AUDIO_MEDIA_TYPE_VOICE` 路径会直接调用 `ai_voice_recoder_open()`；如果打开失败，本次启动直接返回错误，不会在本模式下做20ms轮询重试
- 20ms 延迟重试是 `ai_recorder_try_to_set_func()` 给 A2DP / ESCO 等 `done == 0` 场景准备的兜底逻辑，不是录音模式这条 VOICE 路径的常规行为
- `jlstream_start()` 之后，音频中断开始触发，数据流自动运转

### 4.3 阶段三：音频数据采集与编码

**触发条件**：硬件ADC中断，或DMA完成中断。

```
[硬件ADC中断]
  ▼
[JLStream Pipeline 自动调度]
  │ 音频数据在各节点间流转：
  │
  │ ┌─────────┐   ┌─────────┐   ┌─────────────┐   ┌─────────┐
  │ │ SOURCE  │──→│  CVP    │──→│  ENCODER    │──→│  AI_TX  │
  │ │(ADC节点)│   │(降噪)   │   │(Opus编码)   │   │(输出)   │
  │ └─────────┘   └─────────┘   └─────────────┘   └────┬────┘
  │                                                    │
  │                                                    ▼
  │                                          调用注册的回调函数
  │                                          int func(u8 *buf, u32 len)
  ▼
[recorder_send_for_dev_mic]  // = translator_send_ch_for_dev_mic
  │ ch = source_to_ch_remap(RCSP_TRANSLATOR_AUDIO_SOURCE_DEV_MIC) = 0
  ▼
[ai_recorder_data_send(ch=0, buf, len, offset=0, priv=DEV_MIC)]
  │ 1. 检查充电状态（充电中不发送）
  │ 2. 获取通道互斥锁
  │ 3. 如果rec_ch->state == 0，直接返回
  │ 4. 缓冲区溢出检查
  │ 5. memcpy(rec_ch->cbuf_buf + rec_ch->offset, buf, len)
  │ 6. rec_ch->offset += len
    │ 7. 如果 rec_ch->offset == buf_size（缓冲区满）：
    │    a. rec_ch->offset 清零（回环）
  │    b. 判断TWS角色：
  │       - 主机：调用 recorder_send_by_protocol_layer_host(cbuf_buf, buf_size, ...)
  │       - 从机：调用 recorder_send_by_protocol_layer_slave(...)
  │    c. 如果有剩余数据（left_len），写入缓冲区开头
  ▼
[_translator_op_send_audio_data]
  │ 当前模式 = RCSP_TRANSLATOR_MODE_RECORD
  │ → JL_DATA_send(JL_OPCODE_DATA, 0x04, buf, len, ...)
```

**阶段三关键原理**：
- **Pipeline是全自动运行的**。`jlstream_start()` 之后，从ADC采样到编码输出完全由中断和DMA驱动，不需要APP层主动干预
- **AI_TX节点是数据出口**。它本质上是一个回调钩子，当编码器输出一帧数据时，AI_TX节点调用注册的函数把数据推出去
- **环形缓冲区做发送批次聚合**。AI_TX每次回调的数据长度可能不是固定发送长度，`ai_recorder_data_send` 会先写入环形缓冲区；在默认录音配置下，通常累计到约 `200 bytes`（约 5 帧）后再统一交给协议层发送，而不是每 `40 bytes` 一发

### 4.4 阶段四：数据发送

```
[_translator_op_send_audio_data(buf, len, offset, priv)]
  │ 根据当前翻译模式选择发送方式：
  │
  │ case RCSP_TRANSLATOR_MODE_RECORD:
  │    // 录音模式使用DATA通道直接发送
  │    JL_DATA_send(JL_OPCODE_DATA, 0x04, buf, len,
  │                 JL_NOT_NEED_RESPOND, 0, NULL);
  │    break;
  ▼
[BLE协议栈 → 射频发送]
  ▼
[APP收到音频数据]
```

**阶段四关键原理**：
- `JL_DATA_send` 使用的是 **RCSP DATA通道**。与CMD通道不同，DATA通道：
  - 不需要ACK响应（`JL_NOT_NEED_RESPOND`），降低延迟
  - 独立的传输队列，不会阻塞控制指令
  - subtype `0x04` 标识这是AI翻译录音数据
- 录音模式**不封装op03包头**，直接发送裸Opus帧。因为APP在模式设置阶段已经知道了编码参数，不需要每帧都带元数据

## 第5章 关键原理与机制

### 5.1 TWS同步机制

#### 5.1.1 为什么需要TWS同步

在TWS场景下，左右耳都可能采集到用户语音。AI翻译录音只需要 **一份** 音频数据（通常来自主耳）。SDK通过TWS同步机制确保：
- 只有主耳执行录音和发送
- 从耳保持静默，不重复发送
- 主从切换时（如把右耳放入充电盒），录音状态正确转移

#### 5.1.2 同步命令

`ai_recorder.c` 中定义了4种TWS同步命令：

```c
enum {
    TWS_TRANS_CMD_INIT,         // 初始化
    TWS_TRANS_CMD_DEINIT,       // 注销
    TWS_TRANS_CMD_SET_OPS,      // 同步ops回调指针
    TWS_TRANS_CMD_SYNC_REC_INFO,// 同步录音通道状态
};
```

录音启动时，主机发送 `SYNC_REC_INFO`：
```c
ai_recorder_tws_send_cmd(TWS_TRANS_CMD_SYNC_REC_INFO, rec_ch->ch, rec_ch, sizeof(struct ai_recorder_ch));
```

从机收到后：
```c
case TWS_TRANS_CMD_SYNC_REC_INFO:
    // 1. 复制主机的通道状态
    rec_ch->host_state = rec_ch_src.host_state;
    rec_ch->media_type = rec_ch_src.media_type;
    rec_ch->tws_rec    = rec_ch_src.tws_rec;
    rec_ch->format     = rec_ch_src.format;
    
    // 2. 如果 tws_rec == 1，同步启动/停止录音
    if (rec_ch->tws_rec) {
        if (rec_ch->host_state) {
            ai_recorder_start(ch, &format, media_type, tws_rec);
        } else {
            ai_recorder_stop(ch);
        }
    }
```

#### 5.1.3 主从角色对发送路径的影响

`ai_recorder_data_send()` 中的角色判断：

```c
role = tws_api_get_role();
if (role == TWS_ROLE_MASTER) {
    recorder_hdl.ops.recorder_send_by_protocol_layer_host(...);
} else if (role == TWS_ROLE_SLAVE) {
    recorder_hdl.ops.recorder_send_by_protocol_layer_slave(...);
}
```

在录音模式下，`recorder_send_by_protocol_layer_slave` 被设置为 `NULL`，意味着**从机不会发送数据**。这是正确的，因为翻译音频只需要一份。

### 5.2 环形缓冲区与流控

#### 5.2.1 缓冲区大小设计

录音模式的缓冲区设计遵循以下原则：
- **帧对齐**：缓冲区大小必须是帧大小的整数倍
- **低延迟**：默认约200字节（5帧@40字节/帧），缓存约100ms数据
- **防溢出**：如果写入长度超过剩余空间，直接丢弃溢出数据（打印 `trans send overflow`）

#### 5.2.2 流控策略

录音模式的流控相对简单：
- 编码器以固定速率产生数据（20ms/帧）
- BLE发送速率取决于射频带宽和APP接收能力
- 如果BLE发送慢于编码产生速度，缓冲区会持续累积
- 当缓冲区满时，新数据被丢弃，打印溢出日志

**注意**：录音模式没有使用 `player_inform_cache_free_size` 机制（这是播放侧的流控）。录音侧的处理是"尽力发送，溢出丢弃"。

### 5.3 Opus编码配置来源

录音模式使用Opus编码，其参数来自多处，容易混淆：

| 参数 | 代码中的值 | 实际生效位置 | 说明 |
|------|-----------|-------------|------|
| **采样率** | 16000 Hz | 音频流程图工具的Encoder节点 | 代码中设置仅作参考 |
| **码率** | 16000 bps | 音频流程图工具的Encoder节点 | 代码中设置仅作参考 |
| **帧长** | 20ms (200 dms) | 当前录音路径未显式写入 Encoder 私有参数；实际编码帧长应以音频流程图工具的Encoder节点为准 | `ai_audio_format.frame_dms` 仍会被 `ai_recorder_start` 用来计算理论帧大小与发送批次 |
| **声道** | 1 (单声道) | 音频流程图工具的Encoder节点 | 代码中设置仅作参考 |
| **complexity** | 0 | `ai_voice_recoder_open` 中通过 `NODE_IOC_SET_PRIV_FMT` 设置 | **实际生效** |
| **format** | 0 (百度无头) | `ai_voice_recoder_open` 中通过 `NODE_IOC_SET_PRIV_FMT` 设置 | **实际生效** |

**关键理解**：
- 编码器的核心参数（采样率、码率、帧长）在 **音频流程图工具** 中配置，编译进 `stream.bin`
- 当前录音路径中，代码通过 `jlstream_node_ioctl(..., NODE_IOC_SET_PRIV_FMT, ...)` 显式设置的是 `complexity` 和 `format`；源码注释虽然把 `frame_dms` 也列为可生效私有参数，但这里没有实际赋值
- `bit_rate` 和 `frame_dms` 在 `ai_recorder_start` 中会参与帧大小与缓冲区大小计算，用于决定本地发送批次大小，**不直接决定当前 Encoder 节点的实际编码码率/帧长**

### 5.4 音频Pipeline节点连接

#### 5.4.1 Pipeline来源

`jlstream_event_notify(STREAM_EVENT_GET_PIPELINE_UUID, "ai_voice")` 获取的Pipeline定义在 **音频流程图工具** 中。对应 `src/音频流程/智能语音.x6flow`（或类似文件）。

#### 5.4.2 节点自动选择逻辑

```c
// 按优先级尝试音频源
recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_ADC);      // 模拟麦克风
if (!recoder->stream) {
    recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_PDM_MIC);  // PDM数字麦
    source_uuid = NODE_UUID_PDM_MIC;
}
if (!recoder->stream) {
    recoder->stream = jlstream_pipeline_parse(uuid, NODE_UUID_IIS0_RX);  // IIS输入
    source_uuid = NODE_UUID_IIS0_RX;
}
```

**原理**：同一个Pipeline UUID可能包含多个并行的输入源分支。`jlstream_pipeline_parse(uuid, source_node)` 从指定的源节点开始解析，如果该源节点在Pipeline中存在，则返回可用的stream句柄。

#### 5.4.3 CVP节点的作用

CVP（Clear Voice Processing）节点负责：
- 回声消除（AEC）
- 降噪（NS）
- 自动增益控制（AGC）

在录音模式下，CVP节点默认插入在ADC和Encoder之间：
```
ADC → CVP → Encoder → AI_TX
```

`get_cvp_node_uuid()` 返回CVP节点的UUID。如果Pipeline中存在CVP节点，代码会：
1. 设置CVP的参考采样率（与DAC输出一致，用于AEC）
2. 设置CVP的输入源（告诉CVP数据来自ADC还是PDM）
3. 将ADC中断点数从320改为256（CVP处理要求的采样点数）

### 5.5 异常处理

#### 5.5.1 来电打断

`ai_translator.c` 注册了蓝牙事件处理器：

```c
APP_MSG_HANDLER(ai_translator_bt_event) = {
    .from = MSG_FROM_BT_STACK,
    .handler = ai_translator_bt_event_handler,
};
```

当来电时（`BT_STATUS_SCO_CONNECTION_REQ`）：

```c
case BT_STATUS_SCO_CONNECTION_REQ:
    // 如果当前在录音模式（非通话翻译），暂停录音
    if (tlr_hdl.mode_info.mode == AI_TRANSLATOR_MODE_RECORD) {
        // 停止录音
        ai_recorder_stop(0);
        tlr_hdl.stop_rec_for_call = 1;  // 标记：因来电暂停
        
#if AI_AUDIO_TRANSLATION_CALL_CONTROL_BY_DEVICE
        // 如果配置为耳机控制，直接退出翻译模式
        minfo.mode = AI_TRANSLATOR_MODE_IDLE;
        ai_translator_set_mode(&minfo);
        // 通知APP
        tlr_hdl.ops.translator_inform_mode_info(&minfo);
#endif
    }
```

通话结束后（`BT_STATUS_SCO_DISCON`）：
```c
case BT_STATUS_SCO_DISCON:
    // 如果之前因来电暂停了录音，恢复录音
    if (tlr_hdl.stop_rec_for_call) {
        tlr_hdl.stop_rec_for_call = 0;
        ai_recorder_start(0, &format, AI_AUDIO_MEDIA_TYPE_VOICE, 0);
    }
```

#### 5.5.2 充电状态

`ai_recorder.c` 注册了电池事件处理器：

```c
case CHARGE_EVENT_LDO5V_IN:   // 入舱充电
    recorder_hdl.charge_state = 1;  // 暂停发数
    break;
case CHARGE_EVENT_LDO5V_OFF:  // 出舱
    recorder_hdl.charge_state = 0;  // 恢复发数
    break;
```

`ai_recorder_data_send()` 开头检查：
```c
if (recorder_hdl.charge_state) {
    return 0;  // 充电中直接丢弃数据
}
```

**原理**：入舱充电时TWS主从可能切换，暂停发数避免发送混乱。

#### 5.5.3 A2DP音乐冲突

进入录音模式前，如果正在播放蓝牙音乐：

```c
if (a2dp_player_runing()) {
    ai_translator_stop_a2dp_player();  // 发送AVRCP暂停指令
}
```

`ai_translator_stop_a2dp_player()` 会定时检测A2DP是否真正停止：
```c
static void ai_translator_stop_a2dp_confirm(void *arg)
{
    if (0 == a2dp_player_runing()) {
        sys_timer_del(tlr_hdl.detect_timer_id);
        tlr_hdl.detect_timer_id = 0;
    }
}
```
