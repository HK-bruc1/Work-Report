# WiFi MAC 地址来源分析

## 1. 来源链路

`rdx_app_earphone_pack_readchardata()` 中的 WiFi MAC 地址，**来自 ESP8684 (ESP32-C2) 模组自身的出厂硬件 MAC**，主控代码不做任何计算或修改，直接透传。

```
pack_readchardata()                           [rdx_app.c:715]
  └── xxp_uart_get_wifi_AP_info()             [rdx_app.c:723, 声明为 extern]
        └── 实现在 librdxApp.a (闭源库)
              └── rdx_esp32_spi_send_data()    [rdx_spi.c:703]
                    └── SPI 总线 ───────────── [rdx_spi.c:68-102, 引脚定义]
                          └── ESP8684 模组
                                └── 模组出厂 MAC (固件上报)
```

### 逐层说明

| 层级 | 位置 | 说明 |
|------|------|------|
| **调用入口** | `rdx_app.c:723` | `ApInfo* p = xxp_uart_get_wifi_AP_info();` |
| **数据载体** | `xxpUart.h:72-78` | `ApInfo.mac_bytes[6]` — 6 字节原始数组 |
| **闭源库** | `librdxApp.a` | 封装了与模组的 SPI 通信协议，负责解析模组上报数据并填充 `ApInfo` |
| **SPI 驱动** | `rdx_spi.c:68-102,703` | 开源层，引脚 `MOSI=PA6, SCLK=PA5, CS=PB4, HANDSHAKE=PA3` |
| **模组硬件** | ESP8684 (ESP32-C2) | MAC 由模组固件在初始化时通过 SPI 协议上报给主控 |

### 架构图

```
┌──────────────────────────────────────────────────────┐
│  BR28 主控 (AC701N)                                   │
│                                                      │
│  rdx_app_earphone_pack_readchardata()                 │
│    │                                                 │
│    ├── rdx_vm_get_auth_info()  ──► auth_info         │
│    ├── rdx_vm_get_ep_info()    ──► 已配对耳机 MAC    │
│    └── xxp_uart_get_wifi_AP_info()                   │
│          │                                           │
│          ├── librdxApp.a (闭源库)                     │
│          │     └── rdx_spi.c (开源 SPI 驱动)          │
│          │           └── rdx_esp32_spi_send_data()   │
│          │                                           │
└──────────┼───────────────────────────────────────────┘
           │
    SPI 总线 (MOSI=PA6, SCLK=PA5, CS=PB4, HANDSHAKE=PA3)
           │         引脚定义: rdx_spi.c:68-102
           │
┌──────────┼───────────────────────────────────────────┐
│  ESP8684 (ESP32-C2) WiFi 模组                         │
│                                                      │

```

> **边界条件**（`rdx_app.c:748`）：若 WiFi 模块未初始化导致 `xxp_uart_get_wifi_AP_info()` 返回 NULL，`wifi_mac` 保持零值，拼接时输出 `000000000000`。

> **关于 `WIFI_CTRL_BUS_SELECT`**：`rdx_app_config.h:137` 定义了该宏为 `WIFI_CTRL_BUS_SPI`，但全库无任何代码引用它，属历史遗留。判断当前系统使用 SPI 的可靠证据是 `rdx_spi.c` 的引脚定义和函数实现。

## 2. 在 BLE readchar 中的使用

该函数仅在**耳机仓模式**（`RDX_PRODUCT_IS_CHARGE_CASE = 1`）下写入 WiFi MAC。

### 字段清单与数据来源

每个字段的**存储位置和获取方式**不同，这是理解整个 readchar 机制的关键：

```
C,<ep_mac_L>,<ep_mac_R>,<case_mac>,<auth_key>,<label_sn>,<wifi_mac>
```

| 字段 | 来源 | 存储介质 | 说明 |
|------|------|----------|------|
| `C` | 硬编码 | — | 角色标识，无存储 |
| `ep_mac_L` | `rdx_vm_get_ep_info()` | **VM（Flash）** | 耳机绑定时通过 BLE 传递过来，调用 `rdx_vm_write_ep_info_intoVM()` 持久化保存，重启不丢失 |
| `ep_mac_R` | 计算派生 | — | `ep_mac_L` 末字节 +1，无独立存储 |
| `case_mac` | `rdx_vm_get_auth_info()` | **Flash（出厂烧录）** | 本机 BLE MAC，量产时写入 Flash，初始化时由 `rdx_vm_auth_info_init()` 读取并同时设置到 BLE 协议栈 |
| `auth_key` | `rdx_vm_get_auth_info()` | **Flash（出厂烧录）** | 24 字符授权码，量产时写入 Flash，读取路径同上 |
| `label_sn` | `rdx_vm_get_auth_info()` | **Flash（出厂烧录）** | 16 字符标签序列号，量产时写入 Flash，读取路径同上 |
| `wifi_mac` | `xxp_uart_get_wifi_AP_info()` | **RAM 缓存（不落盘）** | ESP8684 模组出厂 MAC，**每次冷启动时**通过 SPI 从模组实时读取，不经过 Flash/VM，掉电丢失 |

> **关键区分**：
> - **Flash 字段**（`auth_key` / `case_mac` / `label_sn`）：量产时工厂一次性烧录，通过 `rdx_vm_read_product_info_from_flash()` 解析 CSV 格式的出厂数据
> - **VM 字段**（`ep_mac_L`）：运行时绑定行为产生，通过 VM（Flash 上的 key-value 持久化系统）保存
> - **实时字段**（`wifi_mac`）：每次冷启动从 ESP8684 模组重新获取，仅缓存在 RAM 的 `ApInfo` 结构体中

### 拼接代码

WiFi MAC 按**高字节在前（大端）**转为 12 位 HEX 字符串（`rdx_app.c:784-787`）：

```c
sprintf(temp + len, ",%02X%02X%02X%02X%02X%02X",
        wifi_mac[0], wifi_mac[1], wifi_mac[2],
        wifi_mac[3], wifi_mac[4], wifi_mac[5]);
```

> 注意：同函数中 BLE/耳机 MAC 使用低字节在后（小端，`mac[5]` 在前），与 WiFi MAC 顺序相反。这是与手机 APP 端的解析约定。

### 其他消费场景

| 场景 | 文件:行号 | 格式 |
|------|-----------|------|
| **设备基础信息** | `rdx_app.c:836,853,860` | `rdx_app_get_dev_base_info()` 统一组装 `devBaseInfo.wifi_mac` / `wifi_mac_str`，供下方各模块复用 |

## 3. 总结

- **WiFi MAC 来源**：ESP8684 模组的出厂硬件 MAC，由模组固件初始化时通过 SPI 上报，`librdxApp.a` + `rdx_spi.c` 接收并缓存在 RAM 的 `ApInfo.mac_bytes[6]`，不经过 Flash/VM
- **其他字段来源**：`auth_key` / `case_mac` / `label_sn` 量产时烧录到 Flash；`ep_mac_L` 绑定时通过 BLE 获取后存入 VM
- **透传**：WiFi MAC 6 字节原始数组，主控代码不做任何计算或修改
- **字节序**：序列化时采用大端序（与 BLE MAC 的小端序相反）
- **容错**：模组未初始化时 WiFi MAC 回退为 `000000000000`
- **共享**：同一 WiFi MAC 还被 QR 码、SPP 协议、SSID 后缀生成、动态密码派生等场景消费
