# BLE 名称与 BLE 广播包长度的关系

## 1. 背景

2026-06-11 发现一个 BLE 名称 buffer 溢出导致的连锁 Bug：
- BLE 名称显示乱码（APP 端读到垃圾数据）
- 配对后无法重连（手机端 GATT 特征值读取异常）

根因是 `BLE_LOCAL_NAME_MAX_LEN` 宏定义过小，`sprintf` 写入时 null 终止符溢出。

---

## 2. BLE 广播包硬限制

BLE 4.x legacy advertising 的广播数据（ADV_IND / SCAN_RSP）**最大 31 字节**，这是蓝牙协议规范的硬限制，不可突破。

每个 AD Structure 格式：
```
[Length] [Type] [Data...]
  1B      1B     NB
```

典型广播数据构成（以当前设备为例）：
```
Flags:           02 01 0A              → 3 字节
Complete Name:   13 09 "Zenchord Case" → 20 字节 (2+18)
Manufacturer:    放在 SCAN_RSP 中
总计 ADV:        23 字节 (< 31)
```

---

## 3. 三层限制宏

### 3.1 宏定义一览

| 宏名 | 值 | 定义位置 | 作用 |
|------|---|---------|------|
| `BLE_LOCAL_NAME_MAX_LEN` | 24 | `rdx_ble_server.h` | RDX 协议层 BLE 名称 buffer/VM 存储上限 |
| `ADV_RSP_PACKET_MAX` | 31 | `le_user.h` (BLE 栈) | BLE 广播/扫描响应包最大字节数 |
| `BT_NAME_LEN_MAX` | 29 | `le_user.h` (BLE 栈) | **经典蓝牙（EDR）名称长度限制**，非 BLE |

### 3.2 BLE 名称的真正限制

BLE 广播包最大 31 字节，名称需要占用：
```
Flags:           3 字节 (02 01 0A)
Complete Name:   2 字节头 + N 字节名称
总计:            5 + N ≤ 31
因此:            N ≤ 26 字节
```

**约束关系：**
```
BLE_LOCAL_NAME_MAX_LEN (24) ≤ 26 ≤ ADV_RSP_PACKET_MAX (31)
```

当前值 24 满足约束。

### 3.3 关于 `BT_NAME_LEN_MAX` 的澄清

**`BT_NAME_LEN_MAX = 29` 是经典蓝牙（EDR）的名称长度限制**，不是 BLE 的。

在 BLE 场景中：
- `ADV_RSP_PACKET_MAX = 31` 是 BLE 广播包的协议硬限制
- BLE 名称实际可用空间 = 31 - 3(Flags) - 2(Name头) = **26 字节**
- `BT_NAME_LEN_MAX` 在 BLE 代码中被引用，但它是为经典蓝牙设计的

在 RCSP 方案中，`BT_NAME_LEN_MAX` 被用来限制 GAP 名称长度，但这只是代码复用，不是 BLE 协议要求。

---

## 4. RCSP 与 RDX 的区别

### 4.1 RCSP（杰理标准方案）

RCSP 使用 BLE 栈的标准 GAP 名称路径：

```c
// ble_rcsp_server.c
gap_device_name = (char *)bt_get_local_name();  // 从栈获取名称
gap_device_name_len = strlen(gap_device_name);
if (gap_device_name_len > BT_NAME_LEN_MAX) {    // 用 BT_NAME_LEN_MAX 截断（经典蓝牙限制）
    gap_device_name_len = BT_NAME_LEN_MAX;
}
```

- 名称由 BLE 栈统一管理（`bt_get_local_name()`）
- 限制使用 `BT_NAME_LEN_MAX`（29）—— **注意：这是经典蓝牙的限制，不是 BLE 的**
- 广播数据由栈内部的 `make_eir_packet_data` 拼装
- **不需要自定义 `BLE_LOCAL_NAME_MAX_LEN`**

### 4.2 RDX（自定义方案）

RDX 绕过了 BLE 栈的标准 GAP 路径，自己管理名称和广播数据：

```c
// rdx_ble_server.c — 自己管理名称
char* rdx_ble_server_get_local_name(void) {
    syscfg_read(VM_RDX_BLE_NAME, tmp, BLE_LOCAL_NAME_MAX_LEN);  // 从 VM 读
    strncpy(g_rdx_ble_server_info.ble_local_name, tmp, strlen(tmp));
    return g_rdx_ble_server_info.ble_local_name;
}

// rdx_ble_server.c — 自己拼广播数据
static u8 rdx_ble_server_fill_adv_data(u8 *adv_data) {
    const char *name_p = rdx_ble_server_get_local_name();
    int name_len = strlen(name_p);
    if(name_len > BLE_LOCAL_NAME_MAX_LEN){      // 用 BLE_LOCAL_NAME_MAX_LEN 截断
        name_len = BLE_LOCAL_NAME_MAX_LEN;
    }
    offset += make_eir_packet_val(..., HCI_EIR_DATATYPE_FLAGS, 0x0A, 1);
    offset += make_eir_packet_data(..., HCI_EIR_DATATYPE_COMPLETE_LOCAL_NAME, name_p, name_len);
    if (offset > ADV_RSP_PACKET_MAX) {           // 最终校验广播包总大小
        r_printf("***rsp_data overflow!!!!!!\n");
        return 0;
    }
}
```

- 名称存储在自定义 VM key 中，不走栈的 `bt_get_local_name()`
- 限制使用 `BLE_LOCAL_NAME_MAX_LEN`（24），**直接限制，不经过经典蓝牙的 `BT_NAME_LEN_MAX`**
- 广播数据自己拼，最后用 `ADV_RSP_PACKET_MAX` 兜底校验

### 4.3 对比

| 维度 | RCSP | RDX |
|------|------|-----|
| 名称来源 | `bt_get_local_name()`（栈管理） | `syscfg_read(VM_RDX_BLE_NAME)`（自管理） |
| 名称限制宏 | `BT_NAME_LEN_MAX` (29) — **经典蓝牙限制** | `BLE_LOCAL_NAME_MAX_LEN` (24) — **BLE 专用限制** |
| 广播数据拼装 | 栈内部完成 | `fill_adv_data()` 自己拼 |
| 溢出检查 | 栈内部处理 | `ADV_RSP_PACKET_MAX` 兜底 |
| GAP 名称读写 | 栈统一处理 read/write callback | 自定义 `rdx_ble_server_att_read_callback` |

## 5. 实际 Bug 分析

### 5.1 问题描述

`BLE_LOCAL_NAME_MAX_LEN` 原值为 18，而 `"Zenchord Case 60BC"` 恰好 18 字符。

### 5.2 溢出过程

```c
#define BLE_LOCAL_NAME_MAX_LEN  18
char ble_local_name[18];  // buffer 刚好 18 字节

memset(ble_local_name, 0, 18);
sprintf(ble_local_name, "%s %s", "Zenchord Case", "60BC");
// "Zenchord Case 60BC" = 18 字符 + '\0' = 19 字节
// null 终止符写到 ble_local_name[18]，越界 1 字节！
```

### 5.3 连锁影响

**影响 1：GATT 特征值读取返回垃圾数据（已连接状态下）**

配对完成后，手机通过 GATT 读取 GAP Name 特征值（handle 0x0003）。此时 `get_local_name()` 从 VM 读回 18 字节（无 null 终止符），`strlen()` 扫描越界：

```
实际字符串: "Zenchord Case 60BC" (18 字节)
strlen 返回: 32（扫描到内存中下一个 0x00）
att_value_len: 32
手机读到: "Zenchord Case 60BC" + 14 字节垃圾 = 32 字节
```

日志证据（NRF 抓取，设备已连接状态）：
```
read_callback, handle= 0x0003
att_value_len = 32
read gap_name: Zenchord Case 60BC??????=????((    ← 乱码
```

**注意：此时 BLE 连接已经建立**，乱码发生在 GATT 特征值读取阶段，不是广播阶段。

**影响 2：配对后无法重连（由影响 1 直接导致）**

日志分析（COM3_2026-06-11_11-28-59.log）：

**配对过程（溢出触发点）：**
```
[00:02:50.790] *APP#devpair#GSPK99012266A1M00...
[00:02:50.792] _rdx_protocol_parse_device_pair --> auth_code: GSPK99012266A1M0005060BC
[00:02:50.796] rdx_app_device_pair_handle --> get earphone mac address success
[00:02:50.798] rdx_app_earphone_pack_readchardata --> readchardata: C,7878359A614E,...
[00:02:50.801] rdx_ble_server_reset_local_name --> AuthKey:GSPK99012266A1M0005060BC
                ↑ 这里触发溢出！AuthKey 后 4 位是 "60BC"
                BLE名称 = "Zenchord Case 60BC" = 18 字符 + null = 19 字节
                buffer 只有 18 字节，null 终止符溢出 1 字节
```

**第一次连接断开：**
```
11:29:24  设备断开连接 (reason 0x13: Remote User Terminated)
          手机读取 GAP Name 时收到 32 字节垃圾数据
          手机缓存了损坏的 GATT 服务表
```

**第二次连接（失败）：**
```
11:29:42  BLE连接建立
11:29:42  no handle 0x0050  ← 关键错误！
          手机使用缓存的旧 GATT 表，访问不存在的句柄
```

**因果链：**
1. 配对过程中，APP 发送 devpair 命令
2. `rdx_app_device_pair_handle()` 调用 `rdx_ble_server_reset_local_name()`
3. `rdx_ble_server_reset_local_name()` 用 `sprintf` 写入 18 字节 buffer，实际 19 字节（18 字符 + null）
4. null 终止符溢出，踩坏相邻内存
5. 手机读取 GAP Name 特征值（handle 0x0003）时，`strlen()` 越界返回 32
6. 设备返回 32 字节垃圾数据给手机
7. 手机 BLE 栈收到异常数据，判定 GATT 特征值损坏
8. 手机主动断开连接（reason 0x13）
9. **手机缓存了损坏的 GATT 服务表**（包含无效句柄 0x0050）
10. 手机尝试重连，使用缓存的旧 GATT 表发送 ATT 请求
11. 设备端不存在句柄 0x0050，返回 `no handle` 错误
12. 手机无法完成 GATT 服务发现，连接卡住

**影响 3：上电后连接NRF获取GATT仍有乱码（VM 中名称无 null 终止符）**

配对时溢出导致 VM 中存储的名称没有 null 终止符：
```
配对时：sprintf 写入 19 字节到 18 字节 buffer → null 终止符溢出
写入 VM：syscfg_write 只写 18 字节 → VM 中无 null 终止符
```

上电后从 VM 读取：
```
rdx_ble_server_get_local_name():
  tmp[19] 初始化为 0
  syscfg_read 读取 18 字节 → tmp[18] = 0（null 在 tmp 内部）
  strncpy(ble_local_name, tmp, 18) → 只复制 18 字节
  ble_local_name[18] 未初始化 → 没有 null 终止符！
```

GATT 读取时：
```
rdx_ble_server_att_read_callback:
  gap_name = rdx_ble_server_get_local_name()
  att_value_len = strlen(gap_name) → 扫描越界，返回错误长度
  手机读到垃圾数据
```

**日志证据：**
```
[00:02:50.801] rdx_ble_server_reset_local_name --> AuthKey:GSPK99012266A1M0005060BC
                ↑ 配对时写入，null 终止符溢出
上电后连接 NRF 读取 GAP Name → 乱码（strlen 越界）
```

**影响 4：重新上电后恢复正常**

重新上电后：
- **上电时不调用 `rdx_ble_server_reset_local_name()`**，不会触发新的溢出
- `rdx_ble_server_get_local_name()` 从 VM 读取名称（配对时已写入）
- 但由于 VM 中无 null 终止符，`g_rdx_ble_server_info.ble_local_name` 仍可能无 null 终止符
- 手机读取 GAP Name 时仍可能返回垃圾数据
- 手机缓存损坏的 GATT 服务表，重连失败

**关键点：**
1. 溢出只在配对时触发（调用 `rdx_ble_server_reset_local_name()`）
2. 上电后虽然不触发新溢出，但 VM 中已存储无 null 终止符的名称
3. 手机读取 GATT 时仍会返回垃圾数据，导致连接失败

**关键发现：名称乱码和无法连接是同一个 bug 的两个表现**，修复 buffer 溢出后两个问题同时解决。

**影响 4：结构体内存越界**

`ble_local_name` 是 `rdx_ble_server_info_t` 的最后一个字段，null 终止符溢出到结构体后面的内存，可能踩到相邻变量。

### 5.4 修复

1. `BLE_LOCAL_NAME_MAX_LEN`: 18 → 24（给名称+后缀+null 留够空间）
2. `sprintf` → `snprintf`（防止未来名称更长时溢出）

```c
snprintf(g_rdx_ble_server_info.ble_local_name, BLE_LOCAL_NAME_MAX_LEN, "%s %s", BLE_LOCAL_NAME, buf);
```

## 6. 经验总结

1. **`sprintf` 是嵌入式 C 的常见溢出源**。凡是写入固定大小 buffer 的场景，必须用 `snprintf` 并指定大小。
2. **宏定义需要考虑实际内容长度**。`BLE_LOCAL_NAME_MAX_LEN = 18` 看似够用，但名称后缀（AuthKey 后 4 位）使其刚好撞到边界。
3. **区分经典蓝牙和 BLE 的限制宏**。`BT_NAME_LEN_MAX` 是经典蓝牙限制，`ADV_RSP_PACKET_MAX` 是 BLE 广播包限制。RDX 自定义路径用 `BLE_LOCAL_NAME_MAX_LEN`，修改时需确认影响范围。
4. **BLE 广播包 31 字节是硬限制**。名称只是广播数据的一部分，需要给 Flags、Manufacturer Data 等字段留空间。
5. **BLE 名称实际可用空间 = 31 - 3(Flags) - 2(Name头) = 26 字节**，当前 `BLE_LOCAL_NAME_MAX_LEN = 24` 满足约束。
