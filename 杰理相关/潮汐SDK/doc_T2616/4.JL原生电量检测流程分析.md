# JL原生 vs RDX框架 电量检测流程分析

> 本文档基于源码逐行审计，标注了实际工程配置对代码路径的影响。
> 最后更新：2026-06-12（第二轮审计修正）

---

## 1. ADC 模式配置 — 决定第一级滤波是否生效

本工程的关键配置位于 `lib_driver_config.c`：

```c
// SDK/apps/earphone/log_config/lib_driver_config.c:64
const u8 gpadc_battery_mode = WEIGHTING_MODE; //使用IOVDD供电时,禁止使用 MEAN_FILTERING_MODE 模式
```

两种模式定义于 `gpadc.h`：

| 模式 | 宏 | 行为 |
|------|-----|------|
| 均值滤波模式 | `MEAN_FILTERING_MODE` | 20样本环形缓冲区 + 排序去极值 + 中间10个均值，100ms自动刷新 |
| 加权模式 | `WEIGHTING_MODE` | 直接调用 `adc_get_voltage()` 单次采样，无缓冲区滤波 |

**本工程使用 `WEIGHTING_MODE`**，`gpadc_battery_init()` 在此模式下直接 `return 0`（`gpadc_battery.c:126`），不注册回调，环形缓冲区、排序、均值滤波等逻辑**完全不执行**。

`gpadc_battery_get_voltage()` 在两种模式下的实际路径：

```c
// gpadc_battery.c:152-161
u32 gpadc_battery_get_voltage()
{
    if (gpadc_battery_mode == MEAN_FILTERING_MODE) {
        return battery_fifo.voltage;           // ← 本工程不走这里
    } else if (gpadc_battery_mode == WEIGHTING_MODE) {
        return adc_get_voltage(AD_CH_PMU_VBAT) * AD_CH_PMU_VBAT_DIV;  // ← 实际路径
    } else {
        return 0;
    }
}
```

## 2. JL原生电量检测流程

### 2.1 整体架构

JL原生的电量检测由 `battery_level.c` 驱动，采用**快慢双定时器 + 渐变防抖 + 状态机防抖**机制。

### 2.2 快慢定时器协同机制

```c
// battery_level.c:41-44
#define VBAT_DETECT_CNT         6       // 每轮采集6次
#define VBAT_DETECT_TIME        10L     // 快采样间隔10ms
#define VBAT_UPDATE_SLOW_TIME   60000L  // 慢周期（充电中）60s
#define VBAT_UPDATE_FAST_TIME   10000L  // 快周期（放电中）10s
```

**`vbat_slow_timer`**（`sys_timer_add`，重复定时器）：
- 放电中：每10s触发一次 `vbat_check_slow()`
- 充电中：每60s触发一次 `vbat_check_slow()`

**`vbat_fast_timer`**（`usr_timer_add`，重复定时器，priority=1 阻止低功耗）：
- 创建来源：`vbat_check_slow()`（慢周期触发）、`vbat_check_init()`（初始化/充电拔出重置）
- 每10ms触发一次 `vbat_check()`
- **仅在 `VBAT_LOWPOWER` 状态下持续运行**；`VBAT_WARNING` 和 `VBAT_NORMAL` 状态下会被删除

`vbat_check_init()` 的调用场景：
- 系统启动时（`check_power_on_voltage()` → `vbat_check_init()`）
- 充电拔出时（`app_power_event_handler()` 收到 `CHARGE_EVENT_LDO5V_OFF` → `vbat_check_init()`，重新初始化检测定时器）

**生命周期关键逻辑**（`battery_level.c:504-506`）：
```c
if (cur_bat_st != VBAT_LOWPOWER) {
    usr_timer_del(vbat_fast_timer);  // 只要不是 LOWPOWER，就删除
    vbat_fast_timer = 0;
}
```

**协同流程**：
```
vbat_slow_timer (10s/60s)
  └─ vbat_check_slow()
       ├─ 创建 vbat_fast_timer (10ms重复) ← 如果不存在则创建
       └─ 根据充电状态调整自身周期

vbat_fast_timer (10ms, 重复)
  └─ vbat_check()
       ├─ 累加电压，unit_cnt++
       ├─ unit_cnt < 6 → return（继续采样）
       ├─ unit_cnt == 6 → 处理：取平均、更新百分比、判断状态
       └─ cur_bat_st != LOWPOWER → 删除 vbat_fast_timer
```

**不同状态下的实际行为**：

| 状态 | fast_timer 行为 | 每个慢周期采样量 |
|------|-----------------|-----------------|
| `VBAT_NORMAL` | 每个慢周期创建后，完成1次6点采样（~60ms）即被删除 | 6次（60ms窗口） |
| `VBAT_WARNING` | 同上——`cur_bat_st` 被设为 WARNING 后，504行条件成立，删除 fast_timer | 6次（60ms窗口） |
| `VBAT_LOWPOWER` | 持续运行，每个慢周期内可完成多个6点采样窗口 | 密集采样（10ms间隔持续） |

**关键**：`vbat_fast_timer` 是**重复定时器**（`usr_timer_add` 第4参数为 priority，非 one-shot 标志）。常态下（NORMAL/WARNING），每个慢周期仅运行约60ms完成一次6点采样后即被删除，日常采样节奏由 slow_timer 决定（放电10s/充电60s）。仅在 `VBAT_LOWPOWER` 时持续密集采样。

### 2.3 电压采样与平均

```c
// battery_level.c:430-438 (vbat_check 内)
bat_voltage += get_vbat_voltage();  // get_vbat_voltage() 内部可能经过参考电压校准
unit_cnt++;
if (unit_cnt < VBAT_DETECT_CNT) {  // VBAT_DETECT_CNT = 6
    return;                          // 不够6次，继续累积
}
unit_cnt = 0;
cur_battery_voltage = bat_voltage / VBAT_DETECT_CNT;  // 6次取平均
bat_voltage = 0;
```

每10ms采样一次，累计6次（60ms窗口）后取平均值。常态下（NORMAL/WARNING）每个慢周期仅完成一次6点采样。

**参考电压校准**（`TCFG_REFERENCE_V_ENABLE`）：

```c
// battery_level.c:229-238
static u16 get_vbat_voltage(void)
{
    u16 now_voltage = gpadc_battery_get_voltage();
#if TCFG_REFERENCE_V_ENABLE
    if (vbat_full_voltage != 0xffff) {
        now_voltage = (u32)now_voltage * get_charge_full_value() / vbat_full_voltage;
    }
#endif
    return now_voltage;
}
```

当 `TCFG_REFERENCE_V_ENABLE` 使能时，`get_vbat_voltage()` 会基于存储的满电电压 `vbat_full_voltage`（从VM读取）和 `get_charge_full_value()` 做比例校准，补偿个体电池差异。未使能时直接透传 `gpadc_battery_get_voltage()` 的值。

### 2.4 电量百分比渐变防抖 — 核心机制

```c
// battery_level.c:440-449
tmp_percent = battery_calc_percent(cur_battery_voltage);

if (get_charge_online_flag()) {
    if (tmp_percent > cur_battery_percent) {
        cur_battery_percent++;   // 充电：每次最多+1%
    }
} else {
    if (tmp_percent < cur_battery_percent) {
        cur_battery_percent--;   // 放电：每次最多-1%
    }
}
```

电量百分比**不会跳变到ADC计算值**，而是每次最多变化1%。即使电压因负载波动导致计算百分比跳了10%，用户看到的电量也只会平滑变化。

### 2.5 电量曲线

```c
// battery_level.c:241-265
static u16 battery_calc_percent(u16 bat_val)
{
    // 遍历 battery_curve 表，线性插值计算百分比
    for (i = 0; i < (battery_curve_max - 1); i++) {
        if (bat_val <= battery_curve_p[i].voltage) {
            return battery_curve_p[i].percent;
        }
        if (bat_val >= battery_curve_p[i + 1].voltage) {
            continue;
        }
        // 线性插值
        div_percent = battery_curve_p[i + 1].percent - battery_curve_p[i].percent;
        tmp_percent += (bat_val - min) * div_percent / (max - min);
        return tmp_percent;
    }
}
```

支持可配置的电池曲线表（`battery_curve`），通过线性插值得到精确百分比。

**默认曲线生成**（`TCFG_BATTERY_CURVE_ENABLE == 0` 时）：

当未配置外部电池曲线时，`vbat_check_init()` 会自动构建两点式线性曲线：

```c
// battery_level.c:365-393
#if TCFG_BATTERY_CURVE_ENABLE == 0
if (battery_curve_p == NULL) {
    battery_curve_max = 2;
    curve = malloc(battery_curve_max * sizeof(struct battery_curve));
    battery_0 = app_var.poweroff_tone_v;         // 关机电压 → 0%
#if TCFG_CHARGE_ENABLE
    battery_100 = (get_charge_full_value() - 100); // 满电电压-100mV → 100%
#else
    battery_100 = 4100;
#endif
    curve[0] = {0, battery_0};
    curve[1] = {100, battery_100};
    battery_curve_p = curve;
}
#endif
```

因此电量曲线并非总是外部配置表——在默认配置下是关机电压到满电电压的两点线性插值。

### 2.6 低电状态机与防抖

状态机有三个状态：`VBAT_NORMAL` → `VBAT_WARNING` → `VBAT_LOWPOWER`。

每次 `vbat_check()` 完整处理时（unit_cnt 达到6次后），根据当前电压判断状态：

#### 低电关机（`VBAT_LOWPOWER`）

```c
// battery_level.c:456-468
if (adc_check_vbat_lowpower() ||
    (cur_battery_voltage <= app_var.poweroff_tone_v)) {
    low_power_cnt++;
    // ...
    cur_bat_st = VBAT_LOWPOWER;
    if (low_power_cnt > 6) {  // 注意：是 > 6，即第7次才触发
        batmgr_send_msg(POWER_EVENT_POWER_LOW, 0);
    }
}
```

触发条件：`low_power_cnt > 6`，即**连续7个完整处理周期**。

时间窗计算：
- 放电中：slow_timer 周期 10s，每个周期触发一次 fast_timer 窗口，完成一次处理 → **约 70s**
- 充电中：slow_timer 周期 60s → **约 7min**（但充电中通常不会触发低电关机）

#### 低电提醒（`VBAT_WARNING`）

```c
// battery_level.c:469-487
} else if (cur_battery_voltage <= app_var.warning_tone_v) {
    low_voice_cnt++;
    cur_bat_st = VBAT_WARNING;
    if ((low_voice_first_flag && low_voice_cnt > 1) ||    // 首次：> 1，即第2次触发
        (!low_voice_first_flag && low_voice_cnt >= 5)) {  // 后续：>= 5，即第5次触发
        low_voice_first_flag = 0;
        low_voice_cnt = 0;
        batmgr_send_msg(POWER_EVENT_POWER_WARNING, 0);
    }
}
```

| 场景 | 触发条件 | 放电时间窗 |
|------|----------|-----------|
| 首次进入低电 | `low_voice_cnt > 1`（第2次） | ~20s |
| 后续重复提醒 | `low_voice_cnt >= 5`（第5次） | ~50s |

注：代码注释写 "第一次进低电10s后报一次"，但实际 slow_timer 放电周期为 10s，`> 1` 意味着第2个周期，即约 20s。

#### 恢复正常（`VBAT_NORMAL`）

```c
// battery_level.c:488-498
} else {
    power_normal_cnt++;
    if (power_normal_cnt > 2) {  // 第3次才确认恢复
        if (cur_bat_st != VBAT_NORMAL) {
            cur_bat_st = VBAT_NORMAL;
            batmgr_send_msg(POWER_EVENT_POWER_NORMAL, 0);
        }
    }
}
```

连续3个处理周期确认恢复正常（放电约 30s）。

### 2.7 状态转换时的 fast_timer 管理

```c
// battery_level.c:504-506
if (cur_bat_st != VBAT_LOWPOWER) {
    usr_timer_del(vbat_fast_timer);  // 非低电关机 → 删除快速采样
    vbat_fast_timer = 0;
}
```

此逻辑位于 `vbat_check()` 末尾，**每次完整处理周期（unit_cnt 达到6次后）都会执行**。

**执行时序分析**（以首次进入 WARNING 为例）：

```
slow_timer 触发 → vbat_check_slow() → 创建 fast_timer (10ms)
  ↓ 60ms后
fast_timer 累积6次 → vbat_check() 处理：
  ├─ cur_battery_voltage <= warning_tone_v
  ├─ cur_bat_st = VBAT_WARNING          ← 此时状态变为 WARNING
  ├─ low_voice_cnt++, 处理低电提醒逻辑
  └─ 到达 504 行: cur_bat_st(==WARNING) != LOWPOWER → 删除 fast_timer
  ↓
下一个 slow_timer 周期 (10s后) → 重新创建 fast_timer → 重复
```

因此 `VBAT_WARNING` 状态下，fast_timer **不会持续运行**——每个慢周期仅运行约60ms完成一次6点采样后即被删除。`low_voice_cnt` 每10s增加1。

仅 `VBAT_LOWPOWER` 时 fast_timer 持续运行（504行条件不成立），提供密集采样用于精确监测临界电压。

### 2.8 电量变化事件上报

```c
// battery_level.c:507-513
if ((cur_battery_level != old_battery_level) ||
    (charge_online_flag != get_charge_online_flag())) {
    batmgr_send_msg(POWER_EVENT_POWER_CHANGE, 0);
}
```

电量等级变化或充电状态变化时，发送 `POWER_EVENT_POWER_CHANGE`，触发 TWS 同步和 HFP 上报。

## 3. RDX框架电量检测流程

### 3.1 电量百分比计算

```c
// rdx_battery.c:290-304
u8 rdx_battery_get_percent(void)
{
    u8 chr_state = rdx_app_get_charge_state();
    u16 voltage = get_vbat_value();

    // ★ 直接复用JL原生的电量等级接口
    return ((battery_value_to_phone_level() + 1) * 10);

    // ↓↓↓ 以下代码是死代码，永远不会执行 ↓↓↓
    // 包含 RDX_BAT_VOLT_THRESHOLD_xxx / RDX_BAT_VOLT_CHRG_THRESHOLD_xxx 阈值表
}
```

**关键发现**：第304行直接 `return`，后面的电压-百分比映射逻辑是**死代码**。

实际电量计算链路：
```
get_vbat_percent()           → cur_battery_percent (JL原生，0~100，有±1%渐变)
  ↓
battery_value_to_phone_level() → (percent - 5) / 10，返回 0~9 级
  ↓
rdx_battery_get_percent()    → (level + 1) * 10，返回 10, 20, 30, ..., 100
```

RDX 电量被量化为 **10% 档位**（共10档）。当 JL 原生的 `cur_battery_percent` 跨过10的整数倍边界时，RDX 显示会出现 10% 跳变。

### 3.2 RDX充电中电量格数延迟

```c
// rdx_battery.c:99-101
#define INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL3_4   (20 * 60 * 1000)  // 20分钟
#define INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL4_5   (30 * 60 * 1000)  // 30分钟
#define INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL5_6   (15 * 60 * 1000)  // 15分钟
```

但 `rdx_battery_incharge_batLevel_indicator()` 中，**仅 `BAT_LEVEL_4→5` 和 `BAT_LEVEL_5→6` 真正启动延迟定时器**；`BAT_LEVEL_0~3` 直接 `return`，延迟宏虽已定义但未使用。注：宏名 `INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL3_4` 暗示 3→4 档位有延迟，但代码中 `BAT_LEVEL_3` 直接返回不启动延迟，该宏实际未被引用。

```c
// rdx_battery.c:182-214 (简化)
switch(incharge_bat_Info.level){
    case BAT_LEVEL_0:
    case BAT_LEVEL_1:
    case BAT_LEVEL_2:
    case BAT_LEVEL_3:
        return;  // 直接返回，不启动延迟

    case BAT_LEVEL_4:
        delay_time = INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL4_5;  // 30分钟
        break;

    case BAT_LEVEL_5:
        delay_time = INCHARGE_SHOW_DELAY_TIME_BAT_LEVEL5_6;  // 15分钟
        break;

    case BAT_LEVEL_6:
        rdx_app_charge_full();  // 满电直接触发
        return;
}
rdx_battery_delay_show_batLevel_timer_start(delay_time);
```

### 3.3 RDX充电满检测防抖

```c
// rdx_charge.c:412-449 (rdx_app_incharge_full_check_timer_cb)
// 每5秒触发一次 (RDX_APP_INCHARGE_FULL_CHECK_TIMEOUT = 5000ms)
rdx_protocol_update_dev_battery_level();
cur_bat = rdx_battery_get_percent();
full_check = charge_check_is_full();

if (full_check == TRUE) {
    // 硬件满：直接触发，无防抖
    rdx_app_charge_full();
    full_confirm_count = 0;
} else if (cur_bat >= 100) {
    // 软件满：防抖，连续60次确认（5s × 60 = 300s）
    if (full_confirm_count < 255) full_confirm_count++;
    if (full_confirm_count >= 60) {
        rdx_app_charge_full();
        full_confirm_count = 0;
    }
} else {
    // 未满：重置计数器
    full_confirm_count = 0;
}
```

| 触发条件 | 防抖策略 | 确认时间 |
|----------|----------|----------|
| 硬件满 (`charge_check_is_full()`) | 无防抖，直接触发 | 立即 |
| 软件满 (`cur_bat >= 100`) | 连续60次确认 | ~300秒（5分钟） |

### 3.4 rdx_app_charge_full() 条件编译分支

`rdx_app_charge_full()` 存在 `RDX_PRODUCT_IS_CHARGE_CASE` 条件编译，充电仓版本与非充电仓版本逻辑不同：

```c
// rdx_charge.c:329-384 (简化)
void rdx_app_charge_full(void)
{
#if RDX_PRODUCT_IS_CHARGE_CASE
    // 充电仓版本：hw_full 或 cur_bat>=100 即触发
    bool hw_full = charge_check_is_full();
    u8 cur_bat = rdx_battery_get_percent();
    if (hw_full != TRUE && cur_bat < 100) {
        return;  // 二次门限：硬件未满且电量<100则不触发
    }
    orig_bat = 0;
    // ↓ 继续执行后续共同逻辑
#else
    // 非充电仓版本：依赖 BatLevel 格数判断
    BatLevel value = rdx_battery_get_incharge_batLevel();
    if (charge_check_is_full() == TRUE) {
        if (value == BAT_LEVEL_4) return;       // 格数不够 → 提前返回，不执行共同逻辑
        if (value < BAT_LEVEL_5) {
            rdx_battery_set_incharge_batLevel(BAT_LEVEL_6);
            rdx_battery_delay_show_batLevel_timer_stop();
            return;                               // 提前返回，不执行共同逻辑
        }
        orig_bat = 0;
        // ↓ 继续执行后续共同逻辑
    } else {
        return;  // 硬件未满 → 提前返回，不执行共同逻辑
    }
#endif
    // 后续共同逻辑（仅 orig_bat=0 后到达此处）：
    // 设 RDX_CHARGE_FULL 状态、LED灯效、定时关机等
}
```

注意：非充电仓版本中，`value == BAT_LEVEL_4` 和 `value < BAT_LEVEL_5` 两个分支都会提前 `return`，**不执行**后续共同逻辑（设 RDX_CHARGE_FULL 状态、LED灯效、定时关机）。只有 `value >= BAT_LEVEL_5` 且硬件满时，才会设置 `orig_bat = 0` 并进入共同逻辑。

### 3.5 RDX电量数据流与调用链

```
                    ┌─────────────────────────────────────┐
                    │         JL原生底层 (共享)            │
                    │                                     │
ADC → gpadc_battery_get_voltage()                         │
      → vbat_check() 6次平均                              │
      → cur_battery_percent (±1%渐变)                      │
      → battery_value_to_phone_level() → 0~9级            │
                    └──────────────┬──────────────────────┘
                                   │
                    ┌──────────────▼──────────────────────┐
                    │         RDX框架层                    │
                    │                                     │
                    │  rdx_battery_get_percent()          │
                    │    → (level+1)*10 → 10%档位         │
                    │                                     │
                    │  调用入口：                           │
                    │  ├─ rdx_charge.c:424                 │
                    │  │  (充电满检测定时器, 5s周期)         │
                    │  ├─ rdx_app.c:2801                   │
                    │  │  (APP电量查询 PROTOCOL_EVENT)      │
                    │  ├─ rdx_app.c:2809                   │
                    │  │  (APP充电状态查询)                  │
                    │  └─ rdx_ble_server.c:1297            │
                    │     (BLE Battery Characteristic读取)  │
                    └──────────────┬──────────────────────┘
                                   │
                    ┌──────────────▼──────────────────────┐
                    │         上层输出                     │
                    │                                     │
                    │  ├─ rdx_ble_server: BLE上报给APP     │
                    │  ├─ rdx_protocol: 协议层上报          │
                    │  └─ rdx_led_ctrl: 充电灯效更新       │
                    └─────────────────────────────────────┘
```

## 4. 核心区别总结

| 特性 | JL原生 | RDX框架 |
|------|--------|---------|
| ADC采样模式 | `WEIGHTING_MODE`（单次采样） | 继承JL原生 |
| 多次采样平均 | 有（6次×10ms=60ms窗口，常态下每慢周期1次） | 继承JL原生 |
| **放电百分比渐变** | **有（每次最多±1%）** | **无（量化到10%档位，跨档跳变）** |
| 低电状态防抖 | 有（7次关机/2次首次提醒/5次后续提醒/3次恢复） | 复用JL原生 |
| 充电满防抖 | 有（硬件满直接触发） | 有（硬件满直接 + 软件满60次×5s） |
| 充电满条件编译 | 无 | 有（`RDX_PRODUCT_IS_CHARGE_CASE` 分支） |
| 充电显示延迟 | 无 | 有（仅4→5、5→6两档，15~30分钟） |
| 电量精度 | 1%（0~100连续） | 10%（10,20,30,...,100共10档） |
| 电池曲线 | 可配置曲线表 + 线性插值 | 死代码（实际复用JL原生） |

## 6. 潜在问题与改进方向

### 问题1：RDX放电电量跳变

**现象**：放电时电量从80%直接跳到70%，用户感知明显。

**根因**：`rdx_battery_get_percent()` 返回 `(battery_value_to_phone_level()+1)*10`，量化为10%档位，跨档时无渐变。

**改进方案**：
- 方案A：直接使用 `get_vbat_percent()` 获取JL原生的0~100连续百分比
- 方案B：在RDX层增加自己的渐变防抖逻辑

### 问题2：RDX电池曲线是死代码

**现象**：`rdx_battery_get_percent()` 中定义了完整的充电/放电电压阈值表（`RDX_BAT_VOLT_THRESHOLD_xxx`），但第304行的 `return` 使这些代码永远不会执行。

**建议**：如果需要独立的RDX电量曲线，应删除第304行的 `return`；如果确认要复用JL原生曲线，应清理死代码。

### 问题3：WEIGHTING_MODE 下无ADC滤波

**现象**：本工程 `gpadc_battery_mode = WEIGHTING_MODE`，`gpadc_battery_get_voltage()` 直接返回单次ADC采样值，无任何滤波。偶发的ADC毛刺会直接传递到 `vbat_check()` 的6点平均。

**影响**：在电压纹波较大或ADC噪声较高的场景下，电量可能出现短暂波动。

**改进方案**：确认 `WEIGHTING_MODE` 是否为最佳选择；如果IOVDD供电稳定，可考虑启用 `MEAN_FILTERING_MODE`。
