# 利用系统时基模拟 RTC 的原理

> 本文件是项目自行编写的软件模拟 RTC，不依赖芯片硬件 RTC 外设。

---

## 一、核心思路

硬件 RTC 的本质是一个**持续计数的硬件计数器**，只要有时钟源就能自己走时。软件模拟 RTC 没有这种硬件计数器，因此换一种等价思路：

> **记录一个"锚点"：在某个已知时刻（Unix 时间戳），同时记下此刻系统已运行了多少毫秒。此后任何时刻，用"当前系统运行毫秒数 - 锚点毫秒数"得到经过的时间，加到锚点时间戳上，即得当前时间。**

```
当前时间 = 锚点时间戳 + (jiffies_msec() - 锚点jiffies) / 1000
```

这里的 `jiffies_msec()` 就是系统时基，由 GPTIMER5 驱动，每 10 ms 递增一次（见 `1.时钟.md` 第五节）。

---

## 二、形象类比

把两种 RTC 的工作方式类比成两种计时员：

**硬件 RTC** = 一块永远在走的手表

> 你把它放进口袋、锁进抽屉（软关机）、甚至睡着（深度睡眠），它都在走。随时拿出来看，上面的时间就是现在的时间。代价是表芯需要持续供电（~2~5 μA）。

**软件模拟 RTC** = 计时员 + 笔记本

> 出发前，计时员按下秒表，同时在笔记本上记下：`"北京时间 10:00:00，秒表从 0 开始"`（即 `rtc_value` 和 `begin_msec`）。
>
> 之后任意时刻，只要秒表还在跑：
> ```
> 现在时间 = 10:00:00 + 秒表读数
> ```
>
> 但如果计时员睡着了（Soft-Off），秒表会归零。醒来后只能翻笔记本——而笔记本是每 3 分钟才抄一次（NVM 定时写入），所以最多会少记 3 分钟。
>
> 笔记本（NVM）不怕断电，但秒表（jiffies/RAM）断电就清零。

用代码对照这个类比：

| 类比 | 代码对应 |
|---|---|
| 秒表 | `jiffies_msec()` |
| 笔记本上的记录 | `rtc_params.rtc_value`（NVM）+ `rtc_params.begin_msec`（RAM）|
| 每 3 分钟抄一次笔记本 | `sys_timer_add(rdx_rtc_store_timestamp, 3*60*1000)` |
| 计时员睡着（Soft-Off） | RAM 清空，jiffies 归零，笔记本（NVM）完好 |

## 三、数据结构

```c
// rdx_rtc.c:41
typedef struct {
    time_t        rtc_value;    // 锚点时间戳（Unix 秒）
    unsigned long begin_msec;   // 记录锚点时，jiffies_msec() 的值
} RTCParams;

static RTCParams rtc_params;   // 仅存在于 RAM 中，掉电丢失
```

- `rtc_value`：上一次设置/校准时的 Unix 时间戳（精确到秒）
- `begin_msec`：设置该锚点时系统已运行的毫秒数（来自 jiffies）

**两者配合，锚定了"某个 jiffies 时刻对应某个 Unix 时间"的映射关系。**

---

## 三、关键函数逐一分析

### 3.1 设置时间：`rdx_rtc_set_timestamp()`

```c
// rdx_rtc.c:339
int rdx_rtc_set_timestamp(time_t timestamp)
{
    rtc_params.rtc_value  = timestamp;         // ① 记录锚点时间戳
    rtc_params.begin_msec = jiffies_msec();    // ② 记录此刻的 jiffies 值
    syscfg_write(VM_RDX_RTC_INIT_VALUE,
                 &rtc_params, sizeof(time_t)); // ③ 只把 rtc_value 写入 NVM
    return 0;
}
```

**注意**：`syscfg_write` 写入的是 `sizeof(time_t)` 字节（4 字节），而非整个 `RTCParams`（8+ 字节）。因此 **`begin_msec` 从不写入 NVM**，只存在于 RAM 中。

---

### 3.2 读取时间：`rdx_rtc_get()`

```c
// rdx_rtc.c:362
time_t rdx_rtc_get(void)
{
    unsigned long sys_timestamp = jiffies_msec();
    int ms_offset = jiffies_msec2offset(rtc_params.begin_msec, sys_timestamp);
    // ↑ 先用 RAM 中的 begin_msec 计算经过了多少毫秒

    syscfg_read(VM_RDX_RTC_INIT_VALUE,
                &rtc_params, sizeof(time_t));
    // ↑ 从 NVM 刷新 rtc_value（也只读 4 字节，不覆盖 begin_msec）

    return rtc_params.rtc_value + ms_offset / 1000;
    // ↑ 锚点时间戳 + 经过的秒数
}
```

执行步骤：
1. 取当前 `jiffies_msec()`
2. 计算与锚点之间经过的毫秒数
3. 从 NVM 刷新 `rtc_value`（`begin_msec` 不受影响）
4. 返回 `rtc_value + 经过秒数`

---

### 3.3 初始化：`rdx_rtc_init()`

```c
// rdx_rtc.c:544
void rdx_rtc_init(void)
{
    time_t rtc_timestamp = rdx_rtc_get();  // 尝试从 NVM 恢复时间
    if (rtc_timestamp == 0) {
        // NVM 中无有效时间，使用默认时间
        memset(&rtc_params, 0, sizeof(RTCParams));
        rtc_timestamp = rdx_rtc_utc_string_to_timestamp(RTC_DEFAULT_DATE_AND_TIME);
        // RTC_DEFAULT_DATE_AND_TIME = "2025-04-01 16:50:06"
    }
    rdx_rtc_set_timestamp(rtc_timestamp);  // 以此建立新锚点
    rdx_rtc_restore_timer_start();         // 启动 NVM 定期写入定时器
}
```

系统上电时调用，流程：从 NVM 读上次存的时间 → 以此为锚点重新开始计时 → 启动定时持久化。

---

### 3.4 定时持久化：每 3 分钟写一次 NVM

```c
// rdx_rtc.c:34
#define RTC_RESTORE_INTERVAL   (3 * 60 * 1000)  // 3 分钟

// rdx_rtc.c:485
void rdx_rtc_store_timestamp(void)
{
    time_t timestamp = rdx_rtc_get();       // 获取当前准确时间
    rdx_rtc_set_timestamp(timestamp);       // 更新锚点并写入 NVM
}

// rdx_rtc.c:524
void rdx_rtc_restore_timer_start(void)
{
    if (rtc_restore_timer == 0) {
        rtc_restore_timer = sys_timer_add(
            NULL, rdx_rtc_store_timestamp, RTC_RESTORE_INTERVAL);
    }
}
```

每隔 3 分钟：
1. 调用 `rdx_rtc_get()` 得到当前准确时间
2. 重新设置锚点（更新 RAM 中的 `begin_msec` 和 NVM 中的 `rtc_value`）

这相当于每 3 分钟把"活的时间"固化一次到 NVM 里，防止掉电损失过多。

## 四、运行时序图

```
上电初始化
    └─ rdx_rtc_init()
         ├─ NVM 有值 → rtc_value = NVM值，begin_msec = 当前jiffies
         └─ NVM 无值 → rtc_value = 默认时间，begin_msec = 当前jiffies
                            │
                    ┌───────▼────────────────────────────────┐
                    │          正常运行中                      │
                    │  jiffies_msec() 持续递增                │
                    │  rdx_rtc_get() = rtc_value             │
                    │               + (jiffies - begin_msec) │
                    │                 / 1000                  │
                    │                            每3分钟       │
                    │                    ┌───────────────┐   │
                    │                    │ 更新锚点写NVM  │   │
                    │                    └───────────────┘   │
                    └────────────────────────────────────────┘
```

## 五、各场景下的表现

| 场景 | jiffies 行为 | rtc_params 状态 | 软件RTC 结果 |
|---|---|---|---|
| 正常运行 | 持续递增 | RAM 保持 | 精确走时 ✓ |
| Light Sleep | 正常递增（STD 12M 保持） | RAM 保持 | 精确走时 ✓ |
| Deep Sleep | 暂停，唤醒时**有补偿** | RAM 保持 | 走时连续，精度取决于 LRC 补偿精度 ✓ |
| Soft-Off（软关机） | 归零重来 | **RAM 清空，begin_msec = 0** | 重启后时间 = NVM存的值（最多误差 3 分钟）⚠ |
| 硬断电重新上电 | 归零重来 | **RAM 清空，begin_msec = 0** | 重启后时间 = NVM存的值（最多误差 3 分钟）⚠ |

## 六、与硬件 RTC 的本质区别

| 对比项 | 软件模拟 RTC（rdx_rtc.c） | 硬件 RTC（HW_RTC） |
|---|---|---|
| 走时依赖 | jiffies_msec()，依赖 CPU 时钟运行 | P33 RTCVDD 域硬件计数器，独立于 CPU |
| Soft-Off 期间 | **停止**，时间最多丢失 3 分钟 | **继续走时**，无时间损失 |
| Deep Sleep 期间 | 暂停，但唤醒后有补偿，基本连续 | 继续走时 |
| 时间精度 | 依赖 jiffies 精度（10 ms 分辨率） | 依赖 32K 晶振精度（约 30 ppm） |
| 掉电保持 | NVM 存最后一次锚点（3 分钟间隔） | 只要电池有电即持续 |
| 硬件资源 | 无需额外硬件，只用 jiffies + NVM | 需要 RTC 硬件外设（可选接 32K 晶振） |
| 使用场景 | 精度要求不高、无 32K 晶振的方案 | 需要精确时间、Soft-Off 后仍准确的方案 |

---

## 七、功耗对比

| 功耗项 | 软件模拟 RTC | 硬件 RTC |
|---|---|---|
| 正常运行时 | **零额外功耗**（GPTIMER5/jiffies 无论如何都在跑，是 OS 调度必须的）| 额外 ~2~5 μA（P33 RTCVDD 域 + 32K 晶振持续工作）|
| Deep Sleep 时 | **零额外功耗**（GPTIMER5 已暂停，补偿由 LRC 完成，而 LRC 是低功耗模式本身就要用的）| 额外 ~2~5 μA（32K 晶振 + RTC 计数器仍运行）|
| Soft-Off 时 | **零额外功耗**（完全不运行，但时间精度丢失最多 3 分钟）| 额外 ~2~5 μA（P33 RTCVDD 域始终供电）|

**结论：软件模拟 RTC 功耗更低，但精度更差。**

具体来说：

- **软件模拟 RTC 在任何阶段都不产生额外功耗**，因为它完全寄生在 jiffies 体系上，而 jiffies 是 OS 运行的基础设施，不管用不用软件 RTC 它都必须存在。

- **硬件 RTC 在 Soft-Off 期间持续消耗 ~2~5 μA**。以 TWS 耳机典型场景举例：放入充电盒后处于 Soft-Off 状态 8 小时，硬件 RTC 额外消耗约 **16~40 μAh**。对于 50 mAh 的耳机电池，约占 **0.03%~0.08%**，基本可以忽略不计。

- **但若系统本就需要用硬件 RTC 来实现定时唤醒（闹钟功能），则 P33 RTCVDD 域无论如何都要供电**，此时硬件 RTC 的计时功能等于"免费搭车"，不产生任何额外功耗。

**实际选择建议：**

| 场景 | 推荐方案 |
|---|---|
| 系统需要 Soft-Off 定时唤醒（闹钟） | 用硬件 RTC，计时功能顺带免费 |
| 不需要精确走时，对 3 分钟误差可接受 | 软件模拟 RTC，节省 32K 晶振元器件成本 |
| 需要 Soft-Off 后时间依然精确 | 必须用硬件 RTC |

## 八、与公版 SDK RTC 接口的功能对比

公版 SDK 的 RTC 接口定义在 `interface/driver/device/rtc/rtc_dev.h`，以下逐类对比 `rdx_rtc.c` 实现了哪些、缺失了哪些、以及额外扩展了什么。

### 8.1 接口风格差异（贯穿全部）

两者使用了完全不同的时间表示方式：

| | rdx_rtc.c | 公版 SDK |
|---|---|---|
| 时间类型 | `time_t`（Unix 时间戳，秒数） | `struct sys_time`（年月日时分秒结构体） |
| 适合场景 | 网络对时（NTP/蓝牙同步直接给 Unix 时间戳） | 硬件寄存器直接读写，直观 |

```c
// 公版 SDK 的时间结构体（interface/utils/sys_time.h）
struct sys_time {
    u16 year;
    u8  month;
    u8  day;
    u8  hour;
    u8  min;
    u8  sec;
    // 注意：无星期字段
};

// rdx_rtc.c 自定义的时间结构体
typedef struct {
    int year; int month; int day;
    int hour; int minute; int second;
    int weekday;
    const char *weekday_name;   // 多了星期
} DateTime;
```

---

### 8.2 基础时间读写（已实现，接口不同）

| 功能 | 公版 SDK 接口 | rdx_rtc.c 对应接口 |
|---|---|---|
| 初始化 | `rtc_config_init(platform_data*)` | `rdx_rtc_init()` |
| 读取当前时间 | `rtc_read_time(sys_time*)` | `rdx_rtc_get()` → 返回 `time_t` |
| 写入时间 | `rtc_write_time(sys_time*)` | `rdx_rtc_set_timestamp(time_t)` |

功能等价，但接口签名不同。公版 SDK 传结构体指针，rdx_rtc.c 传 Unix 时间戳。

### 8.3 时间转换工具函数（已实现，各有侧重）

| 功能 | 公版 SDK 接口 | rdx_rtc.c 对应接口 |
|---|---|---|
| 时间戳 → 结构体 | `sec_to_datetime(u64, sys_time*)` | `rdx_rtc_timestamp_to_datetime(time_t)` |
| 结构体 → 时间戳 | `datetime_to_sec(sys_time*)` → `u64` | `rdx_rtc_datetime_to_timestamp(DateTime)` |

| 时间差（秒） | `time_diff_for_sec(old*, curr*)` | 无 |
| 时间加减（秒） | `time_add_sec(time*, s64)` | 无 |
| 拷贝时间结构体 | `copy_time(old*, new*)` | 无 |

| 年月日 → 天数 | `ymd_to_day(sys_time*)` | 无 |
| 判断闰年 | 无独立接口 | `rdx_rtc_is_leap_year(year)` |

### 8.4 闹钟功能（完全缺失）

这是 rdx_rtc.c 相对公版 SDK **最大的功能缺口**：

| 功能 | 公版 SDK 接口 | rdx_rtc.c |
|---|---|---|
| 读取闹钟时间 | `rtc_read_alarm(sys_time*)` | **无** |
| 设置闹钟时间 | `rtc_write_alarm(sys_time*)` | **无** |
| 启用/禁用闹钟 | `rtc_alarm_en(u8)` | **无** |
| 查询闹钟状态 | `rtc_get_alarm_en()` | **无** |
| 注册唤醒回调 | `rtc_dev_platform_data.cbfun` | **无** |
| 配置唤醒源 | `rtc_wakup_source()` | **无** |

**连锁缺失：由于没有闹钟功能，rdx_rtc.c 也无法实现"RTC 定时从 Soft-Off 中唤醒系统"**。公版硬件 RTC 支持设定闹钟，时间到时自动唤醒处于 Soft-Off 的芯片，是低功耗定时任务的关键机制，rdx_rtc.c 完全不具备此能力。

---

### 8.5 低功耗电源管理（完全缺失）

| 功能 | 公版 SDK 接口 | rdx_rtc.c |
|---|---|---|
| Soft-Off 前保存时间 | `poweroff_save_rtc_time()` | **无**（只有每 3 分钟定时写 NVM） |
| LRC 频率校准数据读取 | `get_lrc_rtc_trim(_rtc_trim*)` | **无** |
| P11 RAM 时间备份写入 | `write_p11_sys_time(param)` | **无** |
| P11 RAM 时间备份读取 | `read_p11_sys_time(sys_time*, trim*)` | **无** |
| P11 时间初始化 | `p11_sys_time_init()` | **无** |

公版 SDK 在 Soft-Off 前主动调用 `poweroff_save_rtc_time()` 将时间和 LRC 校准数据写入 P11 RAM，重启后精确恢复。rdx_rtc.c 没有这套机制，只有被动的 3 分钟定时 NVM 写入。

### 8.6 rdx_rtc.c 独有的扩展功能

以下功能是 rdx_rtc.c 相比公版 SDK 额外提供的：

| 功能 | rdx_rtc.c 接口 | 说明 |
|---|---|---|
| 时区转换 | `rdx_rtc_timestamp_to_timezone_string(time_t, offset, buf)` | 公版 SDK 无时区概念 |

## 九、已知问题

**1. `rdx_rtc_get_string()` 返回栈上指针（悬空指针 BUG）**

```c
// rdx_rtc.c:384
char* rdx_rtc_get_string(void)
{
    char buffer[64];          // 局部变量，函数返回后栈帧销毁
    time_t timestamp = rdx_rtc_get();
    rdx_rtc_timestamp_to_utc_string(timestamp, buffer);
    return buffer;            // ← 返回已销毁的栈内存，行为未定义
}
```

对比同文件中正确的写法（使用 `static` 缓冲区）：

```c
char* rdx_rtc_get_string_date(void)
{
    static char date_buffer[32];   // static：存放在全局数据区，不随函数返回销毁
    ...
    return date_buffer;            // ← 安全
}
```

**2. 误差随每次 Soft-Off 持续累积，没有上限**

单次 Soft-Off 最多丢失 3 分钟，但这不是一次性的代价——**每次 Soft-Off 都会叠加一次最多 3 分钟的误差，且误差永远不会自动修正**。

典型使用场景下的累积示例（耳机每天放入充电盒 5 次，每次 Soft-Off）：

```
第 1 天结束：误差最多 5 × 3 = 15 分钟
第 7 天结束：误差最多 7 × 15 = 105 分钟（约 1.75 小时）
第 30 天结束：误差最多 30 × 15 = 450 分钟（7.5 小时）
```

误差只会越来越大，且 NVM 里存的时间本身就是"错的"，下次 Soft-Off 会在已经偏差的基础上继续叠加。

**唯一的解决方式是外部时间校正**，例如：
- 手机通过蓝牙连接时将当前时间同步给耳机（调用 `rdx_rtc_set_timestamp()`）
- 用户手动设置时间
- 接入网络时对时（NTP）

如果产品设计中没有任何外部校时机制，软件模拟 RTC 的时间会随使用时间越来越不准，最终完全失去参考价值。

对比之下，**硬件 RTC 只要电池有电就持续走时，Soft-Off 不会带来任何误差损失**，不依赖外部校时也能保持连续性（仍有晶振本身的温漂误差，约 ±30 ppm，每天约 ±2.6 秒，属于另一个量级）。

缩短 `RTC_RESTORE_INTERVAL` 可以减小单次 Soft-Off 的最大误差，但无法消除累积问题，且会加速 flash 磨损。
