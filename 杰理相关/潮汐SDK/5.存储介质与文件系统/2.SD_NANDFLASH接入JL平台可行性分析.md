# 3. JL平台对SD NAND Flash的支持

> **本文档总结 SD NAND Flash（以 XCSD02G-MJ 为例）接入 JL AC701N/BR28 平台的实现方案。SD NAND 已实际焊接到当前项目板并调通。** SD NAND 是内部集成 SD2.0 控制器的 NAND Flash 芯片，对外呈现标准 SD 卡接口。与 SPI NAND 方案相比，SD NAND 无需主机端 FTL——坏块管理、磨损均衡、ECC 全部由芯片内部控制器处理，架构显著简化。

## 1. 能力边界总览

### 1.1 JL已提供的能力（开箱即用）

|--------|------|----------|

| SD 卡检测 | CMD 检测、CLK 检测、IO 检测三种方式 | 框架代码（源码可见）：`sdmmc_cmd_detect`、`sdmmc_0_clk_detect`、`sdmmc_0_io_detect` |

**一句话**：SD NAND 接入 JL 平台**不需要编写任何新驱动代码**。BR28 的 SD/MMC 主机控制器（`JL_SD0`）硬件 + `sd_dev_ops` 驱动已完整覆盖 SD2.0 协议。**当前项目硬件已将 SD NAND 焊接到 PC03/04/05**，通过关闭 SPI1 + 自定义检测函数绕过插拔检测实现挂载；同时启用 `TCFG_SD0_FORMAT_ON_BOOT` 处理首次上电格式化，电源由 PA00 控制 LDO3V 独立供电。完整原理和配置步骤见第 3 节。

### 1.2 与 SPI NAND 方案的关键差异

| 维度 | SPI NAND | SD NAND |
|------|----------|---------|
| 裸驱动 | `nandflash.c`（~1300 行，需从 soundbox SDK 获取） | `sd_dev_ops`（闭源，SDK 自带） |
| FTL 层 | 需要——`ftl.a` 或 `nandflash_ftl.c`（~1400 行） | **不需要**——芯片内置 |
| 设备注册 | 双设备（`nand_flash` + `nandflash_ftl`） | 单设备（`sd0`） |
| 文件系统 | FAT，需手动 `f_format()` 首次格式化 | FAT32，**出厂预格式化** |
| 坏块管理 | FTL 软件管理 | 芯片内部硬件 |
| 磨损均衡 | FTL 软件管理 | 芯片内部硬件 |
| ECC | FTL 软件管理 | 芯片内部 8bit/512B |
| 物理接口 | SPI（CLK/DO/DI/CS，4 线，占用 PC03/04/05） | SDIO（CLK/CMD/DAT0，3 线 + 上拉，**不能与 SPI1/IMU 共用 IO**） |
| 最高读速 | 取决于 SPI 时钟（2MHz → ~250KB/s） | 最高 21MB/s |
| 最高写速 | 取决于 SPI 时钟 + 擦除时间 | ~4.6MB/s |
| 驱动源码可见性 | ✅ 裸驱动 + FTL 源码可见（从 soundbox 获取后） | ❌ `sd_dev_ops` 闭源，在 `device.a` 中 |
| Host CPU 负载 | 高（FTL 磨损均衡/垃圾回收计算） | 低（仅块读写） |
| 初始化时间 | ~100ms（复位 + ID 识别 + FTL 扫描） | ~1s（SD 协议初始化 + FAT 挂载） |
| 首次使用流程 | 获取源文件 → 修复编译 → 配置 SPI → 格式化 → 挂载 | **先解 GPIO 冲突** → 配置引脚 → 使能宏 → 挂载（出厂已格式化） |
| ⚠️ 当前项目风险 | — | **PC03/04/05 三重冲突：SPI1 vs SD0 vs MPU6887P，必须硬件改板** |

### 1.3 SD NAND vs SPI NAND 架构对比

```
SPI NAND 方案（需要 3 层）：
  App → VFS → FAT(fs.a) → FTL(nandflash_ftl.c) → SPI Driver(nandflash.c) → SPI HW → NAND 芯片

SD NAND 方案（只需 2 层）：
  App → VFS → FAT(fs.a) → SD Driver(sd_dev_ops) → SDIO HW(JL_SD0) → XCSD02G-MJ
```

SD NAND 少了一个 FTL 层，因为芯片内部 SD2.0 控制器已完整实现坏块管理、磨损均衡和 ECC。Host 端看到的是一个标准 FAT32 块设备。

### 2.1 设备注册模型

SD NAND 在 JL 平台采用**单设备注册**模式——仅一个 `"sd0"` 设备，直接向上提供块设备接口：

```c
// device_config.c — 硬件设备注册（源码可见）
REGISTER_DEVICES(device_table) = {
#if TCFG_SD0_ENABLE
    { "sd0", &sd_dev_ops, (void *)&sd0_data },
#endif
    ...
};

// dev_reg.c — 逻辑盘注册（源码可见）
#if (TCFG_SD0_ENABLE)
    {
        /*logo*/         "sd0",
        /*name*/         "sd0",
        /*storage_path*/ "storage/sd0",
        /*root_path*/    "storage/sd0/C/",
        /*fs_type*/      "fat",
    },
#endif
```

与 SPI NAND 的"双设备"（`"nand_flash"` + `"nandflash_ftl"`）不同，SD NAND 只有一个逻辑设备。不需要中间 FTL 设备来桥接裸驱动和文件系统。

**文件系统直接挂载在 `"sd0"` 上。**

### 2.2 SD 协议初始化流程

SD NAND 的初始化流程与标准 SD 卡完全一致。SD 协议上电默认 1-bit 模式（仅 DAT0），无需任何特殊配置：

```
Step 1:  上电 → 发送 ≥74 个虚拟时钟周期（CMD 线保持高电平）
Step 2:  CMD0  (GO_IDLE_STATE)        → 复位到 Idle 状态
Step 3:  ACMD41 (SD_SEND_OP_COND)     → 查询工作电压范围，轮询 busy 位
Step 4:  CMD2  (ALL_SEND_CID)         → 获取 CID
Step 5:  CMD3  (SEND_RELATIVE_ADDR)   → 获取 RCA，每次上电随机变化
Step 6:  CMD7  (SELECT_CARD)          → 选中卡，进入传输状态
Step 7:  CMD9  (SEND_CSD)             → 获取 CSD（含容量信息）
Step 8:  ACMD6 (SET_BUS_WIDTH)        → [可选] 切换到 4-bit 模式
```

> **注意**：上述序列为 SD2.0 标准流程。由于 `sd_dev_ops` 闭源，实际初始化序列由驱动内部实现决定。规格书 5.4 节还建议了 `ACMD13`（读 SD Status）和 `CMD10`（再次读 CID）等可选步骤。

**对于 1-bit 模式：Step 8 可省略。** 上电默认就是 1-bit，不需要发送 ACMD6。

### 2.3 模式选择机制

SD NAND 支持 SD 模式和 SPI 模式两种操作模式。DAT3 引脚在上电时的电平决定模式：

| DAT3 电平 | 发送 CMD0 后进入 | 说明 |
|-----------|:---:|------|
| 高电平（上拉） | **SD 模式** | DAT3 通过 10K~100KΩ 上拉到 VDD |
| 低电平 | SPI 模式 | 需要 Host 主动拉低 |

由于 SD NAND 焊死在 PCB 上，DAT3 只接上拉电阻（未接 GPIO），上电后自然为高电平，芯片**自动进入 SD 模式**——完全符合预期。

### 2.4 文件系统支持

SD NAND 只支持 `"fat"` 这一种文件系统类型，这与 SPI NAND 一致。但 SD NAND 有一个关键优势：**出厂预格式化**。

XCSD02G-MJ 出厂时已格式化为 FAT32（规格书 4.2 节），用户可直接使用 240MB（02G 型号）。这意味着：

- SPI NAND：首次使用需要 `f_format("nandflash_ftl", "fat", 4096)` 手动格式化
- SD NAND：通常挂载即用，不需要手动格式化步骤

> **补充**：为兼容实际焊接、测试过程中可能出现的未格式化或文件系统损坏情况，当前项目在 `dev_manager.c` 中实现了 `TCFG_SD0_FORMAT_ON_BOOT` 机制，可在首次上电或 VM 标志位丢失时自动格式化 `sd0`，避免量产后重复格式化用户数据。详见 §3.3.3。

### 3.0 第零步：GPIO 引脚冲突检查（🔴 必须确认）

当前项目的 SD NAND 硬件连接在 **PC03/PC04/PC05**，与以下外设共享引脚：

| 引脚 | SD0 (当前配置) | SPI1 (已关闭) | MPU6887P IMU (已启用) |
|------|---------------|--------------|---------------------|
| PC03 | `TCFG_SD0_PORT_DA0` | ~~`TCFG_HW_SPI1_PORT_DO`~~ | `TCFG_MPU6887P_AD0_SELETE_IO` ⚠️ |
| PC04 | `TCFG_SD0_PORT_CLK` | ~~`TCFG_HW_SPI1_PORT_CLK`~~ | — |
| PC05 | `TCFG_SD0_PORT_CMD` | ~~`TCFG_HW_SPI1_PORT_CS`~~ | — |

**当前解决策略**：

1. **SPI1 已关闭**：`TCFG_HW_SPI1_ENABLE = 0`，释放 SPI1 功能占用，PC03/04/05 不再被 SPI 复用
2. **NAND Flash 已关闭**：`TCFG_NANDFLASH_DEV_ENABLE` 未使能（在 `board_ac701n_demo_cfg.h` 中未定义，默认为 0），SPI NAND 相关注册不编译，避免与 SD0 冲突
3. **MPU6887P AD0 共用 PC03**：IMU 的 I2C 地址选择脚 AD0 仅在初始化时通过 `imu_sensor_ad0_selete()` 设置一次电平，之后不再操作。SD 驱动初始化时会将 PC03 复用为 SDIO DAT0，而 IMU 的 I2C 地址已锁定，因此**运行时无冲突**。但注意——这是"时机分离"的共存，并非真正的引脚复用。如果 IMU 需要在运行中动态切换地址，则必须将 AD0 改到其他 IO

**对其他项目的建议**：

如果硬件允许，最好将 SD NAND 接到不与任何外设冲突的独立 IO。若必须共用，至少确保：
- 共用引脚的外设**不在 SD 通信期间操作**
- 或使用软件 I2C/SPI 走不同 GPIO（当前 IMU 用软件 I2C 走 PG07/PG08）

### 3.1 第一步：硬件连接

#### 3.1.1 最小连接方案（1-bit 模式）

SD NAND 在 1-bit 模式下只需 **5 个物理连接**：GND、CMD、VDD、CLK、DAT0。

```
         ┌─────────────────────┐
  DAT2 ──┤1 (上拉到 VDD)   8├── VDD (3.3V + 2.2μF 去耦)
  DAT3 ──┤2 (上拉到 VDD)   7├── DAT1 (上拉到 VDD)
   CLK ──┤3 (33Ω 串联)     6├── DAT0 → MCU GPIO (上拉)
   GND ──┤4                5├── CMD → MCU GPIO (上拉)
         └─────────────────────┘
```

#### 3.1.2 上拉电阻要求

这是 SD NAND 硬件连接中**最容易出问题的环节**。规格书多处强调（2.1 节、5.4 节、6.5 节）：CMD 和**所有 DAT 线**都必须通过 10K~100KΩ 上拉到 VDD。

**关键是 DAT1~3 即使不使用，也必须上拉。** DAT3 的上拉尤为关键——它不仅是数据线，还承担 SD 模式选择功能（高电平 = SD 模式）。如果 DAT3 因漏接上拉而浮空，芯片可能误入 SPI 模式，导致 SD 协议初始化失败。

| 信号 | 处理 | 备注 |
|------|------|------|
| CMD | 10KΩ 上拉到 VDD | 防止命令线悬浮，必接 |
| DAT0 | 10KΩ 上拉到 VDD | 防止数据线悬浮，必接 |
| DAT1 | 10KΩ 上拉到 VDD | **虽不使用，规格书要求上拉** |
| DAT2 | 10KΩ 上拉到 VDD | **虽不使用，规格书要求上拉** |
| DAT3 | 10KΩ 上拉到 VDD | **虽不使用，规格书要求上拉；此引脚上拉 = 选择 SD 模式** |
| CLK | 33Ω 串联 | 限流/阻抗匹配（规格书 2.1 节推荐 0~120Ω） |
| VDD | 2.2μF 去耦电容到 GND | 规格书 2.1 节建议 |

### 3.2 第二步：配置 SD0 引脚与时钟

在 `sdk_config.h` 中配置 SD0 的硬件参数。以下为推荐配置（引脚需按实际 PCB 连接调整）：

```c
// sdk_config.h — SD0 硬件配置（当前项目实际工作值）
#define TCFG_SD0_ENABLE                  1                // 启用 SD0
#define TCFG_SD0_DAT_MODE                1                // 1-bit 单线模式
#define TCFG_SD0_DET_MODE                SD_IO_DECT       // IO 检测（配合自定义 detect_func）
#define TCFG_SD0_CLK                     1000000          // 1MHz（保守值，稳定后可提升）
#define TCFG_SD0_PORT_CMD                IO_PORTC_05      // CMD → PC05
#define TCFG_SD0_PORT_CLK                IO_PORTC_04      // CLK → PC04
#define TCFG_SD0_PORT_DA0                IO_PORTC_03      // DAT0 → PC03
#define TCFG_SD0_PORT_DA1                NO_CONFIG_PORT   // 1-bit 模式不接
#define TCFG_SD0_PORT_DA2                NO_CONFIG_PORT   // 1-bit 模式不接
#define TCFG_SD0_PORT_DA3                NO_CONFIG_PORT   // 1-bit 模式不接
#define TCFG_SD0_POWER_SEL               SD_PWR_NULL      // 不使用内部 SDPG，由 GPIO 控制外部供电
#define TCFG_SDX_CAN_OPERATE_MMC_CARD    0                // 纯 SD2.0 设备，不需要 MMC 回退
#define TCFG_KEEP_CARD_AT_ACTIVE_STATUS  0                // 允许空闲休眠（TWS 场景无感知影响）
```

#### 卡检测策略

`TCFG_SD0_DET_MODE` 在 `device_config.c` 中有**双重作用**：

```c
.detect_mode = TCFG_SD0_DET_MODE,          // ① 传给闭源驱动，影响内部初始化行为

#if (TCFG_SD0_DET_MODE == SD_IO_DECT)      // ② 选择 detect_func 指针
    .detect_func = sdmmc_0_io_detect,      //    (可替换为自定义函数)
#elif (TCFG_SD0_DET_MODE == SD_CLK_DECT)
    .detect_func = sdmmc_0_clk_detect,
#else  // SD_CMD_DECT
    .detect_func = sdmmc_cmd_detect,
#endif
```

JL SDK 提供三种检测方式（`sdmmc.h:9-11`）：

| 检测方式 | 宏（值） | 原理 | 适用场景 |
|---------|-----|------|---------|
| CMD 检测 | `SD_CMD_DECT`(0) | 发送 CMD0 并等待响应 | 可插拔卡座，无需额外检测引脚 |
| CLK 检测 | `SD_CLK_DECT`(1) | 检测 CLK 引脚电平变化 | 需要 CLK 引脚支持复用检测 |
| IO 检测 | `SD_IO_DECT`(2) | 读取指定 IO 引脚电平判断卡是否在位 | 有专用卡检测引脚（如卡座机械开关） |

**焊死 SD NAND 的三种应对方案**：

| 方案 | 机制 | 修改文件 | DET_MODE 是否生效 | 适用场景 |
|------|------|---------|:---:|---------|
| **A: ALWAY_ONLINE** | `force_set_sd_online("sd0")` 直接标记在线，**完全跳过**检测定时器和 detect_func | `board_ac701n_demo_cfg.h`（加宏），`dev_manager.c`（加 else 定时器修复） | ✗ 被绕过 | 焊死芯片，追求最快启动 |
| **B: 检测定时器 + 标准 detect_func** | `sdx_dev_detect_timer` 轮询，标准库函数判断卡在位 | `dev_manager.c`（加定时器），`dev_status.c`（修复 IOCTL 编译错误） | ✓ | 需要真实插拔检测的场景 |
| **C: 检测定时器 + 自定义 detect_func**（⭐ 当前方案） | 标准库 detect_func 替换为**始终返回 1** 的自定义函数，检测定时器一触发即认为卡在线 | `device_config.c`（加自定义函数） | △ 设为哪个 mode 就用哪个分支，但函数被替换 | 焊死芯片，改动最小，不需要改 `dev_manager.c` |

**当前项目采用方案 C**，在 `device_config.c` 中实现了 `sdmmc_0_io_detect_user()`：

```c
int sdmmc_0_io_detect_user(const struct sdmmc_platform_data *p)
{
    return 1;  // SD NAND 始终在线
}
```

当前 `TCFG_SD0_DET_MODE = SD_IO_DECT`（值为 2），因此自定义函数放在 `#if SD_IO_DECT` 分支中。若改为 `SD_CMD_DECT`，则需将自定义函数移到 `#else` 分支替换 `sdmmc_cmd_detect`，否则会回退到真实的 CMD 检测（可能失败）。

> **总结**：方案 C 的优势是**改动最小**——不需要修改 `dev_manager.c` 和 `dev_status.c`，只需在 `device_config.c` 加一个 3 行的函数即可。代价是检测定时器仍在运行（但立即返回 1，实际开销可忽略）。对于焊死的 SD NAND，这是我们推荐的方案。

**⚠️ 检测方式 vs 休眠控制：两个不同阶段的配置，不要混淆**

| | 检测方式（`DET_MODE` + `detect_func`） | 休眠控制（`KEEP_ACTIVE`） |
|---|---|---|
| 作用阶段 | **初始化/挂载阶段** | **运行时** |
| 解决的问题 | "SD 卡是否存在？是否应该挂载？" | "SD 卡空闲时是否允许进入 Standby 休眠？" |
| 失败后果 | **不挂载，文件系统不可用** | 空闲唤醒时 ~1ms 延迟（不影响功能） |
| 当前值 | `SD_IO_DECT` + 自定义函数返回 1 | `TCFG_KEEP_CARD_AT_ACTIVE_STATUS = 0`（允许休眠） |

**检测方式是必须选的**——无论哪种方案（A/B/C），总要有一个机制让系统知道 SD NAND 已就绪。方案 C 不是"跳过检测"，而是将检测函数替换为始终返回 1，本质上等价于"系统问有没有卡 → 回答有"。`KEEP_ACTIVE` 则完全是另一个维度的问题，它决定卡活跃之后的行为。两者在源码中也是独立变量（`lib_driver_config.c` 第 24、30 行），互不影响。

#### SD 卡电源选择

`TCFG_SD0_POWER_SEL` 控制 SD 卡的供电策略（`sdmmc.h:13-14`）：

| 值 | 宏 | `.power` 回调 | 行为 | 适用场景 |
|---|---|--------------|------|---------|
| `0` | `SD_PWR_SDPG` | `sd_set_power`（闭源库函数） | 操作 BR28 芯片**内部 SDIO 电源门控寄存器**（SDPG），控制 SDIO 引脚（CMD/CLK/DAT）的 IOVDD 输出。驱动在探测前自动上电，检测到拔出后自动断电。驱动电流 ≤64mA，内阻 ≤4Ω。 | 可插拔 SD 卡座，需要热插拔电源管理 |
| `1` | `SD_PWR_NULL` | `sd_set_power_user`（用户自定义） | 驱动**不管理电源**，由用户在 `app_main.c` 中自行上电（通常通过 GPIO 控制外部供电）。适合不需要热插拔的场景。 | 焊死在 PCB 上的芯片，或已通过其他方式常供电 |

> **实测验证**：当前项目 XCSD02G-MJ 由 LDO3V 单独供电，PA00 作为电源使能控制开关。`SD_PWR_NULL` 模式下 `.power` 回调为 `sd_set_power_user()`，由 SD 驱动在探测阶段自动调用，因此不需要也不应该在 `app_main.c` 中再手动调用 `sd_set_power_user(1)`。早期版本曾使用 `TCFG_IO_CFG_AT_POWER_ON` 在开机后一直拉高 PA00，但这样无法配合 SD 驱动的开关时序；当前已将电源控制完全交给驱动回调。
>
> **⚠️ SDPG 的副作用：干扰同一 IOVDD 域的其他外设**
>
> `SD_PWR_SDPG` 模式下，闭源驱动在 SD 探测阶段会**反复开关 SDPG**（上电→探测→断电→重试→……）。SDPG 控制的是整个 SDIO I/O 环的 IOVDD 供电，而同一 IOVDD 域上可能还有别的外设（如 LED、GPIO 等）。每次 SDPG 切换都会导致该域 IOVDD 波动，表现为同域外设供电不稳。
>
> **实际问题案例**：当前项目的 LED 使用 PC02，与 SDIO 引脚（PC03/PC04/PC05）同处 Port C，共享同一个 IOVDD 域。`SD_PWR_SDPG` 模式下 LED 出现人眼可见的闪烁——因为闭源驱动在探测期间反复拉低/拉高 SDPG，LED 的 IOVDD 随之波动：
>
> ```
> SD_PWR_NULL + sd_set_power_user:
>   LDO3V ─→ [PA00 使能] ─→ SD NAND VDD
>   系统 3V ─→ IOVDD ─→ PC02(LED) + PC03/04/05(SDIO)  ← 独立路径，不受 SDPG 影响
>
> SD_PWR_SDPG + sd_set_power:
>   系统 3V ─→ [SDPG门控 ≤64mA] ─→ IOVDD ─→ PC02(LED) + PC03/04/05(SDIO)
>                                     ↑ SDPG 反复开关导致整个域波动
> ```
>
> `sd_set_power_user()` 在 `en=1` 时拉高 PA00 给 SD NAND 上电，在 `en=0` 时默认不拉低 PA00（只记录请求），避免探测阶段的反复断电影响 LDO3V 稳定性。
>
> **结论**：焊死芯片建议使用 `SD_PWR_NULL`，除了避免无意义的热插拔管理外，更重要的是**防止 SDPG 开关干扰同一 IOVDD 域上的其他外设**，同时把 SD NAND 的独立供电开关时机交给驱动回调。

**`sd_set_power()` vs `sd_set_power_user()` 的本质区别**：

| | `sd_set_power`（SDPG） | `sd_set_power_user`（NULL） |
|---|---|---|
| 实现位置 | 闭源库 `device.a` | `app_main.c`，用户自行实现 |
| 控制对象 | BR28 内部 SDIO 电源门控寄存器 | 外部 GPIO（当前为 `IO_PORTA_00`） |
| 调用时机 | 闭源驱动在 SD 探测前自动调用 | 由 SD 驱动在初始化/探测阶段自动调用 |
| 自动断电 | 驱动检测到拔出时自动调 `sd_set_power(0)` | 无自动断电，`en=0` 默认不拉低 GPIO |
| 参数 | 不需要 GPIO 参数（操作寄存器） | 需要显式指定 GPIO |

对应 `device_config.c:54-58`（当前实际代码）：

```c
#if (TCFG_SD0_POWER_SEL == SD_PWR_SDPG)
    .power  = sd_set_power,         // 闭源库函数 → 芯片内部 SDPG 门控
#else
    .power  = sd_set_power_user,    // 用户自定义 → 外部 GPIO 控制
#endif
```

`app_main.c` 中的实现（当前实际代码）：

```c
void sd_set_power_user(u8 en)
{
    if(en){
        gpio_set_mode(IO_PORT_SPILT(IO_PORTA_00), PORT_OUTPUT_HIGH);
#if TCFG_SD0_DIAG_ENABLE
        printf("[SD-PWR] PA00 on\n");
#endif
    } else {
#if TCFG_SD0_DIAG_ENABLE
        /* 保持原始回调行为：默认不在驱动 power-off 请求里拉低 PA00，只记录请求。 */
        printf("[SD-PWR] PA00 off request ignored by default callback\n");
#endif
    }
}
```

> **注意**：`app_main.c` 中原本手动调用的 `sd_set_power_user(1);` 已注释掉。因为 `TCFG_SD0_POWER_SEL = SD_PWR_NULL` 时，闭源 SD 驱动会在合适的时机自行调用 `.power(1)`，手动提前调用反而可能打乱上电时序。

#### MMC 卡支持

`TCFG_SDX_CAN_OPERATE_MMC_CARD` 控制闭源 `sd_dev_ops` 驱动是否尝试 **MMC/eMMC 协议初始化**（`lib_driver_config.c:30-33`）：

```c
#if TCFG_SDX_CAN_OPERATE_MMC_CARD
const int sdx_can_operate_mmc_card = TCFG_SDX_CAN_OPERATE_MMC_CARD;
#else
const int sdx_can_operate_mmc_card = 0;
#endif
```

这个变量被闭源 `sd_dev_ops` 驱动读取。MMC 和 SD 虽然物理接口兼容（CLK/CMD/DAT），但**初始化命令序列不同**——MMC 使用 `CMD1`（SEND_OP_COND）而非 `ACMD41`。开启此选项后，驱动会在 SD 协议初始化失败时，额外尝试 MMC 协议作为回退。

| 值 | 效果 |
|:---:|------|
| `0` | 仅使用标准 SD 协议初始化（CMD0 → ACMD41 → CMD2 → ...） |
| `1` | SD 协议失败后，回退尝试 MMC 协议（CMD0 → CMD1 → CMD2 → ...） |

XCSD 系列 SD NAND 是纯 SD2.0 协议设备，**不需要 MMC 支持**，设为 **`0`**。若设为 `1`，驱动在 SD 初始化失败后会多走一轮 MMC 尝试，无实际作用且增加初始化耗时。

#### 时钟策略

| 阶段 | 时钟范围 | 说明 |
|------|---------|------|
| 识别模式 (Identification) | 100 ~ 400 KHz | 初始化阶段，CMD0~CMD3。SD 驱动自动降速 |
| 数据传输模式 (Default Speed) | 0 ~ 25 MHz | 常规读写 |
| 数据传输模式 (High Speed) | 0 ~ 50 MHz | XCSD02G-MJ 最高支持 50MHz |

当前项目使用 `TCFG_SD0_CLK = 1MHz` 作为初始保守值（已验证稳定）。稳定后可按上表将数据传输频率提升至 12~25MHz。

### 3.3 第三步：设备检测与挂载机制

**当前项目未使用 `TCFG_SD_ALWAY_ONLINE_ENABLE`**（该宏在整个代码库中未定义，`dev_manager.c:1029` 的 `#if` 分支不编译）。实际挂载由**闭源驱动内部检测 + 自定义 detect_func** 触发。

#### 实际挂载流程

SD NAND 的挂载有**两个独立路径**，首次挂载走路径 1：

```
路径 1（开机自动，闭源驱动触发）：
  devices_init()
    └── sd_dev_ops.init(sd0_data)                   [闭源 device.a]
          └── 内部检测逻辑调用 detect_func
                └── sdmmc_0_io_detect_user() → return 1
                      └── sdx_dev_event_to_user(DRIVER_EVENT_FROM_SD0, IN, 0)
                            └── os_taskq_post_type("dev_mg", ...)
                                  └── dev_status.c: dev_manager_add("sd0")
                                        └── mount("sd0", "storage/sd0", "fat", ...)

路径 2（运行时，RDX 协议栈触发）：
  录音/PC模式/DUT 等场景
    └── rdx_app_emmc_poweron(0)              [rdx_app.c]
          └── sd_set_power(1)                [闭源 SDPG 上电]
    └── dev_manager_add("sd0")               [rdx_record.c]
          └── 若已挂载则直接返回成功，无副作用
```

**关键代码**（`device_config.c`）：

```c
extern void sd_set_power_user(u8 en);

int sdmmc_0_io_detect_user(const struct sdmmc_platform_data *p)
{
    return 1;  // SD NAND 始终在线
}
```

闭源驱动 `sd_dev_ops.init()` 内部会调用此函数判断卡是否在位。返回 1 后，闭源驱动通过 `sdx_dev_event_to_user()` 向 `dev_mg` 任务投递 `DEVICE_EVENT_FROM_SD0` 事件，触发 `dev_status.c` 中的挂载逻辑。

#### 备选方案对比

| 方案 | 机制 | 适用场景 |
|------|------|---------|
| **方案 C**（⭐ 当前方案）：自定义 `detect_func` 返回 1 | 闭源驱动内部检测 → 直接返回"卡在位" | 焊死芯片，改动最小（只改 `device_config.c`） |
| **方案 A**：`TCFG_SD_ALWAY_ONLINE_ENABLE = 1` | `force_set_sd_online()` + 直接 `dev_manager_add()` | 焊死芯片，完全跳过检测框架 |
| **方案 B**：检测定时器 + 标准库函数 | `sdx_dev_detect_timer` 轮询 + 真实 IO/CMD/CLK 检测 | 可插拔卡座场景 |

> **当前项目选择方案 C 的原因**：只需要在 `device_config.c` 添加 3 行自定义函数，不需要修改 `dev_manager.c` 或 `dev_status.c`，也不会因 SDPG 反复开关影响同域外设（如 LED）。

#### 电源控制

当前项目 SD NAND 采用**LDO3V 独立供电 + PA00 使能控制**：

| 函数 | 控制对象 | 调用位置 | 实现 |
|------|---------|---------|------|
| `sd_set_power_user(1)` | PA00 GPIO → LDO3V 使能 → SD NAND VDD | 闭源 SD 驱动在探测阶段自动调用 | `gpio_set_mode(IO_PORT_SPILT(IO_PORTA_00), PORT_OUTPUT_HIGH)` |
| `sd_set_power(1)` | BR28 内部 SDPG 门控 → SDIO 引脚 IOVDD | `rdx_app_emmc_poweron()`（运行时场景触发） | 闭源库函数 |

两者互不冲突，形成双重保障。注意 `TCFG_SD0_POWER_SEL = SD_PWR_NULL` 时 `.power` 回调为 `sd_set_power_user`，但 RDX 协议栈通过 `rdx_app_emmc_poweron()` 额外调用 `sd_set_power()`，这是在 `.power` 回调之外的独立调用。`app_main.c` 中已不再手动调用 `sd_set_power_user(1)`，避免干扰驱动回调的上电时序。

#### 上电自动格式化（TCFG_SD0_FORMAT_ON_BOOT）

虽然 XCSD02G-MJ 规格书声称出厂已格式化为 FAT32，但实际到手的芯片/批次可能存在未格式化或文件系统损坏的情况。当前项目在 `dev_manager.c` 中增加了 `TCFG_SD0_FORMAT_ON_BOOT` 机制，用于**首次上电自动格式化**，并避免后续重复格式化：

```
mount("sd0", "storage/sd0", "fat") 失败？
  ├─ 是：读取 VM 标志位 CFG_SD0_FORMAT_DONE_FLAG
  │      ├─ 标志位有效 → 认为文件系统损坏，执行 f_format() 恢复
  │      └─ 标志位无效 → 认为首次上电，执行 f_format() 初始化
  │      格式化成功后写入 VM 标志位，并重新 mount
  └─ 否：直接跳过格式化分支
```

关键代码逻辑（`dev_manager.c:471-531`）：

```c
#if (TCFG_SD0_ENABLE && TCFG_SD0_FORMAT_ON_BOOT)
if (!strcmp(logo, "sd0")) {
    static u8 _sd0_fmt_done = 0;
    if (!_sd0_fmt_done) {
        _sd0_fmt_done = 1;

        if (dev->fmnt) {
            printf("[SD-FMT] sd0 mount ok, skip format on boot\n");
        } else {
            u8 fmt_flag = 0;
            int rd = syscfg_read(CFG_SD0_FORMAT_DONE_FLAG, &fmt_flag, sizeof(fmt_flag));
            u8 vm_formatted = (rd == sizeof(fmt_flag) && fmt_flag == TCFG_SD0_FORMAT_DONE_MAGIC);

            if (vm_formatted) {
                printf("[SD-FMT] fs damaged, recover by format...\n");
            } else {
                printf("[SD-FMT] first boot or vm cleared, format sd0...\n");
            }
            os_time_dly(50);

            int ret = f_format("storage/sd0/C/", "fat", 0);
            if (ret == 0) {
                // 重新 mount 并写入 VM 标志位
            }
        }
    }
}
#endif
```

| 宏 | 作用 | 推荐值 |
|---|---|:---:|
| `TCFG_SD0_FORMAT_ON_BOOT` | 使能开机自动格式化：mount 成功则跳过，mount 失败则根据 VM 标志位决定是否格式化 | `ENABLE_THIS_MOUDLE` |
| `TCFG_SD0_FORCE_FORMAT_ON_BOOT` | 调试用，强制无论 mount 是否成功都走格式化分支 | `DISABLE_THIS_MOUDLE` |
| `TCFG_SD0_FORMAT_DONE_MAGIC` | 写入 VM 的标志位魔数，用于识别是否已经格式化过 | 项目自定义 |

**为什么既需要这个宏，又保留 "出厂已格式化" 的说法？**

- 规格书层面的 "出厂已格式化" 指的是芯片出厂状态；
- 项目级的 `TCFG_SD0_FORMAT_ON_BOOT` 是为了兼容实际焊接、运输、测试过程中可能出现的文件系统异常，同时避免量产后重复格式化用户数据。

**注意事项**：

- 该机制依赖 VM（syscfg）持久化标志位。若后续 OTA/烧录策略会擦除 VM，则升级后可能再次触发格式化，需同步调整策略。
- 只有在 `sd0` mount 失败时才会格式化；mount 成功时完全无感。
- `f_format("storage/sd0/C/", "fat", 0)` 中的第 3 个参数为 0，表示使用默认簇大小。

### 3.4 第四步：注册设备与挂载

设备注册分两层，SDK 已有完整代码，**无需修改**：

**第一层：硬件设备注册**（`device_config.c`）：

```c
REGISTER_DEVICES(device_table) = {
#if TCFG_SD0_ENABLE
    { "sd0", &sd_dev_ops, (void *)&sd0_data },
#endif
};
```

`devices_init()` 在启动时遍历 `device_table`，调用 `sd_dev_ops.init(sd0_data)` 完成硬件初始化（GPIO 功能复用、SD 控制器复位、中断注册）。

**第二层：逻辑盘注册**（`dev_reg.c`）：

```c
#if (TCFG_SD0_ENABLE)
    {
        /*logo*/         "sd0",
        /*name*/         "sd0",
        /*storage_path*/ "storage/sd0",
        /*root_path*/    "storage/sd0/C/",
        /*fs_type*/      "fat",
    },
#endif
```

`dev_manager_add("sd0")` 调用 `__dev_manager_add("sd0", 1)` 遍历 `dev_reg[]` 按 `logo` 字段匹配配置项，然后调用 `mount("sd0", "storage/sd0", "fat", ...)` 挂载。

**完整启动流程**：

```
系统上电
  ↓
devices_init()
  ├── sd_dev_ops.init(sd0_data)
  │   ├── sdmmc_0_port_init()         → 初始化 GPIO 引脚功能复用
  │   ├── SD 控制器硬件复位
  │   └── 注册 SD 中断处理
  ↓
dev_manager_init()
  ↓
dev_manager_task()
  ├── [TCFG_SD_ALWAY_ONLINE_ENABLE 路径] dev_manager.c:1029-1047    ← 可选，当前项目未使用
  │   ├── force_set_sd_online("sd0")    → 绕过插拔检测，强制标记在线
  │   ├── dev_manager_add("sd0")        → 挂载逻辑盘
  │   └── sdx_dev_detect_timer_del()    → 关闭检测定时器
  │
  ├── dev_manager_add("sd0") 内部:
  │   ├── __dev_manager_add("sd0", 1)
  │   │   └── 遍历 dev_reg[]，按 logo "sd0" 匹配配置项
  │   └── mount("sd0", "storage/sd0", "fat", ...)
  │       ├── sd_dev_ops.open("sd0")
  │       │   ├── 发送 ≥74 个虚拟时钟
  │       │   ├── CMD0 → ACMD41 → CMD2 → CMD3 → CMD7 → CMD9
  │       │   └── 卡就绪，接受块读写
  │       ├── FAT 超级块读取 → 验证 BPB 签名
  │       ├── FAT 缓存初始化
  │       └── 挂载成功 → 挂载点: storage/sd0/C/
  ↓
应用层可通过 fopen("storage/sd0/C/...") 访问
```

### 3.5 第五步：通过 VFS 进行文件读写

与 SPI NAND 完全相同的接口。SD NAND 挂载后，通过标准 C 风格的文件操作访问：

```c
// === 写入文件 ===
FILE *fp = fopen("storage/sd0/C/test.txt", "w+");
if (fp) {
    fwrite("Hello SD NAND!", 1, 15, fp);
    fclose(fp);
}

// === 读取文件 ===
fp = fopen("storage/sd0/C/test.txt", "r");
if (fp) {
    char buf[64];
    int len = fread(buf, 1, sizeof(buf), fp);
    fclose(fp);
}

// === 删除文件 ===
fdelete_by_name("storage/sd0/C/test.txt");

// === 获取剩余空间 (单位: KB) ===
u32 free_space;
fget_free_space("storage/sd0/C/", &free_space);

// === 扫描目录 ===
struct vfscan *fs = fscan("storage/sd0/C/", "-tMP3 -r", 3);
if (fs && fs->file_number > 0) {
    FILE *file = fselect(fs, FSEL_FIRST_FILE, 0);
    // 使用 file ...
}
fscan_release(fs);
```

完整的 VFS 接口参考 `SDK/interface/utils/fs/fs.h`。

### 3.6 补充配置项

#### TCFG_KEEP_CARD_AT_ACTIVE_STATUS

控制闭源 `sd_dev_ops` 驱动在**空闲时是否保持卡在活跃（Transfer）状态**（`lib_driver_config.c:24-28`）：

```c
#if TCFG_KEEP_CARD_AT_ACTIVE_STATUS
const int keep_card_at_active_status = 1;
#else
const int keep_card_at_active_status = 0;
#endif
```

这个变量被闭源驱动读取，决定在连续读写间隔超过一定时间后，是否主动发送命令让卡保持活跃。

**SD 卡状态机回顾**：SD 协议中，卡在完成一次读写后，若 Host 持续不发送命令，卡会从 Transfer State 自动进入 Standby State（省电）。下一次读写前需要先发送 CMD7（SELECT_CARD）重新选中才能回到 Transfer State，这个过程约 **1ms**。

| 值 | 行为 | 优点 | 缺点 |
|:---:|------|------|------|
| `0` | 空闲时让卡休眠到 Standby | 更省电 | 下次访问有 ~1ms 唤醒延迟 |
| `1` | 空闲时驱动主动维持活跃 | **无唤醒延迟**，响应更快 | 略增加功耗（约 mA 级） |

**当前项目**：使用 `0`（允许休眠），TWS 耳机场景下 ~1ms 唤醒延迟无感知影响，且节省空闲功耗。若未来发现频繁读写场景下延迟累积影响体验，可改为 `1`。

#### TCFG_RECORD_FOLDER_DEV_ENABLE

| 宏 | 推荐值 | 说明 |
|------|:---:|------|
| `TCFG_RECORD_FOLDER_DEV_ENABLE` | 按需 | 如果同时启用，系统会额外创建 `sd0_rec` 逻辑分区（`dev_reg.c:72-81`），指向 `storage/sd0/C/REC/` 录音专用文件夹 |

#### TCFG_SD0_FORMAT_ON_BOOT

详见 §3.3.3。该宏在 `board_ac701n_demo_cfg.h` 中定义，用于控制开机时是否对 `sd0` 执行自动格式化。量产时保持 `ENABLE_THIS_MOUDLE`，可兼容出厂未格式化或文件系统损坏的芯片，同时通过 VM 标志位避免重复格式化。

#### TCFG_SD0_DIAG_ENABLE

`board_ac701n_demo_cfg.h` 中定义的 bring-up 诊断总开关。开启后会在 `sd0` mount 前后增加以下日志：

- mount 调用与结果
- raw open / ioctl / LBA0 读取状态
- `sd_set_power_user()` 被调用时的 PA00 开关日志

**默认必须关闭**（`DISABLE_THIS_MOUDLE`），避免已调通芯片的正常 mount 时序被额外日志和探测动作干扰。仅在定位 bring-up 问题时临时打开。

## 4. 集成点

### 4.1 USB MSD (U盘模式)

SD NAND 可以作为 USB 大容量存储设备暴露给 PC。前提是使能 `TCFG_USB_SLAVE_MSD_ENABLE = 1`（`sdk_config.h:109`）。SDK 在 `task_pc.c:258-260` 中自动将 `sd0` 注册为 USB mass storage 设备：

```c
#if TCFG_USB_SLAVE_MSD_ENABLE
#if (!TCFG_USB_DM_MULTIPLEX_WITH_SD_DAT0 && TCFG_SD0_ENABLE)\
     ||(TCFG_SD0_ENABLE && TCFG_USB_DM_MULTIPLEX_WITH_SD_DAT0 && TCFG_DM_MULTIPLEX_WITH_SD_PORT != 0)
    msd_register_disk("sd0", NULL);
#endif
```

**注意**：如果 USB DM 引脚与 SD DAT0 复用（`TCFG_USB_DM_MULTIPLEX_WITH_SD_DAT0`），需额外确认 `TCFG_DM_MULTIPLEX_WITH_SD_PORT` 配置。对于 SD NAND 焊死场景，通常不涉及此复用。

### 4.2 OTA 升级

支持从 SD 卡读取升级固件进行 OTA，设备类型为 `SD0_UPDATA`（`update.h:30`，值 `0x5A01`）。与之对应，SPI NAND 的设备类型为 `USER_NANDFLASH_UFW_UPDATA`（`update.h:46`）。

### 4.3 设备管理器

设备管理器的完整生命周期管理：

| 阶段 | 函数调用 | 说明 |
|------|---------|------|
| 初始化 | `dev_manager_init()` → `devices_init()` | 注册所有设备 |
| 挂载 | `dev_manager_add("sd0")` | 自动 mount FAT 文件系统 |
| 检查 | `dev_manager_online_check_by_logo("sd0", 1)` | 检查设备是否在线 |
| 获取根路径 | `dev_manager_get_root_path_by_logo("sd0")` | 返回 `"storage/sd0/C/"` |
| 卸载 | `dev_manager_unmount("sd0")` | 卸载文件系统 |

### 4.4 与 SPI NAND 的共存

两个方案可以通过编译期宏切换：

```c
// board_ac701n_demo_cfg.h
#define TCFG_STORAGE_TYPE_SPI_NAND    0
#define TCFG_STORAGE_TYPE_SD_NAND     1

#if TCFG_STORAGE_TYPE_SD_NAND
  #define TCFG_SD_ALWAY_ONLINE_ENABLE       ENABLE_THIS_MOUDLE   // 可选，当前项目未使用
  #define TCFG_SD0_ENABLE                   ENABLE_THIS_MOUDLE
  #define TCFG_NANDFLASH_DEV_ENABLE         DISABLE_THIS_MOUDLE
#elif TCFG_STORAGE_TYPE_SPI_NAND
  #define TCFG_NANDFLASH_DEV_ENABLE         ENABLE_THIS_MOUDLE
  #define TCFG_SD0_ENABLE                   DISABLE_THIS_MOUDLE
#endif
```

运行时共存（同时挂载两种存储）理论上可行，但需确保 GPIO 无冲突——SPI1 的引脚和 SD0 的引脚分配到不同的 IO 端口组。

## 5. 常见问题

### Q1: SD NAND 和 SD 卡驱动是同一个吗？

是的。`sd_dev_ops` 是通用的 SD/MMC 块设备驱动，对 SD 卡、SD NAND、MMC 卡均使用相同的驱动代码。SD NAND 在协议层与标准 SD 卡完全一致。

### Q2: SD NAND 不需要 FTL，那坏块管理怎么办？

XCSD02G-MJ 芯片内部集成了完整的 NAND 管理控制器，包括 8bit/512B ECC 纠错、坏块管理、磨损均衡、垃圾回收。Host 端看到一个"完美"的块设备，完全不需要关心底层 NAND 物理特性。这是 SD NAND 相对 SPI NAND 最大的架构优势。

### Q3: SD NAND 是否需要手动格式化？

通常不需要。XCSD02G-MJ 出厂已格式化为 FAT32（规格书 4.2 节），用户可用容量 240MB（02G 型号），直接挂载即可使用。

当前项目为进一步提高兼容性，启用了 `TCFG_SD0_FORMAT_ON_BOOT`：若开机 mount 失败（首次上电或文件系统损坏），会自动执行 `f_format()` 并写入 VM 标志位；mount 成功则完全跳过，不会重复格式化已有数据。

### Q4: 1-bit 模式性能够用吗？

写速度 ~4.6MB/s，读速度 ~21MB/s。对于 TWS 耳机的典型应用场景（蓝牙音频 ~320KB/s、固件升级、配置文件读写）完全足够。4-bit 模式需要额外 3 个 GPIO（DAT1~3），在 IO 紧张的小型设备上得不偿失。

### Q5: SD NAND 的 DAT1~3 不接 MCU，只上拉行不行？

可以，而且规格书明确支持这种配置（2.1 节）。上电后 SD NAND 默认使用 1-bit 模式（仅 DAT0），未使用的 DAT1~3 保持输入状态即可。但**上拉电阻必须接**——规格书强调"主机应通过 RDAT 将所有 DAT0-3 线路进行上拉，即使主机在 SD 模式下仅将 SD NAND 用作 1 位模式也是如此"。

### Q6: 闭源的 sd_dev_ops 怎么调试？

虽然 `sd_dev_ops` 在 `device.a` 中闭源，但 SD 协议是完全标准化的。调试时可使用逻辑分析仪同时捕获 CLK、CMD、DAT0 三条线的波形，对照 SD2.0 规范分析每一帧命令和响应。常见问题（上拉漏接、时钟频率过高、供电不稳）都可以从波形中直接定位。

### Q7: SD NAND 和 SPI NAND 可以同时使用吗？

理论上可以。两者使用不同的硬件接口（SDIO vs SPI），GPIO 不冲突的话可以共存。但需要评估是否有实际需求——SD NAND 已经提供了完整的存储能力，同时维护两套存储方案增加复杂度。

### Q8: SD NAND 初始化需要多长时间？

SD 协议初始化约 1 秒（规格书 6.2 节规定初始化超时值为 1 秒）。主要包括：
- ≥74 个虚拟时钟周期
- ACMD41 轮询 busy 位（芯片内部准备就绪）
- FAT 文件系统挂载（读取超级块、初始化 FAT 缓存）

相比 SPI NAND（~100ms 复位 + FTL 扫描），SD NAND 初始化略慢，但对用户感知无影响。

### Q9: SDK 在 SD NAND 场景下有哪些原生缺陷？需要修复吗？

在 SD NAND 接入过程中，发现 SDK 存在以下原生问题。但**在当前项目的实际代码路径下，这些问题不触发**：

| 缺陷 | 文件 | 表现 | 当前项目状态 |
|------|------|------|------------|
| 代码 bug | `dev_status.c:260` | 引用了未定义的 `IOCTL_RESET_DET_STATUS` 和 `event` 变量 | **不触发** — 该行在 `#if defined(TCFG_SD_ALWAY_ONLINE_ENABLE) && ... == DISABLE` 条件编译下，当前项目未定义此宏，分支不编译 |
| 逻辑缺失 | `dev_manager.c:1029` | 关闭 `ALWAY_ONLINE` 后检测定时器 `sdx_dev_detect_timer_add()` 无人调用 | **不影响** — 闭源 `sd_dev_ops.init()` 内部有自己的检测机制，不需要公开 API 定时器。RDX 协议栈中也曾注释掉了该调用（`rdx_app.c:2360`） |

> **重要**：上述缺陷只有在尝试使用 `TCFG_SD_ALWAY_ONLINE_ENABLE` 宏或标准检测定时器路径时才会暴露。当前项目采用的自定义 `detect_func` 方案（方案 C）完全不经过这些代码路径，因此**无需修复 `dev_manager.c` 或 `dev_status.c`**。

## 7. 硬件调试检查清单

如果 SD NAND 初始化失败，按以下顺序排查：

1. **供电**：VDD 引脚电压是否在 2.7~3.6V 范围？2.2μF 去耦电容是否已焊接？PA00 是否通过 LDO 使能脚正确给 SD NAND 供电？可用万用表测量 LDO3V 输出。
2. **上拉电阻**：CMD、DAT0~3 是否全部通过 10K~100KΩ 上拉到 VDD？特别注意 DAT3——如果没有上拉，芯片可能误入 SPI 模式
3. **CLK 串联电阻**：CLK 线上是否串接了 33Ω 左右电阻？
4. **引脚映射**：`TCFG_SD0_PORT_CMD/CLK/DA0` 是否与 PCB 实际连接一致？⚠️ **特别检查 CMD 和 CLK 是否有互换**（这是本项目的实际根因）
5. **GPIO 冲突**：`TCFG_HW_SPI1_ENABLE` 是否已关闭？`TCFG_NANDFLASH_DEV_ENABLE` 是否未使能（默认为 0 即可）？MPU6887P AD0 是否确认仅在初始化时设置一次（不持续操作）？
6. **自定义 detect_func**：`device_config.c` 中是否已实现 `sdmmc_0_io_detect_user()` 并替换 `detect_func`？自定义函数是否放在与 `DET_MODE` 匹配的 `#if` 分支中？
7. **格式化机制**：若首次上电，确认 `TCFG_SD0_FORMAT_ON_BOOT` 已使能，且 VM 区域可正常读写（否则每次 boot 都可能重复格式化）
8. **诊断宏**：bring-up 阶段可临时打开 `TCFG_SD0_DIAG_ENABLE`，查看 `[SD-DIAG]` / `[SD-FMT]` / `[SD-PWR]` 日志定位阶段
9. **逻辑分析仪**：抓 CLK/CMD/DAT0 波形，确认 ≥74 个虚拟时钟 + CMD0 + ACMD41 序列已发出且收到了正确的响应
