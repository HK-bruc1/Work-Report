# 存储方案对接与 FTL 优化经验

## 一、存储设备别名模式：盘符对业务透明

### 背景

JL 平台有一层 `dev_reg` 设备注册表，定义"逻辑盘符（logo）→ 物理设备（name）"的映射。上层业务
（录音库、USB MSD、升级等）通过盘符访问文件系统，不直接感知底层存储类型。

**关键架构**：

```
业务层（录音库）        dev_reg.c               device_config.c         驱动层
─────────────────      ──────────              ────────────────         ────
dev_manager_add(logo)  →  logo → name         →  设备注册表匹配 name   →  ops
fopen(root_path)            logo → root_path      ops.init/open/read/write
```

### 问题

录音/文件业务在预编译库（`.a`）中硬编码了盘符 `sd0` 和路径 `storage/sd0/C/`，
但实际存储方案从 SD 卡换成了 SPI NAND Flash（驱动名 `nandflash_ftl`）。
直接改库不可行，需要在不改业务代码的前提下把 I/O 路由到新驱动。

### 方案：logo 别名映射

在 `dev_reg.c` 中将 logo 设为 `"sd0"`，name 保持为新驱动的注册名：

```c
// dev_reg.c — 映射关系
{
    /*logo*/          "sd0",                    // 业务层用的盘符
    /*name*/          "nandflash_ftl",          // 驱动的设备名（下面那层的注册名）
    /*storage_path*/  "storage/sd0",
    /*root_path*/     "storage/sd0/C/",
    /*fs_type*/       "fat",
},
```

对应的 `device_config.c` 不需要改——驱动注册名保持不变：

```c
// device_config.c — 驱动注册（不改）
{ "nandflash_ftl",  &ftl_dev_ops,  (void *)"nand_flash" },
```

**流转过程**：

```
1. 业务调用 dev_manager_add("sd0")
2. dev_reg 查表：logo="sd0" → name="nandflash_ftl"
3. 根据 name 找到 device_config 中注册的设备节点
4. 调用 ftl_dev_ops->init / open / read / write
5. mount("nandflash_ftl", "storage/sd0", "fat", ...)
6. 业务 fopen("storage/sd0/C/xxx") → 实际走 NAND Flash FTL
```

### 通用性

| 未来换到什么存储 | 只需改 dev_reg.c 的 name 字段 |
|---|---|
| eMMC | `name="emmc0"`, 其他不变 |
| NOR Flash | `name="fat_nor"`, 其他不变 |
| USB U 盘 | `name="udisk0"`, 其他不变 |
| 虚拟 FAT | `name="virfat_flash"`, 其他不变 |

**核心原则**：logo（盘符）是业务接口，name（驱动名）是实现细节。两者在
`dev_reg.c` 中解耦，换存储方案只需改一行映射。

### 注意

- 如果启动流程中同时需要两份挂载点（如 `storage/sd0` 和 `storage/nandflash_ftl` 都指向
  同一设备），需确认文件系统层是否支持同一块设备多次 mount。对于 FAT 通常不支持，
  建议只保留一种盘符。
- `dev_manager_add` 本身是幂等的——重复调用同一个 logo 会返回 "already online" 而非报错。

## 二、FTL 上电初始化优化

### 背景

FTL 在 Flash 上维护一个 `block_info` 表，记录每个物理块的映射关系和擦写次数。
上电时需要把这张表从 Flash 读到 RAM。同时，FTL 保留两份副本（info 区 + log 区）
用于容灾，上电时做一次同步。

### 问题

`ftl_block_info_exchange()` 在上电时同步两张副本，原来的循环边界用了**分配大小**：

```c
int c = block_info_len * block_sec;  // 分配了整块 256KB
for (int i = 0; i < c; i += page_sec) {
    // 读一页 (4KB), 写一页 (4KB)
}
```

- `block_info_len = 1` 个 Block（256KB），这是擦除粒度的分配单位
- 实际数据量 = `flash_size × sizeof(BLOCK_INFO)` = `2048 × 8 = 16KB`
- 循环走了 64 次（256KB / 4KB），其中 60 次在搬运 0xFF 空白填充

每次迭代 = 1 次 SPI 读 4KB + 1 次 SPI 写 4KB ≈ 30ms。
64 次 ≈ 1900ms，构成了上电初始化 90% 的耗时。

**实际测量**：优化前 `[FTL]init: 2170ms`，优化后 `[FTL]init: 150ms`。

### 方案

边界改用**实际数据大小**：

```c
int c = (flash_size * sizeof(BLOCK_INFO) + 511) / 512;
// 2048 × 8 = 16384 bytes → 32 sectors → 4 pages
```

循环从 64 次降到 4 次，SPI 操作从 128 次降到 8 次。

**同时收益**：每次 `ftl_update_block_info`（录音写入时触发）也会调用
`ftl_block_info_exchange`，该路径同样被加速。

### 通用原理

任何块设备 FTL 的元数据管理都面临同一个问题：

> **分配粒度 ≠ 实际数据量**

- 元数据必须按 Block（擦除单位）分配存储空间
- 但实际数据通常远小于一个 Block 的大小
- 同步/扫描操作应该以**实际数据边界**为准，而非分配边界

**适用场景**：

| 元数据类型 | 分配大小 | 实际大小 | 浪费比例 |
|---|---|---|---|
| Block Info 表 (2048 块) | 256KB | 16KB | 16:1 |
| Super Block | 256KB | 512B | 512:1 |
| FAT 表 | 多 Block | 少得多 | 视情况 |
| 日志区 | 多 Block | 动态 | 视情况 |

**检查清单**——换存储方案后，审视 FTL/文件系统初始化路径：

1. 找到所有 "全量复制元数据区" 的循环
2. 确认循环边界是基于分配大小还是实际数据大小
3. 如果 `flash_size` / `block_count` 变了，重新计算实际数据量
4. 用 `sizeof(结构体) × 条目数` 而非 `block_info_len × block_size`

## 三、其他注意事项

### SPI NAND 写入时串口日志噪音

FTL 写入路径（`_ftl_write`）中的 `log_info` 每次写扇区都刷一行到串口。
在大流量场景（录音）下，串口输出本身会拖慢写入，且日志无实际价值。

**建议**：写入路径的日志用 `log_debug`（生产固件不编译），仅保留 `log_error`。

### 启动时 `dev_manager_add` 的顺序

当前 `dev_manager_task()` 中 NAND Flash 在 dev_reg mount 之前就完成了 FTL 初始化。
如果换用需要热插拔检测的存储（如 SD 卡、U 盘），`dev_manager_add` 的调用时机
应推迟到硬件检测完成之后，否则 mount 会失败。

### 测试开关

`TCFG_NAND_TEST_ENABLE` 及相关子开关（`NAND_TEST_RAW/FTL/FORMAT/FS`）应
在量产固件中设为 `DISABLE`，避免上电时执行破坏性测试（如 RAW 测试擦写 Block 0）。

## 四、SD NAND 挂载失败：电源脚未拉高导致芯片没上电

### 现象

`mount("sd0", "storage/sd0", "fat", ...)` 返回 NULL，日志 `fmnt = 0`。
SD 初始化耗时 1140ms（正常 452ms），但即使超时后设备检测走完，mount 依然失败。

### 排查路径

| 步骤 | 对比维度 | 结论 |
|------|---------|------|
| 1 | `sdk_config.h` | 两工程相同，排除主配置 |
| 2 | 预编译 `.a` 库 MD5 | 完全一致，排除库版本 |
| 3 | `board_ac701n_demo_cfg.h` | 有差异但非 SD 相关 |
| 4 | 日志时序 | 相同 `sd io init` 时间点，但 mount 耗时差一倍多 → 怀疑硬件层 |
| 5 | `rdx_app.h` 的 `VDD_POWER_PORT_IO` | 一个指向 `IO_PORTA_00`，另一个指向 `IO_PORTE_05` ← **命中** |

### 根因

SD NAND 芯片的 VDD 电源由 `IO_PORTA_00` 控制，当前工程有三处缺失导致该引脚从未被拉高：

| # | 位置 | 问题 | 影响 |
|---|------|------|------|
| 1 | `rdx_app.h` | `VDD_POWER_PORT_IO = IO_PORTE_05` | `rdx_app_emmc_poweron()` 控制的是错误引脚 |
| 2 | `sdk_config.h` | `TCFG_IO_CFG_AT_POWER_ON = 0` | 开机 GPIO 配置功能被禁用 |
| 3 | `sdk_config.c` | `g_io_cfg_at_poweron` 数组为空 | 即便是正确引脚也无配置项 |

芯片全程断电，SD 驱动对着一块无响应的芯片反复发 CMD0/ACMD41 直到超时。
超时后 `mount()` 读到的全是噪声数据，FAT 驱动找不到有效分区表，返回 NULL。

### 正确的配置

`rdx_app.h`:
```c
#define VDD_POWER_PORT_IO   IO_PORTA_00
#define VDD_POWER_PORT      PORTA
#define VDD_POWER_PIN       PORT_PIN_0
```

`sdk_config.h`:
```c
#define TCFG_IO_CFG_AT_POWER_ON    1   // 启用开机 GPIO 配置
#define TCFG_IO_CFG_AT_POWER_OFF   1   // 启用关机 GPIO 配置
```

`sdk_config.c`:
```c
const struct gpio_cfg_item g_io_cfg_at_poweron [] = {
    { .gpio = IO_PORTA_00, .mode = PORT_OUTPUT_HIGH, .hd = PORT_DRIVE_STRENGT_8p0mA }
};
const struct gpio_cfg_item g_io_cfg_at_poweroff [] = {
    { .gpio = IO_PORTA_00, .mode = PORT_OUTPUT_LOW,  .hd = PORT_DRIVE_STRENGT_8p0mA }
};
```

### 通用教训

外设需要独立电源控制时，三重检查：

1. **宏定义** — 电源控制脚指向正确的 GPIO
2. **开关使能** — `TCFG_IO_CFG_AT_POWER_ON` 等宏为 1
3. **GPIO 配置表** — 数组中确实有对应引脚的拉高/拉低配置

三者缺一不可，且任何一处错误都会导致"外设检测得到但通信失败"的隐蔽故障。
排查时先看日志时序（正常 vs 异常耗时差异是最快线索），再顺着电源路径逐层确认。

### 纠正！！！

PA0的确时VDD电源脚，但是不应该通过TCFG_IO_CFG_AT_POWER_ON设置，这会一直保持！应该通过SD驱动的回调使用！比较合适。
