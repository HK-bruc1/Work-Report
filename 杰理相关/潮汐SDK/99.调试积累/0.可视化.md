# 根据固件大小选JL内置flash型号

```c
-----------------------------------FLASH INFO-----------------------------------
|  PID : AC701N                                                                |
|  VID : 0.01                                                                  |
|  FLASH_BIN_SIZE : 0xea000                                                    |
|  FLASH_NEED_SIZE : 0x1ff000                                                  |
|  FLASH_REAL_SIZE : UNLIMITED                                                 |
|  VM_REAL_SIZE : 0x2000                                                       |
|  VM_START_ADDR : 0x1fc000                                                    |
|  VM_END_ADDR : 0x1fe000                                                      |
|  BTIF_RESERVED_SIZE : 0x1000                                                 |
|  BTIF_RESERVED_START : 0x1fe000                                              |
|  BTIF_RESERVED_END : 0x1ff000                                                |
|  CODE_BOUNDARY_LINE : 0xff000                                                |
|  LAVE_SIZE : 0x15000                                                         |
|  ENTRY_ADDR : 0x6000100                                                      |
|  ERASE MODE : NONE                                                           |
|  UBOOT VERSION : 2025-03-18 17:25:58,uboot-$-@20250318-$88b6c68              |
|  OTA VERSION : 2024-12-26 16:12:52,OTA_LOADER-$-@20241226-$8860ad9           |
--------------------------------------------------------------------------------
```

结论：

根据当前 FLASH INFO 日志分析，如果物理 Flash 为 16Mbit，即 2MB，地址空间为 `0x000000 ~ 0x200000`。

当前 Flash 布局需求为：

```text
FLASH_NEED_SIZE = 0x1FF000
```

因此整片 Flash 剩余空间为：

```text
0x200000 - 0x1FF000 = 0x1000 = 4KB
```

也就是说，当前分区布局在 16Mbit Flash 上可以放下，但整体 Flash 余量只有 4KB。

另外，日志中：

```text
FLASH_BIN_SIZE     = 0xEA000
CODE_BOUNDARY_LINE = 0xFF000
LAVE_SIZE          = 0x15000
```

可计算出代码区剩余：

```text
0xFF000 - 0xEA000 = 0x15000 ≈ 84KB
```

因此这里的 `LAVE_SIZE` 表示的是代码分区内部剩余空间，不是整片物理 Flash 的剩余空间。

最终判断：

1. 如果物理 Flash 是 16Mbit，则当前 Flash 布局整体剩余约 4KB。
2. APP 代码分区内部仍剩余约 84KB。
3. 代码区已经包含在整体 Flash 分区布局中。
4. “剩余 80K”和“剩余 4K”并不矛盾，前者是代码区剩余，后者是整片 Flash 布局后的物理剩余。
5. 当前固件可以适配 16Mbit Flash，但整体 Flash 空间已经非常接近上限，后续如需增加资源区、VM、BTIF 或调整 OTA 布局，需要重新评估分区。

# os_taskq_post_type 死机分析

## 根因

JL 内核 `Q_CALLBACK` 的消息格式是 **`{callback, arg_count, args...}`** —— `msg[1]` 是参数个数，不是参数值。JL SDK 全局所有原生代码统一用 `argc=3`、`msg[1]=1`。

`rdx_os_task_post_callback` 实现时错误地用了 `argc=2`、`{callback, arg}`，把参数值直接塞进 `msg[1]`。内核把 `arg`（一个指针值）当作参数个数解读，然后从 `msg[2]` 取参数。由于旧格式没有合法的 `msg[2]`，回调收到的 `priv` 是队列缓冲区里的残留值。

**这条路径从 day 1 就是错的。** `67c8ecc` 没炸只是因为残留值碰巧指向 CPU 可写内存，`msg->busy = 0` 静默损坏了别处。`1996287` 改了二进制布局后，残留值固定为 `app_core` 栈地址；第二次 BLE 连接时回调被 CPU1 执行 → 写不进 CPU0 栈 → `cpu1 write hmem exception`。

## 修法

两个 port 的 `rdx_os_task_post_callback` 对齐 JL 原生约定：

```c
// 错误
int msg[2];
msg[0] = (int)callback;
msg[1] = (int)arg;
os_taskq_post_type(task_name, Q_CALLBACK, 2, msg);

// 正确
int msg[3];
msg[0] = (int)callback;
msg[1] = 1;
msg[2] = (int)arg;
os_taskq_post_type(task_name, Q_CALLBACK, 3, msg);
```

## 教训

封装底层 OS 原语时，必须以原生调用方的用法为唯一权威参考，而不能根据"看起来能工作"的测试结果反推语义。寄存器残留值会让错误代码在测试中表现为正常 —— 这种幸存者偏差比直接崩溃更危险。
