## 1. 录音文件的产生（前置知识）

在理解播放之前，需要先知道录音文件是如何产生的，因为播放的文件列表完全依赖这个过程。

### 1.1 录音文件的生命周期

```
用户按下录音键
  │
  ▼
rdx_record_run_init()                     ← 录音启动
  │
  ├─ rdx_uxfile_dat_1_gen(scene)          ← 在 DAT 中分配新 SN + 生成文件名
  │     │
  │     └─ 生成形如 "46d3ca.raw" 的文件名（不定长十六进制 SN）
  │        写入到 DAT 缓存的 uxfile_data_t 条目中
  │
  ├─ fopen("storage/sd0/C/xxxx.raw", "w") ← 在 SD 卡上创建物理文件
  │
  ▼
rdx_record_run_data_handle(buf, len)      ← 录音数据持续写入
  │
  └─ rdx_uxfile_raw_write(data, len, scene) ← 原始 OPUS 数据写入文件
  │
  ▼
用户停止录音
  │
  ▼
rdx_record_run_exit()                     ← 录音停止
  │
  ├─ rdx_uxfile_dat_1_save_gen()          ← 将文件长度、CRC32、时间等写入 DAT
  │
  └─ fclose()                             ← 关闭物理文件
```

此时，DAT 缓存中多了一条 `uxfile_data_t` 记录：

```c
typedef struct {
    u32 sn;                            // 序列号，如 0x46d3ca
    char filename[64];                 // 文件名，如 "46d3ca.raw" 或 "/46d3ca.RAW"
    u8  record_scene;                  // CHAT(0) 或 CALL(1)
    u32 data_len;                      // 文件总字节数
    u32 start_time;                    // 录音开始时间戳
    u32 end_time;                      // 录音结束时间戳
    u32 crc32;                         // 文件校验和
    u32 frame_size;                    // 编码帧大小
    // ...更多字段
} uxfile_data_t;
```

### 1.2 DAT 缓存的初始化（开机同步）

```
开机
  │
  ▼
rdx_app_tasks_init()
  │
  └─ rdx_uxfile_init()                 ← 初始化 DAT 系统
       │
       └─ rdx_uxfile_sync_files_with_dat()
            │
            └─ 扫描 SD 卡 "storage/sd0/C/" 目录下所有 .raw 文件
               将文件信息加载到 DAT 缓存的内部数据结构中
               此过程约需 4 秒（23 个文件、~330KB 元数据）
```

开机同步期间，`rdx_uxfile_is_sync_in_progress()` 返回 true。播放模块通过 `rdx_playback_can_start()` 中的互斥检查，在同步完成前拒绝启动播放。

## 2. 文件查找：从 DAT 缓存定位到物理文件

这是播放流程中最关键的环节——如何从一个 SN 号找到 SD 卡上的实际文件。

### 2.1 通过 SN 获取文件元数据

```c
// rdx_playback.c 中的调用
uxfile_data_t *fi = rdx_uxfile_get_file_data_by_sn(sn, 0, 0);
```

`rdx_uxfile_get_file_data_by_sn()` 在 DAT 缓存内部按 SN 查找，返回指向 `uxfile_data_t` 的指针：
- 找到 → 返回 `fi`，其中 `fi->filename` 如 `"46d3ca.raw"`
- 未找到 → 返回 `NULL`

### 2.2 通过固定挂载点路径打开文件

录音文件统一存放在 SD 卡的 `"storage/sd0/C/"` 目录下。DAT 缓存中只记录纯文件名（如 `"46d3ca.raw"`），播放时由 `pb_open_file()` 拼接完整路径后直接打开：

```c
// rdx_playback.c 顶部
#define PB_STORAGE_ROOT  "storage/sd0/C/"

static bool pb_open_file(const char *fname, FILE **out_f)
{
    char path[128];
    snprintf(path, sizeof(path), PB_STORAGE_ROOT "%s", fname);

    FILE *f = fopen(path, "r");
    if (f && flen(f) > 0) {          // 打开成功且非空文件
        *out_f = f;
        return true;
    }
    if (f) fclose(f);
    return false;
}
```

成功条件：`fopen()` 返回非空 **且** `flen(f) > 0`（排除空文件）。

失败处理：如果某个 SN 对应的文件在 SD 卡上不存在或为空（DAT 与文件系统不一致），则**自动跳过**，尝试下一个 SN。

## 3. 播放控制流程

### 3.1 初始化

```
系统启动
  │
  ▼
rdx_app_tasks_init()                    ← rdx_app.c:3232
  │
  ├─ rdx_uxfile_init()                  ← 先初始化 DAT（预编译库）
  │    │
  │    └─ rdx_uxfile_sync_files_with_dat()
  │         │
  │         └─ 扫描 SD 卡 "storage/sd0/C/" 目录下所有 .raw 文件
  │            将文件信息加载到 DAT 缓存的内部数据结构中
  │            此过程约需 4 秒（23 个文件、~330KB 元数据）
  │
  └─ rdx_playback_init()                ← rdx_playback.c:199
       │
       └─ memset(&pb, 0, sizeof(pb))    ← 清零全局 pb 结构体
```

注意：`rdx_playback_init()` **不再读取 `max_sn` / `total_count`**。上电时 DAT 同步尚未完成，缓存为空/不可信；这些值延迟到用户首次按播放键时，由 `pb_cache_ready()` 在 DAT 同步完成后再读取。

此时 pb 状态：
| 字段 | 值 | 含义 |
|------|-----|------|
| `cur_sn` | 0 | 未在播放 |
| `total_count` | 0 | 尚未加载 |
| `fp` | NULL | 无播放器实例 |
| `status` | 0 (STOP) | 停止状态 |
| `intent` | NONE | 无操作意图 |

### 3.2 播放一个文件（PREV / NEXT 的公共入口）

```
rdx_playback_prev() / rdx_playback_next()
  │
  ├─ 1. 残留清理
  │     if (pb.fp && pb.status == STOP)
  │         cleanup_playback()           ← 清理上次"已停未释放"的 player
  │
  ├─ 2. 互斥检查
  │     if (!rdx_playback_can_start())
  │         return                       ← 以下任一条件不满足则直接退出：
  │           • 正在录音 (run == START || RESUME)
  │           • DAT 同步中 (is_sync_in_progress)
  │           • DAT 分包传输中 (is_datFileInfo_loading)
  │           • 文件传输中 (is_file_transfer_active)
  │           • 文件同步忙 (is_file_sync_busy)
  │
  ├─ 3. 检查文件列表是否就绪
  │     if (!pb_cache_ready())
  │         return                       ← 以下任一情况直接退出：
  │           • DAT 同步尚未完成
  │           • DAT 缓存为空（无录音文件）
  │           • 缓存中最大 SN 为 0（数据异常）
  │
  ├─ 4. SN 扫描查找
  │     PREV: for sn = cur_sn-1 downto 1          ← 往前找
  │           如果 cur_sn==0 则从 pb_max_sn 开始
  │           wraparound: for sn = pb_max_sn downto cur_sn
  │
  │     NEXT: for sn = cur_sn+1 to pb_max_sn       ← 往后找
  │            如果 cur_sn==0 则从 1 开始
  │            wraparound: for sn = 1 to cur_sn-1
  │
  │     每次循环:
  │       fi = rdx_uxfile_get_file_data_by_sn(sn, 0, 0)
  │       if (!fi || filename 为空) → 跳过，继续
  │       if (rdx_playback_play_file(fi) == true) → 成功，停止扫描
  │       else → 跳过（文件打不开），继续
  │
  └─ 5. 启动播放 (rdx_playback_play_file)
        │
        ├─ pb_open_file(fi->filename) → FILE* f
        │     失败 → return false（上层继续扫描下一个 SN）
        │
        ├─ cleanup_playback()
        │     ├─ pb.intent = PB_INTENT_STOP
        │     ├─ music_file_player_stop()     ← 停止上一首
        │     │     ├─ jlstream_stop(stream, 50ms 超时)
        │     │     ├─ jlstream_release(stream)
        │     │     └─ music_player_free(player)  ← 释放 player 内存
        │     ├─ pb.fp = NULL
        │     ├─ pb.status = STOP
        │     ├─ pb.intent = NONE
        │     └─ fclose(old_file)              ← 关闭上一首的文件句柄
        │
        ├─ pb_file = f
        ├─ pb.fp = music_file_play_callback(f, &pb, pb_callback, NULL)
        │     │
        │     └─ 内部流程:
        │           music_player_create(f, NULL)   ← zalloc + init file_player
        │           player->priv = &pb
        │           player->callback = pb_callback
        │           music_player_add(player)       ← 入全局链表
        │             └─ music_file_player_start(player)
        │                   ├─ uuid = jlstream_event_notify(
        │                   │      GET_PIPELINE_UUID, "music")
        │                   │      → 返回 PIPELINE_UUID_MUSIC (0x6FEC)
        │                   │
        │                   ├─ player->stream = jlstream_pipeline_parse(
        │                   │      uuid, NODE_UUID_MUSIC)
        │                   │      → 构建 demuxer→decoder→DAC 管线
        │                   │      失败 → 返回 NULL
        │                   │
        │                   ├─ jlstream_set_callback(music_player_callback)
        │                   │      → 注册内部回调，用于将 STREAM_EVENT
        │                   │        转发到 pb_callback
        │                   │
        │                   ├─ jlstream_set_dec_file(stream, player, &music_file_ops)
        │                   │      → 设置文件操作表:
        │                   │         .read  = music_file_read  → fread
        │                   │         .seek  = music_file_seek  → fseek
        │                   │         .close = music_file_close → fclose
        │                   │         .get_fmt = music_file_get_fmt
        │                   │              → jldemuxer 自动检测编码格式 (OPUS/MP3)
        │                   │
        │                   └─ jlstream_start(stream)
        │                         → 开始拉流: fread(file) → demuxer → OPUS decode → DAC
        │
        ├─ pb.cur_sn = fi->sn               ← 更新当前 SN
        ├─ pb.status = PB_STATUS_PLAYING
        └─ pb.intent = PB_INTENT_NONE
```

### 3.3 快进 / 快退（FF / FR）

```
rdx_playback_ff() / rdx_playback_fr()
  │
  ├─ if (pb.fp == NULL) → return          ← 没有在播放，直接返回
  ├─ if (pb.status != PLAYING) → return   ← 已停止/暂停，直接返回
  │
  └─ music_file_player_ff(3, pb.fp)       ← 步长 3 秒
       │
       └─ jlstream_node_ioctl(
              stream,                       ← 当前 stream 实例
              NODE_UUID_DECODER,            ← 目标节点：解码器
              NODE_IOC_DECODER_FF,          ← 命令：快进
              3)                            ← 参数：3 秒
            │
            └─ 解码器内部:
                 计算 3 秒对应的文件偏移
                 → music_file_seek(file, offset, SEEK_CUR)
                 → 从新位置继续解码

  -- FR 同理，NODE_IOC_DECODER_FR 命令调 fseek 往前跳 --
```

### 3.4 停止播放

```
rdx_playback_stop()
  │
  └─ cleanup_playback()
       ├─ pb.intent = PB_INTENT_STOP     ← 标记为人为停止
       ├─ music_file_player_stop()       ← 停止并释放 player
       ├─ pb.fp = NULL
       ├─ pb.status = PB_STATUS_STOP
       ├─ pb.intent = PB_INTENT_NONE
       └─ fclose(pb_file)               ← 关闭物理文件
```

注意 `cleanup_playback()` 的执行顺序：**先 `stop` 解码管线，再 `fclose` 文件**。这与 SDK 官方的 `music_player.c` 顺序一致——如果在管线还在读文件时 `fclose`，会导致 decoder 访问已关闭的 FILE*。

### 3.5 自动下一首（播放结束回调）

```
当前文件播放到末尾
  │
  ▼
jlstream 发出 STREAM_EVENT_STOP
  │
  ▼
file_player.c::music_player_callback()
  │  根据 player_id 找到对应的 file_player
  │  调用 player->callback(priv, player->read_err, STREAM_EVENT_STOP)
  │
  ▼
rdx_playback.c::pb_callback(priv, parm, STREAM_EVENT_STOP)
  │
  ├─ pb_ctx->status = PB_STATUS_STOP
  │
  └─ if (parm == 0                              ← 正常播完，不是读错误
          && pb_ctx->intent != PB_INTENT_STOP    ← 不是人为按停止
          && pb_ctx->intent != PB_INTENT_SWITCH) ← 不是切歌中
        │
        └─ os_taskq_post_type("app_core",
               Q_CALLBACK,
               2, (int[]){rdx_playback_next, 0})
             │
             └─ 投递到 app_core 任务队列（异步）
                 在 app_core 的上下文中执行 rdx_playback_next()
                 这样就避免了在 jlstream 回调中嵌套调用的风险
```

回调中不直接清理 `pb.fp` 和 `pb_file`（因为 `STREAM_EVENT_STOP` 意味着 player 已被 `free`）。残留的 `pb.fp`（悬空指针）在下一次 PREV/NEXT 调用时，由步骤 1 中的"残留清理"逻辑检测 `pb.fp != NULL && status == STOP` 后统一清理。

## 5. 关键数据结构

### 5.1 播放状态 (`rdx_playback_t`)

```c
typedef struct {
    u32  cur_sn;             // 当前播放文件的 SN，0 = 未在播放
    u32  total_count;        // DAT 缓存中的文件总数
    struct file_player *fp;  // music_file_play_callback 返回的句柄
    u8   status;             // 0=PB_STATUS_STOP, 1=PB_STATUS_PLAYING, 2=PB_STATUS_PAUSE
    pb_intent_t intent;      // 区分"正常播完自动下一首"和"人为停止不切歌"
} rdx_playback_t;
```

### 5.2 操作意图 (`pb_intent_t`)

| 值 | 含义 | 影响 |
|----|------|------|
| `PB_INTENT_NONE` | 无特殊意图 | 正常播完后**自动下一首** |
| `PB_INTENT_STOP` | 人为按停止 | 不自动切歌 |
| `PB_INTENT_SWITCH` | 切歌中 | 等新歌开始后才清零，防止切换期间误触发自动下一首 |

### 5.3 播放互斥条件

`rdx_playback_can_start()` 检查的 5 个 guard：

| 条件 | 检查函数 | 含义 |
|------|----------|------|
| 正在录音 | `rdx_record_get_status()->run` | START 或 RESUME 状态不允许播放 |
| DAT 同步中 | `rdx_uxfile_is_sync_in_progress()` | 开机后扫描 SD 卡中 |
| DAT 分包传输中 | `rdx_uxfile_is_datFileInfo_loading()` | BLE 正在传输 DAT 信息到 APP |
| 文件传输中 | `rdx_is_file_transfer_active()` | WiFi/BLE 文件传输进行中 |
| 文件同步忙 | `rdx_is_file_sync_busy()` | 文件同步操作繁忙 |

## 6. 关键设计决策

### 6.1 为什么用 `music_file_play_callback()` 而不是 `music_file_play()`

两个 API 的区别：

```c
// 无回调版本：播放结束后无法感知
struct file_player *music_file_play(FILE *file, struct audio_dec_breakpoint *dbp);

// 回调版本：可接收 STREAM_EVENT_START/STOP 事件
struct file_player *music_file_play_callback(FILE *file, void *priv,
        music_player_cb_t callback, struct audio_dec_breakpoint *dbp);
```

必须用回调版本，因为需要实现"**当前文件播完 → 自动下一首**"。回调接收 `STREAM_EVENT_STOP` + `parm=0`（正常播完）时，投递 `rdx_playback_next` 到 `app_core` 队列。

### 6.2 为什么 SN 扫描而不是直接拿文件列表

`rdx_uxfile` 库没有提供"返回所有 SN 的有序列表"API。现有 API 是按 SN 随机查找（`get_file_data_by_sn(sn, ...)`）。因此 PREV/NEXT 的实现方式是在 SN 空间内线性扫描 + `fopen` 验证。

### 6.3 为什么用固定挂载点路径

录音文件统一由本机生成在 `"storage/sd0/C/"` 目录下，DAT 缓存中只保存纯文件名。因此播放时不需要猜测多个候选路径，直接拼接固定前缀即可：

```c
#define PB_STORAGE_ROOT  "storage/sd0/C/"
snprintf(path, sizeof(path), PB_STORAGE_ROOT "%s", fname);
```

`PB_STORAGE_ROOT` 定义在 `rdx_playback.c` 顶部，如果以后切换挂载点或支持多存储设备，只需修改这一处定义。

### 6.4 为什么 `cleanup_playback()` 先 stop 后 fclose

SDK 的 jlstream 解码管线在播放期间持有 FILE* 的引用。如果先 `fclose` 再 `jlstream_stop`，decoder 的收尾阶段可能尝试 `fread` 一个已关闭的文件句柄，导致未定义行为。因此顺序必须是：
1. `music_file_player_stop()` — 停止管线，释放 stream
2. `fclose()` — 关闭文件句柄

这与 SDK 官方的 `music_player.c` 顺序一致。

### 6.5 为什么自动下一首用 `os_taskq_post` 投递

`pb_callback` 在 jlstream 线程中执行。在回调中直接调用 `rdx_playback_next()` → `music_file_play_callback()` 会导致**在旧 player 的释放回调中创建新 player**，可能触发 jlstream 的重入问题。因此用 `os_taskq_post_type("app_core", Q_CALLBACK, ...)` 投递到 `app_core` 队列，延迟到主任务上下文中执行。
