# PB1 回弹按键 — 拨码开关改造

## 状态摘要

本文记录 PB1 从 JL 原生回弹式多功能电源键改造成拨码式纯电源开关的设计与实现。

### 核心设计原则

**JL 平台默认将 PB1 配置为 IO 回弹按键（键值 KEY_POWER），但这不是硬件约束——可视化工具可以把 IO 口换成其他引脚。** 改造的核心思路极其简单：

1. 通过 JLStudio 可视化工具的按键配置界面，把 `g_iokey_info[]` 中的 PB1 条目删掉（或把 IO 口换成其他引脚）。工具生成文件无需手改。
2. 在芯片层级，把 PB1 的 P33 唤醒路径从 `key_wakeup_callback` 切换到 `rdx_dip_switch_p33_irq`，让 PB1 只响应拨码电平跳变。
3. 按键事件域（key_driver、key tables、UUID 映射、TWS 组合键等）**完全不需要知道 PB1 的存在**——零 DIP 引用。

两条线唯一的交汇点是 `t2620_project_config.h`（定义 `TCFG_DIP_SWITCH_POWER_ENABLE=1`，并强制关闭可能占用 PB1 的 ADKEY/LP_TOUCH 模块），其余完全正交。

- 拨码 `ON = LOW`，`OFF = HIGH`，PB1 通过内部上拉保持 OFF 态高电平
- 关机态 P33 保持 `FALLING_EDGE` 唤醒，拨到 ON 时由 HIGH→LOW 下降沿唤醒
- 运行态由 `rdx_dip_switch_deferred_handle()` 消抖、切边沿并执行关机
- 其他物理按键正常接入 iokey 框架——可以使用 KEY_POWER 或其他键值，不受拨码开关影响

## Part 1：JL 原生 PB1 回弹按键的完整链路

在理解拨码开关改造之前，必须先理解 PB1 在 JL SDK 中原生的工作机制。PB1 不是普通的 GPIO——它是**电源按键 + 多功能按键事件源**的双重角色，深度嵌入在从芯片 P33 唤醒硬件到应用层 earphone.c 的每个环节。

### 1.1 硬件连接与电平约定

PB1（`IO_PORTB_01`）在硬件上是一个回弹式按键，按下接通 GND，松开悬空，由芯片内部上拉维持 HIGH：

```
PB1 → 按键 → GND
        ↓ 松开：IO 悬空 + 内部上拉 10K → HIGH
        ↓ 按下：IO 接通 GND → LOW
```

代码中的配置（`sdk_config.c:86-89`）：

```c
.key_value = KEY_POWER,
.key_io   = IO_PORTB_01,
.detect   = 0,    // 0 = ONE_PORT_TO_LOW（低电平有效）
```

### 1.2 三层架构总览

PB1 的完整处理链路可以拆成三层，自底向上：

```
┌──────────────────────────────────────────────────────┐
│ 第三层：应用消息分发                                   │
│ earphone.c: APP_MSG_KEY_POWER_OFF → sys_enter_soft_poweroff()
│ bt_key_power_msg_remap(): CLICK→播放, HOLD_5SEC→关机    │
│       ▲                                              │
│ app_main.c: app_key_event_remap(bt_mode_key_table)    │
│       ▲                                              │
│ MSG_FROM_KEY 消息 (key_event → msg[0], msg[1])        │
├──────────────────────────────────────────────────────┤
│ 第二层：按键驱动扫描                                   │
│ key_driver.c: 10ms 周期扫描 + 消抖 + 动作识别          │
│       ▲                                              │
│ iokey.c: gpio_read() 读取 PB1 电平 → key_value         │
│       ▲                                              │
│ key_event_handler → multi_clicks_translate            │
│                   → app_send_message_from(MSG_FROM_KEY)│
├──────────────────────────────────────────────────────┤
│ 第一层：P33 硬件唤醒                                   │
│ key_wakeup.c: port0.gpio=PB1, edge=FALLING_EDGE      │
│              .callback=key_wakeup_callback             │
│              → p33_io_wakeup_port_init                │
│              → key_active_set(0)  // 按键活跃标记       │
└──────────────────────────────────────────────────────┘
```

第一层负责"唤醒芯片"，第二层负责"识别按键动作"，第三层负责"执行应用逻辑"。

### 1.3 第一层：P33 唤醒（key_wakeup.c）

`key_wakeup.c` 是 PB1 与芯片睡眠/唤醒硬件之间的唯一接口。配置在 P33 电源域（常开域，即使 CPU 深度睡眠也保持供电）：

```c
// key_wakeup.c:26-36
static struct _p33_io_wakeup_config port0 = {
    .pullup_down_mode = PORT_INPUT_PULLUP_10K,  // 内部 10K 上拉
    .filter           = PORT_FLT_DISABLE,        // 不滤波
    .edge             = FALLING_EDGE,            // 下降沿唤醒
    .gpio             = IO_PORTB_01,             // PB1
    .callback         = key_wakeup_callback,     // ISR 回调
};
```

**为什么是 FALLING_EDGE？** 松键状态 IO 被上拉为 HIGH。按下按键 → HIGH→LOW 下降沿 → P33 唤醒 CPU。松键是 LOW→HIGH 上升沿，不需要唤醒。

**ISR 回调做了什么？** 非常简单：

```c
// key_wakeup.c:21-23
static void key_wakeup_callback(P33_IO_WKUP_EDGE edge)
{
    key_active_set(0);  // 标记按键活跃，阻止立即再次睡眠
}
```

它只设置一个标志位，告诉 key_driver "有按键活动，别急着进低功耗"。真正的按键识别由第二层的 10ms 定时扫描完成。

**ISR → 扫描的时序关系：**

```
P33 中断触发
  → key_wakeup_callback()  // ISR context, ~μs 级
     → key_active_set(0)   // g_is_key_active = 35 (350ms 活跃窗口)
  ... 回到主循环 ...
  → key_driver_scan()      // 10ms 定时器触发（app_core 上下文）
     → iokey get_value()   // gpio_read(PB1)
     → 消抖 + 动作识别
```

### 1.4 第二层：key_driver 扫描 + iokey 驱动

#### iokey 驱动（iokey.c）

iokey 驱动做两件事：
1. 提供 `get_value()` 函数给 key_driver 调用——直接 `gpio_read()` 读取 PB1 电平
2. 从 `g_iokey_info[]` 配置表读取 IO 信息，注册到 `platform_data`

```c
// iokey_config.c:19-27 — 配置表（key_uuid_table 用于蓝牙配网/APP 按键自定义）
static const u16 key_uuid_table[][2] = {
    { 0xaefa, KEY_POWER },   // PB1 → UUID 0xaefa → KEY_POWER
    { 0x368c, KEY_NEXT },
    { 0x842a, KEY_PREV },
    { 0x0bb0, KEY_SLIDER },
    { 0xd212, KEY_MODE },
    { 0x6a23, KEY_PLAY },
};
```

```c
// sdk_config.c:84-93 — 硬件配置表
const struct iokey_info g_iokey_info[] = {
    {
        .key_value = KEY_POWER,
        .key_io    = IO_PORTB_01,
        .detect    = 0,                     // 低电平有效
        .long_press_reset_enable = 0,       // 不启用长按复位
        .long_press_reset_time   = 16
    }
};
```

#### key_driver 扫描引擎（key_driver.c）

`key_driver_scan()` 由 10ms 定时器驱动，是按键事件框架的核心状态机：

```
10ms 定时器到
  → key_driver_scan(&iokey_scan_para)
     → iokey->get_value()          // gpio_read(PB1)，得到当前电平
     → 消抖: filter_cnt 累加，filter_time 次连续相同才认可
     → 状态迁移:
        NO_KEY → 按下 → press_cnt++ → 单击/长按/HOLD 识别
        按下   → 松键 → KEY_ACTION_UP
     → 输出: struct key_event { .event, .value }
     → key_event_handler(&key_event)
```

key_driver 识别的动作类型：

| key_event | 含义 | 典型时间 |
|-----------|------|----------|
| `KEY_ACTION_CLICK` | 单击 | < 1s |
| `KEY_ACTION_DOUBLE_CLICK` | 双击 | 两次点击间隔 < 500ms |
| `KEY_ACTION_TRIPLE_CLICK` | 三击 | |
| `KEY_ACTION_HOLD_1SEC` | 长按 1 秒 | press_cnt > 100 |
| `KEY_ACTION_HOLD_5SEC` | 长按 5 秒 | press_cnt > 500 |
| `KEY_ACTION_UP` | 长按后松键 | |
| `KEY_ACTION_FOURTH_CLICK` | 四击 | |
| `KEY_ACTION_FIRTH_CLICK` | 五击 | |

### 1.5 第三层：key_event → MSG_FROM_KEY → 模式 key table → APP_MSG → earphone.c

这是从"按键动作"到"应用行为"的完整消息链路。

#### Step 1: key_event_handler → MSG_FROM_KEY

```c
// key.c:283-326
void key_event_handler(struct key_event *key)
{
    if (multi_clicks_translate(key)) return;      // 多击识别（双击/三击）
    if (combination_key_translate(key)) return;    // 组合键识别
    if (key->event == KEY_ACTION_NO_KEY) return;

    int msg[2];
    msg[0] = (key->value << 8) | key->event;       // 高8位=KEY_POWER, 低8位=动作
    msg[1] = 0;
    app_send_message_from(MSG_FROM_KEY, 8, msg);   // 发消息到 app_core 消息队列
}
```

`msg[0]` 编码：`0x(KET_VALUE)(KEY_ACTION)`，例如长按 5 秒 → `0x0605`。

#### Step 2: app_main.c 消息分发 → app_key_event_remap

```c
// app_main.c:522-537
if (msg[0] == MSG_FROM_KEY && key_table) {
    struct app_mode *mode = app_get_current_mode();
    int key_msg = app_key_event_remap(key_table, msg + 1);
    // key_table 就是 bt_mode_key_table[]（BT 模式下）
    if (key_msg == APP_MSG_NULL) return 1;
    msg[0] = MSG_FROM_APP;
    msg[1] = key_msg;
}
```

#### Step 3: app_key_event_remap 查表

```c
// app_msg.c:36-55
int app_key_event_remap(const struct key_remap_table *table, int *event)
{
    u8 key_value  = APP_MSG_KEY_VALUE(event[0]);   // 取高8位 = KEY_POWER
    u8 key_action = APP_MSG_KEY_ACTION(event[0]);  // 取低8位 = HOLD_5SEC

    for (int i = 0; table[i].key_value != 0xff; i++) {
        if (table[i].key_value == key_value) {
            if (table[i].remap_func)               // bt_key_power_msg_remap
                return table[i].remap_func(event);
            if (table[i].remap_table)              // 静态映射表
                return table[i].remap_table[key_action];
        }
    }
    return APP_MSG_NULL;
}
```

#### Step 4: bt_key_power_msg_remap 做场景化映射

`bt_key_power_msg_remap()` 是整个 PB1 按键功能的核心决策函数。它根据**当前通话状态**和**按键动作**决定映射到哪个 APP_MSG：

```
通话中 (active_device)     + 双击 → APP_MSG_CALL_HANGUP     (挂断)
来电中 (incoming_device)   + 单击 → APP_MSG_CALL_ANSWER     (接听)
来电中                     + 双击 → APP_MSG_CALL_HANGUP     (拒接)
拨号中 (outgoing_device)   + 双击 → APP_MSG_CALL_HANGUP     (取消)
Siri 激活中                + 双击 → APP_MSG_CLOSE_SIRI      (关闭 Siri)
已连接手机 (非通话):
  + 单击        → APP_MSG_MUSIC_PP          (播放/暂停)
  + 双击(左耳)  → APP_MSG_MUSIC_NEXT        (下一曲)
  + 双击(右耳)  → APP_MSG_MUSIC_PREV        (上一曲)
  + 三击(左耳)  → APP_MSG_LOW_LANTECY       (低延迟模式)
  + 三击(右耳)  → APP_MSG_OPEN_SIRI         (语音助手)
  + 四击        → APP_MSG_VOL_UP            (音量加)
  + 五击        → APP_MSG_VOL_DOWN          (音量减)

所有场景:
  + 长按 1s     → APP_MSG_ANC_SWITCH        (切换 ANC)
  + 长按 5s     → APP_MSG_KEY_POWER_OFF     (关机)
```

对应代码 `bt_key_msg_table.c:114-298`。

#### Step 5: earphone.c 执行关机

```c
// earphone.c:1287-1290
case APP_MSG_KEY_POWER_OFF:
    puts("APP_MSG_KEY_POWER_OFF\n");

```

### 1.6 完整开/关机时序

#### 开机流程

```
电池上电 / 充电插入
  ↓
CPU 上电，执行 setup_arch()
  ├── gpio_longpress_pin0_reset_config(IO_PORTB_01, ...)  ← 长按复位配置 (setup.c:175)
  │   如果 PB1 被按超过 8s → 芯片硬件复位
  └── memory_init() ...
  ↓
key_wakeup_init()
  ├── p33_io_wakeup_port_init(&port0)   ← 配置 PB1 为 P33 下降沿唤醒
  └── p33_io_wakeup_enable(PB1, 1)      ← 使能唤醒
  ↓
check_power_on_key()                     ← app_main.c:291
  ├── while (1):
  │     is_iokey_press_down()            ← 遍历 g_iokey_info[], gpio_read(PB1)
  │     if (PB1 == LOW):                 ← 按键被按住
  │       delay_10ms_cnt++              ← 累计 80×10ms = 800ms
  │       if (delay_10ms_cnt > 80):
  │         app_var.poweron_reason = SYS_POWERON_BY_KEY
  │         return                       ← 开机！
  │     else:                            ← 按键松开了
  │       power_set_soft_poweroff()     ← 轻量关机（boot 阶段专用）
  ↓
app_main loop → 初始化各模块 → 注册 key_driver 10ms 扫描
  ↓
运行时按键处理（见上面 1.5 节链路）
```

关键点：
- `check_power_on_key()` 是一个**阻塞循环**，会一直检测 PB1 直到被按住 800ms 才开机，否则关机
- 只有**长按 800ms** 才被认为是"有意开机"，短按或误触会走关机路径
- `TCFG_AUTO_POWERON_ENABLE=1` 时这个函数被跳过，直接上电开机

#### 运行时关机流程（长按 5s 关机）

```
用户长按 PB1 5 秒
  ↓
key_driver 10ms 扫描 → press_cnt 累加到 500
  ↓
key_event_handler → multi_clicks_translate (不匹配) → 输出 KEY_ACTION_HOLD_5SEC
  ↓
app_send_message_from(MSG_FROM_KEY)
  ↓
app_main.c → app_key_event_remap(bt_mode_key_table)
  → KEY_POWER → bt_key_power_msg_remap()
    → key_action = KEY_ACTION_HOLD_5SEC
    → return APP_MSG_KEY_POWER_OFF
  ↓
msg[0] = MSG_FROM_APP, msg[1] = APP_MSG_KEY_POWER_OFF
  ↓
earphone.c: case APP_MSG_KEY_POWER_OFF:
  → sys_enter_soft_poweroff(POWEROFF_NORMAL)
    → gpio_config_soft_poweroff()
      → soff_gpio_protect(get_iokey_power_io())  ← 保护 PB1 不被设 HIGHZ
    → CPU 深度睡眠
```

### 1.7 PB1 在各子系统中的深度嵌入

PB1 不是"顺便用作电源键"。在当前 JL SDK 中，它是整个按键事件框架的**锚点 IO**，被以下子系统直接硬编码引用：

| 子系统 | 文件 | 代码 | 行为 |
|--------|------|------|------|
| P33 唤醒 | `cpu/br28/power/key_wakeup.c:30` | `port0.gpio = IO_PORTB_01` | P33 下降沿唤醒的默认 IO |
| 长按复位 | `cpu/br28/setup.c:175` | `gpio_longpress_pin0_reset_config(IO_PORTB_01, 0, 0, 1, 1, 0)` | PB1 长按 8s 触发芯片复位 |
| iokey 扫描 | `board/br28/sdk_config.c:84-93` | `g_iokey_info[] = {KEY_POWER, IO_PORTB_01}` | IO 按键扫描配置表 |
| iokey UUID | `board/iokey_config.c:21` | `key_uuid_table[] = {0xaefa, KEY_POWER}` | 蓝牙配网/APP 按键自定义 |
| 开机检测 | `earphone/app_main.c:268` | `is_iokey_press_down()` 遍历找 KEY_POWER | 开机时判断按键是否被按下 |
| 关机 GPIO 保护 | `cpu/br28/power/power_port.c:109` | `soff_gpio_protect(get_iokey_power_io())` | 软关机时保护 PB1 不被设高阻态 |
| BT key table | `mode/bt/bt_key_msg_table.c:399` | `{KEY_POWER, bt_key_power_msg_remap}` | BT 模式下 KEY_POWER 的全部映射 |
| Idle key table | `mode/idle/idle_key_msg_table.c:88` | `{KEY_POWER, key_idle_io_num0_msg_table}` | 空闲模式 KEY_POWER 映射 |
| Music key table | `mode/music/music_key_msg_table.c:76` | `{KEY_POWER, key_music_io_num0_msg_table}` | 音乐模式 KEY_POWER 映射 |
| Linein key table | `mode/linein/linein_key_msg_table.c:89` | `{KEY_POWER, key_linein_io_num0_msg_table}` | 线路输入模式 KEY_POWER 映射 |
| PC key table | `mode/pc/pc_key_msg_table.c:89` | `{KEY_POWER, key_pc_io_num0_msg_table}` | PC 模式 KEY_POWER 映射 |
| TWS 组合键 | `message/adapter/key.c:219` | `if (key_value == KEY_POWER)` | TWS 左右耳组合键同步 |
| 触摸按键 | `board/br28/sdk_config.c:131` | `LPCTMU_CH1_PB1` 通道映射到 `KEY_POWER` | 硬件触摸通道占用 PB1 |
| adkey 默认 IO | `board/br28/sdk_config.c:118` | `g_adkey_data.key_io = IO_PORTB_01` | AD 按键的默认端口（即使不使能） |

这解释了为什么简单地"换一个 IO 给 iokey"是不够的。PB1 的引用散布在 10+ 个文件的各个层级，每一处都需要显式处理。

## Part 2：引入拨码开关的设计理念

### 2.1 回弹按键 vs 拨码开关的本质差异

从系统设计角度看，回弹按键和拨码开关代表了两种根本不同的交互模型：

| 维度 | 回弹按键 (PB1 原生) | 拨码开关 (改造目标) |
|------|---------------------|---------------------|
| 状态模型 | **无状态**：松键恢复，系统不记住 | **有状态**：ON/OFF 位置保持 |
| 时间模型 | **脉冲型**：短按/长按，依赖持续时间 | **电平型**：读取电平即可，不关心时间 |
| 交互语义 | **多功能**：单击/双击/三击/长按各有含义 | **单一功能**：就是电源开关 |
| 事件来源 | key_driver 10ms 扫描 + 动作识别 | P33 中断检测电平跳变 |
| 关机触发 | 长按 5 秒 → APP_MSG_KEY_POWER_OFF | 拨到 OFF → 直接关机 |
| 开机触发 | 按住不放 800ms | 拨到 ON |
| 关机后状态 | 松键自动回到 HIGH | 保持 HIGH（OFF）或 LOW（ON） |

核心矛盾：**回弹按键的整个软件栈（key_driver 扫描、动作识别、多击检测、场景化映射）对拨码开关毫无意义，且会引入误触发风险。**

### 2.2 核心挑战：关机自唤醒

这是从回弹按键切换到拨码开关时最核心的技术问题。

#### 问题机制

在 `ON = LOW`、`OFF = HIGH`（IO 上拉）的电平约定下：

```
运行中，拨到 OFF（LOW → HIGH 上升沿）
  → P33 上升沿中断触发
  → 执行软关机
  → CPU 掉电 / 深度睡眠
  → IO 上拉将 PB1 拉高（HIGH）
  → 如果 P33 唤醒边沿仍为 RISING_EDGE
  → IO 维持 HIGH ✓ 不触发（看似安全）

  但此时用户想把设备重新开机——拨到 ON（HIGH → LOW 下降沿）
  → P33 唤醒配的是 RISING_EDGE ← 方向相反！
  → P33 不触发 ← 设备醒不来
```

更糟的是，如果 P33 唤醒配了 RISING_EDGE 且 IO 处于 HIGH：

```
OFF 态 IO 维持 HIGH
  → 充电插入导致的 IO 微扰、RTC 唤醒重采样
  → 任何微小抖动都可能被 P33 识别为上升沿
  → 误开机 → 读 PB1 = HIGH (OFF) → 关机
  → 死循环：误开机 ↔ 关机
```

**根因**：P33 唤醒边沿方向必须与"拨到 ON"对应的电平跳变方向一致。ON = LOW → 拨到 ON = HIGH→LOW = **FALLING_EDGE**。

**不能简单地用 RISING_EDGE 触发关机然后保持 RISING_EDGE 等下次开机。** 两个原因：
1. 关机后拨 ON 是下降沿，与 RISING_EDGE 方向相反，无法唤醒
2. OFF 态 IO 被上拉维持在 HIGH，RISING_EDGE 容易误触发

#### 实测验证：BOTH_EDGE 不适用于运行时检测

在 JL7018 上实测发现，P33 的 `BOTH_EDGE` **只在芯片处于深度睡眠时有效，运行时不会产生中断**。这意味着运行时无法用 BOTH_EDGE 同时检测 ON→OFF 和 OFF→ON，必须在 ISR 中**动态翻转边沿**。

### 2.3 设计决策一：P33 动态翻转边沿

参考 T2508 示例工程的做法（T2508 用 PB2 拨码开关切换 CALL/CHAT 场景），采用**每次 P33 ISR 触发后切换到反边沿**的策略：

| 阶段 | P33 边沿配置 | 为什么 |
|------|-------------|--------|
| `key_wakeup_init()`（最早期） | `FALLING_EDGE` | 保证关机态拨 ON（HIGH→LOW）能唤醒芯片 |
| `rdx_dip_switch_init()`（RDX 初始化时）| `os_time_dly(2)` 消抖后读电平：ON=LOW → 设 `RISING_EDGE`；OFF=HIGH → 关机 | ON 时等 OFF（LOW→HIGH），OFF 时直接关机 |
| ISR 触发（RISING 沿，即检测到 OFF）| → 设 `FALLING_EDGE`，保持唤醒使能 | 关掉 RISING 沿中断源，防止 OFF 态的 HIGH 自唤醒；保留下次拨回 ON 的下降沿 |
| ISR 触发（FALLING 沿，即检测到 ON） | → 设 `RISING_EDGE` | 等下次拨到 OFF |
| 关机前 | `FALLING_EDGE` + `gpio_set_mode(INPUT_PULLUP_10K)` | 确保 OFF 时 IO 稳定 HIGH，拨 ON 时产生干净的 HIGH→LOW 下降沿 |

#### 2.3.1 核心机制：关机后保持唤醒使能

关机前设 **FALLING_EDGE 但保持唤醒使能（enable=1）**。OFF 状态 IO = HIGH 不会触发 FALLING_EDGE，所以不会误唤醒；而拨到 ON 时 HIGH→LOW 会产生干净的下降沿，触发 P33 唤醒。这就是"关机后还能用同一个 GPIO 唤醒"的机制。

对应代码 `rdx_dip_switch.c:26-50`（截取 OFF 分支）：

```c
// 消抖后确认 level == 1 (OFF)
p33_io_wakeup_edge(TCFG_DIP_SWITCH_POWER_IO, FALLING_EDGE);  // 关 RISING，换 FALLING
gpio_set_mode(IO_PORT_SPILT(TCFG_DIP_SWITCH_POWER_IO), PORT_INPUT_PULLUP_10K);  // 稳定上拉

os_time_dly(1);
if (gpio_read(TCFG_DIP_SWITCH_POWER_IO) == 1) {
    sys_enter_soft_poweroff(POWEROFF_NORMAL);                // 二次确认，执行关机
} else {
    p33_io_wakeup_edge(TCFG_DIP_SWITCH_POWER_IO, RISING_EDGE); // 快速拨回 ON，取消
}
```

#### 2.3.2 跨上下文委托：P33 ISR → os_taskq_post_type → app_core

P33 中断回调 `rdx_dip_switch_p33_irq()` 运行在 P33 域的 ISR 上下文中——不能直接调用 `gpio_read()`（需要时钟）、不能调用 `sys_enter_soft_poweroff()`（会触发任务调度和 flash 操作）。提交采用**函数指针投递**模式，将实际处理逻辑委托到 app_core 任务队列：

```c
// rdx_dip_switch.c:54-65
void rdx_dip_switch_p33_irq(P33_IO_WKUP_EDGE edge)
{
    int msg[2];

    if (!s_init_done) {       // 见 §2.3.3
        return;
    }

    msg[0] = (int)rdx_dip_switch_deferred_handle;  // 函数指针作为消息体
    msg[1] = 0;
    os_taskq_post_type("app_core", Q_CALLBACK, 2, msg);  // 委托到 app_core
}
```

`Q_CALLBACK` 消息类型让 app_core 消息循环直接调用 `msg[0]` 指向的函数——不经过消息分发、不需要消息 ID。从 ISR 到 app_core 的上下文切换完全由 RTOS 任务队列机制完成。

#### 2.3.3 竞态窗口保护：s_init_done 守卫

`key_wakeup_init()` 在早期 boot 就使能了 P33 唤醒，但 `rdx_dip_switch_init()` 要到 `rdx_app_all_init()` 才被调用。两者之间存在一个**初始化时序窗口**：

```
key_wakeup_init()                    ← P33 唤醒已使能，边沿 = FALLING_EDGE (struct 默认值)
      ⋮                             ← 窗口：此时 P33 中断可能触发，但 init 未完成
rdx_app_all_init()
  → rdx_spi_init_irq()
  → rdx_dip_switch_init()           ← 读电平、切边沿、设 s_init_done = true
```

窗口期如果 P33 中断触发，没有 `s_init_done` 守卫的情况下：`os_taskq_post_type` 会把 `rdx_dip_switch_deferred_handle` 投递到尚未完全就绪的 app_core，或者使用尚未被 `rdx_dip_switch_init()` 修正的边沿状态。`s_init_done` 让 ISR 直接 `return`，丢弃窗口期内的中断，等 init 完成后由下一次边沿事件接管。

这是**代价最小的竞态修复**——不需要改动 `key_wakeup_init` 和 `rdx_app_all_init` 的调用顺序。

#### 2.3.4 三条关机路径的分阶段设计

同一个"拨到 OFF → 关机"行为，在不同启动阶段需要走不同的代码路径：

| 阶段 | 代码位置 | 关机函数 | 为什么不能复用 |
|------|----------|----------|---------------|
| 早期 boot | `check_power_on_key()` | `power_set_soft_poweroff()` | app task 未就绪，`sys_enter_soft_poweroff()` 调用后不生效 |
| 运行时 init | `rdx_dip_switch_init()` | `sys_enter_soft_poweroff(POWEROFF_NORMAL)` | app_core 已就绪，走正常关机流程 |
| 运行时 ISR | `rdx_dip_switch_deferred_handle()` | `sys_enter_soft_poweroff(POWEROFF_NORMAL)` | 同上，通过 app_core 队列投递后执行 |

```c
// 路径 1 — 早期 boot（app_main.c:293-299）
if (get_power_on_status()) {
    app_var.poweron_reason = SYS_POWERON_BY_KEY;
} else {
    power_set_soft_poweroff();           // ← 轻量，不依赖 app task
}

// 路径 2/3 — 运行时（rdx_dip_switch.c:41,82）
sys_enter_soft_poweroff(POWEROFF_NORMAL); // ← 完整关机，app task 已就绪
```

`power_set_soft_poweroff()` vs `sys_enter_soft_poweroff()` 的区别是实测踩出来的——早期在 `check_power_on_key()` 中直接调用 `sys_enter_soft_poweroff()` 关机不生效，因为 app task 尚未初始化。这是早期 boot 阶段的硬约束。

> **电源域依赖**：关机后 P33 能检测到 PB1 下降沿唤醒，前提是 soft-off 期间 **IOVDD（VDDIO）保持供电**——IO 域掉电的话，内部上拉不工作，引脚电平不确定，也就没有干净的边沿。
>
> 当前项目 `power_param.nkeep_vddio` 默认 `0`（保持 VDDIO），`TCFG_LOWPOWER_VDDIO_KEEP` 宏为 `0` 但未显式接入 `power_param`，实际行为仍是保持。如果未来有人改成 `nkeep_vddio=1`（不保持），拨码开关将无法唤醒设备。详见 [BR28 电源域说明 §3（TCFG_LOWPOWER_VDDIO_KEEP）](0.BR28电源域说明.md#3-tcfg_lowpower_vddio_keep--关机时是否保持-vddio)。

### 2.4 设计决策二：彻底剥离 PB1 从按键框架

拨码开关作为**纯电源开关**，不需要 key_driver 扫描、不需要动作识别（单击/双击/长按）、不需要场景化映射（通话中双击挂断等）、不需要 TWS 组合键同步。

设计的核心原则：**PB1 退化为一个裸 GPIO + P33 唤醒源，只通过 `gpio_read()` 和 P33 中断与之交互。**

#### 剥离清单与守卫方式

| # | 子系统 | 文件 | 守卫 | 效果 |
|---|--------|------|------|------|
| 1 | 工具生成配置 | `sdk_config.h/c` | 无自定义 DIP 宏/guard | 保持可视化工具可解析、可重生成 |
| 2 | P33 唤醒回调 | `key_wakeup.c:31-32` | `#if TCFG_DIP_SWITCH_POWER_ENABLE` | callback 改为 `rdx_dip_switch_p33_irq` |
| 3 | 长按复位 | `setup.c:174-175` | `#elif !TCFG_DIP_SWITCH_POWER_ENABLE` | 拨码开关不需要 8s 长按复位 |
| 4 | 开机检测 | `app_main.c:264-265` | `#if TCFG_DIP_SWITCH_POWER_ENABLE` | 直接 gpio_read(PB1)，不做 800ms 按住检测 |
| 5 | 关机 GPIO 保护 | `power_port.c:119-121` | `#if TCFG_DIP_SWITCH_POWER_ENABLE` | `soff_gpio_protect(TCFG_DIP_SWITCH_POWER_IO)` 显式保护 PB1 |
| 6 | 模块级禁用 | `t2620_project_config.h:25-30` | `#if TCFG_DIP_SWITCH_POWER_ENABLE → #undef TCFG_ADKEY_ENABLE, #undef TCFG_LP_TOUCH_KEY_ENABLE` | 防止工具重生成后这些模块抢占 PB1 |

#### 为什么 PB1 不需要在按键事件域做任何守卫

核心原则：**正交分离。** PB1 通过 JLStudio 工具从 `g_iokey_info[]` 中删除后，key_driver 完全不知道它的存在，按键事件域所有文件（`iokey_config.c`、`key_uuid_table[]`、`bt_key_power_msg_remap()`、所有 mode key tables、TWS 组合键）**零 DIP 引用**。

这比"源头 + 消费端双重守卫"更简洁：不是"层层过滤"，而是"从不进入"。其他 IO 可以正常注册 KEY_POWER——产生端不拦截，消费端正常处理。

唯一需要防御的是模块级抢占：`t2620_project_config.h` 强制关闭 ADKEY 和 LP_TOUCH（它们的默认 IO 是 PB1），防止工具重生成 sdk_config.h 后这些模块意外启用。

### 2.5 TCFG_DIP_SWITCH_POWER_ENABLE 宏（含编译规范）

#### 宏定义

```c
// t2620_project_config.h
#ifndef TCFG_DIP_SWITCH_POWER_ENABLE
#define TCFG_DIP_SWITCH_POWER_ENABLE              1
#endif

#ifndef TCFG_DIP_SWITCH_POWER_IO
#define TCFG_DIP_SWITCH_POWER_IO                  IO_PORTB_01
#endif
```

设计原则：
- **单宏控制**：`TCFG_DIP_SWITCH_POWER_ENABLE` 是唯一的开关，设为 0 则全部行为回退到原 PB1 回弹按键
- **工具隔离**：定义在 `t2620_project_config.h`，并由 `app_config.h` 在 `sdk_config.h` 后立即包含；`sdk_config.h/c` 保持可视化工具生成格式
- **编译期隔离**：所有新逻辑都在 `#if TCFG_DIP_SWITCH_POWER_ENABLE` 内，出问题改 0 即可回退
- **IO 可配置**：`TCFG_DIP_SWITCH_POWER_IO` 作为独立宏，支持未来换到其他 IO

#### 为什么放在 t2620_project_config.h 而不是 sdk_config.h

这是提交中踩过的一个坑。最初实现把 DIP 宏直接放在 `sdk_config.h` 中：

```c
#define TCFG_DIP_SWITCH_POWER_ENABLE    1
#if TCFG_DIP_SWITCH_POWER_ENABLE
#define TCFG_DIP_SWITCH_POWER_IO        IO_PORTB_01
#endif
```

这个写法导致杰理可视化配置工具直接报错。根因是 `sdk_config.h` 由工具自动生成/解析，文件头部有明确约束：

> 功能块通过 `// ---------xxx------` 和 `#endif // xxx` 是工具识别的关键位置
> 目前工具暂不支持非文件已有的C语言语法

工具解析器遇到不认识的 `#if TCFG_DIP_SWITCH_POWER_ENABLE` 就会失败——即使 SDK 原生块如 `#if TCFG_SD0_ENABLE` 也用了 `#if`，但那是工具已知的宏。自定义宏 `TCFG_DIP_SWITCH_POWER_ENABLE` 不在工具的配置定义中，解析器无法处理。

**解决方案：三层配置架构**

```
工具层（可视化工具管理）
  sdk_config.h  ──→ 仅包含工具认识的标准宏
  sdk_config.c  ──→ 仅包含工具生成的标准数组
       │ include
       ▼
策略层（手写，项目级决策）
  t2620_project_config.h ──→ #define TCFG_DIP_SWITCH_POWER_ENABLE 1
       │                      #define TCFG_DIP_SWITCH_POWER_IO IO_PORTB_01
       │                      #undef TCFG_ADKEY_ENABLE  ← 强制关闭占用 PB1 的模块
       │ include (via app_config.h)
       ▼
适配层（手写，根据策略做过滤和切换）
  iokey_config.c ──→ 运行时跳过 KEY_POWER + TCFG_DIP_SWITCH_POWER_IO
  key_wakeup.c  ──→ callback 切换到 rdx_dip_switch_p33_irq
  key tables    ──→ 编译期排除 KEY_POWER 条目
  power_port.c  ──→ soff_gpio_protect(TCFG_DIP_SWITCH_POWER_IO)
```

每一层的职责和边界：

| 层 | 谁管理 | 可以放什么 | 不能放什么 |
|----|--------|-----------|-----------|
| 工具层 | 可视化工具 | SDK 标准 `TCFG_*` 宏、`#if TCFG_XXX_ENABLE` / `#endif // TCFG_XXX_ENABLE` 原生块 | 自定义宏、不在工具 schema 中的 `#if` 条件编译 |
| 策略层 | 手写 | `#ifndef` 默认值、`#if TCFG_DIP_SWITCH_POWER_ENABLE` 条件块、`#undef` 覆盖工具宏 | — |
| 适配层 | 手写 | `#if TCFG_DIP_SWITCH_POWER_ENABLE` 守卫、运行时过滤循环 | — |

**为什么 iokey 用运行时过滤而不是编译期 `#if` 守卫**

sdk_config.c 中的 `g_iokey_info[]` 是工具生成的。如果在里面加 `#if !TCFG_DIP_SWITCH_POWER_ENABLE`：
1. 工具重生成 sdk_config.c 时会覆盖掉自定义 guard
2. 工具不认识 `TCFG_DIP_SWITCH_POWER_ENABLE`，解析失败

因此工具层保持 PB1 条目原样存在，适配层在 `iokey_config.c:53-58` 用运行时 `continue` 跳过：

```c
for (int i = 0; i < ARRAY_SIZE(g_iokey_info); i++) {
#if TCFG_DIP_SWITCH_POWER_ENABLE
    if (info[i].key_value == KEY_POWER && info[i].key_io == TCFG_DIP_SWITCH_POWER_IO)
        continue;  // 拨码模式下不注册 PB1
#endif
    // copy to iokey_ports[platform_data.num++]
}
```

可视化工具仍然可以正常编辑、生成 PB1 按键配置——固件在初始化时自己决定是否注册它。

**为什么 overlay 里要 `#undef` ADKEY 和 LP_TOUCH**

sdk_config.c 中 `g_adkey_data.key_io` 和 `LPCTMU_CH1_PB1` 默认使用 `IO_PORTB_01`。即使当前 `TCFG_ADKEY_ENABLE=0`、`TCFG_LP_TOUCH_KEY_ENABLE=0`，工具重生成时可能改回 1。overlay 中的 `#undef` + `#define 0` 确保：**只要拨码开关使能，这两个模块就被强制关闭**，不会因为工具操作而意外启用并占用 PB1。

**测试验证**

`tests/host/test_t2620_config_overlay.ps1` 作为结构回归脚本，验证 5 项约束：
1. sdk_config.h 不含 `TCFG_DIP_SWITCH_POWER` — 工具层未被污染
2. sdk_config.c 不含 `TCFG_DIP_SWITCH_POWER` — 同上
3. overlay 文件存在且宏值正确 — 策略层完整
4. app_config.h 在 `sdk_config.h` 后立即 include overlay — include 顺序正确
5. iokey_config.c 存在运行时过滤代码 — 适配层生效

通过 `.vscode/tasks.json` 的 `test: config overlay` 任务可在 VS Code 中一键运行。

#### 与 TCFG_AUTO_POWERON_ENABLE 的联动

拨码模式下 `TCFG_AUTO_POWERON_ENABLE` 从 1 改为 0：

```c
#define TCFG_AUTO_POWERON_ENABLE    0  // 上电后不自动开机，等拨码到 ON
```

原因：拨码开关 OFF 时不应该开机。上电后 `check_power_on_key()` 读取拨码位置，OFF 则立即走轻量关机。

#### 条件编译文件的 #include 顺序规范

`rdx_dip_switch.c` 和 `rdx_dip_switch.h` 都依赖 `TCFG_DIP_SWITCH_POWER_ENABLE` 宏控制整个编译单元。该宏由 `app_config.h → t2620_project_config.h` 提供。如果 `#include "app_config.h"`（宏的定义来源）出现在 `#if TCFG_DIP_SWITCH_POWER_ENABLE` 之后，宏未定义，文件编译为空——不会报链接错误，但所有功能静默缺失。

**正确顺序**（`rdx_dip_switch.c`）：

```c
#include "app_config.h"              // ← 必须第一个，宏定义来源
#include "rdx_dip_switch.h"

#if TCFG_DIP_SWITCH_POWER_ENABLE    // ← 宏已可见

#include "system/includes.h"
// ...

#endif // TCFG_DIP_SWITCH_POWER_ENABLE
```

**正确顺序**（`rdx_dip_switch.h`）：

```c
#include "app_config.h"              // ← 引入 t2620_project_config.h
```

### 2.7 改造后的完整时序

#### 拨 ON 开机

```
电池上电
  ↓
CPU 上电，setup_arch()
  ├── gpio_longpress_reset → 跳过 PB1（TCFG_DIP_SWITCH_POWER_ENABLE=1）
  └── memory_init()
  ↓
key_wakeup_init()
  ├── port0.callback = rdx_dip_switch_p33_irq  ← DIP 专用回调
  ├── port0.edge    = FALLING_EDGE              ← 保证 ON (HIGH→LOW) 能唤醒
  └── p33_io_wakeup_enable(PB1, 1)
  ↓
check_power_on_key()
  ├── gpio_read(TCFG_DIP_SWITCH_POWER_IO)
  ├── PB1 = LOW (ON)  → app_var.poweron_reason = SYS_POWERON_BY_KEY
  └── 继续开机
  ↓
rdx_app_all_init()
  → rdx_dip_switch_init()
    ├── s_init_done = true  ← 启用 ISR→app_core 通路
    ├── gpio_read(PB1) = LOW (ON)
    └── p33_io_wakeup_edge(PB1, RISING_EDGE)  ← 切上升沿等下次拨到 OFF
  ↓
正常运行... (key_driver 不扫描 PB1, bt_mode_key_table 无 KEY_POWER)
  ↓
用户拨到 OFF (LOW → HIGH)
  ↓
P33 RISING_EDGE 触发
  → rdx_dip_switch_p33_irq()
    → os_taskq_post_type("app_core", Q_CALLBACK, rdx_dip_switch_deferred_handle)
  ↓
rdx_dip_switch_deferred_handle():
  1. os_time_dly(2)                        ← 消抖
  2. gpio_read(PB1) = HIGH (OFF)          ← 确认
  3. p33_io_wakeup_edge(PB1, FALLING_EDGE) ← 关 RISING，换 FALLING
  4. gpio_set_mode(PB1, INPUT_PULLUP_10K)  ← 稳定上拉
  5. os_time_dly(1)                        ← 再等一个 tick
  6. gpio_read(PB1) = HIGH (OFF)          ← 二次确认
  7. sys_enter_soft_poweroff()            ← 关机
     → gpio_config_soft_poweroff()
       → soff_gpio_protect(PB1)           ← 保护 PB1
  ↓
CPU 深度睡眠，PB1 = HIGH (稳定上拉)
P33 保持 FALLING_EDGE + 唤醒使能
  ↓
... 任意时间后 ...
  ↓
用户拨到 ON (HIGH → LOW 下降沿)
  ↓
P33 FALLING_EDGE 触发 → CPU 唤醒
  ↓
回到 check_power_on_key() → 检测到 LOW → 继续开机
```

### 3.2 宏生效边界

当前正常构建的关键宏来自两个层级：

```c
// sdk_config.h（工具生成）
#define CONFIG_BT_MODE                            BT_NORMAL
#define TCFG_NORMAL_SET_DUT_MODE                  0
#define TCFG_AUTO_POWERON_ENABLE                  0

// t2620_project_config.h（项目 overlay）
#define TCFG_DIP_SWITCH_POWER_ENABLE              1
#define TCFG_DIP_SWITCH_POWER_IO                  IO_PORTB_01
```

`SDK/apps/earphone/include/app_config.h:303-360` 有一个容易误判的覆盖分支：当不是 `BT_NORMAL`，或进入特殊测试配置时，它会 `#undef TCFG_AUTO_POWERON_ENABLE` 并重新定义为 `1`。当前项目是 `CONFIG_BT_MODE == BT_NORMAL` 且 `TCFG_NORMAL_SET_DUT_MODE == 0`，所以该覆盖分支不生效，`app_poweron_check()` 会继续执行 `check_power_on_key()`。

维护结论：

- 正常量产模式必须保持 `TCFG_AUTO_POWERON_ENABLE = 0`，否则 `app_poweron_check()` 会直接 return，拨码 OFF 上电也可能继续启动
- `TCFG_DIP_SWITCH_POWER_ENABLE` 和 `TCFG_DIP_SWITCH_POWER_IO` 不放在 `sdk_config.h/c` 中，避免可视化工具解析未知自定义宏
- 切 BQB/PER/FCC/DUT 等测试模式时，要重新确认 `app_config.h` 的覆盖逻辑是否绕过了拨码开机检测
- `TCFG_DIP_SWITCH_POWER_ENABLE = 0` 是回退到原回弹按键链路的总开关，但回退前要确认硬件是否仍是回弹按键，而不是拨码开关

### 3.3 残留 PB1 引用与复用风险

改造目标不是消灭仓库里所有 `IO_PORTB_01` 字符串，而是确保 PB1 不再作为回弹 KEY_POWER 事件源。当前仍有一些 PB1 引用需要按风险分层理解：

| 引用 | 当前状态 | 风险判断 |
|------|----------|----------|
| `sdk_config.c:118` 的 `g_adkey_data.key_io = IO_PORTB_01` | `t2620_project_config.h` 在拨码模式下强制 `TCFG_ADKEY_ENABLE = 0` | 低风险；未来若要启用 ADKEY，必须先换离 PB1 |
| `sdk_config.c:154-173` 的 `LPCTMU_CH1_PB1` | 已加 `#if TCFG_LP_TOUCH_KEY_ENABLE ... #endif` 包裹，与 `g_adkey_data` 一致；`t2620_project_config.h` 在拨码模式下强制宏 = 0 | 已修复；即便工具重新打开宏，编译也不会产生未引用的 PB1 死数据 |
| `in_ear_detect.c` / `in_ear_manage.c` 中 `TCFG_KEY_IN_EAR_FILTER_ENABLE` 分支读取 `IO_PORTB_01` | 受入耳检测与按键过滤相关宏保护，当前主配置未发现该宏启用 | 中风险；未来打开入耳按键过滤时会直接读 PB1 电平，可能把拨码 ON 误当成"按键被按住" |
| `eartouch_manage.c` 中 `p33_io_wakeup_enable(IO_PORTB_01, 1)` | 受 `TCFG_OUTSIDE_EARTCH_ENABLE` 等外部入耳触摸宏保护，当前主配置未发现启用 | 中风险；未来启用外部 eartouch 中断时，应改为 `__this->int_port` 或实际入耳中断 IO，不能固定 PB1 |
| demo、注释、外设枚举里的 PB1 | 不参与当前 earphone 量产链路 | 低风险；只作为搜索噪声处理 |

后续如果要新增功能，原则是：PB1 在拨码模式下只能作为 `TCFG_DIP_SWITCH_POWER_IO` 使用，不能再被任何按键、触摸、入耳过滤或测试逻辑当作普通输入复用。

### 3.4 回归验证清单

改动以下任一文件或宏后，至少做一次完整回归：

| 场景 | 操作 | 期望结果 |

### 运行时问题

| 问题 | 根因 | 修复 |
|------|------|------|
| 运行时拨 OFF 无反应 | JL7018 P33 `BOTH_EDGE` 仅在深度睡眠时有效，运行时不会产生中断 | 改用动态翻转边沿：初始 FALLING_EDGE，init 时切 RISING_EDGE，每次 ISR 触发后手工切反边沿 |
