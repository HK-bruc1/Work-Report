# 老 SDK 打新 RTC 补丁指南

## 1. 文档定位

这份文档的目标只有一个：让老 SDK 打上新 RTC 补丁后，具备可用的 RTC 底层能力。

这里说的“具备 RTC 能力”，是指下面这些底层接口和行为能够成立：

- 可以正常包含并调用 `rtc/rtc_dev.h` 中的新 RTC 接口。
- 可以正常读写当前时间与闹钟时间。
- Soft-Off 前后 RTC 状态可以按新驱动的方式保存与恢复。
- 编译和链接能够通过，不再依赖旧版 `rtc.h` 入口。

这份文档**不负责**下面这些内容：

- 不负责把 RTC 融入具体业务协议。
- 不负责做闹钟、日程、UI、同步策略等应用层逻辑。
- 不负责把第三方业务代码改造成某个产品方案。

如果你只是想让老 SDK 先拥有新 RTC 驱动能力，那么看本篇就够了。

如果你还想验证补丁是否真的可用，可以在补丁打完后，再临时接入当前仓库的 RTC 测试模块做验收。测试模块属于“验收工具”，不是 RTC 业务接入。

---

## 2. 先说结论

结合当前仓库代码，老 SDK 打新 RTC 补丁时，最重要的原则有三条：

### 2.1 不要继续走旧 `rtc.h` 入口

当前仓库已经统一切到新头文件入口：

```c
#include "rtc/rtc_dev.h"
```

对应代码可以直接参考：

- `SDK/interface/driver/cpu/br28/asm/includes.h`
- `SDK/apps/earphone/device_config.c`
- `SDK/apps/common/third_party_profile/rdx_protocol/rdx_rtc.c`

这意味着：

- 旧 SDK 里如果还保留 `asm/rtc.h` 一类旧头，不要再拿它当主入口。
- 正确做法是把引用 RTC 的地方统一切到 `rtc/rtc_dev.h`。
- 当前仓库里旧 `rtc.h` 入口已经不存在了，说明这条路最终就是要收敛到新头文件，而不是做旧头兼容层。

### 2.2 新头文件和新 `cpu.a` 是一组

新 RTC 补丁不只是头文件变化，还对应了新的底层实现。

所以你不能只拷贝 `rtc_dev.h`，却继续沿用老 `cpu.a`。否则常见结果是：

- 编译通过，链接失败。
- 新 RTC 接口声明已经存在，但静态库里没有实现。

当前仓库的结论很明确：**头文件替换和 `SDK/cpu/br28/liba/cpu.a` 替换要一起做。**

### 2.3 先让底层 RTC 能跑通，再谈业务

补丁阶段只做四件事：

- 头文件入口切对。
- RTC 平台数据结构切到新驱动要求。
- Soft-Off 电源参数也同步到新 RTC 时钟源配置。
- VM 与 Soft-Off 持久化链路切到新 RTC 方案。

做到这一步，老 SDK 就已经有 RTC 能力了。后面的业务接入，是下一阶段的问题，不应该混进这份补丁文档里。

## 3. 必改项：具体改什么，为什么要改

下面按“当前仓库实际落地的修改”来写。

### 3.1 `SDK/interface/driver/cpu/br28/asm/includes.h`

#### 要改什么

把 RTC 统一入口切到：

```c
#include "rtc/rtc_dev.h"
```

#### 为什么要改

这是 CPU/ASM 层的公共聚合头。它决定了上层代码默认拿到的是哪套 RTC 声明。

如果这里还是旧入口，那么即使其他业务文件手工改了一部分，整个工程的 RTC 入口仍然是不一致的，最终很容易出现：

- 某些文件看到的是旧结构体。
- 某些文件看到的是新接口。
- 编译或链接阶段出现 ABI/声明错配。

这一步的目标不是“临时过编译”，而是把 RTC 的公共入口一次性切干净。

---

### 3.2 `SDK/apps/earphone/device_config.c`

#### 要改什么

这个文件里有两类 RTC 改动。

第一类是头文件入口：

```c
#include "rtc/rtc_dev.h"
```

第二类是 RTC 平台数据结构本身。当前仓库采用的是：

```c
RTC_DEV_PLATFORM_DATA_BEGIN(rtc_data)
    .default_sys_time = &def_sys_time,
    .default_alarm = &def_alarm,
    .rtc_clk = RTC_CLK_RES_SEL,
    .rtc_sel = HW_RTC,
    .cbfun = NULL,
RTC_DEV_PLATFORM_DATA_END()
```

当前仓库为了测试闹钟，还额外做了条件挂接：

```c
#if RTC_TEST_ENABLE && (RTC_TEST_ITEM_MASK & RTC_TEST_ITEM_ALARM)
    .cbfun = rtc_test_alarm_callback,
#else
    .cbfun = NULL,
#endif
```

如果你参考当前仓库的闹钟测试接线，还要注意一件事：`rtc_test_alarm_callback` 不能按普通任务态回调去理解。

它是 RTC 驱动在闹钟中断里调用的 ISR 回调。当前仓库已经验证过：

- 回调里直接做 `RTC_LOG`
- 回调里直接做 `rtc_read_time()`
- 回调里直接做 `rtc_alarm_en(0)`

都会有中断上下文风险。

所以当前测试实现采用的是：

- `cbfun` 里只做 `os_taskq_post_type("app_core", Q_CALLBACK, ...)` 投递
- 真正的 RTC 读写、关闹钟和日志输出都切到 `app_core` 任务态执行

但这部分属于“测试模块接线”，不是 RTC 补丁本身的必需项。对于纯补丁场景，`.cbfun = NULL` 就够了。

#### 为什么要改

老 SDK 常见的是旧字段：

```c
.clk_sel = CLK_SEL_32K
```

而新 RTC 驱动要求的是：

- `.rtc_clk`
- `.rtc_sel`
- `.cbfun`

也就是说，这里不是单纯“换个 include”就完了，而是**平台数据结构已经发生了接口升级**。

如果你还沿用旧字段：

- 轻则编译直接报字段不存在。
- 重则你误以为编译过了，但 RTC 初始化参数其实已经错位。

---

### 3.3 `SDK/apps/earphone/include/app_config.h`

#### 要改什么

当前仓库新增了一个 RTC 时钟源选择宏：

```c
#define RTC_CLK_RES_SEL             CLK_SEL_LRC
```

#### 为什么要改

新 RTC 平台数据不再直接在 `device_config.c` 里写死 `CLK_SEL_32K`，而是引用板级宏：

```c
.rtc_clk = RTC_CLK_RES_SEL
```

这样做的好处是：

- RTC 时钟源选择收敛到一个配置点。
- 同一个工程可以根据硬件条件切 `CLK_SEL_32K` 或 `CLK_SEL_LRC`。
- 不用反复去业务文件里翻 `.clk_sel`。

当前仓库选的是 `CLK_SEL_LRC`，原因也很现实：对于老项目/旧硬件，很多时候并没有稳定可用的外部 32K 晶振，先用 LRC 更容易把 RTC 跑起来。

但要特别注意：`CLK_SEL_LRC` 更适合作为“运行态 RTC bring-up”方案。也就是说，它很适合先把 RTC 的读写、走时、闹钟这些基础能力跑通；如果你后续还要评估更严格的低功耗连续计时能力，应结合硬件上是否存在外部 32K 晶振单独分析，而不要把这件事默认绑定进当前测试模块。

如果你的板子已经确认外挂 32K 晶振可用，那么这里也可以改成：

```c
#define RTC_CLK_RES_SEL             CLK_SEL_32K
```

#### 额外说明

旧文档里提到“低功耗掉电 RAM 从 2 改 3”这一类说法，在当前仓库代码里并没有作为 RTC 补丁的固定必改项落地，因此**不要把它写成 RTC 补丁的硬性步骤**。

本仓库当前 RTC 能力跑通，依赖的是头文件入口、平台数据、VM 和库版本对齐，而不是把某个低功耗 RAM 宏机械改成固定值。

---

### 3.4 `SDK/cpu/br28/power/power_config.c`

#### 要改什么

除了 `device_config.c` 里的 RTC 平台数据，Soft-Off 电源参数里也要同步 RTC 时钟源。当前仓库采用的是：

```c
#if TCFG_APP_RTC_EN
    .rtc_clk        = RTC_CLK_RES_SEL,
#endif
```

对应对象是：

```c
const struct low_power_param power_param = {
    ...
    .osc_type = TCFG_LOWPOWER_OSC_TYPE,
#if TCFG_APP_RTC_EN
    .rtc_clk  = RTC_CLK_RES_SEL,
#endif
    ...
};
```

#### 为什么要改

这是这次复盘里真正暴露出来、而旧文档没写全的一点。

当前仓库里其实有两条 RTC 时钟配置链路：

- `device_config.c` 的 `rtc_data.rtc_clk`，决定运行态 RTC 驱动初始化时用什么时钟源。
- `power_config.c` 的 `low_power_param.rtc_clk`，决定 Soft-Off 时是否按这个 RTC 时钟源保住对应时钟。

如果你只改前者，不改后者，就会出现一个非常典型的现象：

- 开机后 RTC 读写、走时、闹钟看起来都正常。
- 一旦进入 Soft-Off，RTC 在关机期间不连续计时。
- 再开机时会误以为“RTC 补丁没生效”或“关机流程没走到”。

这次仓库实测已经证明，`power_config.c` 里的这条 `.rtc_clk` 不是可有可无的补充项，而是 Soft-Off RTC 连续计时是否成立的直接前提。

所以文档里的正确表述应该是：

- `device_config.c` 负责把 RTC 驱动本身配起来。
- `power_config.c` 负责把 Soft-Off 电源侧的 RTC 时钟保持配起来。

两边都要改，RTC 才是“运行态正常，关机后也继续跑”。

---

### 3.5 `SDK/interface/utils/syscfg_id.h`

#### 要改什么

当前仓库中，新 RTC 驱动使用的是这两个 VM ID：

```c
#define VM_RTC_ALM_TIME             105
#define VM_RTC_TIME_TRIM            106
```

#### 为什么要改

新 RTC 驱动不仅是“能读个时间”这么简单，它还需要持久化与 RTC 相关的数据，例如：

- 闹钟时间
- RTC trim/校准相关数据

如果这里还沿用旧 ID 含义，或者没有切到新定义，那么就会出现两类问题：

- RTC 驱动写 VM 时读写错位。
- 旧 VM 数据含义和新驱动期望不一致。

所以这一步是驱动能力的一部分，不是可有可无的“顺手改一下”。

---

### 3.6 `SDK/cpu/br28/power/power_app.c`

#### 要改什么

在 `power_soff_callback()` 里补上：

```c
poweroff_save_rtc_time();
```

#### 为什么要改

新 RTC 驱动不是只处理“系统正在运行时”的读写，它还要覆盖 Soft-Off 场景。

Soft-Off 前如果不保存 RTC 相关状态，那么下一次上电时：

- RTC 可能无法按新驱动预期恢复。
- 关机前后的连续性验证无法成立。
- 某些依赖 RTC 持久化的数据链路会断掉。

所以这一步的本质是：**把 RTC 的持久化链路接入电源关闭流程。**

这也是为什么旧文档里那句“保存 RTC 时间？？？”实际上不是多余代码，而是新补丁必需的电源流程接点。

---

### 3.7 `SDK/apps/common/third_party_profile/rdx_protocol/rdx_rtc.c` 以及其他业务侧 RTC 调用文件

#### 要改什么

把仍然包含旧 RTC 头的业务文件，也统一切到：

```c
#include "rtc/rtc_dev.h"
```

当前仓库里，`rdx_rtc.c` 已经这样处理。

#### 为什么要改

补丁的目标虽然不是“融入业务代码”，但只要某个现存业务文件还在直接调用 RTC 接口，它就必须跟着切到新头文件入口。

这不属于“业务逻辑改造”，而属于**补丁后的编译面清理**。

换句话说：

- 你不需要在这里新增业务功能。
- 但你必须让旧业务文件能在新 RTC 头文件体系下继续编译。

---

### 3.8 `SDK/cpu/br28/liba/cpu.a`

#### 要改什么

替换为与新 RTC 补丁配套的新版 `cpu.a`。

#### 为什么要改

`rtc_dev.h` 里不仅新增了接口名，还新增了与：

- RTC 读写
- 闹钟
- trim
- Soft-Off 持久化

相关的底层实现入口。

这些实现并不在业务 C 文件里，而是在静态库里。

所以这里的原则非常明确：

- **只换头，不换库，不行。**
- **只换库，不切新头，也不行。**

必须把“头文件声明”和“静态库实现”一起升级。

---

## 4. 当前仓库不再采用的旧思路

为了避免继续误导，下面两种做法不要再写进新文档：

### 4.1 不要靠恢复旧 `asm/rtc.h` 来兼容新补丁

当前仓库已经证明，这条路不是正确方向。

正确方向是：

- 统一切到 `rtc/rtc_dev.h`
- 把调用方和平台数据同步到新接口

而不是恢复一个旧头文件，把问题往后拖。

### 4.2 不要把“测试模块接线”写成“RTC 业务融合”

`rtc_test.c` 只是补丁验收工具。

它的作用是：

- 帮你验证 RTC 在老 SDK 上是否真的跑通。
- 帮你确认走时、闹钟、Soft-Off 连续性是否成立。

它不是产品业务功能的一部分，验收完成后可以关掉，也可以移除。

## 5. 本仓库实测的伴生兼容项

这一节不是 RTC 业务逻辑，而是**在当前仓库中，替换新 `cpu.a` 后额外遇到并解决的构建兼容问题**。

如果你的老 SDK 没有遇到这些报错，这一节可以不跟。

如果你遇到了类似下面这些错误：

- `gpadc_battery_init` 重复定义
- `gpadc_battery_get_voltage` 重复定义
- `lib_adc_clk_max` 未定义
- `gpadc_ch_power` / `gpadc_ch_power_div` 未定义
- `adc_init` 未定义

那么当前仓库的处理方式可以直接参考：

### 5.1 `SDK/apps/earphone/log_config/lib_driver_config.c`

补回新库依赖的 GPADC 配置常量，例如：

```c
const u32 lib_adc_clk_max = 500 * 1000;
const u8 gpadc_battery_mode = WEIGHTING_MODE;
const u32 gpadc_ch_power = ...;
const u8 gpadc_ch_power_div = ...;
```

### 5.2 `SDK/interface/driver/cpu/periph/gpadc.h`

当前接口入口已经是：

```c
void gpadc_init(void);
```

### 5.3 `SDK/apps/earphone/board/sdk_board_config.c`

当前初始化调用已经切到：

```c
gpadc_init();
```

### 5.4 不要再同时编译旧版源码实现和新版库实现

如果老 SDK 里仍然把旧的 `gpadc_battery` 源码一起编译进去，就容易和新 `cpu.a` 里的实现发生重复定义。

这一节的重点是：

- 它是“换新库带来的兼容收口”。
- 它不是 RTC 业务代码。
- 但它确实是当前仓库在“老 SDK 打新 RTC 补丁”过程中实际碰到的问题，所以值得写进文档。

## 6. 打完补丁后，如何利用 RTC 测试模块做验收

这一节是**可选但强烈建议**的。

目的不是把 RTC 接进业务，而是确认补丁是否真的把 RTC 能力打通了。

### 6.1 测试模块由哪些文件组成

当前仓库的 RTC 测试模块主要包括：

- `SDK/apps/earphone/rtc_test.c`
- `SDK/apps/earphone/include/rtc_test.h`

其中：

- `rtc_test.c` 是测试实现。
- `rtc_test.h` 是对外接口和 no-op 桩函数。

### 6.2 最小接入点只有两处

为了做测试，当前仓库只在两处接入测试模块：

#### 1）开机初始化

文件：`SDK/apps/earphone/app_main.c`

调用：

```c
rtc_test_init();
```

作用：上电后自动按配置启动选中的 RTC 测试项。

#### 2）闹钟回调挂接

文件：`SDK/apps/earphone/device_config.c`

挂接点：

```c
.cbfun = rtc_test_alarm_callback
```

说明：

- 这只在开启闹钟测试时需要。
- 如果只是补丁本身，不测闹钟，保持 `.cbfun = NULL` 即可。

### 6.3 测试开关都在 `SDK/apps/earphone/include/app_config.h`

当前仓库已经把 RTC 测试配置集中到这些宏：

```c
#define RTC_TEST_ENABLE                  1
#define RTC_TEST_ITEM_MASK               (...)
#define RTC_TEST_TIMING_SAMPLE_PERIOD_MS (10 * 1000UL)
#define RTC_TEST_TIMING_SAMPLE_COUNT     6
#define RTC_TEST_ALARM_DELAY_SEC         20
```

测试项 bitmask 定义也都在这里，例如：

```c
#define RTC_TEST_ITEM_REG_DUMP      0x00000001UL
#define RTC_TEST_ITEM_BASIC_RW      0x00000002UL
#define RTC_TEST_ITEM_TIMING        0x00000004UL
#define RTC_TEST_ITEM_ALARM         0x00000008UL
#define RTC_TEST_ITEM_UTILS         0x00000010UL
```

当前仓库默认配置是：

```c
#define RTC_TEST_ITEM_MASK (RTC_TEST_ITEM_REG_DUMP | RTC_TEST_ITEM_TIMING | \
                            RTC_TEST_ITEM_BASIC_RW | RTC_TEST_ITEM_ALARM | \
                            RTC_TEST_ITEM_UTILS)
```

也就是当前仓库默认把 Test1~Test5 全部打开，适合作为仓内持续回归时的一次性全量检查。

但如果你是在老 SDK 上第一次做 RTC 补丁验收，**不建议直接照抄这个全开默认值**。

更稳妥的方式是：先按下面的顺序分轮打开测试项，先确认“RTC 补丁链路是否打通”，再扩大到闹钟和工具函数。

### 6.4 建议的验收顺序

#### 第一轮：只开快照、走时

建议先临时把 mask 收窄为：

```c
RTC_TEST_ITEM_REG_DUMP | RTC_TEST_ITEM_TIMING
```

你应该能看到类似下面的日志：

```text
[RTC_TEST] RTC Test Module Start
[RTC_TEST] Config mask=0x05 reg=ON rw=OFF timing=ON alarm=OFF utils=OFF
[RTC_TEST] === Test1: RTC Snapshot ===
[RTC_TEST] Test1 RTC: 2026-05-13 00:00:00
[RTC_TEST] === Test3: Timing Accuracy ===
```

重点看两件事：

- Test1 是否能稳定读出 RTC 时间。
- Test3 里 RTC 是否持续递增，而不是一直停在 0 秒。

这里要注意，Test3 只能证明“运行态 RTC 在走”。如果运行态正常，但 Soft-Off 几分钟后再开机只前进约 1 秒，优先回头检查：

- `device_config.c` 的 `.rtc_clk = RTC_CLK_RES_SEL` 是否已接上。
- `power_config.c` 的 `low_power_param.rtc_clk = RTC_CLK_RES_SEL` 是否也已接上。

如果 Test3 显示 RTC 没走，优先排查：

- `RTC_CLK_RES_SEL` 是否适合当前硬件。
- 外部 32K 晶振是否真的存在并可用。
- 如果没有外部 32K，先切 `CLK_SEL_LRC` 做基础验证。

#### 第二轮：闹钟测试

如果第一轮没问题，再把 `RTC_TEST_ITEM_ALARM` 加进 mask。

当前仓库的 Test4 不再是“只要回调到了就算通过”，而是会同时打印并比较：

```text
[RTC_TEST] === Test4: Alarm Callback Fired! reason=0x00/0x01/... ===
[RTC_TEST]   RTC alarm expected: 2025-06-15 10:30:20
[RTC_TEST]   RTC at alarm:      2025-06-15 10:30:19/20/21
[RTC_TEST]   alarm delta: 0/1 s
[RTC_TEST] Test4 PASS: alarm fired and disabled within 1 s tolerance
```

这里要注意三点：

- `reason` 是底层驱动给出的实测值，可能是 `0x00`、`0x01` 或其他值，不要在文档里写死成固定常量。
- `alarm delta` 允许 `1` 秒容差，这是为了兼容 RTC 秒级采样和任务调度边界，不应该要求“必须精确打印到整秒”。
- 如果闹钟一触发就断言，优先检查 `cbfun` 里是不是直接做了日志或 RTC 操作；当前仓库已经验证，这类操作应从中断回调切到 `app_core` 任务态。

如果到了时间没有回调，优先排查：

- `device_config.c` 的 `.cbfun` 是否真的挂到了 `rtc_test_alarm_callback`
- mask 里是否真的打开了 `RTC_TEST_ITEM_ALARM`

#### 第三轮：基础读写和工具函数

如果前两轮都正常，再把 `RTC_TEST_ITEM_BASIC_RW` 和 `RTC_TEST_ITEM_UTILS` 打开。

这一轮的目的不再是验证“补丁有没有接通”，而是验证 RTC 相关工具函数在当前库版本下的行为是否符合预期。

你应该至少看到：

- Test2 的 `write` 和 `read back` 完全一致
- Test5 最后输出一条总结果日志

例如：

```text
[RTC_TEST] Test5 PASS: all utility checks passed
```

或者：

```text
[RTC_TEST] Test5 WARN: 2 utility check(s) failed; current evidence points to 5d/5g as library defects
```

根据当前仓库的最新实测：

- `5d` 的 `time_add_sec` 在年末跨界时会多进 1 天
- `5g` 的 `month_to_day` 会把闰年和平年 2 月天数判断反掉

如果你看到的是这种“其余项都对，只有 5d / 5g 失败”的结果，更应该把它当成**底层库缺陷候选**继续跟，而不是放宽测试判定把问题掩盖掉。

### 6.5 验收完成后怎么办

验收完成后有两种做法：

#### 做法 A：关闭测试模块，保留补丁能力

最简单的方法是：

```c
#define RTC_TEST_ENABLE 0
```

由于 `rtc_test.h` 在测试关闭时提供 no-op 桩函数，所以当前仓库里的调用点不需要再额外写条件编译。

#### 做法 B：保留测试模块，作为底层回归工具

如果这个老 SDK 后续还会继续演进，保留 RTC 测试模块作为底层回归测试也很有价值。

尤其是下面这几类回归：

- 换板级 RTC 时钟源
- 换 `cpu.a`
- 改 RTC 闹钟回调逻辑

## 8. 如果把硬件 RTC 能力给到 RDX 框架

这一节不再是“老 SDK 打新 RTC 补丁”的必改项，而是补丁打通之后，RDX 业务如何真正吃到硬件 RTC 能力。

如果只看当前仓库代码，结论其实已经很明确：**RDX 现在生效的 RTC 实现，本质上仍然是系统时基模拟 RTC，不是直接以硬件 RTC 作为时间基准。**

### 8.1 当前 RDX 的 RTC 实现到底是什么

当前生效的是 `SDK/apps/common/third_party_profile/rdx_protocol/rdx_rtc.c` 里 `#if 1` 这一支实现。虽然它已经包含了：

```c
#include "rtc/rtc_dev.h"
```

但真正起作用的时间基准并不是硬件 RTC，而是：

- `rdx_rtc_set_timestamp()` 把业务时间戳保存到 `rtc_params.rtc_value`
- 同时记录一次 `jiffies_msec()` 到 `rtc_params.begin_msec`
- `rdx_rtc_get()` 用 `jiffies_msec2offset(...) / 1000` 给这个时间戳补上运行时经过的秒数
- `rdx_rtc_init()` 再起一个 `sys_timer_add()`，每 `180000` 毫秒，也就是每 `180` 秒，把当前时间快照重新写回 `VM_RDX_RTC_INIT_VALUE`

这说明当前 RDX 的 RTC 逻辑本质是：

- 运行时靠系统 jiffies 累加时间
- 掉电前后只靠 VM 保存一个最近快照
- 不是直接读写芯片的硬件 RTC 计数器

所以你前面的判断是对的：**RDX 之前用的是“系统时基模拟 RTC”思路，不具备关机持续跑时能力。**

### 8.2 为什么它不能在关机期间连续跑时

原因也很直接。

`jiffies_msec()` 只在系统运行期间递增。一旦设备进入 Soft-Off：

- CPU 任务停掉
- 软件定时器停掉
- jiffies 不再累计

这时候即使 RDX 在关机前把某个时间点写进了 `VM_RDX_RTC_INIT_VALUE`，它保存的也只是“关机前最后一次快照”，而不是“关机期间仍在持续递增的硬件时间基准”。

所以重新开机后，RDX 最多只能恢复到上次保存时的时间附近，**不会自动把关机这 `300` 秒、`1800` 秒或者更久的离线时长补回来**。

如果 RDX 需要“关机后也继续跑时”，它就不能继续把 `jiffies + VM 快照` 当成主时间源，而必须改成直接吃底层硬件 RTC。

### 8.3 正确的分层应该怎么切

比较稳妥的做法，是把“底层 RTC 驱动初始化”和“RDX 业务取时间”明确分成两层。

第一层是平台层，也就是本篇前面已经讲完的 RTC 补丁链路，负责：

- `device_config.c` 里的 RTC 平台数据
- `power_config.c` 里的 `low_power_param.rtc_clk`
- `power_soff_callback()` 里的 `poweroff_save_rtc_time()`

这一层负责把**系统硬件 RTC 本身**跑起来，并保证 Soft-Off 期间还能持续计时。

第二层才是 RDX 适配层，也就是 `rdx_rtc.c`，负责：

- 把硬件 RTC 读到的 `struct sys_time` 转成 RDX 需要的 Unix 时间戳
- 把 RDX 下发的 Unix 时间戳转回 `struct sys_time` 后写入硬件 RTC
- 保留字符串格式化、时区显示、日期拆解这些业务侧辅助函数

也就是说，**RDX 不应该再自己维护一套“软件 RTC 时间基准”，而应该变成“硬件 RTC 的业务适配器”。**

### 8.4 RDX 真要切到硬件 RTC，建议怎么改

最核心的是把 `rdx_rtc_get()` 和 `rdx_rtc_set_timestamp()` 的时间源替掉。

建议改法如下。

#### 改 `rdx_rtc_get()`

不要再用：

- `jiffies_msec()`
- `jiffies_msec2offset()`
- `rtc_params.begin_msec`

而是直接：

1. 调 `rtc_read_time(&now)` 读取当前硬件 RTC
2. 把 `struct sys_time` 映射到 RDX 的 `DateTime`
3. 再调用 `rdx_rtc_datetime_to_timestamp()` 转成 Unix 时间戳返回

示意写法：

```c
time_t rdx_rtc_get(void)
{
    struct sys_time now = {0};
    DateTime dt = {0};

    rtc_read_time(&now);

    dt.year = now.year;
    dt.month = now.month;
    dt.day = now.day;
    dt.hour = now.hour;
    dt.minute = now.min;
    dt.second = now.sec;

    return rdx_rtc_datetime_to_timestamp(dt);
}
```

#### 改 `rdx_rtc_set_timestamp()`

不要再把时间戳写进 `rtc_params` 后等 jiffies 补秒，而是直接：

1. 先把 Unix 时间戳转成 `DateTime`
2. 再映射成 `struct sys_time`
3. 调 `rtc_write_time(&set_t)` 写入硬件 RTC

示意写法：

```c
int rdx_rtc_set_timestamp(time_t timestamp)
{
    DateTime dt = rdx_rtc_timestamp_to_datetime(timestamp);
    struct sys_time set_t = {
        .year = dt.year,
        .month = dt.month,
        .day = dt.day,
        .hour = dt.hour,
        .min = dt.minute,
        .sec = dt.second,
    };

    rtc_write_time(&set_t);
    return 0;
}
```

#### 改 `rdx_rtc_init()`

`rdx_rtc_init()` 的职责也要跟着收缩。

它应该只负责：

- 首次启动时判断硬件 RTC 里有没有有效时间
- 如果没有，就用默认值或者历史备份值初始化一次硬件 RTC
- 后续正常运行时直接信任硬件 RTC

它**不应该**再做：

- 起 `sys_timer_add()` 每 `180` 秒落一次“软件 RTC 快照”
- 依赖 `jiffies_msec()` 作为主时间源
- 在 RDX 层重新 `rtc_init()` 一次设备

### 8.5 `VM_RDX_RTC_INIT_VALUE` 改造后还要不要保留

可以保留，但定位要变。

改成硬件 RTC 方案后，`VM_RDX_RTC_INIT_VALUE` 更适合做下面两类用途：

- 首次开机时的默认种子时间
- 某些异常恢复场景下的兜底时间备份

它不应该再承担“RDX 当前实时时间”的主存储职责。

换句话说：

- 主时间源应该是硬件 RTC
- VM 只能是初始化种子或异常兜底

### 8.6 有两件事不要做

#### 不要在 RDX 里再起一套 RTC 设备初始化

`rdx_rtc.c` 里现在被关掉的 `#else` 分支，能看出曾经有人尝试把 RDX 直接接到 RTC 上，但那一版还有两个明显问题：

- 它在 RDX 里自己 `rtc_init()`
- 它用的还是旧字段 `.clk_sel`

而当前仓库的新 RTC 平台数据字段已经是：

- `.rtc_clk`
- `.rtc_sel`
- `.cbfun`

所以那一支代码不能直接作为“已经完成的硬件 RTC 方案”来理解，更不能原样拷回当前补丁体系里。

在当前仓库里，RTC 初始化归平台层统一负责，RDX 只负责调用 `rtc_read_time()` / `rtc_write_time()`。

#### 不要保留“双时间基准”

不要搞成下面这种混搭：

- 平时靠硬件 RTC 读时间
- 某些场景又回退成 `jiffies + VM 快照`

这样最后会变成系统里同时存在两套“当前时间”，调试会非常痛苦。

比较稳妥的原则只有一个：**RDX 的当前时间，只认硬件 RTC 一份真值。**

### 8.7 改完之后怎么验收

改完后，RDX 侧的验收最好分两轮。

第一轮看运行态：

- RDX 读到的时间是否和 `rtc_test.c` 打印的 RTC 时间一致
- RDX 连续读两次时间，是否会正常递增

第二轮看 Soft-Off：

1. 开机把 RTC 和 RDX 时间都设到一个明确值
2. 进入 Soft-Off 持续 `300` 秒
3. 再次开机后，同时看系统 RTC 和 RDX 时间

预期结果应该是：

- 系统 RTC 前进约 `300` 秒
- RDX 看到的时间也前进约 `300` 秒

如果运行态正常、但 Soft-Off 后 RDX 仍然只前进 `0` 到 `1` 秒，优先不要先怪 RDX 层，先回头检查本篇前面那条底层链路是不是全接齐了：

- `device_config.c` 的 `.rtc_clk = RTC_CLK_RES_SEL`
- `power_config.c` 的 `low_power_param.rtc_clk = RTC_CLK_RES_SEL`
- `power_soff_callback()` 里的 `poweroff_save_rtc_time()`

只有底层硬件 RTC 自己已经具备“关机后继续跑时”能力，RDX 接上去之后才可能继承这项能力。

## 9. 一句话总结

这次老 SDK 打新 RTC 补丁，真正要做的是：

1. 把 RTC 公共入口统一切到 `rtc/rtc_dev.h`
2. 把 `device_config.c` 的 RTC 平台数据切到新结构
3. 把 `app_config.h` 的 `RTC_CLK_RES_SEL` 和 `power_config.c` 的 `low_power_param.rtc_clk` 同步起来
4. 把 RTC 相关 VM 定义和底层公共接口对齐
5. 在 `power_soff_callback()` 里接上 `poweroff_save_rtc_time()`
6. 同时替换配套的新版 `cpu.a`

做到这六步，老 SDK 就具备了新 RTC 驱动能力。

之后再按“快照/走时 → 闹钟 → 基础读写与工具函数”的顺序用 `rtc_test.c` 做分轮验收，就能比较清楚地区分：

- 是 RTC 补丁链路根本没接通
- 是闹钟回调上下文用错了
- 还是底层 RTC 库本身还有独立缺陷
