# 支持 QSPI 的非 ESP 平台实现读取 `slave_ready` 寄存器的最小移植

## 0. 这篇文档现在的定位

这篇文档现在不是“照着做一定能成功”的操作手册，而是：

1. `esp-hosted-mcu` 在非 ESP Host 上实现最小寄存器访问的**理论要求**
2. 移植前的**能力判定清单**
3. 本项目在 BR28 平台上的**实际验证结论**

本次 BR28 实测已经证明：

1. Slave 端 SPI HD 初始化正常，`slave_ready = 0xEE`
2. Host 端按普通 `SPI1` 分段拼接事务，读不到 `0xEE`
3. Host 端强写 `SLAVE_CTRL = 1`，Slave 端也没有看到控制寄存器变化
4. LCD 专属 `QSPI / IMD` 思路也不能等价构造官方事务

因此，这篇文档必须同时表达两层意思：

1. 官方最小移植需要什么
2. BR28 当前 SDK 公开接口为什么做不到

---

## 1. 最小目标

最小目标只有一个：

**在非 ESP Host 平台上，通过 SPI HD / QSPI 读取从机寄存器 `0x00`，拿到 `0xEE`。**

如果这一步成立，只能说明：

1. 硬件连线正确
2. Host 事务结构正确
3. Slave 端 SPI HD 已正确初始化

这**不代表**完整 Wi-Fi / Bluetooth 功能已经可用。

---

## 2. 官方事务结构的硬要求

`esp-hosted-mcu` 的最小寄存器访问不是“随便发几个字节”，而是一笔**连续的多阶段半双工事务**：

1. `cmd phase`
   - 8 bit
   - 1 线
2. `addr phase`
   - 8 bit
   - 2 线或 4 线
3. `dummy phase`
   - 8 bit
   - 1 线
4. `data phase`
   - 32 bit
   - 2 线或 4 线
5. `CS`
   - 整笔事务期间连续保持有效，不能在 phase 之间抖动或释放

初始化阶段常规要求：

1. 先走 `2 线`
2. 先读 `SLAVE_READY`
3. 读到 `0xEE` 后，再写 `SLAVE_CTRL = 0x01` 打开 datapath

---

## 3. 与仓库对齐后的关键寄存器

结合当前仓库与本项目适配代码，对齐后的关键寄存器如下：

1. `SLAVE_READY = 0x00`
2. `SLAVE_CTRL = 0x14`
3. `SLAVE_READY` 就绪值 = `0xEE`
4. `DATAPATH_ON` 控制位 = `0x01`

注意：

**本项目当前已确认 `SLAVE_CTRL` 应为 `0x14`，不是 `0x04`。**

`0x04` 对应的是 `MAX_TX_BUF_LEN`，不是控制寄存器。

相关代码可对照：

1. `code/sdk/apps/watch/include/common/esp32_host_qspi.h`
2. `code/sdk/apps/watch/common/esp32_host_qspi.c`

---

## 4. 先做可行性判定

在动手移植前，先确认 Host 控制器是否满足以下最小能力。

### 4.1 必须满足

1. 能在**单笔事务**内表达：
   - `cmd(1线)`
   - `addr(2/4线)`
   - `dummy(1线)`
   - `data(2/4线)`
2. 支持 `dummy bits = 8`
3. 支持半双工方向切换
4. 支持初始化先 `Dual SPI`
5. 支持 `CS` 在整个事务中持续有效

### 4.2 结论判定

1. 全部满足
   - 可以继续做最小移植
2. 只满足“普通收发 + 切总线位宽”
   - 高风险，不建议乐观推进
3. 只提供“发字节 / 收字节 / DMA 收发”
   - 通常不满足官方事务结构
4. 只有 LCD 专属 QSPI / 推屏接口
   - 需要额外判断其是否支持通用 phase 事务，不能默认可用

---

## 5. BR28 本次实际能力判定结果

### 5.1 普通 SPI 驱动

BR28 当前公开出来的普通 SPI 接口主要是：

1. `spi_open`
2. `spi_set_bit_mode`
3. `spi_send_byte`
4. `spi_recv_byte`
5. `spi_dma_send`
6. `spi_dma_recv`

这套接口表达的是：

1. 当前 SPI 控制器工作在 `1bit / 2bit / 4bit`
2. 发一个字节
3. 做一段 DMA 收发

它**没有公开**以下事务级能力：

1. `cmd_bits`
2. `addr_bits`
3. `dummy_bits`
4. `cmd_lines / addr_lines / data_lines` 各 phase 独立配置
5. 硬件级 `cs_keep_active`
6. 单事务连续多 phase 状态机

### 5.2 LCD 专属 QSPI / IMD

BR28 SDK 里还有一条看起来更强的路线：

1. LCD QSPI 宏配置
2. `lcd_write_cmd / lcd_read_cmd`
3. IMD 专属推屏硬件

但这条线的能力模型更偏：

1. 屏驱命令写入
2. 面板寄存器读取
3. 像素流输出
4. 面板专属 `submode`

它依然**没有公开**通用事务级能力：

1. `addr phase` 独立配置
2. `dummy phase` 独立配置
3. `cmd/addr/data` 各自独立线宽
4. 通用 `cs_keep_active`
5. 通用 `SPI HD` 风格 phase 编排

### 5.3 结论

本次 BR28 平台实测结论是：

1. 普通 `SPI1` 路线不满足官方事务结构
2. LCD 专属 `QSPI / IMD` 路线也不等价于官方事务结构
3. 当前 SDK 公开给应用层的能力，无法准确表达 `esp-hosted-mcu` 所需的官方 QSPI / SPI HD 事务

10. `_h_msleep`

## 7. 最小寄存器访问的协议参数

### 7.1 固定参数

1. 读寄存器命令：`RDBUF + 2线掩码 = 0x52`
2. 地址：`0x00`
3. dummy bits：`8`
4. 数据长度：`32 bits`
5. SPI Mode：`0`
6. 建议起步时钟：`1 ~ 5 MHz`

### 7.2 时序要求

必须满足：

`cmd(1线) -> addr(2线) -> dummy(1线) -> data(2线)`

且 `CS` 在整个过程中持续有效。

---

## 8. 本项目已经验证过的失败路径

本项目已经做过以下尝试，结论均为失败。

### 8.1 普通 SPI1 分段拼接事务

做法：

1. 手动拉低 `CS`
2. `cmd` 阶段用 `1bit`
3. `addr` 阶段切到 `2bit / 4bit`
4. `dummy` 阶段切回 `1bit`
5. `data` 阶段再切回 `2bit / 4bit`
6. 使用 `spi_send_byte / spi_dma_recv / spi_dma_send` 拼出完整流程

结果：

1. Host 端读不到 `0xEE`
2. 强写 `SLAVE_CTRL = 1` 后，Slave 端控制寄存器也不变化

### 8.2 事务中手动切换线宽

做法：

1. 在一笔逻辑事务中多次调用 `spi_set_bit_mode`
2. 尝试模拟“同一事务内部 phase 线宽不同”

结果：

1. `spi_set_bit_mode` 只是切整个 SPI 外设模式
2. 不能等价替代官方事务描述中的 phase 级配置

### 8.3 借 LCD 专属 QSPI / IMD 能力

做法：

1. 分析 LCD 层 `QSPI` 宏
2. 分析 `lcd_write_cmd / lcd_read_cmd`
3. 评估 IMD 硬件能否借用来表达官方事务

结果：

1. 这套能力更偏面板命令 / 推屏
2. 不具备通用 `SPI HD` 所需的公开 phase 描述能力
3. 实测也不通

---

## 9. 本项目当前的实际验收结果

当前实际日志已经证明：

1. Slave 端正常启动
2. Slave 端日志显示 `slave_ready = 0x000000EE`
3. Slave 端等待 `Host` 打开 datapath
4. Host 端无论读 `SLAVE_READY` 还是强写 `SLAVE_CTRL`
5. Slave 端都没有看到有效命中

因此可以明确判断：

**失败原因不是从机没起来，而是 Host 事务结构没有被 Slave 识别为合法官方事务。**

---

## 10. 什么时候该停止继续硬拼

如果平台只能做到：

1. 发字节
2. 收字节
3. DMA 收发
4. 运行时切整个总线线宽

但做不到：

1. 单事务 phase 编排
2. phase 级线宽控制
3. 硬件级持续 `CS`
4. 独立 `dummy phase`

那么就不建议继续在该公开接口上硬拼 `esp-hosted` SPI HD。

这时更合理的选择是：

1. 继续下探更底层私有寄存器或闭源实现
2. 换 `SPI Full Duplex / SDIO / UART`
3. 换能表达完整 QSPI / OSPI phase 事务的 Host 平台

---

## 11. 一句话总结

非 ESP 平台最小移植的关键不是“能不能发 SPI”，而是：

**能不能严格生成 `esp-hosted-mcu` 所要求的那笔连续多阶段 QSPI / SPI HD 事务。**

BR28 当前 SDK 公开出来的普通 SPI、LCD QSPI、IMD 三条路，本项目已经实际验证过，均**不能等价构造官方事务结构**。
