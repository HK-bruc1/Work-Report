# BR28 SDK中SPI/QSPI/IMD能力边界与ESP-Hosted事务差异结论

## 1. 结论先行

当前 BR28 SDK 公开给应用层的三类能力：

1. 普通 `SPIx` 驱动接口
2. LCD 场景下的 `QSPI` 配置宏与屏驱接口
3. LCD 专属 `IMD` 推屏硬件接口

都**不能直接构造** `esp-hosted-mcu` 官方要求的那种 `SPI HD / QSPI` 主机事务结构。

已经实际验证过的方式包括：

1. 使用 `SPI1` 手动拼接各阶段事务
2. 在事务中手动切换 `1bit/2bit/4bit` 线宽
3. 在 `CS` 保持有效期间分段发送 `cmd/addr/dummy/data`
4. 尝试借助 LCD 侧的 QSPI / IMD 思路理解底层能力

结果均失败。表现为：

1. Slave 端已经正常初始化，`slave_ready = 0xEE`
2. Host 端始终读不到 `0xEE`
3. Host 强行写 `SLAVE_CTRL = 1` 后，Slave 端 `slave_ctrl` 仍保持 `0x00000000`

这说明失败原因不是某个寄存器值写错，而是 **Host 侧事务语义与官方要求不等价**。

---

## 2. ESP-Hosted 官方要求的事务结构

`esp-hosted-mcu` 的 `SPI HD / QSPI` 事务不是“连续发几个字节”这么简单，而是**一笔连续的多阶段半双工事务**。

典型寄存器访问结构为：

1. `cmd phase`
   - 1 线
   - 8 bit
2. `addr phase`
   - 2 线或 4 线
   - 8 bit
3. `dummy phase`
   - 1 线
   - 8 bit
4. `data phase`
   - 2 线或 4 线
   - N bit
5. `CS`
   - 整个事务期间必须连续保持有效

并且 Slave 是按这套 phase 结构去识别：

1. `RDBUF`
2. `WRBUF`
3. `RDDMA`
4. `WRDMA`
5. `CMD8`
6. `CMD9`

也就是说，**重点不是 opcode 对不对，而是事务 phase 结构必须对**。

## 3. BR28 普通 SPI 驱动当前提供的能力

相关源码：

1. `code/sdk/include_lib/driver/cpu/br28/asm/spi.h`
2. `code/sdk/cpu/br28/spi.c`

### 3.1 已公开接口

普通 `SPI` 驱动当前主要提供：

1. `spi_open()`
2. `spi_close()`
3. `spi_set_bit_mode()`
4. `spi_send_byte()`
5. `spi_recv_byte()`
6. `spi_dma_send()`
7. `spi_dma_recv()`
8. `spi_set_baud()`

支持的模式主要是：

1. `SPI_MODE_UNIDIR_1BIT`
2. `SPI_MODE_UNIDIR_2BIT`
3. `SPI_MODE_UNIDIR_4BIT`
4. `SPI_MODE_BIDIR_1BIT`

### 3.2 这套接口真正表达的能力

这套接口表达的是：

1. 打开一个 SPI 控制器
2. 选择当前总线是 `1bit / 2bit / 4bit`
3. 发送一个字节
4. 做一段 DMA 收发

也就是说，它表达的是**“当前总线工作模式”**，不是**“单事务内部各 phase 的工作模式”**。

### 3.3 缺失的关键能力

当前普通 `SPI` 驱动没有公开以下能力：

1. 单事务级别的 `cmd_bits`
2. 单事务级别的 `addr_bits`
3. 单事务级别的 `dummy_bits`
4. `cmd phase / addr phase / data phase` 分别独立配置线宽
5. 硬件级 `cs_keep_active`
6. 由硬件状态机连续执行一笔多 phase 事务

### 3.4 为什么“手动拼接 + 手动切线宽”不行

我们当前做法本质上是：

1. `spi_send_byte()` 发 `cmd`
2. `spi_set_bit_mode()` 切到 `2bit/4bit`
3. 再发 `addr`
4. 再切回 `1bit`
5. 再发 `dummy`
6. 再切到 `2bit/4bit`
7. 最后 `spi_dma_recv()/spi_dma_send()` 处理数据

这在逻辑上“像”官方事务，但在控制器语义上不是一回事。

问题在于：

1. `spi_set_bit_mode()` 是**切整个 SPI 外设模式**
2. 不是在**同一笔事务内部切 phase 线宽**
3. `spi_send_byte()` 只是普通发字节
4. `dummy` 只是“又发了 8 bit”
5. 不是硬件事务描述中的独立 `dummy phase`

因此最终虽然 `CS` 可以手动维持不释放，但 Slave 并不会把它稳定识别为官方要求的合法 `SPI HD` 事务。

## 4. SDK 中 LCD 使用的 QSPI 到底是什么

相关源码：

1. `code/sdk/apps/watch/include/ui/lcd_spi/lcd_drive.h`
2. `code/sdk/cpu/br28/ui_driver/lcd_drive/lcd_spi/lcd_qspi_SD3302_466x466.c`
3. `code/sdk/cpu/br28/ui_driver/lcd_drive/lcd_drive.c`

### 4.1 LCD 侧“QSPI”提供了什么

LCD 驱动里的 `QSPI` 主要通过这些宏配置：

1. `SPI_IF_MODE(config)`
2. `OUT_FORMAT(config)`
3. `PIXEL_TYPE(config)`
4. `QSPI_RGB565_SUBMODE1_1T2B`
5. `QSPI_RGB565_SUBMODE2_1T2B`
6. `QSPI_RGB565_SD3302_1T2B`
7. `QSPI_RGB565_ST77903_1T2B`

这些宏表达的是：

1. 屏接口类型
2. 像素输出格式
3. 像素打包方式
4. 某些面板专属 `submode`

不是通用 `SPI HD` 事务描述。

### 4.2 LCD 驱动对外暴露的接口模型

LCD 驱动最终调用的是：

1. `lcd_write_cmd(cmd, buf, len)`
2. `lcd_read_cmd(cmd, buf, len)`

也就是“写一个面板命令 + 可选参数”“读一个面板命令 + 返回数据”。

这套模型更适合：

1. LCD 初始化命令
2. `CASET/PASET/RAMWR/RAMWRC`
3. 面板寄存器读取
4. 像素流输出

不适合表达：

1. `cmd phase`
2. `addr phase`
3. `dummy phase`
4. `data phase`
5. 每个 phase 独立线宽
6. `cs_keep_active`

### 4.3 结论

LCD 侧虽然名字叫 `QSPI`，但它本质是：

**“面板协议型 QSPI”**

而 `esp-hosted` 要的是：

**“通用 SPI HD 多阶段事务型 QSPI”**

两者不是同一层能力。

## 5. IMD 硬件当前公开出来的能力

相关头文件：

1. `code/sdk/include_lib/system/ui_new/ui/cpu/br28/asm/imd.h`

### 5.1 IMD 是什么

`imd.h` 中已经明确说明：

1. IMD 是专门用于推屏的独立硬件模块
2. 与普通硬件 `SPI` 无关

也就是说，LCD 侧很多 `QSPI` 场景其实走的是 `IMD`，不是 `spi.c` 这一套普通 SPI 驱动。

### 5.2 IMD 已公开接口

IMD 当前公开了以下与 LCD SPI/QSPI 相关的接口：

1. `lcd_init(struct imd_param *param)`
2. `lcd_write_cmd(u32 cmd, u8 *buf, u8 len)`
3. `lcd_read_cmd(u32 cmd, u8 *buf, u8 len)`
4. `lcd_draw(...)`
5. `lcd_draw_rect(...)`
6. `lcd_set_align(...)`
7. `lcd_set_param(mode, priv)`
8. `imd_custom_clk_set(cmd_clock, dat_clock, dat_clock_default)`
9. `imd_spi_io_enable()`
10. `imd_spi_io_disable()`

并且 `imd_param.spi` 中还公开了：

1. `write_cmd`
2. `read_cmd`
3. `submode0_cmd`
4. `submode1_cmd`
5. `submode2_cmd`
6. `caset_cmd`
7. `paset_cmd`
8. `ramwr_cmd`
9. `ramwrc_cmd`

### 5.3 IMD 比普通 SPI 强在哪里

IMD 至少多提供了这些能力：

1. 面板命令读写接口
2. 命令和数据可分离处理
3. 命令位宽模式，如 `CMD_8BIT / CMD_16BIT / CMD_24BIT`
4. QSPI 子模式命令码配置
5. 命令时钟与数据时钟可分开设置

### 5.4 IMD 依然缺失什么

从公开头文件看，IMD 仍然没有向应用层公开这些关键能力：

1. 通用 `addr_bits`
2. 通用 `dummy_bits`
3. `cmd_lines / addr_lines / data_lines` 的独立显式配置
4. `CS` 在单事务内保持的通用控制
5. 一笔事务中 `cmd->addr->dummy->data` 的通用 phase 编排能力

因此，IMD 虽然比普通 `SPI1` 更强，但它暴露出来的仍是：

**“LCD 面板命令/推屏接口”**

而不是：

**“可构造任意 SPI HD 多阶段事务的通用主机接口”**

---

## 6. 为什么 IMD 也不能满足 ESP-Hosted

IMD 当前暴露给应用层的核心模型是：

1. `lcd_write_cmd(cmd, buf, len)`
2. `lcd_read_cmd(cmd, buf, len)`

这种模型默认的思路是：

1. 发送一个面板命令
2. 后面跟若干参数
3. 或者读这个命令的返回值

但 `esp-hosted` 的要求不是“命令+参数”，而是：

1. `cmd phase`
2. `addr phase`
3. `dummy phase`
4. `data phase`

并且 `addr phase` 和 `data phase` 的线宽还可能与 `cmd phase` 不同。

IMD 公开接口里没有把这些 phase 拆开，因此即使它内部可能做了某些 LCD 场景专用处理，也不能等价推出：

“它就能稳定构造出 `esp-hosted` 所需的通用 QSPI/SPI HD 事务”

此前已经实际试过 IMD 路径，结果也验证了这一点：**不行**。

---

## 7. 已验证过的失败路径

### 7.1 普通 SPI1 分段拼接事务

已验证内容：

1. 手动拉低 `CS`
2. `cmd` 阶段先走 `1bit`
3. `addr` 阶段切到 `2bit/4bit`
4. `dummy` 阶段切回 `1bit`
5. `data` 阶段再切到 `2bit/4bit`
6. 用 `spi_send_byte()/spi_dma_recv()/spi_dma_send()` 拼完整流程

结果：

1. 不能稳定读到 `slave_ready = 0xEE`
2. 强写 `SLAVE_CTRL = 1` Slave 端也未响应

### 7.2 事务中手动切换线宽

已验证内容：

1. 在一笔逻辑事务中多次调用 `spi_set_bit_mode()`
2. 切换 `1bit / 2bit / 4bit`

结果：

1. 只能切整个外设模式
2. 不能等价替代官方要求的“单事务内部 phase 线宽配置”
3. Slave 不识别为合法 `SPI HD` 事务

### 7.3 IMD / LCD QSPI 路径

已验证内容：

1. 分析 LCD 的 `QSPI` 宏配置
2. 分析 `lcd_write_cmd()/lcd_read_cmd()` 接口模型
3. 实际尝试借用 IMD/QSPI 思路

结果：

1. LCD/IMD 的能力模型偏面板协议
2. 不具备官方事务结构所需的通用 phase 描述能力
3. 实测也无法打通 `0x00 -> 0xEE`

---

## 8. 最终结论

当前 BR28 SDK 中公开给应用层的相关能力可以分成两类：

1. 普通 `SPIx` 控制器接口
2. LCD 专属 `QSPI/IMD` 接口

它们都不等价于 `esp-hosted-mcu` 官方要求的 Host 侧事务能力。

失败的根本原因不是：

1. 某个寄存器值写错
2. 某个 GPIO 没接好
3. 某个 opcode 不对

而是：

**现有 SDK 公开出来的接口模型，本身就无法准确表达官方要求的 QSPI/SPI HD 多阶段事务结构。**

因此当前可以确认：

1. 普通 `SPI1` 分段拼接方案失败
2. 手动切换线宽方案失败
3. 借 LCD 专属 `QSPI/IMD` 能力构造官方事务的方案失败

---

## 9. 当前状态建议

基于目前验证结果，后续只有两条真正有意义的方向：

1. 继续向下挖 BR28 更底层的私有寄存器/闭源实现，确认是否存在未公开的通用 phase 事务能力
2. 接受当前 SDK 能力边界，认为 BR28 现有公开接口无法完成 `esp-hosted-mcu` 官方 QSPI Host 事务移植

在没有新的底层能力证据之前，不建议继续在现有公开 `SPI/QSPI/IMD` 接口上反复尝试“拼事务”。
