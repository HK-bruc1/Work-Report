# ESP32-P4 外接 ESP32-C2 作为 Slave 的切换与引脚映射手册

## 1. 文档目的

这份文档专门解决一个容易踩坑的问题：

ESP32-P4-Function-EV-Board 默认是围绕板载 ESP32-C6 或官方配套 EV Board 来设计的，而你现在要把 slave 换成自己手里的 ESP32-C2 开发板，所以 host 侧不能机械沿用默认板型映射，必须根据你的接线方式切换到正确的 GPIO 映射方案。

这份文档重点回答 4 件事：

1. 什么时候可以沿用板级 preset，什么时候必须改成手工映射。
2. host 和 slave 两边在 menuconfig 里到底要开什么。
3. ESP32-P4 外接 ESP32-C2 时推荐用哪套引脚。
4. 第一阶段如何先验证链路，再切到 TCP/UDP 吞吐测试。

---

## 2. 关键结论

1. ESP32-P4 作为 host、ESP32-C2 作为 slave 是支持的。
2. ESP32-C2 不支持 SDIO，所以不要走 P4 板载 C6 默认的 SDIO 方案。
3. 这套组合推荐使用 SPI Full-Duplex。
4. 如果你是用 ESP32-P4-Function-EV-Board 的 J1 排针外接一块独立 ESP32-C2 开发板，host 侧推荐使用“外接其他 MCU”的引脚映射，不要直接沿用板载 C6 的默认板级配置。
5. 如果你在 host 的 menuconfig 里选了开发板 preset，那么 Kconfig 会把部分 P4 GPIO 约束到板级 preset 的固定值；这种情况下你可能根本改不到你跳线外接 C2 所需要的那套引脚。
6. 对于 J1 外接独立 C2 的场景，host 侧应优先选择 `No development board`，然后手工填写 SPI-FD 的 GPIO。

---

## 3.1 Host 工程到底在哪里

这是这类仓库最容易误解的地方。

### `host/` 目录不是一个可直接编译的 ESP-IDF 应用工程

仓库根目录下的 `host/` 只是 ESP-Hosted 的 host 侧组件代码和头文件目录，不是一个可以直接执行 `idf.py build` 的独立 app。

原因可以直接从仓库结构看出来：

1. `host/` 目录下没有 app 级别的 `CMakeLists.txt`、`main/`、`project(...)` 这些典型 ESP-IDF 工程入口。
2. 仓库顶层 `CMakeLists.txt` 是通过 `idf_component_register(...)` 把 `host/` 下面的源码注册成组件来参与构建的。

这就是为什么 VS Code 不会把 `host/` 识别成一个独立 IDF 工程。

### 仓库里真正能直接打开并编译的 host 工程在 `examples/host_*`

你应该把下面这些目录理解成“真正的 host 应用工程”：

1. `examples/host_peer_data_transfer`
2. `examples/host_transport_config`
3. `examples/host_network_split__power_save`
4. 其他 `examples/host_*` 目录

这些目录才有标准 ESP-IDF app 的工程骨架，例如：

1. `CMakeLists.txt`
2. `main/`
3. `sdkconfig.defaults`
4. `project(...)`

### 对你当前场景，host 工程应该怎么选

对“P4 外接独立 C2”这条路径，推荐这样分阶段：

1. 第一阶段 transport 验线：使用 `examples/host_peer_data_transfer`
2. 第二阶段编程式 transport 参考：可看 `examples/host_transport_config`
3. 第三阶段 TCP/UDP 吞吐测试：不要去 `host/` 目录，也不要继续沿用 `host_peer_data_transfer`，而是使用 ESP-IDF 自带的 `examples/wifi/iperf`

### 为什么吞吐测试的 host 工程不在本仓库里

因为仓库官方文档给出的 host 吞吐测试方案，本来就是：

1. 选用 ESP-IDF 自带的 Wi-Fi 示例工程
2. 给这个工程添加 `esp_hosted` 和 `esp_wifi_remote`
3. 让它以 Hosted 模式跑在 P4 上

所以吞吐测试的 host 工程通常在：

`$IDF_PATH/examples/wifi/iperf`

这也解释了为什么你在当前仓库工作区里看不到一个现成的 “host iperf app”。

### 如果你想让 VS Code 识别吞吐测试工程

有两个实际可行的办法：

1. 直接把 `$IDF_PATH/examples/wifi/iperf` 这个目录单独打开，或加入当前 VS Code 工作区。
2. 把 `$IDF_PATH/examples/wifi/iperf` 复制一份到你自己的工作目录，再在这份副本上添加 Hosted 依赖并做配置。

对当前这套 P4 + C2 bring-up，我建议你先不要着急打开 iperf 工程。先把 `examples/host_peer_data_transfer` 跑通，等 transport 稳定后，再单独处理 iperf 工程。

## 4. 先分清你属于哪一种接法

## 4.1 场景 A：P4 Function EV Board 通过 J1 跳线外接独立 C2 开发板

这是你当前最可能的场景，也是本文重点。

这种情况下：

1. host 侧不要沿用板载 C6 的 SDIO 路线。
2. host 侧不要默认相信开发板 preset 给你的 GPIO。
3. 应该在 P4 host 上切换到 SPI Full-Duplex。
4. 应该在 host 上选择 `No development board`，然后手动填写你接线对应的 P4 GPIO。

## 4.2 场景 B：你使用的是官方 EV Board 配套连接器或与板级 preset 完全一致的连接方式

这种情况下，可以考虑直接用板级 preset。

对于 `ESP32-P4-Function-EV-Board with on-board C6 or C5/C2 on EV board` 这一类 preset，Kconfig 对 P4 + C2 约束出来的 host 侧 SPI-FD 引脚是：

1. MOSI = 23
2. MISO = 22
3. CLK = 33
4. CS = 4
5. Handshake = 21
6. Data Ready = 32
7. Reset = 53

这套值只适合你真的在用对应板级走线或对应 EV Board 连接方式时使用。

如果你是 J1 跳线外接独立 C2 开发板，不要直接套这组值。

---

## 5. 推荐接线方案：P4 J1 外接独立 C2

## 5.1 为什么推荐这套方案

`docs/esp32_p4_function_ev_board.md` 第 8 节已经给出 P4 在“外接其他 MCU”时的推荐 GPIO。它的目的就是避开板载 C6 的默认路径，改成适合外接设备的 SPI-FD 映射。

这套方案最适合：

1. 你手里是 ESP32-P4-Function-EV-Board。
2. 你手里另有一块独立 ESP32-C2 开发板。
3. 你打算用跳线把两块板连接起来。

## 5.2 Host 侧推荐引脚

P4 侧推荐如下：

| 功能 | P4 GPIO |
|---|---:|
| MOSI | 4 |
| MISO | 5 |
| CLK | 26 |
| CS | 6 |
| Handshake | 20 |
| Data Ready | 32 |
| Reset | 2 |

补充说明：

1. 官方文档提醒优先使用 36 以下 GPIO，以避免 LDO 供电域问题。
2. 官方文档还提醒不要碰 35 和 36，因为它们会影响 boot mode。

## 5.3 C2 侧推荐引脚

C2 侧采用 `docs/spi_full_duplex.md` 给出的默认 co-processor 映射：

| 功能 | C2 GPIO |
|---|---:|
| CLK | 6 |
| MOSI | 7 |
| MISO | 2 |
| CS | 10 |
| Handshake | 3 |
| Data Ready | 4 |
| Reset In | EN/RST |

## 5.4 实际接线表

| P4 侧 | C2 侧 | 说明 |
|---|---|---|
| GPIO26 | GPIO6 | SPI CLK |
| GPIO4 | GPIO7 | MOSI |
| GPIO5 | GPIO2 | MISO |
| GPIO6 | GPIO10 | CS |
| GPIO20 | GPIO3 | Handshake |
| GPIO32 | GPIO4 | Data Ready |
| GPIO2 | EN/RST | Reset |
| GND | GND | 共地 |

建议：

1. 至少接 1 根地线。
2. 最好额外再补 1 根地线。
3. 第一轮验证时跳线尽量短。

---

## 6. Host 侧 menuconfig 该怎么切

## 6.1 最重要的一个开关：开发板 preset 选择

在 P4 host 侧 menuconfig 中，找到：

`Configure GPIOs for Development Board`

如果你的接法是“J1 跳线外接独立 C2 开发板”，这里应该选：

`No development board`

原因很直接：

1. `Kconfig` 明确说明，只有选 `No development board`，你才能自由修改 transport GPIO。
2. 如果你选了 `ESP32-P4-Function-EV-Board with on-board C6 or C5/C2 on EV board`，Kconfig 会把 host 侧多个 SPI GPIO 约束到板级 preset 范围内。
3. 这样你就没法手工改成 P4 J1 外接 C2 所需的那一套引脚。

如果你在 menuconfig 菜单里一时找不到这一项，可以直接使用菜单搜索，搜索关键字：

`Configure GPIOs for Development Board`

## 7. Slave 侧必须开启的配置

进入 slave 工程：

```bash
cd slave
idf.py set-target esp32c2
idf.py menuconfig
```

### 7.1 必选项

路径：

```text
Example Configuration
└── Bus Config in between Host and Co-processor
    └── Transport layer
        └── SPI Full-Duplex
```

这一步必须这样选，不能选 SDIO。

### 7.2 必改 GPIO

在 slave 侧 SPI-FD 配置里，把 C2 侧 GPIO 改成和实际接线一致：

1. CLK = 6
2. MOSI = 7
3. MISO = 2
4. CS = 10
5. Handshake = 3
6. Data Ready = 4
7. Reset 连接 EN/RST

### 7.3 第一阶段验线时建议额外开启的配置

如果你准备先跑 `host_peer_data_transfer`，slave 侧还应打开：

```text
Example Configuration
└── Additional higher layer examples to run
    └── Select Examples to run
        └── [*] Peer Data Transfer Example
```

### 7.4 第一阶段不要开的项

1. 不要开 Network Split。
2. 不要开 Host Power Save。
3. 不要混入其他高层 feature。
4. 不要在第一轮 bring-up 就切到 SDIO、UART 或 SPI-HD。

---

## 8. Host 侧第一阶段：先跑 peer_data_transfer 验线

第一阶段推荐使用 `examples/host_peer_data_transfer` 作为 P4 host 工程。

进入工程：

```bash
cd examples/host_peer_data_transfer
idf.py set-target esp32p4
idf.py menuconfig
```

### 8.1 必选项

在 host 侧 menuconfig 中，至少确认这些项：

```text
Component config
├── Wi-Fi Remote
│   └── Choose slave target
│       └── esp32c2
└── ESP-Hosted config
    ├── Configure GPIOs for Development Board
    │   └── No development board
    ├── Transport layer
    │   └── SPI Full-duplex
    └── SPI Configuration
```

### 8.2 SPI Configuration 里必须改的值

如果你按本文的 J1 外接方案接线，host 侧应设置：

1. Host MOSI = 4
2. Host MISO = 5
3. Host CLK = 26
4. Host CS = 6
5. Handshake = 20
6. Data Ready = 32
7. Reset slave = 2

### 8.3 建议项

1. SPI Checksum 建议启用。
2. 第一轮 SPI 时钟先保守一点，建议先从 10 MHz 或 20 MHz 开始。
3. peer_data_transfer 稳定通过后，再逐步提升时钟。

### 8.4 第一阶段不要做的事

1. 不要一上来就追 40 MHz。
2. 不要把 host 的 GPIO 改成另一套而 slave 仍然保留默认值。
3. 不要在 peer_data_transfer 还没稳定前就切到 Wi-Fi iperf。

---

## 9. Host 侧第二阶段：切到 ESP-IDF wifi/iperf 做 TCP/UDP 测试

这一步不要继续用 `host_peer_data_transfer`。应切到 ESP-IDF 自带的 `examples/wifi/iperf`，再按 Hosted 模式接入 `esp_hosted` 和 `esp_wifi_remote`。

### 9.1 Host menuconfig 必选项

P4 host 侧仍然保持：

```text
Component config
├── Wi-Fi Remote
│   └── Choose slave target
│       └── esp32c2
└── ESP-Hosted config
    ├── Configure GPIOs for Development Board
    │   └── No development board
    ├── Transport layer
    │   └── SPI Full-duplex
    └── SPI Configuration
```

也就是说，第二阶段只是 host 工程换了，不是 transport 思路换了。

### 9.2 Host 第二阶段的 GPIO 仍然保持一致

host 侧仍然用同一套引脚：

1. MOSI = 4
2. MISO = 5
3. CLK = 26
4. CS = 6
5. Handshake = 20
6. Data Ready = 32
7. Reset = 2

不要在第一阶段和第二阶段切换两套不同的 host GPIO。

### 9.3 第二阶段性能配置到底加在哪里

这组性能参数是加在第二阶段 host 工程里，不是加在当前仓库根目录，也不是加在 `host/` 或 `slave/` 目录。

更准确地说，它应该加在你第二阶段实际编译的那个 `wifi/iperf` 工程根目录下。

如果你是直接使用 ESP-IDF 自带的 `examples/wifi/iperf`，那么应先进入这个工程，再设置 target：

```bash
cd %IDF_PATH%\examples\wifi\iperf
idf.py set-target esp32p4
```

然后把性能项放到这个工程根目录下的默认配置文件里。推荐顺序如下：

1. 优先写入 `sdkconfig.defaults.esp32p4`。
2. 如果该工程没有 `sdkconfig.defaults.esp32p4`，就写入工程根目录的 `sdkconfig.defaults`。
3. 如果你已经生成过 `sdkconfig`，改完 defaults 后最好执行一次 `idf.py fullclean` 再重新 build，避免旧配置残留。

也就是说，这些参数的归属是“第二阶段 host app 自己的默认配置文件”，而不是本仓库里第一阶段示例工程的配置文件。

### 9.4 第二阶段建议加入的性能配置

在第二阶段的 `wifi/iperf` host 工程里，建议加入 P4 + C2 的官方优化配置：

```text
CONFIG_SLAVE_IDF_TARGET_ESP32C2=y
CONFIG_ESP_HOSTED_CP_TARGET_ESP32C2=y

CONFIG_WIFI_RMT_STATIC_RX_BUFFER_NUM=10
CONFIG_WIFI_RMT_DYNAMIC_RX_BUFFER_NUM=32
CONFIG_WIFI_RMT_DYNAMIC_TX_BUFFER_NUM=32
CONFIG_WIFI_RMT_AMPDU_TX_ENABLED=y
CONFIG_WIFI_RMT_TX_BA_WIN=6
CONFIG_WIFI_RMT_AMPDU_RX_ENABLED=y
CONFIG_WIFI_RMT_RX_BA_WIN=6

CONFIG_LWIP_TCP_SND_BUF_DEFAULT=16384
CONFIG_LWIP_TCP_WND_DEFAULT=32768
CONFIG_LWIP_TCP_RECVMBOX_SIZE=20
CONFIG_LWIP_UDP_RECVMBOX_SIZE=20
CONFIG_LWIP_TCPIP_RECVMBOX_SIZE=20

CONFIG_LWIP_TCP_SACK_OUT=y
```

这是第二阶段吞吐测试更接近官方数据的关键前提。

如果你更习惯走 menuconfig，也可以手工改，但建议把最终值落回 defaults 文件，这样后面重新配置或迁移工程时不会丢。

### 9.5 第二阶段如何触发测试

第二阶段的 iperf 测试不是烧录后自动跑流量，而是：

1. P4 host 侧先启动 `wifi/iperf` 工程并进入串口控制台。
2. 在 P4 控制台把 Hosted Wi-Fi 连上你的 AP。
3. 再分别在“P4 控制台”和“外部 PC”两边手工启动 iperf client/server。

推荐按下面顺序做：

#### 第一步：编译、烧录、监视第二阶段 host 工程

```bash
cd %IDF_PATH%\examples\wifi\iperf
idf.py set-target esp32p4
idf.py build
idf.py -p COMx flash monitor
```

如果你不是直接在原始 ESP-IDF 示例目录上改，而是复制了一份工程，那么就在你的那份 `wifi/iperf` 副本目录执行这些命令。

#### 第二步：在串口控制台连 Wi-Fi

烧录启动后，进入 `iperf>` 提示符，先执行：

```text
sta_connect <SSID> <password>
sta_ip
```

这里的目的有两个：

1. 让 P4 作为 Hosted host 连上路由器。
2. 记下 P4 这边拿到的 IP，后面做 Host RX 测试时要用。

补充一点：外部 PC 也必须在同一个局域网里。

#### 第三步：在外部 PC 准备 iperf

外部 PC 建议使用 iperf 2.x，不要直接拿 iperf 3 的命令来套，因为这个示例兼容的是 iperf 2.x 语法。

#### 第四步：按方向触发 TCP/UDP 测试

下面这张表里，`P4 控制台命令` 是你在串口 `iperf>` 提示符里输入的，`PC 命令` 是你在电脑终端里输入的。

| 测试方向 | P4 控制台命令 | PC 命令 |
|---|---|---|
| TCP Host TX | `iperf -c <PC_IP> -t 60 -i 3` | `iperf -s -i 3` |
| TCP Host RX | `iperf -s -i 3` | `iperf -c <P4_IP> -t 60 -i 3` |
| UDP Host TX | `iperf -u -c <PC_IP> -t 60 -i 3` | `iperf -u -s -i 3` |
| UDP Host RX | `iperf -u -s -i 3` | `iperf -u -c <P4_IP> -t 60 -i 3` |

其中：

1. `Host TX` 表示流量从 P4 发到 PC。
2. `Host RX` 表示流量从 PC 发到 P4。
3. `P4_IP` 是你刚才执行 `sta_ip` 查到的地址。
4. `PC_IP` 是你电脑在同一局域网里的地址。

如果你只想先确认第二阶段链路和 Wi-Fi 路径是通的，建议先跑一条最简单的 TCP Host TX：

1. PC 先执行：`iperf -s -i 3`
2. P4 控制台再执行：`iperf -c <PC_IP> -t 60 -i 3`

只要这一条能稳定出带宽数字，再继续做 UDP、反向 RX、以及更高 SPI 时钟下的吞吐对比。

## 10. 最简操作顺序

如果你的目标是先把 P4 从默认板载 C6 路线切到外接 C2，推荐严格按下面顺序来：

1. 确认自己是 J1 跳线外接独立 C2，而不是官方 EV Board 板级连接。
2. 在 P4 host 侧选择 `No development board`。
3. 在 host 和 slave 两边都切到 SPI Full-Duplex。
4. 在两边都填入与跳线一致的 GPIO。
5. slave 侧打开 `Peer Data Transfer Example`。
6. 先跑 `examples/host_peer_data_transfer` 做 transport 验线。
7. transport 稳定后，再切到 ESP-IDF `wifi/iperf` 做 TCP/UDP 测试。

## 11. 你最容易踩的坑

### 11.1 坑一：以为 P4 默认板型可以直接换成外接 C2

不一定可以。

如果你是 J1 跳线外接独立 C2，而 host 侧还保留开发板 preset，那么很多 GPIO 范围会被 Kconfig 固定住，你就改不到你想要的外接映射。

### 11.2 坑二：host 改了 GPIO，slave 还用另一套默认值

SPI-FD 这套链路需要 host 和 slave 双方完全一致。只改一边没有意义。

### 11.3 坑三：第一轮就做 Wi-Fi 吞吐测试

错误顺序会把 transport 问题和网络问题混在一起。正确顺序是先 peer_data_transfer，再 iperf。

### 11.4 坑四：拿 Network Split 例程做 C2 方案

C2 不在 Network Split 支持列表里。不要把那套例程当成 C2 的端到端方案。

---

## 12. 一句话结论

如果你是把 ESP32-P4-Function-EV-Board 上默认面向板载 C6 的 host 改成外接独立 ESP32-C2，那么最关键的一步不是“换 slave target”本身，而是：

在 host menuconfig 里把开发板 GPIO 方案切到 `No development board`，然后用外接 MCU 的 SPI-FD 映射手工改 GPIO。

对 J1 外接独立 C2 的推荐值就是：

1. P4: 4 / 5 / 26 / 6 / 20 / 32 / 2
2. C2: 7 / 2 / 6 / 10 / 3 / 4 / EN-RST

只有这一步理顺了，后面的 peer_data_transfer、Wi-Fi 连通和 TCP/UDP 测试才有意义。
