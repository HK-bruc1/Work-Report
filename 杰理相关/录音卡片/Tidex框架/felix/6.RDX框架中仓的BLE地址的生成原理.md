# JL 平台经典蓝牙与 BLE 的名称/地址关系及调用链

本文分两层说明：先讲 JL 平台原生，再讲 RDX 框架在原生之上的覆盖。

```
┌────────────────────────────────────────────────────┐
│  RDX 框架层                                         │
│  rdx_app_config.h / rdx_ble_server.c / rdx_vm.c    │
│  → 覆盖平台默认行为，提供独立的 BLE 名称与地址管理      │
├────────────────────────────────────────────────────┤
│  JL 平台原生层                                      │
│  earphone.c / user_cfg.c / bt_ble.c / le_user.h    │

│  → 地址：SAME_ADDR 是唯一交汇点                       │
└────────────────────────────────────────────────────┘
```

## 1. 地址关系

### 1.1 结论：只有一个交汇点

`earphone.c:450-461` —— BT 和 BLE 地址**仅在此处**发生一次关系：

```c
#if TCFG_USER_BLE_ENABLE
    u8 tmp_ble_addr[6];
#if TCFG_BT_BLE_BREDR_SAME_ADDR
    memcpy(tmp_ble_addr, (void *)bt_get_mac_addr(), 6);        // =1: 直接复制
#else
    bt_make_ble_address(tmp_ble_addr, (void *)bt_get_mac_addr());  // =0: 派生
#endif
    le_controller_set_mac((void *)tmp_ble_addr);
#endif
```

| `SAME_ADDR` | 行为 | BLE 地址 |
|-------------|------|---------|
| `1` | `memcpy` 直接复制 | BLE = BT MAC（完全相同） |
| `0` | `bt_make_ble_address()` 派生 | BLE ≠ BT MAC，但确定性地从 BT MAC 算出 |

`bt_make_ble_address()` 实现在预编译库中，源码不可见，但行为是确定性的——同一个 BT MAC 总是派生出同一个 BLE 地址。

这一步之后，BT 和 BLE 地址各自独立管理：

```
BT MAC ─[SAME_ADDR 交汇一次]→ le_controller_set_mac()
                                    │
                                    ▼
                               BLE 控制器地址
                                    │
              ┌─────────────────────┴─────────────────────┐
              ▼                                           ▼
    le_controller_get_mac()                    le_controller_set_mac()
    （后续所有 BLE 地址读取                    （特殊场景重写，如 AI 模式）
      都从这里取）
```

图标广播通道同理（`bt_ble.c:608`），但那是广播数据中 MAC 字段的刷新，不是新的地址生成点。

当前项目配置（`app_config.h:125-126`）：`TCFG_BT_BLE_BREDR_SAME_ADDR = 0x0`（独立地址）。

### 1.2 BT 地址的来源

BT 地址是这条链的起点，它的获取链**源码完全可见**（`user_cfg.c`）：

```c
// user_cfg.c:96-98 —— 对外查询接口
const u8 *bt_get_mac_addr()
{
    return bt_cfg.mac_addr;
}
```

`bt_cfg.mac_addr` 在 `cfg_file_parse()` 中赋值（`user_cfg.c:339-374`）：

```c
u8 mac_buf[6];
u8 mac_buf_tmp[6]  = {0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF};
u8 mac_buf_tmp2[6] = {0, 0, 0, 0, 0, 0};

// 读 VM 中的 BT MAC
ret = syscfg_read(CFG_BT_MAC_ADDR, mac_buf, 6);
// 无效则随机生成并持久化
if ((ret != 6) || !memcmp(mac_buf, mac_buf_tmp, 6) || !memcmp(mac_buf, mac_buf_tmp2, 6)) {
    get_random_number(mac_buf, 6);
    syscfg_write(CFG_BT_MAC_ADDR, mac_buf, 6);
}
memcpy(bt_cfg.mac_addr, mac_buf, 6);
```

完整链路：

```
首次开机:
  CFG_BT_MAC_ADDR (VM 空/全F)
    → get_random_number()           ← 库函数，随机源不可见
    → syscfg_write(CFG_BT_MAC_ADDR) ← 持久化到 VM
    → bt_cfg.mac_addr               ← 运行期缓存

后续开机:
  syscfg_read(CFG_BT_MAC_ADDR)      ← 从 VM 读回
    → bt_cfg.mac_addr

对外:
  bt_get_mac_addr()                 ← 源码可见，直接返回 bt_cfg.mac_addr
```

注意：`get_random_number()` 是库函数（实现在 `libbtstack.a` 或 ROM 中），具体随机算法不可见。但除此之外的整个链路（VM 读写、有效性判断、缓存赋值）都在 `user_cfg.c` 中完全可见。

### 1.3 交汇之后各自独立

| 链路 | 来源 | 读取入口 | 可见性 |
|------|------|---------|--------|
| BT 地址 | `CFG_BT_MAC_ADDR`（VM）→ 随机生成 | `bt_get_mac_addr()` | 源码可见 |
| BLE 地址 | `VM_RDX_BLE_MAC` / 控制器 | `le_controller_get_mac()` | 库函数 |
| BLE 地址派生 | `bt_make_ble_address()` | — | 库函数，不可见 |

## 2. 名称关系

### 2.1 结论：JL 平台原生 BLE 名称没有统一开关，常见模块多以 BT 名为源

JL 平台对地址有 `TCFG_BT_BLE_BREDR_SAME_ADDR` 统一控制，但对名称**没有**同等级的全局开关。BLE 名称是否与 BT 名一致，取决于具体 BLE 模块的实现。按当前 SDK 可见代码，更准确的分类应是下面三类。

#### 类型 A：直接复用 `bt_get_local_name()` 返回指针

`dma_le_port.c:806`——DMA 协议适配层，BLE 名称指针直接指向 BT 名称：

```c
gap_device_name = (char *)bt_get_local_name();
gap_device_name_len = strlen(gap_device_name);
```

这里不是 copy，而是指针赋值。意味着 BT 名称运行时被修改，BLE 名称也会随之变化——完全绑定。

这是“最强绑定”的一种实现方式。

#### 类型 B：以 BT 名为源，copy 后再使用或加工

LE Audio 不是指针复用，而是把 BT 名 copy 到本地缓冲：

```c
char le_audio_name[LOCAL_NAME_LEN] = "le_audio_";
memcpy(&le_audio_name[tem_len], (u8 *)bt_get_local_name(), LOCAL_NAME_LEN - tem_len);
```

这里的关键点是：BLE 名称来源仍然是 BT 名，但关系是“初始化 copy”，不是 DMA 那种共享同一指针。

`le_trans_data.c` / `ll_demo.c` 更进一步，是 copy BT 名后再按编译开关决定是否追加后缀：

```c
name_p = bt_get_local_name();
gap_device_name_len = strlen(name_p);
memcpy(gap_device_name, name_p, gap_device_name_len);

#if DOUBLE_BT_SAME_NAME == 0
    // 增加后缀，区分名字
    memcpy(&gap_device_name[gap_device_name_len], "(BLE)", ext_name_len);
    gap_device_name_len += ext_name_len;
#endif
```

`DOUBLE_BT_SAME_NAME == 1` 时 BLE 名 = BT 名；`== 0` 时 BLE 名 = BT 名 + `"(BLE)"`。

这类实现的共同点是：BLE 名称以 BT 名为源，但最终对外名字不一定与 BT 名完全相同。

#### 类型 C：独立固定名，不走 BT 名称链

真正能支撑这类结论的例子，应该使用一直独立维护 `gap_device_name` 的 demo，例如 `att_over_edr_demo.c`：

```c
static char gap_device_name[20] = "jl_att_test";
```

这类实现没有走 `bt_get_local_name()`，BLE 名称完全独立于 BT 名称。

#### 小结

三类实现共存于同一个 SDK 中，说明 JL 平台对名称**没有统一强制策略**。但很多常见 BLE 模块确实会把 `bt_get_local_name()` 作为名称来源，只是来源强度不同：有的是直接共享指针，有的是 copy 后使用，有的是 copy 后再加后缀。因此更准确的概括应是：

```text
JL 原生 BLE 名称经常以 BT 名为源，但不是平台级强绑定
```

### 2.2 BT 名称的主链路

```
BT_NAME 宏（编译期默认入口）
  → syscfg_read(CFG_BT_NAME)     ← 稳定存储
  → bt_cfg.edr_name              ← 运行期缓存
  → bt_get_local_name()          ← 对外查询接口
  → lmp_hci_write_local_name()   ← 写入 HCI 控制器
```

运行时改名：

```
bt_modify_name(new_name)
  → syscfg_write(CFG_BT_NAME, new_name)
  → bt_cfg.edr_name = new_name
  → lmp_hci_write_local_name()
```

`BT_NAME` 宏不是常规运行时链路，它是默认值入口和 DUT 等特殊场景的直接来源。

## 3. RDX 框架：做了什么、没做什么

先明确一个容易误解的点：**RDX 没有自己写一套 BLE 协议栈。** RDX 完全站在 JL 平台 BLE 框架之上——所有 HCI 操作、ATT 协议、L2CAP 传输、连接管理、广播控制都通过 JL 平台 API 完成。RDX 替换的是**应用层内容**：GATT profile 定义、广播数据格式、名称来源、协议数据处理逻辑。

这跟 JL 自己的 RCSP 参考实现是同一模式——都用 `app_ble_*` 系列高层 API，各自注入自己的 profile 和回调。

### 3.1 BLE：沿用平台框架，替换应用层内容

RDX 在 `rdx_ble_server_init()` 中沿用的 JL 平台 API：

```
rdx_ble_server_init()
  ├─ le_controller_get_mac()                      ← 平台，读 BLE 控制器地址
  ├─ app_ble_set_mac_addr()                       ← 平台，设置 MAC 到协议栈
  ├─ app_ble_profile_set(rdx_profile_data)        ← 平台，注册 GATT profile
  ├─ app_ble_att_read_callback_register()         ← 平台，注册读回调
  ├─ app_ble_att_write_callback_register()        ← 平台，注册写回调
  ├─ app_ble_att_server_packet_handler_register() ← 平台，注册 ATT 事件处理
  ├─ app_ble_hci_event_callback_register()        ← 平台，注册 HCI 事件处理
  ├─ app_ble_l2cap_packet_handler_register()      ← 平台，注册 L2CAP 事件处理
  └─ rdx_ble_server_adv_enable(1)                 ← 最终调 app_ble_adv_enable()
```

RDX 替换的应用层内容：

| 层面 | 平台提供 | RDX 替换 |
|------|---------|---------|
| BLE 协议栈（HCI/ATT/L2CAP） | JL 平台全部 | 不动 |
| GATT profile 数据库 | 无（需自行定义） | `rdx_profile_data[]`——4 个 RDX 专用服务 |
| 广播数据格式 | `make_eir_packet_*` 工具函数 | RDX 自定义 AD（标志 + 名称 + 自定义制造商数据） |
| 名称来源 | 默认走 `bt_get_local_name()` | `BLE_LOCAL_NAME` → `VM_RDX_BLE_NAME`，完全绕开 |
| 名称存储 | 无独立槽位 | `VM_RDX_BLE_NAME`（RDX 自己注册的 VM ID） |
| ATT 读/写处理 | 回调注册框架 | RDX 自己的 `rdx_ble_server_att_read/write_callback` |
| 连接事件处理 | 回调注册框架 | RDX 自己的状态机（录制、电源、WiFi、LED） |
| MAC 存储 | 控制器寄存器 | `VM_RDX_BLE_MAC`（RDX 缓存） |
| 安全/加密 | 平台默认 | 不动（不调任何 SM API） |

**对比总结：**

| 维度 | 经典蓝牙 | BLE（RDX） |
|------|----------|-----------|
| 配置入口 | `BT_NAME` | `BLE_LOCAL_NAME` |
| 运行期存储 | `CFG_BT_NAME` | `VM_RDX_BLE_NAME` |
| 读取函数 | `bt_get_local_name()` | `rdx_ble_server_get_local_name()` |
| 对外出口 | `lmp_hci_write_local_name()` | 广播 Complete Local Name + GATT 0x2A00 |
| 地址存储 | `CFG_BT_MAC_ADDR` | `VM_RDX_BLE_MAC` |
| 恢复默认 | `CFG_BT_NAME` 链 | `rdx_ble_server_reset_local_name()` |

### 3.2 BT 名称：轻量触碰

RDX 没有替换经典蓝牙名称的任何链路。`CFG_BT_NAME` 的读写、`bt_cfg.edr_name` 的缓存、`bt_get_local_name()` 的对外接口、`bt_modify_name()` 的改名流程——RDX **全部沿用平台原生**。

RDX 对 BT 名称的触碰只有三处：

**① DUT 模式直写 HCI（`rdx_dut.c:970`）：**

```c
lmp_hci_write_local_name(BT_NAME);
```

#### 为什么只写 HCI 在这里是正确做法

RDX 的 CC（充电仓）产品**日常不需要经典蓝牙**，蓝牙模块大部分时间处于关闭状态。经典蓝牙只在进入 DUT 模式时才临时开启——每次都是冷启动，HCI 控制器处于空白状态。

在这种场景下，直接 `lmp_hci_write_local_name(BT_NAME)` 不是"绕近路"，而是**最合理的做法**：

1. 每次进 DUT 蓝牙刚上电，HCI 必然为空，不存在"旧值需要覆盖"
2. 不需要走 `CFG_BT_NAME` → `bt_cfg.edr_name` 缓存链——日常不开蓝牙，平台那套 BT 设施可能根本未初始化
3. 退出 DUT 后蓝牙再次关闭，下次进来又是新的冷启动循环
4. `BT_NAME` 宏直接来自当前品牌编译配置，每次 DUT 保证写入正确的名字

```
DUT 模式（CC 产品唯一开蓝牙的场景）：
  蓝牙上电（冷启动）→ HCI 空白
    → lmp_hci_write_local_name(BT_NAME)       // 一次写入，正确到位
    → 退出 DUT → 蓝牙关闭
    → 下次 DUT → 蓝牙重新上电 → HCI 又是空白 → 再写一次
  每次都是干净循环，不需要持久化
```

对比耳机的正常链路（`bt_modify_name` → `CFG_BT_NAME` + `bt_cfg` + `HCI` 三者同步），那条链适合"蓝牙一直开着且需要持久化改名结果"的场景。CC 产品的使用模式完全不同，走那条链反而多此一举。

**② 解绑/恢复默认时读写 `CFG_BT_NAME`（`rdx_vm.c:279-280, 844-845`）：**

```c
syscfg_read_string(CFG_BT_NAME, name, sizeof(name), 0);
syscfg_write(CFG_BT_NAME, name, LOCAL_NAME_LEN);
```

这与平台原生行为一致，只是触发时机由 RDX 的业务逻辑决定。

**③ TWS 同步 BT 名称（`rdx_app.c:498-502`）：**

```c
case APP_TWS_RDX_SYNC_BT_NAME:
    memcpy(rdx_sync_info.bt_name, rdx_info, LOCAL_NAME_LEN);
```

仅在 TWS 主从之间同步名称字符串，不改变名称来源。

### 3.3 BT 地址：基本不动

RDX 没有替换或接管 BT 地址的任何逻辑。BT 地址的来源、缓存、读取完全沿用平台原生链（见 1.2 节）。BLE 地址的派生也直接使用平台 SAME_ADDR 机制（见 1.1 节）。RDX 只是运行时通过 `rdx_ble_server_get_ble_mac()` 读取和持久化控制器地址，不做二次派生。

### 3.4 RDX 代码的可见性边界：源码 vs 闭源库

RDX 的代码不是"全部开源"或"全部闭源"——有一条清晰的边界：

```
源码可见（15 个 .c 文件）           闭源库（librdxApp.a，372KB）
─────────────────────────────      ─────────────────────────────
"怎么用 BLE 传输"                   "传什么内容"
                                     rdx_protocol_packet_recv()    ← 协议包解析
rdx_ble_server.c   GATT/广播/连接     rdx_protocol_packet_send()    ← 协议包封包
rdx_app.c          应用层入口         rdx_protocol_task_create()    ← 协议任务调度
rdx_vm.c           VM 持久化          rdx_protocol_ota_handle()     ← OTA 协议
rdx_dut.c          DUT 模式          rdx_protocol_handle_custom_cmd() ← 命令解析
rdx_ota.c          OTA 传输          ...
rdx_record.c       录音              （rdx_protocol.h 中声明但无 .c 文件的全部函数）
rdx_charge.c       充电管理
rdx_battery.c      电量
...                按键/LED/SPP/...
```

| 边界 | 源码可见 | `librdxApp.a` 闭源 |
|------|---------|---------------------|
| 用什么传输 | 全部 GATT/BLE 操作代码 | — |
| 传输什么内容 | — | 协议格式、命令解析、封包/解包 |
| 怎么调用 | 可见：通过 `rdx_protocol.h` 声明的接口 | 不可见：内部实现 |
| 可修改 | 可以改 BLE 行为、GATT profile、广播格式 | 只能改 `rdx_app_config.h` 的宏配置 |

这条边界的设计意图很清楚：RDX 把 BLE 传输层开放给集成方（可以适配不同硬件、调整广播策略），但把协议格式本身保护在库中（保证多品牌/多设备之间的协议兼容性）。

### 3.5 两套配置入口的设计意图

每个品牌块同时定义 `BT_NAME` 和 `BLE_LOCAL_NAME`，不是重复配置：

```c
#define BT_NAME         "AI Note"    // 经典蓝牙
#define BLE_LOCAL_NAME  "AI Note"    // BLE
```

- `BT_NAME`：RDX 仅在两处使用——DUT 模式直写 HCI、作为编译期默认名入口
- `BLE_LOCAL_NAME`：RDX 的完整运行时链路——`rdx_ble_server_get_local_name()` 的默认值，VM 空时写入并持久化

当前项目所有品牌块 `BT_NAME == BLE_LOCAL_NAME`，这是产品命名策略（对外统一设备名），不是代码强制。

### 3.6 BLE 名称调用链

```
rdx_ble_server_get_local_name()
  ├─ syscfg_read(VM_RDX_BLE_NAME)           ← 优先 VM
  │   ├─ [有值] → 返回 VM 值
  │   └─ [空]  → BLE_LOCAL_NAME 宏          ← VM 空时用宏
  │              → syscfg_write(VM_RDX_BLE_NAME) ← 持久化
  │
  └─ 消费方：
      ├─ 广播 Complete Local Name（扫描时名称）
      └─ GATT Characteristic 0x2A00 read callback（连接后读取）
```

**关键：RDX 的 BLE 名称链完全绕过了 `bt_get_local_name()`。**

名称修改与恢复：

```
rdx_ble_server_set_local_name(name, len)     ← APP 下发改名
  → syscfg_write(VM_RDX_BLE_NAME)

rdx_ble_server_reset_local_name()            ← 恢复出厂
  → BLE_LOCAL_NAME 宏
  → syscfg_write(VM_RDX_BLE_NAME)
```

解绑流程分别处理两套名称（`rdx_vm.c`），是"两条独立链路"的直接证据：

```
rdx_vm_unbound_cb() / rdx_vm_restore_default()
  ├─ CFG_BT_NAME → 经典蓝牙默认链
  └─ rdx_ble_server_reset_local_name() → BLE 默认链
```

### 3.7 BLE 地址的真实来源：Flash 认证码覆盖

BLE 地址在 RDX 框架下有**三层**来源，按时间顺序逐层覆盖：

```
时间线 ──────────────────────────────────────────────────────────→

第 1 层：JL 平台初始化（earphone.c:450-461）
  BT MAC → [SAME_ADDR] → le_controller_set_mac()
  决定控制器的初始 BLE 地址

第 2 层：Flash 认证码覆盖（rdx_vm.c:686）          ← 这是关键！
  rdx_vm_auth_info_init()
    → rdx_vm_read_product_info_from_flash()        ← 读外部烧录的认证码
    → rdx_vm_parse_product_info()
    → le_controller_set_mac(flash_mac)             ← 覆盖！用认证码中的 MAC

第 3 层：运行时读取与持久化（rdx_ble_server.c:1849）
  rdx_ble_server_get_ble_mac()
    → syscfg_read(VM_RDX_BLE_MAC)                  ← 先读缓存
    → [无效] → le_controller_get_mac()              ← 从控制器读（已是 flash 值）
    → syscfg_write(VM_RDX_BLE_MAC)                 ← 持久化到 VM
```

#### Flash 认证码的格式

外部通过产线工具烧录到 flash 的认证码是逗号分隔的三元组：

```
AuthKey,MAC_Address,FactorySN
```

例如模拟数据（`rdx_vm.c:710`）：

```
MCMEMOAA2025031800004045,50EE407836FD,01A012530A000001
 └─── AuthKey(24) ───┘ └─ MAC(12) ─┘ └── FacSN(16) ──┘
```

#### 解析与覆盖代码（`rdx_vm.c:629-704`）

```c
// 1. 从 flash 读原始字符串
flash_ret = rdx_vm_read_product_info_from_flash(temp, size);

// 2. 按逗号拆开
auth_key_part = strtok(temp, ",");     // AuthKey
mac_addr_part = strtok(NULL, ",");     // MAC 地址
fac_sn_part   = strtok(NULL, ",");     // 工厂 SN

// 3. 解析 MAC（12 字符 HEX → 6 字节，反转字节序）
rdx_util_str_hexstr2hexarray(mac_addr_part, len, mac_hex);
rdx_util_reverse_byte(mac_hex, 6);

// 4. 存入认证信息结构体
memcpy(rdx_auth_info.ble_mac_hex, mac_hex, 6);
strncpy(rdx_auth_info.ble_mac_str, t_buf, 12);

// 5. 覆盖 BLE 控制器地址！  ← 这是关键一步
le_controller_set_mac((u8 *)mac_hex);           // [rdx_vm.c:686]
```

#### 为什么是这个设计

Flash 认证码中的 MAC 是产线为每台设备**独立分配的**，保证全球唯一。而平台原生的 BT MAC 是随机生成的（见 1.2 节），不满足 RDX 产品的身份要求。因此 RDX 必须在初始化时用 flash 认证码中的 MAC 覆盖控制器地址。

这也解释了为什么 RDX 不关心 `TCFG_BT_BLE_BREDR_SAME_ADDR` 的值——无论它设置成 copy 还是 derive，RDX 随后都会用自己的认证码 MAC 覆盖，两者最终都会被替换。

#### 覆盖后的缓存

`rdx_ble_server_get_ble_mac()`（`rdx_ble_server.c:1849`）在 flash 覆盖之后才运行。它从控制器读到的已经是 flash 认证码中的 MAC，然后写入 `VM_RDX_BLE_MAC` 缓存。后续启动如果 VM 有效就直接用缓存。

#### Flash 读取失败时的回退

```
如果 flash 读取失败（产线未烧录认证码）：
  rdx_auth_info 保持全零
  le_controller_set_mac() 不被调用
  → BLE 地址保留为 JL 平台初始值（SAME_ADDR 链的结果）
  → rdx_ble_server_get_ble_mac() 回退到 le_controller_get_mac()
```

### 3.8 BLE 读特征的字段来源：Flash 与 VM 的分工

RDX 的 BLE 读特征（95 字节固定格式）是理解"什么数据从哪来"的最佳切面。实测日志：

```
rdx_app_earphone_pack_readchardata --> readchardata:
  C,7878359A1340,7978359A1340,50EE407836FD,ZENCHORDE202507070004928,ZCE1250825004928,000000000000
```

对照 flash 认证码 `MCMEMOAA2025031800004045,50EE407836FD,01A012530A000001` 和 VM 耳机信息：

| 字段 | 值 | 来源 | 说明 |
|------|-----|------|------|
| role | `C` | 固定写死 | 仓设备固定为 C |
| Lmac | `7878359A1340` | **VM** `EarphoneInfo.ep_mac_str` | 绑定后持久化的左耳 MAC |
| Rmac | `7978359A1340` | **VM 派生** | Lmac 最后一字节 +1 |
| **Cmac** | `50EE407836FD` | **Flash** `rdx_auth_info.ble_mac_str` | 认证码中的仓 MAC |
| SN | `ZENCHORDE202507070004928` | **VM** `EarphoneInfo.sn` | 绑定后持久化的耳机 SN |
| label sn | `ZCE1250825004928` | **VM** `EarphoneInfo.label_sn` | 绑定后持久化；VM 空时回退 flash `fac_sn` |
| wifi Mac | `000000000000` | **运行时** | Wi-Fi 未就绪时合法占位 |

```
Flash 认证码的三段各司其职：
┌─────────────────────────────────────────────────┐
│ MCMEMOAA2025031800004045,50EE407836FD,01A012... │
│ └── AuthKey (24) ─────┘└─ MAC (12) ─┘└ FacSN ─┘│
│        │                     │           │       │
│        ▼                     ▼           ▼       │
│  devBaseInfo.auth        Cmac         label sn   │
│  (协议鉴权，不出现        (仓 BLE 地址)  (盒标 SN   │
│   在读特征中)                          备用回退)  │
└─────────────────────────────────────────────────┘
```

Flash 认证码的三个字段各有用处：

| Flash 字段 | 写入位置 | 是否出现在读特征 | 作用 |
|-----------|---------|:---:|------|
| AuthKey | `rdx_auth_info.AuthKey` | ❌ | 设备鉴权，`devBaseInfo.auth` |
| MAC | `rdx_auth_info.ble_mac_hex/str` + `le_controller_set_mac()` | ✅ Cmac | 仓的 BLE 地址（控制器 + 读特征） |
| FacSN | `rdx_auth_info.fac_sn` | ❌ 条件 | label sn 备用回退，仅 VM `label_sn` 为空时启用 |

**核心结论：flash 认证码中真正体现在 BLE 读特征里的只有 MAC（Cmac 字段）。** AuthKey 用于协议鉴权但不出现在读特征中；FacSN 仅在 VM 的 label_sn 缺失时才作为回退参与拼包。

这与 VM 的分工非常清晰：

```
Flash：仓自己的身份（MAC、AuthKey、出厂盒标）
VM：   绑定耳机的身份（SN、左耳 MAC、label_sn）
BLE 读特征：Flash 仓信息 + VM 耳机信息 + 运行时 Wi-Fi → 混合拼装
```

## 5. 关键结论

1. **BT 地址来源可见**：`bt_get_mac_addr()` 源码在 `user_cfg.c:96`，链路为 `CFG_BT_MAC_ADDR`（VM）→ 首次随机生成 → `bt_cfg.mac_addr`。仅随机数生成和硬件 MAC 读在库中。
2. **BLE 地址三层覆盖**：第 1 层 — JL 平台 SAME_ADDR 初始值（`earphone.c:450`）；第 2 层 — Flash 认证码 MAC 覆盖（`rdx_vm.c:686`），产线烧录的全球唯一 MAC；第 3 层 — `VM_RDX_BLE_MAC` 缓存（`rdx_ble_server.c:1849`）。正常设备上，第 2 层会覆盖第 1 层。
3. **平台原生名称无统一绑定**：JL 平台没有对应的全局"名称统一开关"，但很多常见 BLE 模块会以 `bt_get_local_name()` 为名称来源。
4. **RDX 对 BLE 和 BT 的干预不对称**：BLE 侧——沿用 JL 平台协议栈，应用层替换（GATT profile、广播、名称、Flash 认证码 MAC 覆盖）；BT 名称侧——仅 DUT 直写 HCI + 解绑读写 `CFG_BT_NAME`；BT 地址侧——基本不动。
5. **当前项目默认同名、允许分名**：`BT_NAME == BLE_LOCAL_NAME` 是产品策略，关闭 `TCFG_BT_BLE_BREDR_SAME_ADDR` 后两者即可独立设置。
6. **DUT 直写 HCI 是 CC 产品的最优做法**：CC 产品日常不开蓝牙，DUT 时冷启动直接 `lmp_hci_write_local_name(BT_NAME)` 一次到位。
7. **`rdx_ble_server_get_ble_mac()` 不是派生函数**：它是读取并持久化当前控制器 BLE 地址的函数，此时控制器已被 flash 认证码覆盖。
