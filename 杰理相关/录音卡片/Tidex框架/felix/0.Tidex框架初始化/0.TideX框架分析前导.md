# 0. 框架原理与入口

TideX AICAP 中间件以"第三方协议"的身份注入杰理 SDK，在不修改 SDK 核心代码的前提下，通过**消息拦截**与**回调桥接**两套机制，实现对 SDK 原生行为的接管与扩展。整个框架的入口是 `port/app/tdx_app.c`。

## 0.1 第三方协议注入

杰理 SDK 在 `app_config.h` 中通过 `THIRD_PARTY_PROTOCOLS_SEL` 宏选择启用的第三方协议。TideX 通过 `TDX_EN` 位被选中后，SDK 的协议通用层（`app_protocol_common.c`）会为 TideX 预留消息通道与初始化时机。TideX 的所有代码位于 `apps/common/third_party_profile/tidex_aicap/` 目录下，与 SDK 核心代码物理隔离。

## 0.2 消息拦截机制

`tdx_app.c` 在编译期通过 `APP_MSG_PROB_HANDLER` 和 `APP_MSG_HANDLER` 宏，将消息处理器注册到 `.app_msg_prob_handler` 和 `.app_msg_handler` 两个链接段中。SDK 的消息分发循环在投递消息前，会先遍历 `PROB` 段中的处理器：

| 消息源              | 处理器                            | 注册段 | 拦截策略                                        |
| ------------------- | --------------------------------- | ------ | ----------------------------------------------- |
| `MSG_FROM_APP`      | `tdx_app_msg_handler`             | PROB   | 选择性拦截（`true` = 中断分发，`false` = 放行） |
| `MSG_FROM_KEY`      | `tdx_app_key_msg_handler`         | PROB   | 完全拦截（永远返回 `true`）                     |
| `MSG_FROM_BT_HCI`   | `tdx_app_hci_event_handler`       | HANDLER| 监听放行                                        |
| `MSG_FROM_BT_STACK` | `tdx_app_bt_status_event_handler` | HANDLER| 监听放行                                        |
| `MSG_FROM_TWS`      | `tdx_app_bt_tws_event_handler`    | HANDLER| 监听放行                                        |
| `MSG_FROM_OTA`      | `tdx_app_ota_event_handler`       | HANDLER| 监听放行                                        |

**PROB 段与 HANDLER 段的核心区别：**

- **PROB 段**（Probe，探针）：遍历方式是**短路求值**。处理器返回 `true`，消息分发立即中断，后续处理器（包括 SDK 默认处理器）永远收不到这条消息。返回 `false` 则放行，继续遍历下一个 PROB 处理器。
- **HANDLER 段**（Handler，处理器）：遍历方式是**全部执行**。所有匹配消息源的处理器都会被依次调用，返回值被忽略，不存在"拦截"概念。

**TideX 是第三方协议生态中最早、最主要使用 `APP_MSG_PROB_HANDLER` 的协议。** 其他所有第三方协议（GFPS、Realme、MMA、DMA、Ximalaya、Swift Pair、RCSP 等）都只使用 `APP_MSG_HANDLER`，它们的设计目标是作为 SDK 能力的**插件**与默认逻辑共存；而 TideX 的设计目标是在用户交互层作为**代理层**——把自己放在 SDK 和用户之间，让 SDK 默认逻辑"看不到"原始按键和某些应用消息。此外，TideX 内部的部分子系统（如充电管理 `tdx_charge.c`、WiFi 总线 `tdx_hal_wifi_bus_jl7018.c`）也复用了 `APP_MSG_PROB_HANDLER` 进行模块级消息过滤。

按键消息被完全接管后，会重新映射为 TideX 的业务动作，再选择性同步给对耳。

## 0.3 回调桥接机制

SDK 核心（`libtdx_sdk.a`）完全不依赖任何 `port/` 目录的头文件，它只通过 4 张回调表向外界请求服务：

- `tdx_app_callbacks_t` — 设备信息、鉴权、绑定状态、蓝牙名称等 Getters / Setters / Actions
- `tdx_ui_callbacks_t` — 显示相关（OLED 画面、电池、二维码等）
- `tdx_cbuf_ops_t` — 环形缓冲操作（bulk transfer）
- `tdx_event_callbacks_t` — BLE / WiFi / 录音 / 文件传输 / OTA 事件通知（**当前未注册**，预留接口）

`tdx_app_callbacks_impl.c::tdx_app_register_all_callbacks()` 在启动时一次性把**已实现的 3 张回调表**注册给 SDK，实现 SDK 核心与应用层的完全解耦。

## 0.4 协议事件独立处理

除了杰理 SDK 的消息循环，TideX 还维护了一套独立的协议事件通道。`tdx_app_all_init()` 中调用 `tdx_protocol_task_create(tdx_app_protocol_event_handler)` 创建一个独立的协议任务，所有 BLE 命令、文件传输请求、OTA 指令等都通过 `tdx_app_protocol_event_handler` 统一入口处理，与杰理的消息循环并行运行、互不阻塞。

# 1. 框架注入时机

TideX 不是作为一个独立固件运行，而是**以"第三方协议"的身份寄生在杰理 SDK 内部**。它的注入遵循"先等基础设施就绪，再借势接管"的策略——只有在 SDK 的消息循环、蓝牙协议栈、BLE/SPP 通道等核心基础设施全部初始化完成后，TideX 才开始注册自己的处理器和回调，从而能够**借助这些现成的基础设施完成消息截胡**。

## 1.1 启动时序

整个注入过程的调用链如下：

```
上电 → app_main() → app_core 任务
    └── 进入 APP_MODE_BT（蓝牙模式）
            └── earphone.c:bt_init()
                    ├── 蓝牙控制器初始化
                    ├── bt_ble_init()          ← BLE 协议栈就绪
                    └── multi_protocol_bt_init()  ← 【第三方协议注入点】
                            ├── multi_protocol_profile_init()
                            │       ├── app_ble_init()       ← BLE SM/GATT 就绪
                            │       ├── app_spp_init()       ← SPP 通道就绪
                            │       └── ble_op_multi_att_send_init() ← ATT 发送缓冲就绪
                            └── tdx_app_all_init()           ← 【TideX 初始化】
                                    ├── tdx_app_register_all_callbacks()
                                    ├── tdx_transport_init_all()
                                    └── tdx_app_tasks_init() → tdx_protocol_task_create()
                    └── bt_init_ok_search_index()
                    └── APP_MSG_STATUS_INIT_OK ← 通知应用层：蓝牙就绪
```

关键时机：**`tdx_app_all_init()` 被调用时，BLE GATT Server、SPP、ATT 发送缓冲区、蓝牙状态机已经全部就绪**。这意味着 TideX 不需要自己从零初始化蓝牙，而是直接复用 SDK 建好的通道，在上面叠加自己的业务逻辑。

## 1.2 为什么必须等基础设施就绪

TideX 的消息拦截机制（见 0.2 节）依赖于 SDK 的**消息分发循环**。这个循环在 `app_main.c` 的 `app_core` 任务中运行，其代码逻辑如下（`app_main.c:461-475`）：

```c
int app_get_message(int *msg, int max_num, const struct key_remap_table *key_table)
{
    const struct app_msg_handler *handler;
    app_core_get_message(msg, max_num);

    // 消息截获, 返回1表示中断消息分发
    for_each_app_msg_prob_handler(handler) {
        if (handler->from == msg[0]) {
            int abandon = handler->handler(msg + 1);
            if (abandon) {
                return 0;  // ← 截胡！消息不再往下传
            }
        }
    }
    // ... 后面再遍历 HANDLER 段
}
```

分发顺序：

1. 启动时遍历 `.app_msg_handler` 和 `.app_msg_prob_handler` 两个链接段，收集所有注册的消息处理器
2. 运行时从任务队列 `os_taskq_post_type()` 中取出消息
3. 先遍历 `PROB` 段中的处理器，如果任一返回 `true`（非零），消息被拦截，`return 0` 直接退出，HANDLER 段永不会执行
4. 如果 `PROB` 全部放行（所有处理器都返回 `false`），再遍历普通 `HANDLER` 段，所有匹配处理器都会被调用

如果 TideX 在消息分发循环启动**之前**就注册了自己的处理器，那么这些处理器会被 SDK 的收集逻辑自然纳入；如果 TideX 在消息分发循环启动**之后**才注册，处理器虽然写入了链接段，但 SDK 可能已经完成了遍历，导致拦截失效。

因此，`multi_protocol_bt_init()` 被放在 `bt_ble_init()` **之后**、消息循环大规模运转**之前**，确保：

- SDK 的 BLE/SPP 基础设施已就绪（TideX 可以注册自己的 GATT Service、SPP 回调）
- SDK 的消息分发系统已构建（链接段收集完成）
- 但用户交互尚未开始（消息队列几乎为空，TideX 有时间完成初始化）

## 1.3 "后发先至"的截胡原理

TideX 虽然在启动顺序上是"后到者"，但在消息分发链路上却是"先到者"——因为 `APP_MSG_PROB_HANDLER` 宏把处理器放进了 `.app_msg_prob_handler` 段，而 SDK 的消息循环在每次投递时都会**从头遍历**这个段。只要 TideX 的处理器返回 `true`，消息就不会到达 SDK 默认的处理器。

### 为什么不是"所有回调都触发"

容易混淆的是：PROB 段和 HANDLER 段的遍历语义完全不同：

```
收到消息
  │
  ▼
┌─────────────────────────────┐
│  遍历 .app_msg_prob_handler  │  ← TideX 注册在这里
│  (PROBE 探针段)              │
│                              │
│  每个处理器：                 │
│    返回 true  → 拦截！       │
│    return 0; 消息到此为止    │
│    返回 false → 放行         │
└─────────────────────────────┘
  │ (全部放行才继续)
  ▼
┌─────────────────────────────┐
│  遍历 .app_msg_handler       │  ← SDK 默认处理在这里
│  (普通处理段)                │
│                              │
│  每个处理器都会被调用         │
│  没有返回值控制是否中断       │
└─────────────────────────────┘
```

- **PROB 段 = "谁先返回 true 谁垄断"**：只要有一个处理器返回非零值，后续所有处理器（包括 SDK 默认的）都被跳过。这是**截胡**。
- **HANDLER 段 = "大家都有份"**：所有匹配消息源的处理器依次执行，互不影响。这是**广播/监听**。

### TideX 不是全盘拦截，而是分层接管

TideX 的注册策略是**混合的**：

| 层级 | 消息源 | 注册段 | 策略 | 原因 |
|------|--------|--------|------|------|
| **用户交互层** | `MSG_FROM_KEY` | PROB | 完全拦截 | 按键必须彻底接管，否则 SDK 默认逻辑会播放按键音、执行默认动作，与 TideX 冲突 |
| **用户交互层** | `MSG_FROM_APP` | PROB | 选择性拦截 | 拦截需要改写的控制消息（如关机、模式切换），放行不需要改写的 |
| **底层蓝牙层** | `MSG_FROM_BT_HCI` | HANDLER | 监听放行 | 只需要知道蓝牙连接/断开状态，不能拦截（否则蓝牙协议栈会挂） |
| **底层蓝牙层** | `MSG_FROM_BT_STACK` | HANDLER | 监听放行 | 同上 |
| **底层蓝牙层** | `MSG_FROM_TWS` | HANDLER | 监听放行 | 监听对耳状态变化，不阻断 TWS 同步逻辑 |
| **底层蓝牙层** | `MSG_FROM_OTA` | HANDLER | 监听放行 | 监听 OTA 事件 |

如果 TideX 把蓝牙 HCI 事件也放进 PROB 段并拦截，SDK 的蓝牙协议栈就收不到连接完成/断开完成事件，整个蓝牙功能会崩溃。因此 **TideX 只拦截用户交互层消息，对底层蓝牙事件只监听不拦截**。

### 与其他第三方协议的对比

| 协议 | 注册方式 | 定位 |
|------|----------|------|
| GFPS、Realme、MMA、DMA、Ximalaya、Swift Pair、RCSP 等 | 全部使用 `APP_MSG_HANDLER` | **插件**——扩展 SDK 能力，与默认逻辑共存 |
| **TideX** | `APP_MSG_PROB_HANDLER`（KEY/APP）+ `APP_MSG_HANDLER`（蓝牙事件） | **代理层**——在用户交互层拦截并重新映射，同时监听底层状态 |

这种设计让 TideX 实现了**最小入侵**：

- 不改 SDK 核心代码（不碰 `app_main.c`、`earphone.c` 的默认逻辑）
- 不改蓝牙协议栈（复用 `bt_ble_init()` 建好的 GATT 连接）
- 不改消息循环（只往链接段里塞自己的处理器）
- 却能完全接管按键、监听蓝牙状态变化、选择性拦截应用消息

## 1.4 一句话总结

**TideX 把自己伪装成一个普通的第三方协议，在 SDK 建完所有基础设施后"搭便车"入驻，然后借助 SDK 自己的消息分发系统反客为主——在用户交互层（按键、应用控制消息）完成截胡，在底层蓝牙层完成监听。它是第三方协议生态中最主要使用 `APP_MSG_PROB_HANDLER` 的协议，且其内部子系统（充电、WiFi 总线等）也复用了同一机制。**

# 2. tdx_app_all_init 的作用

`tdx_app_all_init()` 是 TideX 中间件的**总初始化入口**，由杰理 SDK 的第三方协议框架在蓝牙协议栈初始化完成后调用。它的职责是把 TideX 运行所需的全部子系统从冷态带到可工作状态。

## 2.4 两种启动路径

`tdx_app_all_init()` 会根据上次关机是否正常，走两条不同的路径：

- **正常启动**：OLED 显示主页 → 初始化按键/振动 → EMMC 检查 → 启动应用任务

# 3. TideX 的能力不是凭空产生的

## 3.1 核心原则

`libtdx_sdk.a` 是闭源预编译静态库，它**不包含任何芯片驱动代码、不直接操作硬件寄存器、不包含杰理 SDK 头文件**。它的每一行代码能做什么，完全取决于 port 层给它提供了什么能力。

## 3.2 三大能力来源

SDK 核心只能通过以下三类接口与外界交互：

### 来源一：HAL 接口（必须，链接时符号绑定）

HAL 是 SDK 核心操作硬件的唯一途径。SDK 核心通过**链接时的符号解析**调用这些函数，不需要 port 层注册——只要 port 层实现了同名函数，链接器就会自动绑定。

```
libtdx_sdk.a 编译时：    extern void tdx_hal_vm_write(...);  ← 声明，不实现
port 层编译时：          void tdx_hal_vm_write(...) { ... } ← 实现
链接时：                 符号解析，绑定在一起
```

HAL 覆盖的域（26 个头文件）：

| HAL 域 | 头文件 | 提供给 SDK 核心的能力 |
|--------|-------|---------------------|
| OS 抽象 | `tdx_hal_os.h` | 任务创建、mutex、sem、延时、tick |
| 软件定时器 | `tdx_hal_timer.h` | 单次/周期定时器 |
| 日志输出 | `tdx_hal_log.h` | printf、原始写 |
| BLE Server | `tdx_hal_ble_server.h` | GATT 初始化、广播数据设置、ATT 收发、连接控制 |
| 经典蓝牙 | `tdx_hal_bt.h` | EDR MAC、TWS/通话/A2DP 状态、**名称读写** |
| VM 持久化 | `tdx_hal_vm.h` | 按 ID 读写非易失存储 |
| 文件系统 | `tdx_hal_fs.h` | 文件 open/read/write/seek/remove |
| GPIO | `tdx_hal_gpio.h` | 高低电平、高阻控制 |
| SPI / UART / I2C | `tdx_hal_spi.h` 等 | 总线收发 |
| 音频 | `tdx_hal_audio.h` | MIC 数据获取、编解码控制 |
| 电源 | `tdx_hal_power.h` | 电量、充电状态、关机 |
| WiFi 总线 | `tdx_hal_wifi_bus.h` | SPI/UART 到 WiFi 模组通信 |

### 来源二：回调表（必须，运行时注册）

SDK 核心**不依赖 port 层头文件**，它通过回调表向 port 层"提问"或"通知"：

| 回调表 | 注册函数 | 方向 | 内容 |
|--------|---------|------|------|
| `tdx_app_callbacks_t` | `tdx_register_app_callbacks()` | SDK ↔ App | 设备信息、BT/BLE 名称、鉴权、电源、绑定状态、OTA 通知等 ~40 个回调 |
| `tdx_ui_callbacks_t` | `tdx_register_ui_callbacks()` | SDK → UI | 显示画面、电池、二维码、OTA 进度 |
| `tdx_cbuf_ops_t` | `tdx_register_cbuf_ops()` | SDK → 缓冲 | 环形缓冲区 create/write/read/destroy |
| `tdx_event_callbacks_t` | `tdx_register_callbacks()` | SDK → App | 异步事件（当前未注册） |

### 来源三：操作结构体（按模块，运行时注册）

部分模块需要更复杂的平台操作，通过专用的 ops 结构体注册：

| 结构体 | 模块 | port 层提供的操作 |
|--------|------|------------------|
| `tdx_auth_ops_t` | 鉴权 | Flash 容量查询、地址映射、BLE MAC 设置、CRC16 |
| `tdx_record_ops_t` | 录音 | 音频通路开关、MIC 增益读写、存储挂载、状态变化通知 |
| `tdx_spi_hw_ops_t` | SPI | 14 个 HAL 函数（总线、CS、握手、DMA、喂狗等） |
| `tdx_spi_callbacks_t` | SPI | 4 个应用回调（数据接收、传输状态等） |

## 3.3 SDK 核心无法做到的事

反过来说，如果 port 层没有提供对应的 HAL 或回调，SDK 核心就**绝对无法**做到：

- **无法直接操作 GPIO** — 必须通过 `tdx_hal_gpio_*`
- **无法直接读写 Flash** — 必须通过 `tdx_hal_flash_*` 或 `tdx_hal_vm_*`
- **无法直接操作 BLE 广播** — 必须通过 `tdx_hal_ble_srv_adv_data_set()` 等
- **无法直接获取芯片唯一 ID** — 必须通过回调 `get_dev_base_info()`
- **无法直接读电池电量** — 必须通过 `tdx_hal_power_*`

## 3.4 反向推理方法

因为能力边界是明确的，拿到 SDK 源码之前就可以做合理的反向推理。

**步骤**：

1. 列出相关 HAL 接口（如 BLE 名称相关 → `tdx_hal_bt.h`、`tdx_hal_vm.h`、`tdx_hal_ble_server.h`）
2. 列出相关回调（如 `get_ble_name`、`set_ble_name`、`get_bt_name`）
3. SDK 核心的实现只能是这些接口的**排列组合**

**以 BLE 名称为例**：

```
已知能力:
  HAL:  tdx_hal_vm_read/write()         ← VM 持久化
  HAL:  tdx_hal_ble_srv_adv_data_set()  ← 设置广播数据（原始字节包）
  HAL:  tdx_hal_bt_get_ble_name()       ← 获取 BLE 名称（返回 BT_NAME 宏）
  HAL:  tdx_hal_bt_set_ble_name()       ← 设置 BLE 名称（空实现）
  回调: get_ble_name / set_ble_name     ← 已注册但可能未使用
  宏:   BLE_LOCAL_NAME                  ← 编译期常量

推断的库内部实现:
  tdx_ble_server_init()
    ├─ tdx_hal_vm_read(VM_KEY, ...)     ← 必须走 HAL 读 VM
    │   ├─ 有值 → 用 VM 值
    │   └─ 空 → 用 BLE_LOCAL_NAME 宏   ← 不走 hal_bt_get_ble_name()，不走回调
    ├─ 手动组装 AD Structure（嵌入名称）
    └─ tdx_hal_ble_srv_adv_data_set()   ← 必须走 HAL 发广播
```

**拿到源码后的验证清单**：

| 验证项 | 搜索关键词 | 预期结果 |
|--------|----------|---------|
| 回调是否被调 | `tdx_get_app_callbacks()->get_ble_name` | 命中数 ≤ 1（可能为 0） |
| HAL 是否被调 | `tdx_hal_bt_get_ble_name` | 命中数（确认库是否走了 HAL） |
| 宏是否直接引用 | `BLE_LOCAL_NAME` | 命中位置（确认是否在 VM 写入分支上） |
| VM key 是多少 | `tdx_hal_vm_write` 附近上下文 | 确认 BLE 名称的 VM ID |

## 3.5 常见误判与排查原则

### 误判："宏定义了就应该生效"

**错误**。宏只是预处理器文本替换，生效与否取决于**谁在引用它、那个引用者什么时候编译的**。如果引用者在预编译库里，port 层改宏不影响它。

**排查**：全文搜索宏名，确认所有引用位置都在 port 层源码中（不在 `.a` 中）。

### 误判："回调注册了就应该被调用"

**错误**。回调只是"提供了可能性"，是否调用取决于 SDK 核心的实现选择。

**排查**：如果回调是 wrapper 调 SDK 内部函数（如 `_cb_get_ble_name` → `tdx_ble_server_get_local_name()`），说明它只是读取路径，不是设置路径。真正设置名称的路径可能在库内部直接完成。

### 误判："HAL 实现了就等于 SDK 在用它"

**错误**。HAL 函数实现了只是"提供了能力"，SDK 核心可以选择不用。

**排查**：拿到源码后，搜索 HAL 函数名在 `sdk/src/` 中的引用，确认实际调用路径。

### 排查一般原则

1. **先看 HAL 接口列表** → 确定 SDK 核心"能做什么"
2. **再看回调表** → 确定 SDK 核心"能问什么"
3. **全文搜索宏的引用位置** → 确定是在 port 层还是库内部消费
4. **如果库内部消费** → 宏值取决于库编译时的配置，不是当前 port 层配置
5. **运行时日志** → 最有说服力的证据（如 `VM empty, set brand default: AI Note`）
