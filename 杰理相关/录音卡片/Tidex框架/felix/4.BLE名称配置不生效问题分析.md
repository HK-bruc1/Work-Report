# 3. BLE 名称配置不生效问题分析

## 现象

Makefile 中配置了板级 (`TDX_BOARD`) 和品牌 (`TDX_CONFIG`)，但 BLE 广播名称始终是固定的默认值，不是品牌配置文件中设定的名称。即使烧录时擦除整个 Flash，问题依然存在。

日志证据：

```
[00:00:03.618] tdx_hal_vm_read  --> id: 150, len: 30
[00:00:03.618][I][TDX.BLE:1625] ===> tdx_ble_server_get_local_name --> VM empty, set brand default: AI Note
[00:00:03.620] tdx_hal_vm_write --> id: 150, len: 8
[00:00:03.624][I][TDX.BLE:1638] tdx_ble_server_get_local_name --> write local name success: AI Note
```

## 配置链路回顾

先确认品牌配置链路本身是正确的：

```
make TDX_BOARD=cc_v1_jl7018 TDX_CONFIG=DEEPMINER
  → -include .../boards/cc_v1_jl7018/board.h    (定义 TDX_PRODUCT_CC)
  → -DTDX_CONFIG_DEEPMINER                       (定义品牌宏)
  → board.h → product_select.h → cc/product_def.h
  → board.h → tdx_config_select.h → #undef BLE_LOCAL_NAME → tdx_config_deepminer.h
      → #if defined(TDX_PRODUCT_CC)
        #define BLE_LOCAL_NAME "DEEPMINER CC"    ← 宏定义正确
```

到这一步，`BLE_LOCAL_NAME` 宏在 **port 层编译期**已经正确解析为品牌配置的值。问题不在配置链路。

## 根因分析

### 1. 定位消费点：谁调用了 `BLE_LOCAL_NAME`？

全文搜索整个 `SDK/` 目录：

- `BLE_LOCAL_NAME` 只出现在**头文件**中（品牌配置文件、`tdx_sdk_config.h`、`tdx_config_select.h`、`tdx_api_transport.h`）
- **没有任何 port 层 `.c` 文件引用它**

这意味着消费这个宏的代码不在 port 层源码中，而在 `libtdx_sdk.a`（预编译静态库，1MB，2026-04-23 签入）里。库的编译时机早于固件编译，当时的 `BLE_LOCAL_NAME` 是库自己的默认值。

### 2. 排除替代路径：逐条验证

TideX 框架有三类机制可以让 SDK 核心获取 BLE 名称，以下逐一排除：

```
SDK 核心 (libtdx_sdk.a)
  │
  ├── 回调表路径 ────────────────────────────────
  │   ✗ tdx_get_app_callbacks()->get_ble_name    ← 回调已注册但不被调用
  │   ✗ tdx_get_app_callbacks()->set_ble_name    ← 回调已注册但不被调用
  │   ✗ tdx_get_app_callbacks()->get_bt_name     ← 经典蓝牙，不同域
  │   ✗ tdx_get_app_callbacks()->set_bt_name     ← 经典蓝牙，不同域
  │
  ├── BT HAL 路径 ───────────────────────────────
  │   ✗ tdx_hal_bt_get_ble_name()                ← HAL 已实现（返回 BT_NAME）但不被调用
  │   ✗ tdx_hal_bt_set_ble_name()                ← HAL 已实现（空函数）但不被调用
  │
  ├── Port 层 wrapper 路径 ─────────────────────
  │   ✗ tdx_app_ble_server_name_set()            ← 调库内部函数，不走回调
  │   ✗ tdx_app_ble_server_name_get()            ← 调库内部函数，不走回调
  │
  └── 协议事件路径 ──────────────────────────────
      ✗ TDX_PROTOCOL_EVENT_BLE_NAME_SET          ← 手机 APP 改名命令，启动时不触发

  日志确认：上述函数入口全部无输出，SDK 核心不走其中任何一条。
```

SDK 核心实际走的路径（日志确认）：

```
[00:00:03.618] tdx_hal_vm_read  → id: 150              ← VM 读取 BLE 名称槽位
[00:00:03.618] TDX.BLE:1625     → VM empty, default: AI Note   ← 空，用宏
[00:00:03.620] tdx_hal_vm_write → id: 150              ← 持久化
[00:00:03.624] TDX.BLE:1638     → write success: AI Note
```

对应的代码路径：

```
BLE_LOCAL_NAME 宏（编译期常量，库编译时展开为 "AI Note"）
    │
    ├── tdx_hal_vm_read(id=150)           ← 读 VM
    │   ├── 有值 → 返回 VM 值
    │   └── 空 → BLE_LOCAL_NAME 宏值
    │        └── tdx_hal_vm_write(id=150) ← 持久化
    │
    ├── 组装 AD Structure                  ← 库内部闭环
    │
    └── tdx_hal_ble_srv_adv_data_set()    ← 推送广播
```

**结论：SDK 核心的 BLE 名称初始化不走回调、不走 BT HAL、不走协议事件。唯一的外部输入就是编译期宏 `BLE_LOCAL_NAME`，其余全部在库内部闭环。**

### 3. 确认根因：宏展开 → 二进制写死

`libtdx_sdk.a` 是预编译二进制，库编译时预处理器阶段就将宏替换为字符串字面量：

```
源代码:  tdx_ble_server_init() {
             name = BLE_LOCAL_NAME;              ← 宏引用
         }

预处理:  tdx_ble_server_init() {
             name = "AI Note";                   ← 字符串字面量
         }

二进制:  41 49 20 4E 6F 74 65 00              ← .rodata 段字节序列
```

库编译完成后，"AI Note" 就从"可配置的宏"变成了"不可变的二进制数据"。此后 `strlen(name)`、`vm_write(name)`、BLE 广播数据组装——所有下游操作都基于这个常量。

用户修改 `port/brands/tdx_config_*.h` 中的 `#define BLE_LOCAL_NAME`，**改的是头文件文本**，改不到 `.a` 文件里已经生成的字节序列。这是 C 预处理器宏固有的**编译期绑定**特性。

### 4. 后果：三重循环，无论怎么操作名称都无法改变

**第一重（不擦除 Flash，常规启动 / OTA 升级）：**

```
启动 → tdx_ble_server_get_local_name()
     → VM 有值（上次写入的 "AI Note"）
     → 直接返回 VM 值，不经过默认值分支
     → 品牌配置的 BLE_LOCAL_NAME 完全不被访问
```

**第二重（擦除 Flash 后首次启动）：**

```
启动 → tdx_ble_server_get_local_name()
     → VM 为空
     → 走默认值分支 → BLE_LOCAL_NAME 宏
     → 宏已在库编译时展开为 "AI Note"（二进制常量）
     → vm_write("AI Note") → 回到第一重
```

**第三重（BLE-APP恢复出厂 → `tdx_app_ble_server_name_reset()`）：**

```c
// tdx_app.c — BLE-APP恢复出厂
void tdx_app_sys_reset_to_defaults(void)
{
    // ...
    tdx_app_ble_server_name_reset();  // → 调库函数，库内部仍用 BLE_LOCAL_NAME 宏
}
```

恢复出厂调用的 `tdx_ble_server_reset_local_name()` 也在 `libtdx_sdk.a` 中，它从库自己的 `BLE_LOCAL_NAME` 宏取默认值，同样走 VM 写入路径。**五击同样无法改变名称**。

三重的根相同：**默认值来源是编译进 `libtdx_sdk.a` 的二进制常量，不是品牌配置。**

### 5. 设计评价

TideX 框架的核心架构原则（文档 `1.通用层初始化详解.md` 1.2 节）：

> SDK 核心完全不依赖任何 `port/` 目录的头文件，它只通过 4 张回调表向外界请求服务

`tdx_ble_server_init()` 使用编译宏 `BLE_LOCAL_NAME` 而不是回调 `_cb_get_ble_name()`，违背了这一原则：

| 模块 | 获取方式 | 是否符合回调原则 |
|------|----------|-----------------|
| 设备信息 | `_cb_get_dev_base_info()` | ✅ |
| 经典蓝牙名称 | `_cb_get_bt_name()` | ✅ |
| MIC 增益 | `_cb_get_mic_gain()` | ✅ |
| 鉴权密钥 | `_cb_get_auth_key()` | ✅ |
| **BLE 初始名** | **`BLE_LOCAL_NAME` 编译宏** | **❌** |

`_cb_get_ble_name` 回调已经注册好了，框架其他模块都走回调，唯独 BLE 名称初始化走了宏。这是库设计上的不一致。

关于 `TDX_PROTOCOL_EVENT_BLE_NAME_SET`：它是**手机 APP 通过 BLE 下发改名命令**时触发的运行时事件。在 `tdx_app_tasks_init()` 中，`tdx_ble_server_init()` 在第 2 步执行，而协议任务在第 3 步才创建——BLE 名称在协议任务还不存在时就已经确定了。

```
tdx_app_tasks_init()
  ├── 1. tdx_file_init()
  ├── 2. tdx_app_ble_server_init()        ← BLE 名称在此确定
  └── 3. tdx_protocol_task_create()       ← 协议事件在此后才存在
```

## 解决方案

在 `port/app/tdx_app_ble_server.c` 的 `tdx_app_ble_server_init()` 中，库初始化完成后，立即用 port 层编译时的 `BLE_LOCAL_NAME` 覆盖：

```c
void tdx_app_ble_server_init(void)
{
    tdx_ble_server_init();
    tdx_app_ble_server_name_set(BLE_LOCAL_NAME, strlen(BLE_LOCAL_NAME));  // ← 新增

#if !TDX_HAS_FILE_STORAGE
    tdx_ble_server_adv_enable(1);
#endif
}
```

### 为什么这是正确的

| 原则 | 说明 |
|------|------|
| **Wrapper 层封装** | `tdx_app_ble_server_name_set()` 是 port 层已有的 wrapper |
| **宏在 port 层消费** | `BLE_LOCAL_NAME` 在 port 层编译期解析为品牌值 |
| **不破坏 VM 机制** | 首次启动写入品牌名；后续启动写入相同值（幂等） |
| **覆盖所有路径** | 五击恢复出厂后重启 → 再次进入 `init` → 品牌名重新写入 |

### 为什么不选其他方案

| 方案 | 问题 |
|------|------|
| 在 `tdx_app_all_init()` 尾部追加 | 跨文件修改，容易被重构遗漏 |
| 在 `_cb_get_ble_name` 中改 | 那是读取回调，不是设置入口 |
| 在 Factory Reset 路径中改 | 只覆盖五击场景，首次启动不受影响 |

## 如果有机会拿到 `libtdx_sdk.a` 源码

### 验证清单

| 步骤 | 搜索关键词 | 预期发现 |
|------|----------|---------|
| 1. 定位宏引用 | `BLE_LOCAL_NAME` | 在 `sdk/src/tdx_ble_server.c` 中直接使用 |
| 2. 确认 VM 路径 | `tdx_hal_vm_write` 上下文 | 宏值出现在 VM id=150 的写入分支上 |
| 3. 确认回调未用 | `tdx_get_app_callbacks()->get_ble_name` | 命中数 0 |
| 4. 确认 HAL 未用 | `tdx_hal_bt_get_ble_name` | 命中数 0 |
| 5. 排查同类问题 | `BT_NAME`、`TDX_WIFI_AP_SSID` 等 | 确认其他品牌宏是否也有同样问题 |

**辅助验证**：`strings libtdx_sdk.a | grep "AI Note"` 确认字符串已编译进二进制。

### 库源码修改建议

```c
// 修复前：
int tdx_ble_server_init(void)
{
    if (vm_empty()) {
        name = BLE_LOCAL_NAME;   // 宏，编译期绑定
        vm_write(name);
    }
}

// 修复后：
int tdx_ble_server_init(void)
{
    if (vm_empty()) {
        const tdx_app_callbacks_t *cbs = tdx_get_app_callbacks();
        if (cbs && cbs->get_ble_name) {
            cbs->get_ble_name(name, sizeof(name));  // 回调，运行时获取
        }
        vm_write(name);
    }
}
```

### 修改清单

| 文件 | 改动 | 说明 |
|------|------|------|
| `sdk/src/tdx_ble_server.c` | `BLE_LOCAL_NAME` → `tdx_get_app_callbacks()->get_ble_name()` | 去掉宏直接引用 |
| `sdk/include/tdx_sdk_config.h` | 保留 `#ifndef BLE_LOCAL_NAME` 作为 fallback | 向后兼容 |
| `sdk/include/tdx_callbacks.h` | 无需改动 | `get_ble_name` 字段已存在 |

---

**结论**：当前修复在 port 层通过 wrapper 传递品牌宏值是符合框架规范的最小改动。根本解决需要闭源库将 BLE 名称初始化从编译期宏改为运行时回调，与框架其他模块保持一致。
