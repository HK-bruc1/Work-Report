# HCI 抓包在 TWS 耳机中最能发挥作用的总结

## 一、按优先级排序（覆盖 80–90% 常见问题）

**手机侧协议栈 + 耳机主耳交互的完整可见性（最强项）**

- 连接建立/断开全流程（Inquiry、Page、LE Create Connection、Disconnect 原因码）
- 配对/绑定/SMP/链路密钥交换是否正常
- AVRCP 控制命令（上一曲/下一曲/播放/暂停/音量）
- HFP 通话控制（接听/挂断/拒接/语音助手触发，如 AT 命令 ATA / +CHUP）
- A2DP 音频流启动、Suspend、MTU 协商、Codec 选择（SBC/AAC/LDAC/aptX 等）
- 耳机上报事件（电量、充电盒状态、入耳检测、触摸事件等，通常走 GATT、HFP vendor 扩展，**或厂商自定义 L2CAP 动态信道**）
- 连接参数更新（Connection Interval、Latency、Supervision Timeout）
- 断连原因分析（本地超时、远程超时、LMP/LL 错误码、加密失败等）

**手机蓝牙芯片/协议栈 Bug 定位**

- 手机是否正确发出 HCI Command
- Controller 返回的 Event 是否异常（Command Status、Command Complete、Hardware Error）
- Buffer overflow、Flush timeout、ACL 数据重传率高等音频卡顿间接证据

**TWS 特有功能验证（主从耳机协作的"手机可见部分"）**

- 主耳与手机的交互是否稳定
- 副耳断开/重连时主耳是否向手机报告角色变化
- 共享链路或多设备切换时的行为
- 厂商自定义功能（低延迟模式切换、ANC 档位、EQ 调节）是否通过 GATT/HFP 正确传输

**快速回归测试 & 复现问题**

- 不需要额外硬件，几分钟就能抓到 log，用 Wireshark 过滤 btavrcp / bthfp / btatt 等就能看到明文指令
- 适合大量重复操作（频繁切歌、接挂电话、开关盒子）

------

## 二、日志分析补充的注意事项

**1. 关于厂商私有 L2CAP 信道**

部分耳机厂商不走 GATT，而是走 L2CAP 动态信道上的私有协议。典型特征是大量末字节为 0xFF 的动态 CID（如 0x0DFF、0x1AFF……）持续推送数据，每个 CID 包数规律一致，接收方向流量远大于发送方向。这属于正常的私有多路复用行为，不要误判为异常；如需解析业务含义，需结合设备厂商 SDK。

**2. 关于"HCI 层看到明文"的前提**

HCI 层看到明文的前提是 Controller 将解密后的 ACL payload 上报给 Host，对标准蓝牙连接成立。但对于把加解密完全放在芯片内部处理的私有协议，Host 拿到的 payload 本身就是私有格式，"明文"指协议格式明文而非业务明文，做深层数据分析时需注意这一区别。

**3. 关于 Event status 字段的解读**

部分 Event 的 status 字段需结合后续命令行为一起判断，不能只看单个 status 码。例如 Remote Name Request Complete 返回 FAIL(0xFF) 但仍然附带了设备名，连接流程实际继续推进——只看 status 会误判为完全失败。

## 三、与专业蓝牙分析仪（空中抓包）的核心区别

| 维度              | 手机 HCI 抓包                                           | 专业蓝牙分析仪（空中）                                   | TWS 场景谁更实用       |
| ----------------- | ------------------------------------------------------- | -------------------------------------------------------- | ---------------------- |
| 抓包位置          | Host ↔ Controller（手机内部接口）                       | Over-The-Air（空中 PHY → LMP/LL）                        | HCI 更日常，空中更彻底 |
| 能看到的内容      | 手机视角的所有命令/事件/ACL 数据（明文 payload）        | 两设备间真正的空中无线包（跳频、信道、RSSI、丢包、重传） | 空中看到"真相"         |
| TWS 主副耳通信    | 完全看不到（主副耳用私有协议）                          | 大多数 TWS 用私有链路，多数也不可见                      | HCI 和空中都大多盲     |
| 空中干扰/信号质量 | 只能间接推测（重传多、Supervision Timeout）             | 直接看到（信道利用率、PER、RSSI、CRC 错误）              | 空中远胜               |
| 加密处理          | 看到标准连接的明文 payload；私有协议 payload 为私有格式 | 需捕获配对过程才能解密，否则只看到加密后包               | HCI 对标准协议更友好   |
| 手机/双侧视角     | 只看到手机这一侧                                        | 双向对等视角（谁先发、谁拒绝、空中谁丢包）               | 空中更客观             |
| 成本 & 便利性     | 免费、手机自带、随时抓                                  | 硬件几百到几十万，需专门环境                             | HCI 碾压               |
| 典型覆盖比例      | 手机端 80–90% 问题                                      | 剩余 10–20% 难问题                                       | 先 HCI，后空中         |
| Wireshark 解析    | ★★★★★                                                   | ★★★★☆                                                    | HCI 更易上手           |

------

## 四、实际 TWS 调试推荐策略

1. **先用 HCI 抓包** → 90% 问题在这里定位或排除（控制指令、通话、音频卡顿、连接不稳）

2. HCI 看不出手机侧问题

    → 再考虑空中抓包，重点排查：

   - 环境干扰（Wi-Fi / 2.4GHz 拥挤）
   - 距离/穿墙/人体遮挡导致的真实丢包
   - 主副耳同步异常（如主副耳使用标准链路）
   - 极少数耳机芯片固件 Bug（空中才能看到耳机主动发的 LL 包异常）

------

## 五、一句话总结

**手机 HCI 抓包**是 TWS 日常开发/售后/用户问题定位的主力武器，让你清楚看到"手机和主耳之间到底在聊什么、聊得对不对"；**空中分析仪**是终极杀手锏，用来解决"为什么聊着聊着就聊丢了、聊卡了"的深层无线原因。

大多数 TWS 问题（切歌不灵、挂电话失败、单耳无声、延迟高）用 HCI log 就够了，只有顽固的"莫名断连""特定距离才卡""干扰环境下不行"才需要空中抓包。

# 小米

1. 在我的设备中连续点击七次OS版本进入开发者模式
2. 开启蓝牙数据包日志
3. 拨号盘输入* # * # 5959 # * #*，统一开始抓包日志
4. 重启蓝牙
5. 进行蓝牙操作问题复现
6. 拨号盘输入* # * # 5959 # * #*，生成日志，日志放到wireshark中分析。
7. 日志在内存存储设备-MIUI-debug_log

# 可以从日志得出的信息有哪一些？

**一、本机信息**

- 手机蓝牙名称：`纵使相逢应不识`
- 抓包总时长：约 228 秒（两份合计）
- 本机角色：始终作为 Central/Master 发起连接

------

**二、连接行为**

- 手机反复尝试连接两个设备：
  - `C0:5C:78:23:53`：被连接了 5 次，全部以 `0x0B`（连接已存在）失败，始终未成功建立新连接，说明该设备的旧连接状态没有被正常清理
  - `21:A0:AD:DB:EC`（设备名 `1111`）：第一份日志中成功建立连接并触发了完整配对流程，第二份日志中快速失败并主动拒绝了对方的连接请求
- 第二份日志开头手机主动执行了 Reject Connection Request，说明上层逻辑判断此时不应接受该设备连接

------

**三、配对流程细节**

- 与111的配对走的是 SSP（简单安全配对）流程，完整经历了：
  - IO Capability 协商
  - User Confirmation Request（数字对比码：151696009）
  - Simple Pairing Complete（配对步骤完成）
  - Link Key 生成并写入本地
  - 但最终 Authentication Failure（0x03）——密钥生成了但认证没过
- Remote Name Request 返回 status=0xFF 但仍携带了设备名 `1111`，是一个"状态码失败但数据有效"的典型案例

------

**四、数据流量**

- Handle `0xEDC`：两份日志合计接收超过 420KB，发送接近 0，是一个后台持续活跃的单向推送连接，L2CAP CID 呈厂商私有模式（大量末字节 0xFF 的动态信道，每 CID 恰好 3 包），推测是已配对的耳机或手环在持续上报状态数据
- Handle `0x005` / `0x003`：与 `1111` 设备的短暂交互，双向流量均很小（各约 4~7KB），L2CAP 层有大量信令帧，说明双方在协商信道但未完成业务数据传输

------

**五、可以排除的问题**

- 手机蓝牙芯片本身没有异常：HCI Command/Event 序列完整，无 Hardware Error，无异常 Controller Event
- 不是扫描/发现阶段的问题：设备地址在日志开始就已知，直接发起连接，没有经历搜索流程
- 不是空中信号问题：HCI 层无法判断，但从连接能够建立、配对流程能够走完来看，基本信道质量没有问题

------

**六、无法从日志得出的信息**

- `1111` 是什么设备、为什么认证失败（是用户没确认数字、还是对端拒绝、还是密钥不匹配）
- `C0:5C:78:23:53` 为什么始终处于"连接已存在"状态、那条旧连接在哪里
- Handle `0xEDC` 对应的是哪台设备、推送的数据业务含义是什么
- 空中信道质量、干扰情况、实际丢包率
